## 四、发现的问题（按归属分类，基于代码审查实际验证）

### 4.1 后端数据库项目问题

#### 问题 B1：slot_category 数据与模型注释不一致

- **现象**：`models/AdSlot.js` 注释说明 slot_category 有效值为 `display` 和 `feed`，但 ID:14（market_feed）和 ID:15（exchange_feed）在数据库中存储的是 `standard`
- **影响**：如果代码层做 `slot_category === 'feed'` 判断会漏掉这两个广告位
- **修复方案**：直接更新数据库：`UPDATE ad_slots SET slot_category='feed' WHERE ad_slot_id IN (14,15)`

#### 问题 B2：抽奖页存在两个功能重复的弹窗位

- **现象**：`lottery_popup`（ID:3）和 `lottery_popup_ad`（ID:7）的 slot_type、position 完全相同，仅 slot_key 和定价不同
- **ID:3 从未被使用**：0 条关联计划
- **修复方案**：直接 DELETE（已决策，见第七节）

#### 问题 B3：ad_creatives.image_url 绕过了 image_resources 体系

- **现象**：3 条 ad_creatives 的 image_url 存的是 Sealos 对象 key（如 `popup-banners/xxx.jpg`），直接从旧系统迁移而来，没有对应的 `image_resources` 表记录
- **影响**：无法通过 ImageService 管理这些图片（无缩略图、无排序、无业务关联追踪）
- **但不阻塞功能**：ad-delivery 接口直接透传 image_url 给前端，前端通过 `/api/v4/images/{objectKey}` 代理访问即可
- **注意**：`image_resources` 表的对象 key 列名为 `file_path`（非 `object_key`）

#### 问题 B4：ad-delivery 接口返回的 image_url 是对象 key 而非完整 URL

- **代码验证**：`routes/v4/system/ad-delivery.js` 第 79 行 `image_url: creative.image_url || null`，直接透传对象 key，无 URL 转换
- **影响**：前端需要知道拼接规则，不符合行业惯例
- **修复方案**：ad-delivery 路由返回 `${PUBLIC_BASE_URL}/api/v4/images/${objectKey}`（Step B5）

#### 问题 B5：createSystemCampaign() 不自动创建 AdCreative

- **代码验证**：`services/AdCampaignService.js` 第 659-704 行，`createSystemCampaign()` 只创建 AdCampaign，不创建 AdCreative。路由层（第 290-304 行）虽然已传入 `image_url`、`content_type`、`text_content`、`display_mode`，但 Service 层未使用这些字段
- **对比**：`createOperationalCampaign()`（第 615-631 行）**已正确实现**自动创建 AdCreative
- **修复方案**：在 `createSystemCampaign()` 中参照 `createOperationalCampaign()` 第 615-631 行的逻辑，添加 AdCreative 自动创建

#### 问题 B6：管理员创建商业广告路由白名单缺少图片字段

- **代码验证**：`routes/v4/console/ad-campaigns.js` 第 104-116 行，`ALLOWED_ADMIN_CREATE_FIELDS` 白名单不包含 `image_url`、`content_type`、`text_content`、`display_mode`
- **对比**：同文件中 POST `/operational`（第 225-243 行）和 POST `/system`（第 290-304 行）**已正确透传**这四个字段
- **影响**：管理员通过 Web 后台创建商业广告时，即使上传了图片并选择了展示模式，后端也不会接收这些字段
- **修复方案**：在 `ALLOWED_ADMIN_CREATE_FIELDS` 白名单中添加这四个字段

#### 问题 B7：createCampaign()（商业广告）不自动创建 AdCreative

- **代码验证**：`services/AdCampaignService.js` 第 234-333 行，`createCampaign()` 只创建 AdCampaign，不创建 AdCreative
- **对比**：`createOperationalCampaign()` 已正确实现
- **修复方案**：参照 `createOperationalCampaign()` 第 615-631 行逻辑，在 `createCampaign()` 事务内添加 AdCreative 创建，但 `review_status` 设为 `'pending'`（商业广告需审核）

#### 问题 B8：用户端创建广告路由白名单缺少图片字段

- **代码验证**：`routes/v4/user/ad-campaigns.js` 第 83-93 行，`ALLOWED_CAMPAIGN_CREATE_FIELDS` 白名单不包含 `image_url`、`content_type`、`text_content`、`display_mode`
- **影响**：小程序端广告主自助创建广告时无法提交图片素材
- **修复方案**：在白名单中添加这四个字段

#### 问题 B9：ad-delivery 路由 slot_type 校验不包含 feed

- **代码验证**：ad-delivery 路由校验 `slot_type ∈ ['popup', 'carousel', 'announcement']`，不包含 `'feed'`，但 `models/AdSlot.js` 第 27 行 `VALID_SLOT_TYPES = ['popup', 'carousel', 'announcement', 'feed']`
- **影响**：feed 类型广告位（market_feed、exchange_feed）无法通过 ad-delivery 接口下发
- **需要拍板**：见决策 6

## 五、补全方案（后端为权威，前端适配后端）

### 5.1 总体原则

1. **后端字段名即前端字段名**：前端直接使用 `ad_campaign_id`、`image_url`、`slot_type` 等后端字段，不做映射

### 决策 6：feed 类型广告位的下发机制

#### 问题说明

数据库里有 2 个信息流广告位（`market_feed` 和 `exchange_feed`，slot_type=feed），但 ad-delivery 接口（`routes/v4/system/ad-delivery.js` 第 59 行）的校验写死了 `['popup', 'carousel', 'announcement']`，不允许 `feed`。而 `models/AdSlot.js` 第 27 行 `VALID_SLOT_TYPES` 已包含 `'feed'`。模型层已支持，路由层卡住了。

feed 广告与 popup/carousel/announcement 的区别：feed 嵌入到列表页商品流中（如交易市场商品列表里每隔几条穿插一条广告），不是独立的 UI 覆盖物。

#### 行业做法对比

| 公司/平台 | 广告下发方式 | feed 广告如何处理 |
|---|---|---|
| **Google Ads** | 统一 Ad Manager API，slot_type 参数区分格式 | Native/Feed 和 Banner/Interstitial 用同一个 API，format 参数不同 |
| **Meta Ads** | 统一 Marketing API | Feed ad、Stories ad、Reels ad 全部通过同一 placement 参数区分 |
| **阿里妈妈/淘宝联盟** | 统一广告服务 API | 信息流广告、横幅广告、开屏广告都是同一接口的 ad_type 参数 |
| **腾讯广告（广点通）** | 统一 SDK/API | Banner、Native（信息流）、Splash 均通过 slot_id 区分，SDK 内统一请求 |
| **美团广告** | 统一投放引擎 | 不同 position 对应不同广告位，信息流和弹窗只是 slot 配置不同，下发接口相同 |
| **穿山甲（字节跳动）** | 统一 SDK | 信息流原生广告和开屏/插屏广告用同一个 loadAd() 方法，code_id 不同 |
| **Steam / Epic** | 统一 Storefront API | 推荐位、轮播位、列表穿插位都走同一内容下发，position 参数不同 |
| **闲鱼/转转** | 统一推荐引擎 | 搜索列表穿插广告和首页 Banner 走同一广告服务 |
| **小型虚拟物品交易平台** | 通常一个接口搞定 | 不会因为 feed 类型单独建接口，slot 配置区分即可 |
| **游戏内广告平台（Unity Ads / AdMob）** | 统一 SDK | Rewarded、Interstitial、Banner、Native 四种格式一个 SDK 入口 |

**行业共识**：没有公司为不同的广告格式（feed/popup/carousel/banner）建独立接口。统一入口 + 参数区分 是绝对的行业标准。穿插逻辑由前端（或前端 SDK）负责，不是广告下发接口的职责。

#### 两种方案对比

| 维度 | A. 扩展 ad-delivery | B. 新建独立接口 |
|---|---|---|
| 后端改动量 | **1 处**：第 59 行数组加 `'feed'` | 新建 1 个路由文件，约 50 行代码 |
| 前端对接成本 | **零额外成本**：调 `getAdContent('feed', 'market_list')` | 需要新增 1 个 API 封装函数 |
| 是否符合行业标准 | **是**：与 Google/Meta/阿里/腾讯做法一致 | 不符合，违背统一下发原则 |
| 后端已有架构兼容性 | **完全兼容**：模型层已支持 feed，竞价引擎已支持，响应结构通用 | 需要重复实现竞价选中 + 扁平化逻辑 |
| 长期维护成本 | **低**：1 个入口维护 | 高：2 个入口，逻辑同步成本 |
| 穿插逻辑归属 | 前端负责（调 API 拿广告列表，自己决定插入位置） | 同左，与接口设计无关 |
| 未来扩展 | 再加新广告类型只需在数组加 1 个值 | 每种新类型都要考虑是否建新接口 |

#### 本项目实际情况

### 决策 7：小程序端广告主创建广告时如何上传图片

#### 问题说明

小程序用户（广告主）创建广告计划时需要上传图片素材，但现有图片上传接口 `POST /api/v4/console/images/upload` 挂在 console 路径下。

代码验证：`routes/v4/console/images.js` 第 96-98 行使用 `authenticateToken` 中间件（不检查角色），但它挂载在 `/api/v4/console/` 路径下，按项目约定 console 是管理员专属路径。

用户端（`/api/v4/user/`）目前没有图片上传接口。修复 B7/B8 后，用户端创建广告可以带 `image_url` 字段，但需要先有地方上传图片拿到 object_key。

#### 行业做法对比

| 公司/平台 | 广告主上传素材的方式 | 是否与管理端共用接口 |
|---|---|---|
| **Google Ads** | 广告主通过 Google Ads API 的 `MediaService.upload()` 上传 | **否**：广告主 API 和 Admin API 完全隔离，各有独立的认证和限流 |
| **Meta Ads** | 广告主通过 Marketing API 的 `POST /{ad-account-id}/adimages` 上传 | **否**：独立端点，绑定广告主账户 |
| **阿里妈妈** | 广告主通过淘宝客 API 的素材上传接口上传 | **否**：商家端和管理端完全隔离 |
| **腾讯广告** | 广告主通过 `marketing_api/images/add` 上传 | **否**：有独立的广告主 API，与平台管理 API 分离 |
| **美团** | 商户通过商户版 APP/Web 的独立上传接口 | **否**：商户端和平台运营端接口隔离，各有独立限流和文件大小限制 |
| **穿山甲** | 广告主通过 Ocean Engine API 独立上传 | **否**：完全独立的广告主 API 体系 |
| **Steam/Epic** | 开发者通过 Partner API 上传 | **否**：Partner 端和 Admin 端分离 |
| **闲鱼/转转** | 用户通过 APP 内上传接口 | **否**：用户端和运营端的上传走不同接口，用户端有额外的内容审核和限流 |
| **小型平台** | 多数有独立的用户上传端点，少数共用但加角色判断 | 共用的做法虽然节省代码但**违反最小权限原则** |
| **游戏平台** | 用户 UGC 上传和管理后台上传走不同路由 | **否**：用户端通常有更严格的文件大小/频率限制 |

**行业共识**：广告主/用户的素材上传和管理端的上传必须走不同接口。原因不仅是权限隔离，还因为用户端通常需要：更严格的文件大小限制、请求频率限制、内容安全检查、不同的业务类型白名单。

#### 两种方案对比

| 维度 | A. 新增 user 端上传接口 | B. 放宽 console 上传权限 |
|---|---|---|
| 后端改动量 | 新建 `routes/v4/user/images.js`，约 30 行（复制 console/images.js upload 部分） | 改 1 行权限中间件 |
| 权限模型 | **清晰**：console = 管理员，user = 普通用户，各自独立 | **混乱**：普通用户可以调 console 路径，破坏命名约定 |
| 安全风险 | **低**：用户只能访问 user 路径下的功能 | **高**：console 路径放开后，可能误放开其他 console 功能 |
| 限流/限制 | **可独立控制**：用户端可设置更严格的频率限制、文件大小上限 | 与管理端共享限制，无法差异化 |
| 业务类型控制 | **可限制**：用户端只允许 `business_type: 'uploads'` + `category: 'ad_creative'` | 共享管理端白名单（lottery/exchange/trade/uploads 全部可用） |
| 符合行业标准 | **是**：所有大厂都隔离 | 不符合 |
| 复用现有代码 | 直接调用 `ImageService.uploadImage()`，只需写路由层 | 无新代码 |
| 长期维护成本 | **低**：路由层 30 行，逻辑在 ImageService 不重复 | 低但留下架构隐患 |
| 审计/追踪 | **清晰**：用户上传记录 source_module='user'，管理员上传记录 source_module='admin' | 混在一起，无法区分来源 |

#### 本项目实际情况

- 后端已有清晰的三层路由体系：`/api/v4/console/*`（管理）、`/api/v4/user/*`（用户）、`/api/v4/system/*`（系统）
- `ImageService.uploadImage()` 是独立的服务层方法，不依赖路由层，可被任何路由调用
- 图片上传核心逻辑（multer 配置 + ImageService 调用 + 响应格式）约 30 行，复制成本极低
- 用户端上传可以硬编码 `business_type: 'uploads'`、`category: 'ad_creative'`、`source_module: 'user'`，比管理端更严格
- 项目未上线，现在建立正确的架构约定，长期受益

**已决策：方案 A — 新增 `POST /api/v4/user/images/upload` 用户端图片上传接口**

改动：新建 `routes/v4/user/images.js`（约 30 行），在 `routes/v4/user/index.js` 中挂载 `router.use('/images', imagesRoutes)`

理由：保持 console/user/system 三层路由的权限语义清晰，符合所有大厂的广告主/管理端隔离标准。代码量极小（复用 ImageService），但建立了正确的架构约定，用户端可独立控制限流、文件大小、业务类型白名单。方案 B 省了 30 行代码但埋下了架构隐患，从降低技术债务角度不可接受。
