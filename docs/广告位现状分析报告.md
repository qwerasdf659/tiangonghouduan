## 四、发现的问题（按归属分类）

### 4.1 后端数据库项目问题

#### 问题 B1：slot_category 数据与模型定义不一致

- **现象**：`models/AdSlot.js` 注释说明 slot_category 有效值为 `display` 和 `feed`，但 ID:14（market_feed）和 ID:15（exchange_feed）在数据库中存储的是 `standard`
- **影响**：代码层做 `slot_category === 'feed'` 判断时会漏掉这两个广告位
- **修复方案**：直接更新数据库：`UPDATE ad_slots SET slot_category='feed' WHERE ad_slot_id IN (14,15)`

#### 问题 B2：抽奖页存在两个功能重复的弹窗位

- **现象**：`lottery_popup`（ID:3）和 `lottery_popup_ad`（ID:7）的 slot_type、position、description 完全相同（"抽奖页面弹窗广告位"），仅 slot_key 和定价不同
- **ID:3 从未被使用**：0 条关联计划
- **修复方案**：删除 ID:3 或设为 is_active=0，保留 ID:7（已有 1 条计划关联）

#### 问题 B3：ad_creatives.image_url 绕过了 image_resources 体系

- **现象**：3 条 ad_creatives 的 image_url 存的是 Sealos 对象 key（如 `popup-banners/xxx.jpg`），直接从旧系统迁移而来，没有对应的 `image_resources` 表记录
- **影响**：无法通过 ImageService 管理这些图片（无缩略图、无排序、无业务关联追踪）
- **但不阻塞功能**：ad-delivery 接口直接透传 image_url 给前端，前端通过 `/api/v4/images/{objectKey}` 代理访问即可

#### 问题 B4：ad-delivery 接口返回的 image_url 是对象 key 而非完整 URL

- **现象**：`GET /api/v4/system/ad-delivery` 返回的 `image_url` 值为 `popup-banners/xxx.jpg`（对象存储 key），不是完整可访问的 URL
- **影响**：前端需要知道拼接规则，不符合行业惯例（Google/Meta/美团/阿里均返回完整 URL）
- **存储层不变**：数据库继续存对象 key（切换 CDN/存储不需要改数据库），但**接口返回层**应拼接完整代理 URL
- **修复方案**：ad-delivery 路由返回 `${PUBLIC_BASE_URL}/api/v4/images/${objectKey}`（Step B5）

#### 问题 B5：创建广告计划时不自动创建 AdCreative 记录

- **现象**：后端三个创建方法（`createCampaign`、`createOperationalCampaign`、`createSystemCampaign`）只创建 AdCampaign，不自动创建 AdCreative
- **影响**：创建计划后还需要单独调用 AdCreativeService 创建素材，流程割裂
- **修复方案**：在三个创建方法中，如果传入了 `image_url` 或 `text_content`，自动创建一条 AdCreative

#### 问题 B6：console 创建路由未透传图片相关字段到 Service

- **现象**：`POST /api/v4/console/ad-campaigns/operational` 和 `/system` 路由的请求体没有将 `image_url`、`content_type`、`text_content`、`display_mode` 传递给 Service
- **影响**：即使前端上传了图片并提交了 URL，后端也不会保存
- **修复方案**：三个 POST 路由添加这四个字段到白名单

### 4.2 Web 管理后台前端问题

#### 问题 W1：创建广告计划弹窗无图片上传 UI

- **现象**：`admin/ad-management.html` 的创建弹窗中没有 `<input type="file">`、没有图片预览、没有拖拽上传区域
- **影响**：所有需要图片素材的广告类型（popup、carousel、feed）无法上传图片
- **可复用**：`imageUploadMixin` 已在组件中引入（`...imageUploadMixin()`），有 `uploadImage()` 方法可直接调用

#### 问题 W2：saveCampaign() 不发送图片相关字段

- **现象**：`admin/src/modules/content/pages/ad-management.js` 的 `saveCampaign()` 提交纯 JSON，不包含 `image_url`、`content_type`、`display_mode`
- **影响**：即使有了上传 UI，提交时也不会把图片 URL 发给后端

## 五、补全方案（后端为权威，前端适配后端）

### 5.1 总体原则

1. **后端字段名即前端字段名**：前端直接使用 `ad_campaign_id`、`image_url`、`slot_type` 等后端字段，不做映射
2. **不兼容旧接口**：不保留 popup-banners / carousel-items / announcements 旧路由
3. **image_url 存对象 key**：前端负责拼接完整 URL，后端不改存储策略
4. **先上传后提交**：前端先调 `POST /api/v4/console/images/upload` 获取 object_key，再提交给创建接口

### 5.2 后端修改清单

#### Step B1：修复 slot_category 数据

```sql
UPDATE ad_slots SET slot_category = 'feed' WHERE ad_slot_id IN (14, 15);
```

#### Step B2：清理重复广告位

```sql
UPDATE ad_slots SET is_active = 0 WHERE ad_slot_id = 3;
```

#### Step B3：console 创建路由透传图片字段

**文件**：`routes/v4/console/ad-campaigns.js`

- `POST /operational` 路由：从 `req.body` 提取 `image_url`、`content_type`、`text_content`、`display_mode` 传给 Service
- `POST /system` 路由：同上
- `POST /`（商业广告）路由：将这四个字段加入白名单

#### Step B4：Service 层创建计划时自动创建 AdCreative

**文件**：`services/AdCampaignService.js`

- `createOperationalCampaign()`：创建 AdCampaign 后，如果 `data.image_url` 或 `data.text_content` 有值，在同一事务内自动创建 AdCreative（`review_status: 'approved'`，运营类型免审核）
- `createSystemCampaign()`：同上
- `createCampaign()`（商业广告）：同上，但 `review_status: 'pending'`

**复用现有能力**：直接调用 `AdCreativeService.createCreative()`，无需新建方法

#### Step B5：ad-delivery 返回完整代理 URL（决策 4 已定）

**文件**：`routes/v4/system/ad-delivery.js`

在 `flatItems.map()` 中将 `image_url` 从对象 key 转为完整代理 URL：

```javascript
image_url: creative.image_url
  ? `${process.env.PUBLIC_BASE_URL}/api/v4/images/${creative.image_url}`
  : null,
```

修改后前端直接 `<image src="{{item.image_url}}">`，不需要任何 URL 拼接。

### 决策 1：重复广告位 lottery_popup（ID:3）如何处理？

#### 行业做法对比

| 公司/平台类型 | 做法 | 原因 |
|---|---|---|
| Google Ads / Meta Ads | 软删除（is_deleted + deleted_at） | 已上线系统有审计追踪和报表数据关联需求 |

| 小型二手/虚拟物品交易平台 | 直接 DELETE | 数据量小，无合规审计要求 |

- ID:7 功能完全相同（同 slot_type、同 position、同 description），已有 1 条计划
- 项目未上线，无生产数据

**已决策：直接 DELETE**

DELETE FROM ad_slots WHERE ad_slot_id = 3;

理由：项目未上线，零外键依赖，保留只会造成混淆。上线后如需删除广告位，应改用软删除（加 `is_deleted` 字段）。

### 决策 4：ad-delivery 返回的 image_url 是对象 key 还是完整 URL？

#### 行业做法对比

| 公司/平台 | 返回什么 | 前端怎么用 |
|---|---|---|
| Google Ads | 完整 CDN URL（`https://tpc.googlesyndication.com/...`） | 前端直接放 `<img src>` |
| Meta/Facebook Ads | 完整 CDN URL（`https://scontent.xx.fbcdn.net/v/...`） | 前端直接放 `<img src>` |
| 阿里妈妈 / 淘宝联盟 | 完整 CDN URL（`https://img.alicdn.com/...`），含图片处理参数 | 前端直接使用，URL 中嵌入裁剪/压缩参数 |
| 美团广告 | 完整 CDN URL | 前端直接使用 |
| 腾讯广告 | 完整 URL（必须在小程序域名白名单中） | 前端直接使用 |
| Steam / Epic | 完整 CDN URL | 前端直接使用 |
| 闲鱼 / 转转 | 完整 CDN URL | 前端直接使用 |
| 小型平台 | 多数返回完整 URL；少数返回 key + 配置 base URL | 返回 key 的前端需要自己拼 |

**行业共识**：所有大公司和绝大多数小公司都是 **后端返回完整的、可直接使用的 URL**。对象 key 是内部存储实现细节，不应暴露给前端。

#### 两种方案对比

| 维度 | A. 返回对象 key | B. 返回完整可用 URL |
|---|---|---|
| 前端复杂度 | 高：每个前端都要知道拼接规则 | **低：直接 `<img :src="item.image_url">`** |
| URL 构造逻辑分散度 | 分散在每个前端项目中 | **集中在后端一处** |
| 代理路径变更影响 | 需要更新所有前端 | **只需改后端一行** |
| CDN 切换影响 | 需要更新所有前端 | **只需改后端一行** |
| 对象存储迁移影响 | 需要更新所有前端 | **只需改后端一行** |
| 行业一致性 | 不符合行业惯例 | **符合所有大公司做法** |
| 长期维护成本 | 高 | **低** |

#### 本项目的特殊约束

Sealos 对象存储的公网 URL 会强制 `Content-Disposition: attachment`，导致微信小程序 `<image>` 组件无法渲染。项目已有解决方案：`/api/v4/images/{objectKey}` 代理路由。因此：

- 小程序必须走代理：`{PUBLIC_BASE_URL}/api/v4/images/{objectKey}`
- Web 管理后台也可以走代理（浏览器同样兼容）

**已决策：后端返回完整代理 URL**

ad-delivery 接口将 `image_url` 从对象 key 转为完整代理 URL：`{PUBLIC_BASE_URL}/api/v4/images/{objectKey}`

**实现方式**：修改 `routes/v4/system/ad-delivery.js` 的 `flatItems.map()` 中一行：

```javascript
image_url: creative.image_url
  ? `${process.env.PUBLIC_BASE_URL}/api/v4/images/${creative.image_url}`
  : null,
```

前端（小程序 + Web 管理后台）直接 `<img :src="item.image_url">`，不需要任何 URL 拼接逻辑。
