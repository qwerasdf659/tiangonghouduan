## 一、需求描述

后端运营希望能**动态控制小程序前端的版本升级**：当发布新版后，运营在后台改一个配置，旧版本用户进入小程序时被提示/强制升级，无需改动并重新发布前端代码。

---

## 二、整体原理（为什么需要业务层版本闸门）

微信原生 `wx.getUpdateManager` 有两个固有缺陷：

1. 只在**冷启动**时检查更新；
2. 拦不住「检测到新版本但用户不点更新、继续用旧版」的情况。

因此前端额外实现了一层**业务层版本闸门**，逻辑链路如下：

```
小程序冷启动（onLaunch）
  → 取真机实际版本号 curVersion = wx.getAccountInfoSync().miniProgram.version
  → 开发版/体验版无版本号（'' 或 'develop'）→ 直接放行（不参与比对）
  → 请求后端 GET /system/app-version 拿 { min_version, force_update, update_message, platform }
  → 绝不误拦判断：未开强更 / 缺 min_version / 平台不匹配 → 放行
  → 逐段语义化比较 compareVersion(curVersion, min_version)
       curVersion < min_version → 进入强制更新模式（全屏遮罩，连首页都进不去）
       否则 → 正常放行
```

## 三、前端链路核查（直读前端代码，逐条验真，全部已就绪）

### 3.1 冷启动调用闸门

`app.ts` `checkVersionGate()`（onLaunch 内调用），取真机版本号：

```typescript
curVersion = wx.getAccountInfoSync().miniProgram.version || ''
// 开发版/体验版无正式版本号（'develop' 或空）→ 不参与比对，直接放行
if (!curVersion || curVersion === 'develop') {
  log.info('版本闸门跳过（开发版/无版本号，放行）:', curVersion)
  return
}
```

→ 关键：闸门比对的版本号取自 `wx.getAccountInfoSync().miniProgram.version`，即**微信平台实际发布的版本号**（如线上版 `0.0.2`），不是代码常量 `globalData.version = '5.2.0'`。

### 3.2 接口封装（公开、不阻塞启动）

`utils/api/system.ts` `getAppVersionGate()`：

```typescript
async function getAppVersionGate() {
  return apiClient.request('/system/app-version', {
    method: 'GET',
    needAuth: false,    // 无需登录
    showLoading: false,
    showError: false    // 接口异常静默，绝不阻塞启动
  })
}
```

### 3.3 版本比较

`utils/util.ts` `compareVersion()`，逐段语义化比较（`compareVersion('5.2.0','5.3.0') === -1`）。`app.ts` 中：

```typescript
if (Utils.compareVersion(curVersion, min_version) < 0) {
  this.enterForceUpdateMode(update_message)
}
```

### 3.4 命中强更 → 全屏遮罩

`app.ts` `enterForceUpdateMode()` 置 `globalData.isForceUpdateMode = true` 并通知所有已注册的 `maintenance-overlay` 组件调用 `showForceUpdate()` 显示「立即更新」按钮（不可绕过）；无遮罩时降级为不可取消的 `wx.showModal`。

### 3.5 重启更新

`applyPendingUpdate()`：有已下载新版本则 `applyUpdate()` 重启；否则提示用户彻底关闭小程序后重进，由微信下次冷启动拉取最新代码。

### 3.6 绝不误拦分支（已全部覆盖）

| 放行场景 | 前端处理 |
| --- | --- |
| 取小程序版本失败 | `catch` 后 `return`，放行 |
| 开发版/无版本号（`''` 或 `develop`） | 直接 `return`，放行 |
| 接口异常/超时 | `catch` 后放行（`showError:false`） |
| 响应不成功 / `data` 为空 | `return`，放行 |
| `force_update !== true` | `return`，放行（总开关关闭） |
| 缺 `min_version` | `return`，放行 |
| `platform` 存在且 `!== 'miniprogram'` | `return`，放行 |

### 4.3 字段说明（前端实际读取的字段）

| 字段 | 类型 | 必填 | 运营含义 | 前端行为 |
| --- | --- | --- | --- | --- |
| `min_version` | string | 是 | 最低可用版本（语义化，如 `"0.0.3"`） | `curVersion < min_version` 时强制更新；**缺失则放行** |
| `force_update` | boolean | 是 | 强更总开关 | `!== true` 时一律放行（即使版本低于 min_version 也不拦） |
| `update_message` | string | 否 | 强更遮罩提示文案 | 显示在「立即更新」遮罩；缺省用「检测到新版本，请更新后继续使用」 |
| `platform` | string | 否 | 平台标识，固定 `"miniprogram"` | 存在且不等于 `miniprogram` 时放行（多端隔离） |
| `latest_version` | string | 否 | 当前线上最新版本 | 当前前端未参与比对，仅展示/备查用，建议返回 |

### 4.4 运营动态控制的使用方式（接口已就绪，仅需配置）

运营有两种等价的配置入口，**保存后写库 + 清 Redis 缓存，小程序下次冷启动（最长 300s 缓存 TTL）生效**：

- **入口 A（推荐）**：Admin 后台页面 `app-version-config.html`（需 ≥100 级管理员），可视化编辑 `min_version`/`latest_version`/`force_update`/`update_message`/`platform` 并保存。
- **入口 B**：带管理员 token 直接调 `PUT /api/v4/console/system/app-version-config`。

典型操作流程：

1. 前端发布新版本（微信平台上传并审核通过，例如线上版从 `0.0.2` → `0.0.3`）。
2. 运营在后台把 `min_version` 改为 `"0.0.3"`、`force_update` 设为 `true`。
3. 仍停留在 `0.0.2` 的用户下次冷启动 → `0.0.2 < 0.0.3` → 弹全屏强更遮罩，拦住不让用。
4. 已升到 `0.0.3` 的用户 → `0.0.3 >= 0.0.3` → 正常放行。
5. 临时关闭强更：把 `force_update` 设为 `false` 即可全员放行。

> **版本号格式约束（后端硬校验，已确认）**：`PUT` 接口对 `min_version`（必填）与 `latest_version`（可空）采用严格语义化校验 `SEMVER_PATTERN = /^\d+\.\d+\.\d+$/` —— **必须是 x.y.z 三段格式，不收 `v` 前缀、不收四段**，否则返回 400 `INVALID_APP_VERSION_CONFIG`。微信线上版本号体系当前为 `0.0.x`，天然满足三段格式。**运营今后发版版本号须始终保持 x.y.z 三段。**

> **版本号体系约束（重要）**：后端 `min_version` 必须填**微信平台那套发布版本号**（当前为 `0.0.x`），不是前端代码常量 `5.2.0`。因为闸门比对的是 `wx.getAccountInfoSync().miniProgram.version`（真机实际发布版本）。运营/后端配置时务必以微信公众平台「版本管理 → 线上版本」显示的版本号为准。

### 4.5 实际数据存储（已落库，非建议）

配置实际存于 `system_settings` 表，`setting_key='app_version_gate'`、`value_type='json'`、`category='feature'`，全局一份，无需按用户存储。读取经 `AdminSystemService.getConfigValue('app_version_gate')`（Redis 缓存 300s），写入经 `upsertConfig`（写后清缓存）。该范式与平台其它自由 JSON 配置（`exchange_page`、`app_theme`、`campaign_placement`）一致。

## 六、联调验证清单

> ⚠️ **前提：必须用真机的体验版/正式版测试。** 开发版（微信开发者工具）`version` 为空或 `develop`，闸门直接放行，**测不出强更效果**。

1. **正常放行**：`force_update=false` 或不返回 `min_version` → 任何版本用户均正常进入，无遮罩。
2. **命中强更**：上传体验版（版本号 `0.0.2`）→ 后端设 `min_version="0.0.3"`、`force_update=true` → 真机扫体验版冷启动 → 应弹全屏「立即更新」遮罩，无法绕过、连首页都进不去。
3. **升级后放行**：把体验版升到 `0.0.3` → 同样配置下冷启动 → 正常进入。
4. **文案生效**：后端配 `update_message="为保障兑换功能，请升级到最新版"` → 遮罩显示该文案。
5. **平台隔离**：后端 `platform="web"` → 小程序端放行（不拦）。
6. **绝不误拦**：后端接口返回 500 / 超时 / 关停 → 小程序端正常放行进入，不被锁死。
7. **关闭强更**：线上正在强更时把 `force_update` 改为 `false` → 用户下次冷启动恢复正常进入。
