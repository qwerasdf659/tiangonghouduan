## 一、这个文档要解决什么问题

### 1.1 现象（真实报错）

微信小程序「我的提交」页（`my-submissions.ts`，**在小程序仓，不在本后端仓**）在 `onShow` 时调用两个**后端真实存在**的接口（路由实测见 `routes/v4/shop/consumption/merchant-query.js` 的 `/list`、`/stats`）：

- `GET /api/v4/shop/consumption/merchant/stats`（未带 `store_id`）
- `GET /api/v4/shop/consumption/merchant/list?page=1&page_size=20`（未带 `store_id`）

用当前测试账号（`13612227910`）打开页面，两个接口都返回：

```jsonc
{ "success": false, "code": "ADMIN_STORE_ID_REQUIRED",
  "message": "管理员执行门店相关写操作必须指定 store_id（不允许无门店写入，以保证审计可定位）" }
```

> ✅ 真实库核对（2026-06-28，Node.js 直连 `restaurant_points_dev`，手机号经 `PiiCrypto.blindHash` 命中 `users.mobile_hash`，非明文）：`13612227910` → `user_id=32`，`user_roles` 关联 `admin(role_level=100)` + `super_admin(role_level=110)` 且均 `is_active=1`；同时在 `store_staff` 是 `store_id=7` 的 `manager(active)`。即该账号 `role_level=110`（≥100），命中 `resolveStoreContext` 管理员分支，未带 `store_id` 必报 `ADMIN_STORE_ID_REQUIRED`，与现象一致。

页面因此报「提交记录加载失败」。

### 1.2 根因（直读后端真实代码，非推测）

两个接口都挂了门店上下文中间件 `resolveStoreContext({ storeIdParam: 'query' })`，其管理员分支强制要求 `store_id`：

```94:116:middleware/resolveStoreContext.js
      // 2. 管理员分支：可跨任意门店，但写操作仍必须带 store_id（审计可定位）
      if (isAdmin) {
        if (!storeId) {
          if (!required) { /* 放行：store_id=null */ }
          return res.apiError('管理员执行门店相关写操作必须指定 store_id（不允许无门店写入，以保证审计可定位）', 'ADMIN_STORE_ID_REQUIRED', null, 400)
        }
```

而列表/统计服务把「看本人 vs 看门店」这一数据维度**绑死在角色 `is_manager` 上**，前端无法切换：

```83:86:services/consumption/MerchantService.js
    // 店员只能查自己录入的（merchant_id=经手人本人）
    if (!is_manager) {
      whereClause.merchant_id = user_id
    }
```

> 结论：这不是单纯的前端漏传参数，而是**后端缺少「查询视角」这一显式契约**——「我的提交」语义上应能按「本人 / 门店 / 员工」切换，但后端只有「按角色隐式决定」一种维度，且管理员被强制要求 `store_id`。因此正确做法是**后端补齐「视角参数 + 分层数据范围校验」**，前端按视角传参，而不是前端硬塞一个 `store_id` 绕过。

## 二、目标层级模型（本次已拍板）

业务规则：**本人只能看本人；店长可看本店全部 + 可按员工分类；管理员可看全部 + 可按门店分类 + 可按人员分类；一层只能看一层（下级范围由后端递归兜底，前端无法越权）。**

| 角色 | role_level | 可用视角 | 可选门店 | 可选人员 | 数据范围（后端强制） |
| --- | --- | --- | --- | --- | --- |
| 本人 / 店员 | <40 | 仅 `self` | 否 | 否 | 锁死 `merchant_id=自己` |
| 店长 | 40~99 | `store`（默认）、`staff` | 否（仅本店/辖区） | 是（本店员工） | 本店（含辖区下级）全部 / 本店某员工 |
| 管理员 | ≥100 | `all`（默认）、`store`、`staff` | 是（分类门店） | 是（分类人员） | 全局 / 某门店 / 某员工 |

> 「一层只能看一层」由后端 `DataScopeService.getAccessibleStoreIds(自己)` 兜底：店长拿到的是本人在职门店 + 递归辖区下级门店集合；选门店/选人员若超出该集合，后端返回 403 越权，前端传什么都越不过去。

## 三、已拍板的三点（2026-06-28 定稿）

| 拍板项 | 定稿结论 | 说明 |
| --- | --- | --- |
| **1. 默认视角** | 店员=`self`（唯一）；**店长默认 `store`**（本店全部）；**管理员默认 `all`**（全局） | 进页面先看全貌，符合各角色主诉求；店员无切换 |
| **2. 统计口径统一** | **统一**：`getMerchantStats` 改为与列表 `getMerchantRecords` 同一套数据范围口径（支持 `self`/`store`/`staff`/`all` 与多店聚合） | 现状统计强制单门店，与列表口径不一致，切视角后「列表」和「统计」会对不上，属隐藏 bug，本次一并修正 |
| **3. 店长「分类员工」范围** | **含在职 + 离职历史**：按 `target_user_id` 过滤 `merchant_id`，不限制该员工当前是否在职 | 要看的是「该员工提交过的历史记录」，离职不应丢失历史 |

## 四、接口契约（后端权威定义，前端直接用同名参数，不做映射）

涉及两个**只读**接口，均新增以下查询参数：

| 参数 | 类型 | 必填性 | 说明 |
| --- | --- | --- | --- |
| `view` | string 枚举 | 可选（不传按角色取默认） | `self`/`store`/`staff`/`all`，后端按角色校验是否允许 |
| `store_id` | number | `view=store`/`staff` 时需要（店长可省=本店，管理员必带） | 目标门店，必须 ∈ 调用者可见门店集合 |
| `target_user_id` | number | `view=staff` 时必填 | 要查看的员工 user_id，必须在该门店有任职记录 |

视角与默认值解析规则：

- `view` 缺省时：店员→`self`；店长→`store`；管理员→`all`。
- `self`：任何角色可用；强制 `merchant_id=自己`，忽略 `store_id`/`target_user_id`；不再强制 `store_id`（消除 `ADMIN_STORE_ID_REQUIRED`）。
- `store`：要求角色 ≥ 店长；`store_id` 必须 ∈ 可见集合，否则 403。
- `staff`：要求角色 ≥ 店长；`store_id` ∈ 可见集合 且 `target_user_id` 在该店有任职记录；过滤 = `store_id` + `merchant_id=target_user_id`。
- `all`：仅管理员；不加门店过滤（全局）。

### 4.1 响应格式（沿用现有 ApiResponse 统一封装，不新增结构）

> ✅ 已核对 `utils/ApiResponse.js`：`res.apiSuccess(data, message, code)` 顶层确实输出 `success/code/message/data/timestamp/version/request_id`（`timestamp` 为北京时间 `YYYY-MM-DD HH:mm:ss`，`version` 固定 `v4.0`，`request_id` 由中间件注入），与文档描述一致。

⚠️ **字段名已锁定为 `view`/`view_note`（决策1，见 §13.2.1）**：
- 后端服务层把现有回显键 `query_scope`/`query_note`（列表）、`stats_scope`（统计）**一次性改名为 `view`/`view_note`**，使「请求参数名 == 响应回显名」对称（业界一致实践）。
- 前端**直接消费 `view`/`view_note`**，零映射。改名只此一处、无旧消费方，随方式2 分支一次性完成。

```jsonc
// GET /api/v4/shop/consumption/merchant/list?view=staff&store_id=1&target_user_id=31&page=1
{
  "success": true, "code": "SUCCESS", "message": "查询成功",
  "data": {
    "records": [ /* ConsumptionRecord.toAPIResponse()：consumption_record_id/user_id/merchant_id/store_id/... */ ],
    "pagination": { "page": 1, "page_size": 20, "total": 0, "total_pages": 0 },
    "view": "staff",                  // 与请求参数同名，零映射
    "view_note": "店长模式：门店1 员工31 的提交记录"
  },
  "timestamp": "2026-06-28 04:10:00", "version": "v4.0", "request_id": "req_xxx"
}
```

越权时返回 `403` + 明确 `code`：`VIEW_NOT_ALLOWED`（角色不够）/ `STORE_OUT_OF_SCOPE`（门店越权）/ `STAFF_NOT_IN_STORE`（员工不属该店）。

## 五、后端数据流（纯文本）

```
小程序请求：view + (store_id?) + (target_user_id?)
   │
   ▼
① resolveStoreContext：view∈{self,all} 时放开 store_id 强制（只读、按本人/全局维度天然可审计）
   │
   ▼
② DataScopeService.getAccessibleStoreIds(自己)
     管理员 → {scope:'all'}
     店长/区域 → {scope:'stores', store_ids:[本人门店 + 递归辖区下级门店]}
   │
   ▼
③ 按 view 校验越权（核心，后端强制）
     self  → 强制 merchant_id=自己（忽略其它筛选）
     store → 角色须≥店长；store_id ∈ store_ids，否则 403 STORE_OUT_OF_SCOPE
     staff → 角色须≥店长；store_id ∈ store_ids 且 target_user_id 在该店有任职；
             过滤 = store_id + merchant_id=target_user_id
     all   → 仅管理员；不加门店过滤
   │
   ▼
④ 落到 MerchantService 现有查询（已支持 store_scope/store_ids/merchant_id 过滤）
     列表 getMerchantRecords 与 统计 getMerchantStats 共用同一套范围解析（口径统一）
```

## 八、验收口径（以后端真实行为为准）

- 店员（role<40）：仅 `self`；带 `view=store/staff` 一律 403 `VIEW_NOT_ALLOWED`。
- 店长（40~99）：`self` 看自己；`store` 看本店全部；`staff` 看本店指定员工（含其离职后历史）；选非本店 store_id → 403 `STORE_OUT_OF_SCOPE`。
- 管理员（≥100）：`all` 看全局；`store` 看任意门店；`staff` 看任意门店任意员工；默认 `all` 时不再报 `ADMIN_STORE_ID_REQUIRED`。
- 列表与统计在同一视角下范围一致（口径统一，拍板项 2）。
- 全程不改写 nonce/库结构/写操作，仅扩展两个只读接口。

## 十四、业界怎么设计「多视角/分层数据权限查询」——对标与选型

> 你问的是同一个本质问题：「一个列表接口，按调用者身份返回不同数据范围」业界各类公司怎么做，哪种最适合本项目。下面按公司类型横向对比，最后给出**为什么本项目应选「显式 `view` 视角参数 + 服务端数据范围强制（DataScopeService）」**。

### 14.1 横向对比（5 类玩家）

| 玩家类型 | 典型做法 | 范围如何决定 | 优点 | 缺点 | 与本项目契合度 |
| --- | --- | --- | --- | --- | --- |
| **大厂·美团/阿里（数据权限中台）** | 显式「数据范围 scope」参数 + 服务端**强制**按组织树过滤；前端只表达「想看哪个维度」，能不能看由后端 RBAC+组织树裁决 | 组织架构树（部门/门店）递归 + 角色数据权限规则 | 前后端契约清晰、越权由后端兜底、可审计、易扩维度 | 需要一套数据范围服务（组织树递归） | **最高**：本项目 `DataScopeService` 正是这套（递归门店 + role_level 门槛），只差一个显式 `view` 参数 |
| **大厂·腾讯（游戏/社交后台权威服务）** | 「服务端权威（server-authoritative）」：客户端只发意图，服务端校验+裁决，绝不信客户端传的范围 | 服务端会话身份 + 权限位 | 防作弊/防越权最强 | 前端灵活性低（一切以服务端为准） | **高**：与你「以后端为准、前端适配、不信前端传 store_id 绕过」完全一致 |
| **小公司/早期 SaaS** | 角色隐式决定范围（`if(isManager) 看全店 else 看自己`），无显式参数 | 角色布尔判断 | 实现最快 | **正是本项目现状的痛点**：维度绑死在角色、不能切视角、加维度就改 if、易出越权 | 低（这就是要被替换掉的旧模式） |
| **游戏公司（虚拟物品/背包/交易行查询）** | 「视角/Tab」枚举（我的挂单/全服在售/某玩家）+ 服务端按 owner_id/scope 过滤；权威库为准 | owner_id + 可见性规则（服务端） | 视角语义直观、可扩 Tab、防刷 | 需枚举管理 | **高**：和你的 `self/store/staff/all` 四视角 Tab 思路同构 |
| **活动策划/二手平台（闲鱼类小众交易）** | 「我的发布/我的门店/全部」视角 + **特权数据访问留痕**（看他人数据写访问日志）；数据范围服务端强制 | owner + 商户/门店归属 + 审计 | 合规、可追溯、隐私可控 | 审计有存储成本 | **高**：直接支撑本项目 D2（看他人记录是否留痕） |

### 14.2 三种「范围决定方式」的本质区别（这才是关键分歧）

1. **隐式角色驱动（小公司/本项目现状）**：范围 = 角色的副作用。`if(!is_manager) merchant_id=self`。加一个维度就加一个 if，维度爆炸、口径漂移（列表和统计各写一套，正是本项目 §步骤2 的 bug 来源）。技术债最高。
2. **显式视角参数 + 客户端可信（反面教材）**：前端传 `store_id` 就给哪个店——**等于把数据权限交给前端**，越权风险极高。你文档里「前端硬塞 store_id 绕过」正是要否决的做法。
3. **显式视角参数 + 服务端强制范围（大厂主流，本方案）**：前端用 `view` 表达**意图**，服务端用 `DataScopeService` 裁决**能不能看**、看多大。意图与权限解耦，前端越不过后端。**这是美团/阿里数据权限中台、腾讯服务端权威、游戏交易行的共同收敛点**。

### 14.3 为什么方式3（显式 view + 服务端强制）最适合本项目

- **你已有半套基础设施**：`DataScopeService`（递归门店+role_level 门槛）= 大厂数据权限中台的「组织树+数据权限规则」内核；只缺一个显式 `view` 参数把意图标准化。选方式3 = **零新建、把现有能力补成业界标准形态**，而不是另起炉灶。
- **契合你的硬约束**：未上线、可一次性投入、不兼容旧接口、长期低债——方式3 一次性把「角色隐式」升级为「视角显式+服务端强制」，根除维度爆炸与口径漂移，正是降债方向。
- **技术栈零冲突**：仍是 Express 路由 + Sequelize where + ServiceManager + ApiResponse，无新框架、无新表、无新服务。
- **可扩展**：未来加「按区域」「按时间段」只是 `view` 枚举 + 复用 `DataScopeService`，对标游戏交易行「加 Tab」。

> 结论：**采用方式3（显式 `view` 视角参数 + 服务端 `DataScopeService` 强制数据范围）**，这是大厂与游戏/交易平台的收敛最佳实践，且最贴合你现有技术栈与降债目标。小公司的「角色隐式」是现状要被替换的对象；「客户端可信范围」是要明确否决的反模式。

## 十五、执行方式（采纳：方式2 引擎+回滚 + 方式3 回归测试）

> 你已选定「**方式2（引擎+回滚）+ 方式3（回归测试）**」作为落地执行方式。下面把它落成贴合本项目现有技术栈的具体步骤；本文档仅记录方案，不在此处改代码。

### 15.1 方式2：引擎化改造 + 可回滚

**「引擎」= 把「视角→数据范围」沉淀为一个统一解析单元，而非散落在路由 if 里。**

- **新增一个纯函数解析层 `resolveView(rawView, roleLevel)`**（放 `services/consumption/` 或 `utils/`，静态/纯函数，零副作用）：输入前端 `view` + 角色，输出标准化 `{ view, allowed }`；缺省按「店员 self / 店长 store / 管理员 all」。这是「引擎」的入口，列表与统计**共用同一个解析**，根除口径漂移。
- **数据范围仍由 `DataScopeService.getAccessibleStoreIds` 提供**（已是单一事实源），引擎只负责「把 view 映射成对 `store_scope`/`store_ids`/`merchant_id` 的约束」，不重复造范围逻辑。
- **可回滚设计**（关键）：
  - 所有改动走 **Git 分支 + 单一 feature 开关**。建议加一个轻量开关（如环境变量 `FEATURE_MERCHANT_VIEW=on/off` 或 config 项）：`off` 时 `/list`、`/stats` 走**旧逻辑**（角色隐式 + 强制 store_id），`on` 时走新引擎。
  - 字段改名（`query_scope`→`view`）与 `staff/list` 修复同分支提交，但**各自独立 commit**，便于单点回退。
  - DB 层**零结构变更**（不加列、不加表、不动索引——§13.2.1 已确认现有索引够用），所以「回滚」只需关开关 / `git revert`，无数据迁移回滚负担。
  - 上线顺序：先后端带开关部署（默认 `off`，等价现状）→ 小程序仓适配新契约 → 灰度把开关切 `on` → 观察 → 稳定后移除开关与旧分支代码（清债）。

### 15.2 方式3：回归测试（连真实环境，以后端真实行为为准）

在现有 Jest 体系（`jest.config.js` + `tests/`）下补**回归用例**，覆盖四视角 × 三角色 × 越权：

| 用例 | 账号/角色 | 请求 | 期望 |

| 店长 store | role 40~99 | `view=store` | 本店（含辖区）全部；选非可见店→403 `STORE_OUT_OF_SCOPE` |
