# 微信小程序 · 成长等级页对接说明（给前端）

> 文档版本：v1.0 &nbsp;|&nbsp; 编写日期：2026-07-12 &nbsp;|&nbsp; 面向：微信小程序前端开发
> 依据：后端实际接口 `routes/v4/user/growth-level.js`、`services/lottery/UserGrowthLevelService.js` 真实返回（非推测）

---

## 一、本次变更摘要

小程序「成长等级」页此前**不显示各等级的积分加成倍率**。经业务确认，成长等级倍率属营销激励信息
（同淘宝/京东/有赞会员倍率），对用户公开可增强升级动力，故后端已在成长等级接口中**新增倍率字段**。

**前端需要做的事**：读取接口新增的 `earn_multiplier` / `current_earn_multiplier` 字段，
在等级阶梯每行、当前等级卡片上展示出来。接口路径与鉴权方式不变，仅新增字段。

> ⚠️ 数据边界：本接口只下发「成长等级积分倍率」这类营销信息。
> 抽奖中奖概率、分层权重等属商业机密，后端不会下发，前端也不应尝试获取或展示。

---

## 二、接口契约

### 请求

```
GET /api/v4/user/growth-level
Header: Authorization: Bearer <access_token>   // 需登录
```

- 无需任何查询参数或请求体。
- 未登录（无有效 token）会返回鉴权失败，前端需先完成登录。

## 三、字段说明

### data 顶层字段

| 字段 | 类型 | 含义 | 备注 |
|---|---|---|---|
| `current_level_key` | string | 当前等级业务码（v1~v9） | 稳定标识，勿用于展示 |
| `current_level_name` | string | 当前等级中文名（如"荣耀殿堂"） | 用于展示 |
| `current_earn_multiplier` | number | **【本次新增】** 当前等级积分加成倍率 | 如 1.5，展示"积分加成 1.5 倍" |
| `history_total_points` | number | 用户累计获得积分（决定等级的数值） | 只增不减 |
| `thresholds_confirmed` | boolean | 等级门槛是否已定稿 | 见下方"注意事项" |
| `next_level` | object \| null | 距下一级信息 | 顶档时为 null，见下方 |
| `levels` | array | 完整等级阶梯（升序，共 9 档） | 见下方 |

### data.levels[] 每档字段

| 字段 | 类型 | 含义 | 备注 |
|---|---|---|---|
| `level_key` | string | 等级业务码 | 稳定标识，勿展示 |
| `level_name` | string | 等级中文名 | 用于展示 |
| `min_history_points` | number \| null | 达到该档所需累计积分 | 未定稿时为 null（见注意事项） |
| `earn_multiplier` | number | **【本次新增】** 该档积分加成倍率 | 如 1.05，展示"积分 ×1.05" |

### data.next_level 字段（非顶档时为对象，顶档为 null）

| 字段 | 类型 | 含义 |
|---|---|---|
| `level_key` | string | 下一级业务码 |
| `level_name` | string | 下一级中文名 |
| `points_needed` | number | 还需多少累计积分升级（1 积分 ≈ 消费 1 元） |

`next_level` 非 null 时，前端可渲染"再消费约 X 元升级至 {level_name}"。

## 四、前端改动点（成长等级页）

在截图那个「成长等级」页面，做两处展示即可（数据后端已给，前端只需渲染）：

1. **当前等级卡片**：在等级名/累计积分附近，增加一行显示 `current_earn_multiplier`。
   - 建议文案：`积分加成 1.5 倍` 或 `消费多得 50% 积分`

2. **等级阶梯列表**：每一档在"累计 X 积分"旁边，增加显示该档 `earn_multiplier`。
   - 建议文案：`积分 ×1.05` 或 `消费多得 5% 积分`

> 文案仅为建议，具体以设计稿为准。核心要求是把 `earn_multiplier` / `current_earn_multiplier` 这两个字段渲染出来。

---

## 五、注意事项

1. **数值类型**：`earn_multiplier` 是数字（如 `1`、`1.05`、`1.5`），`1` 表示 1.0 倍（无加成）。
   前端展示时如需统一两位小数（`1.00`、`1.05`），自行 `toFixed(2)` 格式化，不要向后端要字符串。

2. **`min_history_points` 可能为 null**：当 `thresholds_confirmed = false`（门槛未定稿）时，
   各档 `min_history_points` 会下发 null，此时前端**只显示等级名、不显示门槛数字**，避免误导。
   当前生产数据 `thresholds_confirmed = true`，门槛均为正常数字。

3. **顶档 `next_level = null`**：用户已达最高等级（v9）时 `next_level` 为 null，
   前端应显示"已达最高等级"而非"再消费 X 元升级"。

4. **字段命名 snake_case**：解构时注意 `const { earn_multiplier } = level`，
   不要写成 `earnMultiplier`（后端是 snake_case，写驼峰会取到 undefined）。

5. **等级共 9 档**：`levels` 固定返回 9 档（v1 铜卡 ~ v9 荣耀殿堂），升序排列。

## 六、数据边界（安全提醒）

- ✅ **本接口下发**：等级名、门槛积分、积分加成倍率（营销激励，公开）。
- ❌ **本接口不下发、前端也不应展示**：抽奖中奖概率、分层权重、风控参数等（商业机密）。
  如前端有"抽奖概率展示"需求，属另一套合规公示逻辑，需单独与后端确认，不在本接口范围。

## 七、术语对照

| 术语 | 说明 |
|---|---|
| 成长等级 | 按用户累计获得积分（history_total_points）实时派生的会员等级，共 9 档 |
| earn_multiplier | 积分加成倍率：该等级下消费获得积分的倍数（如 1.5 = 消费多得 50% 积分） |
| history_total_points | 用户历史累计获得的积分总额（只增不减），决定当前等级 |
| level_key | 等级业务码（v1~v9），稳定标识，用于逻辑判断，不用于界面展示 |
