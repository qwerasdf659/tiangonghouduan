## 一、平台定位与商业模式

一句话：**商户营销平台** —— 顾客到店消费拿积分，积分用来抽奖/兑换，奖品与兑换品做成"到店核销"的券，把顾客再次引流回店消费。形成闭环：

```
消费 → 积分 → 抽奖/兑换 → 核销 → 再消费
```

参与角色四类：
- **顾客（C 端微信小程序用户）**：消费得积分、抽奖、兑换、核销。
- **商家（店员 / 店长）**：录入顾客消费、审核、到店核销。
- **运营 / 管理员（Web 后台）**：配置活动、审核终审、看板监控。
- **系统（资产中枢）**：账本记账、双录、幂等、风控。

```
顾客到店消费

   ▼
消费记录(pending) ──审核链──> 店长审 → 管理员终审 ──通过──> 给顾客发积分(points)
   │                                                          │
   │                                                          ▼
   │                                              顾客积分余额(account_asset_balances)
   │                                                          │
   │                          ┌───────────────────────────────┼────────────────────┐
   │                          ▼                               ▼                     ▼
   │                       抽奖(消耗积分)                  兑换(消耗材料)          DIY/竞拍
   │                          │ 100%中奖                     │                     │
   │                          ▼                               ▼                     ▼
   │                  奖品: 积分/材料/券/物品 ───────────> 物品进背包(items) <───────┘
   │                                                          │
   │                                          ┌───────────────┴──────────────┐
   │                                          ▼                              ▼
   │                                  生成核销码(到店核销)              线上直接使用(use)
   │                                          │
   └◄────────────────────────────────────────┘  顾客带码到店核销 → 再次消费
```

## 三、五大业务流程详解

### 流程 1：商户消费录入与审核（积分入口）

1. **录入**：店员/店长扫顾客二维码（V2 动态码，含 exp+nonce 防重放）+ 填消费金额 → `POST /api/v4/shop/consumption/submit`，生成 `pending` 消费记录。
   - 权限：`consumption:create`（scope=store，门店归属校验）；强制 Header `Idempotency-Key`。
   - 文件：`routes/v4/shop/consumption/submit.js`。
2. **审核链**：消费记录走审核链「店长审本店 → 管理员终审」（2026-06-25 重配，模板5）。
   - 店长节点 `assignee_type='submitter_manager'`（按提交人门店动态派在职店长，审本店）。
   - 当事人回避（`exclude_parties=1`）：店长不能审自己提交的；admin(lv≥100) 可兜底越级（留痕）。
   - 引擎：`services/ApprovalChainService.js`（`processStep` / `_verifyOperatorPermission`）。
3. **发积分**：终审通过 → `services/consumption/CoreService.js` `approveConsumption()` → 给**顾客**发积分（`business_type=consumption_reward`，1 元 = 1 分）。

### 流程 2：用户抽奖（积分主消耗口）

1. **入口**：`POST /api/v4/lottery/draw`，强制幂等键，支持 1~10 连抽。
   - 中间件：鉴权 → 请求去重(5s) → 限流(20次/分) → 数据访问控制。
   - 文件：`routes/v4/lottery/draw.js`。
2. **扣费**：`UnifiedLotteryEngine.execute_draw()` 扣顾客 `points`（连抽折扣 `LotteryPricingService`）+ 扣每日抽奖配额（`LotteryQuotaService` 四维度：全局/活动/角色/用户）。
3. **抽奖管线**：统一管线 `NormalDrawPipeline` 11 个 Stage：
   - 加载活动 → 资格 → 决策来源(preset/guarantee/normal) → 预算 → 定价 → 建奖池 → 保底 → 选档位 → 选奖 → 决策快照 → 结算。
   - **100% 中奖**（奖池按 `reward_tier` 分 high/mid/low/fallback，必有兜底）。
4. **奖品发放**（`SettleStage._distributePrize`，4 类）：
   - `points` 积分 → 进积分余额（counterpart=SYSTEM_MINT）
   - `material` 材料（星石/碎片） → 进资产余额
   - `coupon`/`item` 券/物品 → `ItemService.mintItem` 进背包
5. 全程同一事务，任一环节失败全回滚（积分退回、奖品撤回、保底计数恢复）。

### 流程 3：兑换 / 竞拍 / DIY（材料消耗口 → 产出物品）

- **兑换（Exchange）**：用**材料资产**（碎片/星石，非积分）兑换商品（SPU `exchange_items` → SKU `exchange_item_skus` → 渠道价 `exchange_channel_prices`）。
  - 复合门槛 `_assertRedeemRequirement`：成长等级 + 额外资产余额 + 持有指定道具。
  - 核心：`services/exchange/CoreService.js` `exchangeItem()`。实物走发货链（强制地址快照），虚拟/券即时 `completed`。
  - 订单状态机：`pending → approved → shipped → received → rated → completed`（含退款/取消分支；prop 道具单买入即消耗，禁退）。
- **竞拍（Bid）**：官方对纯虚拟道具（prop）发起竞价，用户用白名单资产（星石等）**冻结**出价，价高者得。
  - 核心：`services/exchange/BidService.js` `placeBid()` / `settleBidProduct()`（定时结算→铸造入背包）。
  - 入口：`routes/v4/exchange/bid.js`。
- **DIY 饰品设计**：用户基于模板选珠子/宝石素材自己设计饰品，确认冻结资产 → 完成铸造成实物物品。
  - 作品状态机：`draft → frozen → completed/cancelled`。
  - 核心：`routes/v4/diy.js` + `services/diy/`（WorkService 等）。

> 三者最终都落到 `ExchangeRecord` + 铸造 `items` 进背包，复用同一资产中枢与物品中枢。

### 流程 4：背包与核销（物品出口，引流回店）

- **背包双轨**（`routes/v4/backpack/index.js` + `services/BackpackService.js`）：
  - `assets[]`：可叠加资产（来自 `account_asset_balances`）。
  - `items[]`：不可叠加物品实例（来自 `items` 表）。
- **物品两种出口**：
  - **生成核销码**（`POST /backpack/items/:item_id/redeem` → `RedemptionService.createOrder`）：美团 O2O 模式，12 位 Base32 码（明文只返一次、库存 SHA-256 哈希）+ 动态 QR（默认 5 分钟）。物品被 `holdItem` 锁定。
  - **线上使用**（`POST /backpack/items/:item_id/use` → `ItemService.consumeItem`）：数字物品直接消耗。
- **到店核销**（`routes/v4/shop/redemption/`）：
  - 扫码核销 `POST /shop/redemption/scan`（验签 + 哈希前缀防篡改）或文本码 `POST /shop/redemption/fulfill`。
  - 校验：商家一致性（物品 merchant 必须 == 核销门店 merchant）、门店专属券范围、状态/有效期。
  - 核销 → 物品 `consumeItem(business_type='redemption_use')` 消耗，单 `status=fulfilled`。
  - 状态机：`pending → fulfilled / cancelled / expired`。

### 流程 5：资产中枢（贯穿一切的账本）

- **三层模型**：`Account`（账户：user / system 如 SYSTEM_BURN/SYSTEM_MINT/SYSTEM_ESCROW）+ `AccountAssetBalance`（余额：available/frozen）+ `AssetTransaction`（流水）。
- **核心收口**：`services/asset/BalanceService.js` `changeBalance/freeze/unfreeze/settleFromFrozen`（可叠加资产）；`services/asset/ItemService.js` `mintItem/holdItem/releaseHold/consumeItem`（不可叠加物品，三表 Item/ItemHold/ItemLedger）。
- **铁律**：
  - **双录记账**：每笔变动对手方写反向流水，全局 `SUM(delta_amount)=0`。
  - 强制事务 + 行级锁；`idempotency_key` 唯一兜底；禁负余额；单笔与余额 ≤ 10 亿。
  - `BUDGET_POINTS` 按 `lottery_campaign_id` 隔离；`history_total_points` 仅正向入账时累加（成长等级单一数据源）。
- **抽奖一次的资产流转示意**：
```
扣积分:   用户.points  -N  ⇄  SYSTEM_BURN.points  +N   (lottery_consume)
发奖(积分): SYSTEM_MINT.points -M ⇄ 用户.points +M       (lottery_reward)
发奖(材料): SYSTEM_MINT.<mat> -K ⇄ 用户.<mat> +K          (lottery_reward_material)
发奖(券/物): items + item_ledger 双录                       (lottery_mint)
```

## 五、关键设计特征

1. **零映射直读**：前后端统一 snake_case，前端直读后端字段，不做映射。
2. **资产安全**：双录记账 + 强制事务 + 幂等键，杜绝资损与重复发放。
3. **数据脱敏**：C 端经 `DataSanitizer` 出口，屏蔽 `budget_points`、`star_stone_quota` 等商业机密。
4. **配置化运营**：版本闸门、臻选门槛、审核链节点、活动等大量参数存 DB，运营后台可调，不改代码。
5. **审核链通用化**：消费、商家积分共用一套可配置审核引擎（按 `auditable_type` + `match_conditions` + `priority` 选模板）。
6. **物品/资产双轨**：可叠加资产走 BalanceService，不可叠加物品走 ItemService 三表模型，职责清晰。
7. **分层与收口**：路由不直连 models，读写收口到 Service，事务边界由路由层 `TransactionManager` 统一管理。

## 七、商业模型与薪酬体系设计（操盘手视角）

> 本章为业务/经营设计分析，非代码实现。基于操盘手提供的真实经营参数：收入=用户到店消费额(GMV)的 10% 佣金；实物奖品=GMV 的 8%，由平台从佣金中支出。

### 7.1 营收与成本模型（每 100 元消费）

```
用户到店消费 100 元(GMV)
  → 平台佣金收入 = GMV×10%      = 10.00 元   ← 全部收入
  → 实物奖品支出 = GMV×8%(平台出) =  8.00 元   ← 从佣金里掏（占佣金 80%）
  → 服务器/技术（一年约4000，自理） ≈  0.03 元
  ───────────────────────────────────────────
  平台净利     ≈ 1.97 元/百元消费   （净利率 ≈ 20% of 佣金 / 2% of GMV）
```

**关键认知**：
- 奖品口径已确认为 **GMV 的 8%、由平台从佣金支出**（不是商家出、也不是佣金的8%）。
- 奖品吃掉 **80% 的佣金**，平台只剩约 **2%（占 GMV）** 作为"利润 + 全部人力"的总空间。
- 本质是**薄利多销 + 规模放大**：奖品猛补贴 → 商家/用户爱用 → 易铺量 → 靠 `月流水 × 2%` 赚钱。

### 7.2 提成基数铁律：按佣金，不按 GMV

| 方式 | 算法（店促成本店月流水10万为例） | 提成额 | 占平台该店佣金(1万) |
| --- | --- | --- | --- |
| 按 GMV 提 2% | 100,000 × 2% | 2,000 | 掏走佣金 20% 😱 |
| 按佣金提 2% | 10,000 × 2% | 200 | 掏走佣金 2% ✅ |

> 同写"2%"，按 GMV 算是按佣金算的 10 倍。**所有提成必须以佣金为基数**；系统 `consumption_records` 存的是 GMV，需先算 `佣金=GMV×10%` 再提成。

### 7.3 规模放大逻辑（员工是收入放大器，不是成本）

```
平台总收入 = 商户数 × 每店月流水 × 10%佣金
单干：自己谈，商户数是瓶颈
招地推：一个地推月谈数十家店 → 商户数翻倍 → 收入翻倍
```

- 员工带来**增量佣金**，提成从其"自己创造的增量"里分，盘子本身变大，不是从固定盘子里挤。
- 一个地推自养示例：其拉的店合计月流水 50 万 → 佣金 5 万 → 扣奖品(GMV8%=4万) → 剩 1 万，供"地推提成 + 平台利润"分。**要求地推走量**。

### 7.4 两套薪酬模型

**模型一·单干期（现在）**：无员工工资，无需薪酬体系，只有平台利润（约 2 元/百元消费）。任务是**冲流水 + 控奖品真实成本**。三个要盯：流水量级、奖品真实履约成本、现金流时间差。

**模型二·团队期（将来招人）**：核心矛盾——奖品占佣金 80%，只剩 20% 不够养团队。两条路：
- **路线A**：维持 GMV 8% 奖品，靠地推狂拉店冲流水，靠走量摊薄（利润薄、提成空间小）。
- **路线B**：招人时把奖品降到 GMV 5-6%，腾出空间养团队（补贴变少、利润变厚）。

> 奖品比例是两阶段的总开关。月利润 = 月消费总流水 × 净利率；流水 1000 万 × 2% = 20 万/月才支撑得起团队。

### 7.5 各岗位激励设计（数据平台均可取）

| 岗位 | 提成挂钩 | 平台数据来源 |
| --- | --- | --- |
| 地推 | 拉新店数 + 新店首月佣金（绑出单防僵尸店） | `merchants`/`stores` 创建归属 |
| 店员 | 本店"审核通过的消费佣金" + 核销量 | `consumption_records`、`redemption_orders` |
| 店长 | 本店整体业绩达标奖 | 本店聚合 |
| 区域负责人 | 辖区总佣金阶梯奖 | `DataScopeService` 递归辖区 |

### 7.6 地推提成三模式（核心拍板项）

| 模式 | 做法 | 长期成本 | 防躺赚 | 逼拉新 |
| --- | --- | --- | --- | --- |
| 一次性拉新奖 | 新店出单给固定奖 | 低可控 | 强 | 中 |
| 持续分成 | 老店一直出单就月月抽成 | **高/累积/危险** | 弱（老地推躺赚） | 弱 |
| **混合（推荐）** | 拉新奖 + 新店前3月分成 + 之后转走 | 中可控 | 强 | **强** |

**推荐混合模式**（匹配"规模化拉店"诉求）：
```
地推薪酬 = 低底薪（合规兜底）
        + 拉新奖：新店首月出单，X 元/店
        + 扶店分成：新店第1-3月，该店佣金的 Y%
        + 第4月起该店转维护团队，地推去拉下一家
封顶：分成有时间窗（3个月），不累积成永久负债
```

### 7.7 三条防坑/合规铁律

1. **合规兜底**：实发不得低于当地最低工资，纯提成也要兜底。
2. **防刷单**：提成只认"审核通过"的消费 + 平台风控（大额/高频/跨店检测）。
3. **防僵尸店**：拉新奖绑"新店首月产生 X 佣金"才全额发，防只签不活。

### 7.8 待校准的关键数（强烈建议下一步做）

奖品是"积分→抽奖/兑换→实物核销"，**大量会沉淀未核销**（`redemption_orders` 有 pending/expired）。所以**账面 8% 是预算上限，真实履约成本可能更低**。

> 建议连真实库 `restaurant_points_dev` 统计：已发放实物奖品总价值 vs 真实核销（实际发货）vs 沉淀未核销 → 算出"账面 8%"实际花了几个点。该数决定养团队的真实空间与走路线 A/B。

> 本章为经营设计分析，结合平台真实数据能力梳理，未改动任何代码与数据库。各比例/金额为待操盘手填入的真实决策值。

## 八、薪酬体系落地方案（招员工执行版）

> 已确认前提：① 团队总成本锁死 = **GMV 的 2%**（操盘手原话）；② 操盘手 3 年不拿钱、长期主义、前期利润全让团队；③ 提成以佣金为基数；④ 用佣金分成自动过滤"出几单就不动"的僵尸店。

### 8.1 各类公司地推/分销薪酬对比（设计依据）

| 公司类型 | 地推/分销薪酬核心 | 本方案是否采用 |
| --- | --- | --- |
| 美团/饿了么铁军 | 低底薪+阶梯提成，按"有效交易商户"考核，签约不算、出单才算 | ✅ 按出单佣金算 |
| 阿里中供铁军 | 底薪+高提成，签约后客户成功团队接手维护 | ✅ 拉新与维护分离 |
| 腾讯/游戏分销 | CPS 按效果分成，带时间窗衰减 | ✅ 前3月分成、到期转走 |
| 小公司/创业 | 低底薪+纯提成，控成本 | ✅ 预算紧，低底薪 |
| 二手/虚拟交易平台 | 拉新小奖 + 成交持续分成 | ✅ 混合 |

> 共同铁律：不按签约数给大钱、按真实交易算、低底薪高浮动、定义"有效商户"。
> 本方案最终选型：**低底薪 + 纯佣金分成为主 + 首单小额防白干奖**（预算锁死 GMV2% 永不超支、最防僵尸店、地推自驱省管理）。

### 8.2 GMV 2% 团队预算分配（每百元消费 = 2 元）

```
GMV 2.0% 团队总盘
  ├─ 地推       GMV 1.2%（拉新+扶店，主力）
  ├─ 店员/核销  GMV 0.5%（做厚单店流水、引流回店）
  └─ 店长/区域  GMV 0.3%（管理层）
合计 = GMV 2.0%（锁死不超）
```

> 注：GMV 2% = 佣金扣 8% 奖品后的全部留存。即团队期操盘手把单干利润全让给团队、自己 3 年不拿钱，平台打平/微亏，赌 3 年规模 + 奖品沉淀红利。与长期主义前提一致。

### 8.3 地推薪酬（第一个要招的岗位）

```
月收入 = 底薪 + 首单奖 + 扶店分成

① 底薪    = 当地最低工资（合规底线，示例 2200 元）
② 首单奖  = 新店首月真实出单，100 元/店（只防白干，不给多，避免骗签约）
③ 扶店分成 = 新店第 1-3 个月，从该店产生的「佣金」抽 12%
            第 4 个月起转维护团队，地推不再分成 → 逼其持续拉新
```

### 8.4 其他岗位（随规模逐步招）

| 岗位 | 薪酬结构 | 平台数据来源 |
| --- | --- | --- |
| 店员 | 底薪 + 本店"审核通过消费佣金"的 5% + 核销单×小额单价 | `consumption_records`、`redemption_orders` |
| 店长 | 底薪 + 本店月佣金达标奖 | 本店聚合 |
| 区域负责人 | 底薪 + 辖区总佣金阶梯奖 | `DataScopeService` 递归辖区 |

### 8.5 "出几单就不动"如何被自动解决

```
僵尸店：首月出 3 单共 600 元 → 佣金 60 元 → 地推抽 12% = 7.2 元
→ 地推发现拉这种店赚不到钱 → 自动改拉优质店
→ 利益机制自动过滤，无需人工查僵尸店
```

> 这是"按佣金分成"相对"一次性签约奖"的核心优势：地推收入与店是否持续出单天然绑定。

### 8.6 三条防坑铁律（写进劳动合同/提成规则）

1. **合规兜底**：底薪不低于当地最低工资，纯提成也要兜底（否则违法）。
2. **防刷单**：提成只认"审核通过"的消费 + 平台风控（大额/高频/跨店检测）。
3. **防僵尸店 / 防躺赚**：首单奖绑真实出单；扶店分成只给前 3 个月，到期转走。

### 8.7 待调数字（默认值已给，可直接用或微调）

| 参数 | 默认值 | 说明 |
| --- | --- | --- |
| 地推底薪 | 当地最低工资 | 合规底线 |
| 首单奖 | 100 元/出单新店 | 防白干，不鼓励骗签约 |
| 扶店分成率 | 佣金的 12% | 前 3 个月 |
| 扶店分成时间窗 | 3 个月 | 到期转维护团队 |
| 店员消费佣金提成 | 5% | 审核通过的消费佣金 |
| 团队总成本上限 | GMV 2% | 硬约束，所有岗位提成总和不超 |

> 本章为薪酬落地方案，数字为基于行业实践 + 操盘手约束（GMV2%、3年不拿钱）拟定的可执行默认值，操盘手可按真实情况微调。未改动任何代码与数据库。后续若需"按平台真实消费/核销数据自动算薪"，可基于 `consumption_records`/`redemption_orders`/`DataScopeService` 落地结算模块。

> 操盘手已拍板：**奖品维持 GMV 8%、团队维持 GMV 2%（占佣金 20%），均不调整** —— 认可"激进补贴换增长 + 长期主义"的定位。本章为对标依据存档。

| --- | --- | --- | --- |
| 美团地推铁军 | 营收 15%–25% | 占营收（佣金） | 本平台占佣金 20%，区间中部 ✅ |
| 阿里中供铁军 | 带来营收 10%–20% | 占其带来营收 | 扶店分成 12% 在区间 ✅ |
| SaaS/企业销售 | 首年合同额 10%–30% | 占合同额 | 偏高（获客难） |

### 9.3 本平台方案 vs 行业（速览）

| 维度 | 本平台设定 | 行业区间 | 评价 |
| --- | --- | --- | --- |
| 奖品补贴 | GMV 8% | 补贴型常态 1–5% | 偏高（激进换增长，已拍板维持） |
| 团队总成本 | GMV 2%（佣金 20%） | 营收 15–25% | 合理（已拍板维持） |
| 地推扶店分成 | 佣金 12% / 前 3 月 | 10–20% | 合理 |

> 结论：本平台"团队分成比例"行业合理，"奖品补贴比例"行业偏高。一紧一松，是"3 年不拿钱换增长"战略的直接体现。下一步建议连真实库核算奖品真实履约成本率（见 7.8），校验 8% 的真实利润空间。

## 十、东莞用工成本与落地薪酬方案（结合官方数据）

> 数据来源：东莞市政府/人社局官方公布（2026 年度），已联网核实。本章把抽象提成方案落到东莞真实用工成本上。

### 10.1 东莞官方数据（2026 年度）

**最低工资**（2025-03-01 起执行，延续 2026）：
- 全日制：**2080 元/月**（日 95.63 元 / 时 11.95 元）
- 非全日制：**19.8 元/小时**

**社保最低缴费基数 = 4775 元**，单位承担部分：

| 险种 | 单位比例 | 单位月缴（基数4775） |
| --- | --- | --- |
| 养老 | 16% | 764.00 |
| 医疗（无个账） | 3.8% | 181.45 |
| 失业 | 0.8% | 38.20 |
| 工伤 | 0.2%（一类行业最低） | 9.55 |
| 生育 | 0.7% | 33.43 |
| **单位社保合计** | | **≈ 1026 元/月** |

> 医疗若选"有个账"单位 4.3%，单位社保合计约 1050 元/月。个人部分（养老8%+医疗0.5%+失业0.2%）从员工工资代扣，不算企业成本。

### 10.2 一个全职地推的真实用工成本

```
底薪（最低工资）      2080 元
+ 单位社保           1026 元
──────────────────────────
全职固定成本       ≈ 3106 元/月（未含提成）
```

⚠️ **预算冲突预警**：团队总成本预算 = GMV 2%。单个全职地推光"底薪+社保"3106 元，需其每月拉来 `3106 ÷ 2% ≈ 15.5 万元消费流水`才能覆盖自身固定成本（未含提成）。新人前 1-2 月大概率达不到 → 早期会实打实贴钱。**固定成本（社保）是早期最大压力，用工形式的选择比提成比例更影响成本。**

### 10.3 三种用工形式对比（核心决策点）

| 形式 | 月固定成本 | 社保 | 合规风险 | 适合 |
| --- | --- | --- | --- | --- |
| 全职（标准劳动合同） | 底薪2080 + 社保1026 ≈ **3106** | 必须缴 | 低 | 核心骨干 |
| 非全日制（小时工） | 时薪≥19.8，**月内每天≤4h、不超规定时长** | 仅需缴工伤 | 中（超时长即违规） | 兼职地推 |
| 劳务/合伙人（纯佣金） | **0 底薪**，纯按佣金分成结算 | 不缴（对方自理） | 中（需真实劳务关系，防"假外包真用工"） | 提成制地推（早期最常用） |

### 10.4 推荐方案（结合 3 年不拿钱 + 预算极紧 + 规模化拉店）

**地推 → 走"纯佣金劳务/合伙人"模式（零固定成本，完全按效果付费）**：

```
地推（劳务/合伙人，无底薪不缴社保）：
  收入 = 首单奖 100 元/出单新店
       + 扶店分成：新店第 1-3 月，该店佣金的 12%
       + 第 4 月起转维护，去拉下一家
  → 零固定成本，拉到店才付钱，与"佣金分成"设计天然契合，最适合早期
  ⚠️ 须签真实劳务/合作协议，避免被认定为事实劳动关系
```

**核心骨干（将来规模起来，要留住的人）→ 转全职**：
```
全职 = 底薪 2080（东莞最低工资）+ 单位社保约 1026 + 提成
  → 此时单人固定成本 3106 元/月，需其稳定贡献 ≥15.5 万/月流水覆盖
  → 规模和现金流起来后再转，早期不背这个固定成本
```

### 10.5 落地节奏（资金安全优先）

| 阶段 | 用工形式 | 你的成本敞口 |
| --- | --- | --- |
| 起步（验证模式） | 全部纯佣金劳务 | 零固定成本，拉到店才付佣金 |
| 增长（流水起规模） | 骨干转全职 + 兼职用非全日制 | 开始承担社保，但流水已能覆盖 |
| 成熟（3年后） | 标准全职体系 | 规模化，操盘手启动提取 |

> 关键：**早期绝不背全职社保固定成本**，用纯佣金劳务把风险降到零；等单人能稳定贡献 15 万+/月流水（覆盖 3106 固定成本）再转全职。这样即便招人也不会击穿你的现金流。

> 本章基于东莞 2026 官方最低工资/社保数据 + 操盘手既定约束拟定，未改动任何代码与数据库。劳务/合伙人用工须签合规协议，建议咨询本地人力/法务确认事实劳动关系认定边界。

## 十一、用工路线选择与"会不会亏钱"测算

> 本章解决两个核心顾虑：① 纯佣金 vs 全职怎么选、试用期能不能用；② 招地推会不会亏。

### 11.1 两条用工路线详细对比

| 维度 | 路线A 纯佣金合伙人 | 路线B 全职+合法试用期 |
| --- | --- | --- |
| 关系性质 | 劳务/合作（非劳动关系） | 劳动关系 |
| 你的固定成本 | **0** | ≈3106/月/人（底薪2080+社保1026） |
| 法律风险 | 极低（无试用期/辞退纠纷） | 中（程序错即违法解除赔钱） |
| 社保 | 不缴（对方自理） | 必须缴 |
| 招人难度 | 难（无保障） | 易（有底薪社保） |
| 能否考勤/KPI管理 | **不能**（管了就成事实用工） | 能 |
| 忠诚度/稳定性 | 低 | 高 |
| 筛人方式 | 市场自动淘汰（拉不到没收入自己走） | 试用期考核（需合法程序） |
| 适合阶段 | **早期验证（现在）** | 成熟期（3年后留骨干） |

### 11.2 试用期合规红线（若走全职）

直接"25天没拉到8万就说试用不合格辞退"有违法解除风险。合法前提**三条缺一不可**：
1. **合同期限决定试用期上限**：3个月-1年→试用期≤1个月；1年-3年→试用期≤2个月。要设2个月试用期，合同必须签1年以上。
2. **录用条件须入职前书面约定并签字**：如"25个工作日内拉≥8万元有效消费流水（审核通过+无刷单）"，事后口头说"不合格"无效。
3. **试用期工资≥正式80%且≥最低工资2080，社保照缴**；解除须留业绩数据证据（平台 `consumption_records` 审核通过流水即铁证）。

> 建议：早期把"8万"作为**纯佣金合伙人的合作续约标准**（首月没拉到就不续合作），而非"员工试用期考核"——同样筛人，完全避开劳动法风险。

### 11.3 推荐：分阶段混合用工

```
第1阶段（现在/验证期）：全部纯佣金合伙人
  - 零固定成本、零法律风险；"拉不到没收入"用市场机制自动筛选，不碰试用期
第2阶段（跑出能稳定拉8万+/月的人）：转全职
  - 已验证能出量，转全职值得；签1年合同+合法试用期+阶梯提成绑住
第3阶段（3年规模成型）：标准全职团队+完整提成体系
```

### 11.4 "招地推会不会亏" — 结论：纯佣金不会亏

**核心**：纯佣金下，付给地推的每一分都来自他已赚到的佣金；他没拉到，你付 0。

```
地推拉来一家店，当月消费 1 万元：
  佣金 1000 - 奖品(8%)800 - 地推分成(佣金12%)120 = 你剩 80 元 ✅ 正
地推当月颗粒无收：
  你付 0 元 ✅ 不亏
```

两种情况都不亏。"会亏"只来自两种**非地推本身**的情况：
- **全职且没出量** → 每月白贴 3106 → 故早期用纯佣金消除此风险。
- **奖品 8% 给太狠** → 不是"亏"是"赚得薄"，且 8% 很可能虚高（见 11.5）。

### 11.5 利润其实可能比账面厚（待真账校验）

奖品是"积分→抽奖/兑换→实物核销"，大量用户抽到但**不会真核销/兑换**，那部分不发生成本。
```
账面：奖品 = GMV 8%
实际：若大量沉淀未核销，真实履约可能仅 GMV 4-5%
  → 佣金10 - 真实奖品5 - 地推1.2 = 你剩 3.8 元（远高于账面 0.8 元）
```

> 强烈建议下一步：连真实库 `restaurant_points_dev` 统计「已发放奖品总价值 vs 真实核销 vs 沉淀未用」，算出账面 8% 实际花几个点。该数直接决定真实利润空间与养团队的宽裕度。

> 本章为用工与经营测算分析，未改动任何代码与数据库。用工合规涉具体法律，正式签约前建议本地劳动法律师审阅协议（尤其路线A防"假外包真用工"边界）。

## 十二、社保公积金强制性与全职到手测算（东莞）

> 本章厘清"社保公积金是否必缴"，并把全职用工的真实成本与员工到手金额算清。数据为东莞 2026 官方口径。

### 12.1 社保 / 公积金强制性

| 项目 | 是否强制 | 说明 |
| --- | --- | --- |
| 社保（五险） | **法定强制** | 只要建立劳动关系，自用工起30天内须缴；员工签"自愿放弃"承诺书也无效，事后可要求补缴+赔偿 |
| 公积金 | 法律也要求 | 《住房公积金管理条例》规定单位应缴存；但执行力度比社保松，初创常暂缓或按最低5%缴（有合规瑕疵） |

**唯一合法"不缴"路径 = 不构成劳动关系**：纯佣金劳务/合作关系（地推是合作方非员工）→ 社保公积金都不用平台缴，对方自理。这是早期地推走纯佣金的关键优势之一。

### 12.2 各用工形式的缴纳义务

| 用工形式 | 社保 | 公积金 |
| --- | --- | --- |
| 纯佣金合伙人（劳务） | 不用缴（无劳动关系） | 不用缴 |
| 全职（劳动关系） | 必须缴（强制） | 法律要求，初创多先只缴社保、公积金暂缓 |
| 非全日制（小时工） | 仅需缴工伤 | 一般不缴 |

### 12.3 全职单人固定成本（东莞，含/不含公积金）

```
底薪（最低工资）          2080 元
+ 单位社保               1026 元
────────────────────────────
不含公积金成本         ≈ 3106 元/月
+ 单位公积金（最低5%，基数4775 → 约239）
────────────────────────────
含公积金合规全成本     ≈ 3345 元/月
```

### 12.4 员工到手测算（个人代扣从应发工资中扣，非额外加）

```
应发工资（最低工资）       2080 元
- 个人社保（按社保基数4775算，非按2080）：
    养老8% + 医疗0.5% + 失业0.2% = 8.7% × 4775 ≈ 415 元
────────────────────────────
员工实际到手             ≈ 1665 元/月
```

**两个合规要点（避免误解与纠纷）**：
1. 最低工资 2080 指**应发工资**（扣个人社保/个税前）；依法代扣后到手低于 2080 **是合法的**。
2. **社保基数倒挂**：工资 2080 但社保下限基数 4775，个人仍按 4775 扣（约415），到手约 1665。员工易有"说好2080怎么到手1665"的情绪，部分地区稽查也会质疑"工资低于社保基数"。

> 处理选择：① 工资与社保基数都按最低（到手低、易有情绪）；② 把工资提到接近社保基数 4775（到手体面、但你成本更高）。

### 12.5 对用工策略的印证

全职不仅有 1026 单位社保（或含公积金 3345 总成本），员工到手还低（1665）、易有纠纷与稽查质疑。**纯佣金合伙人则无最低工资约束、无社保代扣、佣金分成直接到手、数字清晰** —— 再次印证"早期纯佣金、规模起来再转全职"对操盘手最省成本、最合规、最少纠纷。

> 本章基于东莞 2026 官方最低工资/社保数据，未改动任何代码与数据库。社保稽查力度各地不同，正式用人前建议本地人力/财税顾问确认实际执行口径。

## 十四、薪酬体系范围界定（谁是平台的人、谁是商家的人）

> 2026-06-28 确认：店员、店长是**商家自己的人**，平台不发工资。平台唯一需设计薪酬的岗位是**地推**。

### 14.1 角色归属与用工成本

| 角色 | 系统角色 | 是谁的人 | 平台发工资？ | 在平台干什么 |
| --- | --- | --- | --- | --- |
| 地推 | （平台员工） | **平台雇的** | ✅ 是（全职 2080+社保） | 拉商户入驻平台 |
| 店长 | merchant_manager(lv40) | 商家的人 | ❌ 否 | 审核本店消费、管店员 |
| 店员 | merchant_staff(lv20) | 商家的人 | ❌ 否 | 扫码录消费、扫码核销 |
| 运营/客服 | （操盘手本人） | 操盘手 | ❌ 否（3年不拿钱） | 平台运营 |

### 14.2 店员/店长的真实定位（基于真实代码核查）

- **店员（merchant_staff, lv20）**：商家门店的服务员/收银员，平台只给账号+权限（`consumption:create` 录消费、`consumption:scan_user` 扫码核销），**工资由商家自己发**。例：老良记(store_id=7) 店员 user_id=12796。
- **店长（merchant_manager, lv40）**：商家门店负责人，有 `staff:manage`（管店员）、审核本店消费（submitter_manager 节点）等权限，**也是商家的人**，平台不发工资。
- 二者均通过 `store_staff` 表挂在商家门店下（`role_in_store` = staff/manager），由店长自助增减（平台提供账号管理能力）。

### 14.3 结论：平台薪酬体系 = 地推一个岗位

- 平台唯一的用工成本与薪酬设计对象是**地推**（已定稿：全职、底薪2080、25天8万试用、阶梯提成、合同三件套齐备）。
- 店员/店长是商家用工，平台**零用工成本**；平台对这块要做的是**产品体验**（让店长能自助管理店员、扫码录入/核销顺手），而非薪酬。
- 运营/客服前期由操盘手本人承担（3年不拿钱）。

> 本章为薪酬范围界定，基于真实角色数据（merchant_staff/merchant_manager + store_staff 表）核查，未改动任何代码与数据库。
