## 一、项目背景

餐厅积分抽奖系统 V4，核心业务流程：用户消费 → 获得积分 → 抽奖/兑换 → 获得奖品。

- 技术栈：Node.js 20+ / Express 4.18 / Sequelize 6.35 / MySQL (mysql2) / Redis (ioredis) / Socket.io 4.8
- API 风格：RESTful 扁平化 `/api/v4/{resource}`，认证 JWT + RBAC (`requireRoleLevel`)
- 抽奖引擎：Pipeline 架构（11个阶段），通过「预算档位(B0-B3) × 压力档位(P0-P2)」矩阵动态调整奖品概率
- 策略配置表：`lottery_strategy_config`，唯一键 `(lottery_campaign_id, config_group, config_key, priority)`
- 配置加载器：`DynamicConfigLoader`（5分钟缓存 + 活动级隔离），位于 `services/UnifiedLotteryEngine/compute/config/ComputeConfig.js`
- 配置 CRUD 服务层：`QueryService.getStrategyConfig()` + `CRUDService.updateStrategyConfig()`，通过 `LotteryStrategyConfig.upsertConfig()` 写入
- Web 管理后台：Vite + Alpine.js + Tailwind CSS，策略开关由 `admin/src/modules/lottery/composables/strategy.js` 的 `toggleActivityStrategySwitch()` 统一处理
- 微信小程序前端：独立仓库，不在本项目中；后端通过 `WXBizDataCrypt` + `platformDetector` 支持微信登录

### 2.4 发现的 bug：diamond_quota 存在两条断裂路径

**路径 A（抽奖引擎 — 死代码）**：`BuildPrizePoolStage.js` 第 112-132 行：

```javascript
const strategy_config = campaign_data.strategy_config || {}
const diamond_quota_enabled =
  strategy_config.diamond_quota_enabled === true ||
  strategy_config.diamond_quota_enabled === 'true'
```

`campaign_data` 来自 `LoadCampaignStage.data`，其返回结构仅包含 `campaign`、`prizes`、`tier_rules`、`fallback_prize`、`pick_method`、`budget_mode`，**没有 `strategy_config` 字段**。`lottery_campaigns` 表 41 个列中也无此字段。

结果：`strategy_config` 始终为 `{}`，`diamond_quota_enabled` 永远为 `undefined`，**抽奖引擎中的钻石配额过滤从未执行**。

**路径 B（消费服务 — 绕过 DynamicConfigLoader）**：`services/consumption/CoreService.js` 第 714-754 行 `_getDiamondQuotaConfig()`（注：此方法因代码增长已从早期的第 538 行移至当前位置）：

```javascript
const config = await LotteryStrategyConfig.findOne({
  where: { config_key: 'diamond_quota_enabled', is_active: true },  // 无 config_group 筛选
  transaction
})
```

此方法直接用 Sequelize 查询 `LotteryStrategyConfig`，绕过 `DynamicConfigLoader` 缓存体系。三个问题：

1. **缺少 `config_group` 筛选**：当前因 `diamond_quota_enabled` 在全表中唯一，不会误匹配。但若 key 重命名为 `enabled`，`config_key: 'enabled'` 会匹配到 pity/anti_empty 等所有 group 的记录，造成严重错误。
2. **不走 DynamicConfigLoader**：没有缓存、不走 `getParsedValue()` 标准转换，自行实现 `=== 'true' || === true` 逻辑。
3. **用途是消费金额 × ratio 发放钻石配额**：`quotaAmount = Math.floor(consumption_amount * quotaConfig.ratio)` — 这是消费流程而非抽奖流程，但配置共用同一张表。

DB 中 `diamond_quota.diamond_quota_enabled = true` 的记录（4 个活动各一条）被路径 B 读取但路径 A 未读取。

### 2.5 三种读取路径的对比

| 路径 | 使用者 | 位置 | 是否工作 |
|---|---|---|---|
| `DynamicConfigLoader.getValue(group, 'enabled', ...)` | pity/anti_empty/anti_high/luck_debt/pressure_tier/matrix/management/guarantee | 抽奖引擎各 Stage | 正常 |
| `campaign_data.strategy_config.xxx` | diamond_quota（抽奖引擎侧） | `BuildPrizePoolStage.js` | **已断裂（bug）** |
| `LotteryStrategyConfig.findOne({ config_key })` | diamond_quota（消费服务侧） | `CoreService._getDiamondQuotaConfig()` | 能工作但绕过缓存体系 |

根因：策略配置从 `lottery_campaigns` 表迁移到 `lottery_strategy_config` 表时，`BuildPrizePoolStage` 的读取路径没有同步更新；`CoreService` 在后续开发中直接操作模型绕过了 `DynamicConfigLoader`。

## 三、行业参考

### 3.1 大厂（美团/腾讯/阿里）

- 使用集中式配置中心（Apollo / Nacos / Diamond）
- 核心原则：**全系统只有一种方式查询「某策略是否开启」**，如 `configCenter.isEnabled('feature', context)`
- Feature Toggle（布尔开关）和 Configuration（参数值）是两层概念，所有开关遵循统一的 key 命名（`enabled`）
- 配置按 namespace / group / key 三级结构，支持按活动维度隔离

本项目的 `lottery_campaign_id + config_group + config_key` 设计与此模式完全吻合。

### 3.2 游戏公司（米哈游/网易游戏）

- 抽卡系统（原神 / 阴阳师）的概率引擎，每个保底机制（pity / guarantee / soft_pity）都有标准 `enabled` 开关
- 不存在有的策略有开关、有的没有的情况 — 监管要求概率可审计，每个影响概率的机制必须可独立关闭、可追溯
- 配置热更新 + 缓存（本项目的 `DynamicConfigLoader` 5分钟缓存 + `clearCache()` 已实现此模式）

### 3.3 小公司 / 二手平台

- 起步时把配置硬编码或放在主表列里，后期迁移到独立配置表
- 本项目已走过这个阶段（已有 `lottery_strategy_config` 表 + `DynamicConfigLoader`），架构90%正确
- 当前的问题属于「迁移未完成的最后10%技术债」

### 3.4 共同结论

所有成熟系统都只有一种读取路径。不存在一半策略通过 `DynamicConfigLoader.getValue()` 读、另一半通过 `campaign_data.strategy_config` 读。两种路径混用是典型的迁移未完成技术债。

## 五、方案：统一开关体系

### 5.1 选择依据

| 因素 | 结论 |
|---|---|
| 项目未上线 | 零兼容负担，可以一步到位 |
| diamond_quota 是 bug | 不论选什么方案都必须修 |
| 现有 DynamicConfigLoader 已是正确基础设施 | 只需在上面加一层薄封装 |
| 长期维护成本 | 统一读取路径，每个接手的工程师认知负担最低 |

### 5.2 具体改动清单

#### 改动1：DynamicConfigLoader 增加 `isStrategyEnabled` 方法

在 `services/UnifiedLotteryEngine/compute/config/ComputeConfig.js` 的 `DynamicConfigLoader` class 中新增一个静态方法，内部调用已有的 `getValue()`，统一布尔转换逻辑。

所有 stage 中散落的开关判断统一替换为调用此方法。

#### 改动2：修复 diamond_quota 两条断裂路径

**2a. 抽奖引擎侧**：`BuildPrizePoolStage.js` 中删除 `campaign_data.strategy_config` 的读取逻辑，改为通过 `DynamicConfigLoader.getValue()` 读取 `diamond_quota` 组的 `enabled` 和 `exhausted_action`（重命名后的 key）。

注意：`_filterByDiamondQuota()` 方法仅使用 `quota_exhausted_action` 和 `BalanceService.getBalance('DIAMOND_QUOTA')`，不使用 `diamond_quota_ratio`。ratio 仅在消费流程中使用。

**2b. 消费服务侧**：`CoreService._getDiamondQuotaConfig()` 改为通过 `DynamicConfigLoader.getValue()` 读取，统一到唯一的配置读取路径。改后：
- 有缓存（5分钟 TTL），消费审核场景可接受此粒度
- 走 `getParsedValue()` 标准类型转换，消除自行实现的 `=== 'true'` 逻辑
- 自动适配重命名后的 key（`enabled`、`ratio`、`exhausted_action`）

#### 改动3：补 tier_fallback.enabled 的 DB 记录 + 引擎读取

在 `LoadCampaignStage._getFallbackPrize()` 开头增加开关检查：通过 `DynamicConfigLoader.isStrategyEnabled('tier_fallback', true, { lottery_campaign_id })` 判断，关闭时直接 return null。

#### 改动4：一个 migration 文件处理所有 DB 变更

migration 做四件事：

1. 为所有活动（1, 25, 26, 27）插入 `config_group='tier_fallback', config_key='enabled', config_value=true`
2. 将所有活动的 `config_group='diamond_quota', config_key='diamond_quota_enabled'` 更新为 `config_key='enabled'`
3. 批量修正 `value_type = 'boolean'` 且 `JSON_TYPE(config_value) = 'STRING'` 的记录：将 JSON 字符串 `"true"/"false"` 更新为 JSON 布尔值 `true/false`（命中 4 条记录）
4. 修正 `value_type = 'number'` 且 `JSON_TYPE(config_value) = 'STRING'` 的记录：活动1 的 `grayscale.pity_percentage` 从 `"52"` 更新为 `52`（命中 1 条记录）

### 5.3 改动后最终状态

#### 可开关策略（11个）— 全部通过 `DynamicConfigLoader` 读取

| # | config_group | 开关 key | 默认值 | 本次改动 |

### 决策1：diamond_quota 是否纳入统一开关页面 — **已决策：不纳入**

### 决策2：diamond_quota 的子参数 config_key 是否也统一命名

当前 diamond_quota 组的 key 为 `diamond_quota_enabled`、`diamond_quota_ratio`、`quota_exhausted_action`。`diamond_quota_enabled` 已在改动中重命名为 `enabled`。

| 选项 | 说明 |
|---|---|
| **A. 保持现状** | `diamond_quota_ratio` → 不改，仅改 enabled 的 key | 
| **B. 统一去前缀** | `diamond_quota_ratio` → `ratio`，`quota_exhausted_action` → `exhausted_action` |

#### 行业对比：config_key 命名规范

**核心问题**：当配置已经按 group 分组后，key 里是否还需要重复 group 名？

| 类别 | 代表 | config_key 命名方式 | 是否带 group 前缀 |
|---|---|---|---|
| 大厂配置中心 | 携程 Apollo | namespace=`app`, group=`cache`, key=`timeout` | 不带。key=`timeout`，不是 `cache_timeout` |
| 大厂配置中心 | 阿里 Nacos | group=`payment`, dataId=`retry_count` | 不带。不是 `payment_retry_count` |
| 大厂配置中心 | 阿里 Diamond | group + key 二级，key 干净 | 不带 |
| 游戏公司 | 米哈游（原神） | pity 组: `threshold`, `multiplier`, `enabled` | 不带。不是 `pity_threshold` |
| 游戏公司 | 网易（阴阳师） | guarantee 组: `interval`, `prize_id` | 不带 |
| 虚拟物品交易平台 | 典型设计 | 有 group 列的配置表，key 不带前缀 | 不带 |
| 小公司 / 早期项目 | 常见模式 | 单表无 group 列，key 自带前缀用于区分 | **带**（因为没有 group 列，前缀是唯一命名空间） |

**本项目现状**：`lottery_strategy_config` 表已有 `config_group` 列，group 已经提供了命名空间。其他 13 个 group 的 key 全部不带前缀：

| config_group | config_keys（均不带 group 前缀） |
|---|---|
| guarantee | `enabled`, `prize_id`, `threshold` |
| pity | `enabled`, `hard_guarantee_threshold`, `min_non_empty_cost`, `multiplier_table` |
| anti_high | `enabled`, `high_streak_threshold`, `recent_draw_window` |
| luck_debt | `enabled`, `expected_empty_rate`, `min_draw_count` |
| **diamond_quota** | **`diamond_quota_enabled`, `diamond_quota_ratio`, `quota_exhausted_action`** ← 唯一例外 |

**原因分析**：diamond_quota 的配置最早可能写在 `lottery_campaigns` 表的 JSON 列里（扁平结构，必须带前缀区分），迁移到独立配置表后前缀没有同步清理。这和大厂、游戏公司从扁平配置迁移到配置中心时的经典遗留问题一模一样。

**两种方案的区别**：

| 对比维度 | A. 保持前缀 | B. 去掉前缀 |
|---|---|---|
| 代码阅读 | `getValue('diamond_quota', 'diamond_quota_ratio')` — 读起来 `diamond_quota` 出现两次，冗余 | `getValue('diamond_quota', 'ratio')` — group+key 语义清晰 |
| 与项目内其他 group 的一致性 | 唯一一个带前缀的 group，需要记住例外 | 全部 14 个 group 规则统一，零例外 |
| 后续新增 key 时 | 新开发人员可能困惑：新 key 要不要带 `diamond_quota_`？ | 规则明确：所有 key 都不带 group 前缀 |
| 改动成本 | 0 | migration 改 2 个 key + `BuildPrizePoolStage._filterByDiamondQuota()` 改 2 处引用 |
| 长期维护成本 | 每个接手的人都要理解为什么这个 group 和别的不一样 | 统一规则，无认知负担 |

**已决策：选 B（去前缀）**。`diamond_quota_ratio` → `ratio`，`quota_exhausted_action` → `exhausted_action`。影响范围：
- migration 改 2 个 key（3 处 config_key UPDATE，含 `diamond_quota_enabled` → `enabled` 已在改动4中）
- `BuildPrizePoolStage._filterByDiamondQuota()` 改 1 处引用（`exhausted_action`）
- `CoreService._getDiamondQuotaConfig()` 改 3 处引用（改动2b 统一解决）

### 决策3：是否一并修复所有活动的 JSON 类型不一致

真实数据库查询 `SELECT lottery_campaign_id, config_group, config_key, JSON_TYPE(config_value) FROM lottery_strategy_config WHERE value_type = 'boolean'`，发现 STRING 类型问题不只活动1：

|---|---|---|---|---|
| 1 | anti_empty | enabled | **STRING** `"true"` | BOOLEAN `true` |
| 1 | guarantee | enabled | **STRING** `"false"` | BOOLEAN `false` |
| 1 | pity | enabled | **STRING** `"true"` | BOOLEAN `true` |
| 26 | pity | enabled | **STRING** `"true"` | BOOLEAN `true` |

共 **4 条记录**（活动1 有 3 条，活动26 有 1 条），活动25 和 27 全部干净。

| 选项 | 说明 |

#### 行业对比：数据类型一致性修复策略

| 类别 | 代表 | 做法 |
|---|---|---|
| 大厂 | 美团/腾讯/阿里 | 配置中心写入时强制类型校验（Apollo/Nacos 的 schema 校验），不可能出现类型不一致。如果存量数据有问题，上线前必须全量清洗。 |
| 游戏公司 | 米哈游/网易 | 概率引擎的配置值有严格的类型约束（boolean 必须是 boolean），不一致会触发报警。修复一定是全量。 |
| 虚拟物品 / 二手平台 | 典型做法 | 数据量小时全量修，数据量大时分批修。但核心配置表（影响交易逻辑的）一定是全量一次性修。 |
| 小公司 | 常见做法 | 上线前能修就全修；上线后怕影响线上才选择渐进修。 |

**两种方案的区别**：

| 对比维度 | A. 仅修活动1 | B. 批量修所有 |
|---|---|---|
| SQL 复杂度 | 3 条精确 UPDATE | 1 条 WHERE 条件 UPDATE |
| 遗漏风险 | 活动26 的 pity.enabled 仍为 STRING，未来可能导致 `=== true` 判断异常 | 零遗漏 |
| 未来新活动 | 如果创建新活动的代码也有类型问题（通过旧 migration seed 创建的），还会产生新的 STRING 记录 | 同上，但至少存量数据干净 |
| 维护认知 | 需要记住「只修了活动1」 | 全部干净，无需记忆 |

**已决策：选 B（批量修）**。migration 中用 `WHERE JSON_TYPE(config_value) = 'STRING' AND value_type = 'boolean'` 一条语句修全部 4 条记录。

### 决策4：CoreService._getDiamondQuotaConfig() 如何获取 lottery_campaign_id — **已决策：选 A**

#### 问题背景

当前 `CoreService._getDiamondQuotaConfig()` 不接收 `lottery_campaign_id` 参数。它的做法是：

```javascript
const config = await LotteryStrategyConfig.findOne({
  where: { config_key: 'diamond_quota_enabled', is_active: true },
  transaction
})
// 然后用 config.lottery_campaign_id 查后续 key
```

这个设计隐含了一个假设：**只有一个活动的 diamond_quota 是开启的**。如果多个活动同时开启，`findOne` 取到哪一条是不确定的（取决于 DB 默认排序）。

改为 `DynamicConfigLoader.getValue()` 后必须传入 `lottery_campaign_id`。问题是：消费审核流程中，消费记录（`consumption_records` 表 22 个列）**没有 `lottery_campaign_id` 列**。

#### 真实数据佐证

- `consumption_records` 表无任何活动关联字段
- 消费流程的预算分配已有先例：`lottery_campaign_id: 'CONSUMPTION_DEFAULT'`（硬编码虚拟 campaign ID，非真实活动 ID）
- 4 个活动的 diamond_quota 配置完全相同（enabled=true, ratio=1, action=filter）— 从未被区分使用
- `system_configs` 表已存在（22 条记录，7 个 config_category：batch_operation/feature/general/ad_system/backpack/customer_service/ad_pricing），但无 diamond/quota 相关条目

#### 三个选项

| 选项 | 说明 | 改动量 |
|---|---|---|
| **A. 查询当前活跃活动** | `_getDiamondQuotaConfig()` 内部查 `lottery_campaigns WHERE status = 'active'` 取第一个 ID，传给 `DynamicConfigLoader` | 最小（+5行） |
| **B. 消费记录关联活动 ID** | 在 `consumption_records` 表加 `lottery_campaign_id` 列，消费审核流程传递 | 大（DDL + 流程改造） |
| **C. 配额配置独立于活动** | 将 diamond_quota 的 `ratio` 从 `lottery_strategy_config` 移到 `system_configs` 表 | 中（拆分 config_group） |

#### 行业对比：消费金额 → 虚拟货币转换比例的配置归属

**核心问题**：「用户消费100元，发放多少钻石配额」这个比例(ratio)，在架构上属于哪一层？是跟着活动走的（活动A比例1:1，活动B比例1:0.5），还是全平台统一的？

| 类别 | 代表 | 消费→货币转换比例的配置归属 | 是否按活动隔离 | 配置表位置 |
|---|---|---|---|---|
| **大厂** | 美团（吃饭→积分） | **平台级**。用户在任何餐厅消费，积分比例都走平台统一规则（如1元=1分），不因当前哪个营销活动在跑而变化。比例调整是平台级商业决策，需财务审批。 | **否** | 平台配置中心（非活动表） |
| **大厂** | 腾讯（充值→Q币/绿钻） | **平台级**。Q币充值比例1元=1Q币全平台统一，不因游戏活动改变。绿钻月卡定价也是平台级。 | **否** | 计费中心独立配置 |
| **大厂** | 阿里（消费→淘金币） | **平台级**。淘金币发放比例走平台规则（如消费100元获100淘金币），不绑定双11或618具体活动。特定活动可能有额外加倍，但基础比例是全局的。 | **否** | 平台级别，活动加倍走活动配置 |
| **游戏公司** | 米哈游（充值→创世结晶→原石） | **平台级**。创世结晶兑换原石 1:1 全游戏统一。充值648元=8080创世结晶，不因当前池子（雷神/钟离）而变化。首充双倍是独立开关。 | **否** | 全局配置，不在卡池配置里 |
| **游戏公司** | 网易（消费→勾玉） | **平台级**。消费转勾玉比例固定，不因阴阳师当前卡池而改变。 | **否** | 平台商城配置 |
| **虚拟物品平台** | Steam（充值→钱包余额） | **平台级**。汇率/充值比例全平台统一。 | **否** | 系统配置 |
| **二手平台** | 闲鱼/转转 | 不涉及虚拟货币发放 | — | — |
| **小公司/早期项目** | 常见做法 | 最初混在活动表里（因为只有一个活动）。上线后发现消费比例不应该跟着活动走，再迁出到系统配置。这是经典的「过度耦合→解耦」技术债务路径。 | 初期耦合→后期解耦 | 活动表→系统配置表 |

**共同结论**：**所有成熟系统的消费→货币转换比例都是平台级配置，不按活动隔离**。原因：

1. **财务合规**：比例变更影响平台成本核算，属于财务级决策，不能让运营随活动随意调
2. **用户一致性预期**：用户消费100元获得的配额不应该因为当天跑的是A活动还是B活动而不同
3. **审计可追溯**：监管要求虚拟货币发放比例有明确的全局规则和变更记录

#### 三个方案对本项目的具体影响

| 对比维度 | A. 查询活跃活动 | B. 消费记录关联活动 | C. 配额配置独立于活动 |
|---|---|---|---|
| **符合行业惯例** | 部分。比例实际存在活动表里，只是代码里自动选了活跃的那个 | 不符合。消费→配额比例不应该按活动隔离，强行关联是过度设计 | **完全符合**。与美团/腾讯/米哈游一致 |
| **本项目已有先例** | 有。消费流程已用 `'CONSUMPTION_DEFAULT'` 虚拟 ID 规避活动关联 | 无。需要改 consumption_records DDL + 审核流程 | 有。`system_configs` 表已存在，已有 7 个 category |
| **代码改动量** | 最小。`_getDiamondQuotaConfig()` 内加 1 条活动查询 | 最大。DDL + migration + 审核流程改造 | 中等。migration 挪 2 条记录到 system_configs + CoreService 改读取源 |
| **diamond_quota 的 3 个 key 归属** | 全部留在 `lottery_strategy_config`（enabled/ratio/action 都按活动） | 全部留在 `lottery_strategy_config` | **拆分**：`ratio` 移到 system_configs（平台级），`enabled` + `exhausted_action` 留在 lottery_strategy_config（抽奖引擎按活动需要） |
| **拆分 config_group 的代价** | 无拆分 | 无拆分 | diamond_quota 一个 group 横跨两张表，需要在两个地方管理 |
| **未来多活动** | 4 个活动配置完全相同→实际无区分效果。未来若需不同比例，可以改 | 支持，但设计上不合理（消费比例不应分活动） | 天然支持。ratio 全局统一，各活动独立控制 enabled/action |
| **DynamicConfigLoader 统一性** | 全部走 DynamicConfigLoader，统一读取路径 ✓ | 全部走 DynamicConfigLoader ✓ | 消费流程读 system_configs（新路径），抽奖流程走 DynamicConfigLoader（现有路径）— **两条路径** |
| **长期维护成本** | 低。改动最小，但 ratio 放在活动表里语义不精确 | 高。改了 DDL 和流程，且设计方向错误 | 中。拆分增加复杂度，但语义最清晰 |

#### 本项目的特殊情况

回到本文档的核心原则：**「全系统只有一种方式查询某策略是否开启」**。

选 C 会引入一个新问题：diamond_quota 的 `ratio` 走 `system_configs`，而 `enabled`/`exhausted_action` 走 `DynamicConfigLoader`，变成两条读取路径。这正是本文档要消灭的问题。

选 A 保持所有 diamond_quota 配置都在 `lottery_strategy_config`，全部通过 `DynamicConfigLoader` 读取，**零新路径**。虽然 ratio 放在活动表里语义不够精确，但：

1. 当前 4 个活动配置完全相同 — 实际效果等同于全局配置
2. `DynamicConfigLoader` 的缓存机制（5分钟 TTL）已足够消费审核场景
3. 项目未上线，单活动运营，零实际影响
4. 未来真正需要区分活动比例时（如促销期间2倍配额），按活动隔离的设计反而更灵活

**已决策：选 A**。`_getDiamondQuotaConfig()` 改为接收可选的 `lottery_campaign_id` 参数，未传入时查询 `lottery_campaigns WHERE status = 'active' LIMIT 1` 获取。全部配置走 `DynamicConfigLoader`，零新路径。

### 决策5：是否在 migration 中同步修复 grayscale.pity_percentage 类型异常 — **已决策：选 A（一并修复）**

活动1 的 `grayscale.pity_percentage` 存储为 STRING `"52"`，其余活动为 INTEGER `100`。在同一 migration 中增加一条 `UPDATE ... WHERE value_type = 'number' AND JSON_TYPE(config_value) = 'STRING'` 一并修复。+1 行 SQL，零额外风险，migration 执行后全表类型干净。

### 决策6：是否在 `upsertConfig()` 写入路径增加类型强制转换 — **已决策：选 A（增加转换）**

当前 `LotteryStrategyConfig.upsertConfig()` 将 `config_value` 直接存入 JSON 列，不校验 JSON 类型是否匹配 `value_type` 声明。选 A：写入前根据 `value_type` 做 `JSON.parse()` / `Boolean()` / `Number()` 强制转换。改动量 `LotteryStrategyConfig.js` +10行。

根因治理：migration 修存量，upsertConfig 堵增量，两者配合彻底解决类型不一致问题。

### 12.3 微信小程序前端对接说明

**微信小程序前端不涉及策略开关管理，无需任何改动。**

策略配置是纯服务端逻辑，小程序仅调用以下接口，不需要感知策略开关参数：

| 接口 | 方法 | 说明 |
|---|---|---|
| `/api/v4/lottery/draw` | POST | 抽奖（策略引擎在服务端自动应用所有开关） |
| `/api/v4/lottery/campaigns/:id` | GET | 获取活动信息（不含策略配置详情） |
| `/api/v4/lottery/campaigns/:id/prizes` | GET | 获取奖品列表（已按配额/库存过滤） |
| `/api/v4/lottery/history` | GET | 抽奖历史记录 |
| `/api/v4/auth/login` | POST | 微信登录（`WXBizDataCrypt` + `platformDetector`） |

策略开关对小程序的影响是透明的：

- **diamond_quota**：抽奖引擎在 `BuildPrizePoolStage` 自动过滤配额耗尽的钻石奖品，小程序获得的奖品池已是过滤后的结果
- **tier_fallback**：兜底奖品由引擎自动处理，小程序不需要知道兜底逻辑
- **pity/anti_empty/anti_high/luck_debt**：体验平滑机制在服务端自动应用，影响概率但不影响接口契约
- **管理干预（management）**：force_win/force_lose 由管理员通过 Web 后台操作，小程序无感知

**小程序抽奖接口的请求/响应格式不变**，无需任何适配。
