## 〇、拍板决议（2026-07-10 终稿，执行以本节为准）

架构结论：现有等级体系（累计积分单调递增、实时派生、不衰减、阈值表驱动）即"腾讯型成长值 + 游戏VIP阈值曲线"组合，为长期维护成本最低方案，**架构不动，只定数字**。不采用电商大厂滑动窗口衰减方案（需数仓/快照/降级链路，技术债过重）。

业务前提（已核实与系统实际行为一致）：
- **1 元 = 1 积分**：真实兑换率是 `system_settings` 的 `points_award_ratio=1`（消费审核通过时 `积分 = round(消费金额 × 比例)`，`services/consumption/CoreService.js`），无单笔截断逻辑；
- **注册不送积分、全员 v1 起步**：注册链路无任何发分代码（`system_settings.initial_points=200` 仅为管理后台展示项，无发放逻辑）；
- **"累计消费"采用代理口径（A 口径）**：等级由 `users.history_total_points` 派生，积分来自消费小票审核，天然≈累计消费，零代码改动。

| ⑯ | 倍数公示口径 + 换物运费 | **(a) 等级回馈倍数对外公示**：会员权益页明码写"铜卡 1.0 倍 / 银卡 1.05 倍 / … / 荣耀殿堂 1.5 倍"（京东PLUS 公开 10 倍同款做法）——C 端明细"等级加成 +X"本就可推算倍数，公示化为升级动力；升级通知文案含倍数（⑬-(d) 口径一致）。注意：公示的只是**发放倍数**，抽奖概率类数值仍守脱敏红线不下发；**(b) 换物产出快递包邮**：延续现状（全库无运费概念、实物兑换即平台包邮），零开发——换物旧物抵新物已有等价/向下价值门槛，运费成本可控 |

## 一、以物易物（Barter，即"以物换物"）

### 1.1 现状：代码完整，配置为空

**业务模式**：B2C 官方合成（合规设计）。用户旧物核销销毁 → 官方库存发放新物，全程用户↔官方，无用户间转移、无货币回流。四条守卫已焊死：旧物真销毁（status='used'）、产出禁货币型资产、方向仅等价或向下、幂等防重。

**已有代码资产**：

| 层 | 位置 | 说明 |
|---|---|---|
| 核心服务 | `services/exchange/BarterService.js` | getRecipes / executeBarter / saveRecipes |
| C 端 API | `GET /api/v4/exchange/barter/recipes`<br>`POST /api/v4/exchange/barter` | 需登录 + Idempotency-Key 头 |
| 管理端 API | `GET/PUT /api/v4/console/exchange/barter-recipes` | role_level ≥ 100 |
| 管理后台前端 | `admin/src/modules/market/composables/barter-recipes.js`<br>`admin/src/modules/market/pages/exchange-market.js` | 配方管理界面已有 |
| 测试 | `tests/api-contracts/barter.contract.test.js` | 契约测试 |

**配方存储**：`system_settings`（category='exchange'，setting_key='barter_recipes'，JSON 数组），不建新表。配方结构：

```json
{
  "recipe_code": "old_towel_to_new_towel",
  "name": "旧毛巾换新毛巾",
  "required_item_template_id": 16,
  "required_quantity": 1,
  "output_exchange_item_id": 20,
  "is_enabled": true,
  "per_user_limit": 1,
  "total_limit": 100
}
```

其中 `per_user_limit`（每人限换次数）/`total_limit`（配方总量）为拍板⑬-(c) 新增可选字段（缺省=不限）：`executeBarter` 校验时按 `exchange_records` 中 source='barter' + recipe_code（存于 item_snapshot）计数，超限拒绝——堵住"无限循环刷换物"漏洞。

### 1.3 启用步骤

1. **【开发】修复物品模板关联（前置阻断项，拍板⑤：一次性全修复）**
   - 排查发放链路（抽奖发奖、兑换发货）铸造 `items` 实例时未写 `item_template_id` 的原因并修复；
   - 对存量 12 件物品按名称补挂对应 `item_templates`；
   - legacy 系列模板补录 `reference_price_points`（当前全 0.00 会绕过方向守卫）。
2. **【运营】准备产出标的**：为产出的 `exchange_items` 商品关联 `item_template_id`。
3. **【运营】配置试点配方（拍板④）**：管理后台"以物易物配方"页先写 1 条（毛巾礼盒×2 → 新毛巾礼盒），产出价值 ≤ 投入价值，验证后再扩充。
4. **【开发】产出快递到家改造（拍板⑩，见 §1.4）**。
5. **【测试】完整链路验证**：测试账号查询配方 → 执行换物 → 核对：旧物 status='used'、`exchange_records` 生成 source='barter' 订单（order_no 前缀 BT）、快递履约走 pending→shipped→received。
6. **【前端】确认小程序端换物入口页面已对接上述两个 C 端 API（本仓库仅含后端+管理后台）。**

### 1.4 产出快递到家改造方案（拍板⑩，新增开发项）

**现状与目标的差距**：换物订单创建时直接 `status='completed'`、不收地址，产出物只能背包+到店核销；而快递链（地址快照 `address_snapshot`、运单号 `shipping_no`、`ShippingTrack` 轨迹、快递100/快递鸟双通道、7 天自动确认收货）全部挂在 `ExchangeRecord` 的 pending→approved→shipped→received 生命周期上，换物订单从未进入。

**改造原则：复用不新建**——换物与普通实物兑换共用同一张订单表和同一条发货链，改动点收口在 BarterService 与其路由：

1. `POST /api/v4/exchange/barter` 增加 `address_id` 参数（实物产出必填，校验地址归属本人，复用 `UserAddressService.buildSnapshot()` 写入 `address_snapshot`）；
2. `BarterService.executeBarter` 订单状态改：产出为实物（模板 `item_type='product'`）→ `status='pending'` 走发货链；产出为券/道具 → 维持 `completed` 即时到账；
3. 实物产出**不再 mint 进背包**（对齐普通实物邮寄兑换的处理：`mint_instance` 与发货链二选一），防止"背包里能核销 + 快递又寄一件"双重履约；
4. 发货/轨迹/确认收货全部复用现有管理端发货操作与 webhook，零新增接口；运费按拍板⑯-(b) 平台包邮（延续现状，无运费字段无新逻辑）。

**投入边界（拍板⑩前半）无需改动**：`_loadAndValidateOldItems` 已强制 `status='available'`——已核销实物（used）、核销订单锁定中（held）的物品天然不可投入，与"电子态才可换"的业务规则完全一致。

## 二、会员成长等级（UserGrowthLevel，即"会员等级"）

### 2.1 现状：代码完整，阈值占位、权益零配置

**业务模式**：独立成长等级体系，由 `users.history_total_points`（累计积分）**实时派生**，无 per-user 同步字段。注意与 `users.user_level`（enum: normal/vip/merchant，身份类型）区分，二者不是一回事。

**已有代码资产**：

| 层 | 位置 | 说明 |

### 2.4 发放线九档阶梯改造方案（拍板②，新增开发项）

**改造点收口在消费审核链路（`services/consumption/CoreService.js`）与积分累加口径（`services/asset/BalanceService.js`）**：

1. **等级倍数配置落位**：`user_growth_levels` 表加 `earn_multiplier DECIMAL(4,2)` 列（一次迁移），九档写入 §0-② 阶梯值——倍数与等级定义同表，管理后台调档零散配置不分家；
2. **可用积分——发放组合器（拍板⑮-(e) 升级）**：审核通过发分改为"基础笔 + N 个加成笔"（同事务）——基础笔 `round(消费金额 × points_award_ratio)`（business_type='consumption_reward'，计入 `history_total_points`）+ 等级加成笔 `round(基础 × (earn_multiplier - 1))`（business_type='level_bonus_reward'，**不计** `history_total_points`）+ 未来活动加成笔（business_type='activity_bonus_reward'，同样不计）。各加成独立按基础分计算（加法叠加），总倍数封顶 3.0，加成规则做成列表——新增活动加成即插规则，不碰主逻辑；
3. **防复利实现**：`BalanceService.changeBalance` 的 `history_total_points` 累加处（该字段唯一写收口）增加 business_type 排除名单（`level_bonus_reward`）——等级永远由"基础分≈真实消费"驱动，杜绝"高等级→加成→更快升级"复利；
4. **预算积分同步（含拍板⑮-(c) 硬封顶）**：预算分配处 `round(消费金额 × budget_allocation_ratio × 加法叠加总倍数)`，且注入率硬封顶 16%（`min(计算值, 消费金额 × 0.16)`，运营手动发放除外）——预算变厚 → 预算分层升档 → 现有 BxPx 闸门自然放行更高奖品档位；
5. **等级倍数提交时锁定（拍板⑬-(a) 修订）**：小票**提交时点**用 `UserGrowthLevelService.resolveUserLevel` 派生等级、锁定倍数并随 `points_to_award` 一并存入消费记录——与 `points_award_ratio` 现有"提交时锁定，保证用户承诺一致"原则完全对齐，审核快慢不影响到账金额；
6. **升级即时通知（拍板⑬-(d)）**：审核发分后对比发分前后等级（两次 `resolveLevelKey`），跨档则经 `NotificationService` 发站内通知（如"恭喜升级金卡，您的积分回馈已提升至 1.1 倍"）——等级实时派生无升级事件，此处是唯一可靠的触达点。

**应急回滚预案（拍板⑬-(b)）**：`earn_multiplier` 九档全部改回 1.00 → 加成笔金额为 0 不再产生、预算注入回到 8% 基准，系统行为**完全回到现状**。成本失控或发现缺陷时回滚只需改九个数字（管理后台等级管理页操作），无需回滚代码、无需 feature flag。

**用户可见性口径（2026-07-10 确认）**：
- **预算积分对用户完全不可见（现有机制，零改动）**：`BUDGET_POINTS` 在 `DataSanitizer` 全局前端资产黑名单（`FORBIDDEN_FRONTEND_ASSET_CODES`）中——余额接口拒查、资产列表剔除、流水记录整条过滤；加成后的预算笔同样被拦，用户只感知结果（更容易抽中好奖）不见机制；
- **加成笔主动展示（权益感知点）**：`level_bonus_reward` 为独立流水，C 端积分明细可显示"会员等级加成 +X"（京东PLUS"多返京豆"同款展示），用户每次消费直观看到等级价值。

**明确不做**：`level_probability` 概率倍数不配置（保持默认 1.0）；`TierPickStage` 的 B 线代码保留不动（未来若需活动级概率活动可复用）。

### 2.5 兑换商城等级区间门槛（拍板⑪，新增开发项）

**现有一半**：`exchange_redeem_requirement.min_growth_level_key`（及以上）已实现——按商品/SKU 配置（SKU 精确优先回退商品级）、启停 + 生效时间窗、`CoreService._assertRedeemRequirement` **扣费前**校验并给出明确提示。

**扩展1——等级上限（及以下）**：
1. 表加列 `max_growth_level_key VARCHAR(32) NULL`（一次迁移，NULL=不限）；
2. 校验：`resolveLevelKey` 派生用户当前等级，按 `sort_order` 比较 `min ≤ 用户等级 ≤ max`；超上限报 `REDEEM_GROWTH_LEVEL_EXCEEDED`（文案如"银卡及以下会员专享"）；
3. 组合语义：min 单配=及以上（拍板③划线）、max 单配=及以下（新人专享）、min=max=仅某等级（v9 专属纪念品）、双 NULL=不限；
4. 管理端写入校验：min/max 必须为启用中 level_key，min.sort_order ≤ max.sort_order（防倒挂）。

**扩展2——C 端"锁住但可见"**：商品列表/详情（`QueryService`）为有门槛商品附加脱敏摘要 `level_requirement: { min_level_name, max_level_name, satisfied }`——`satisfied` 后端按当前用户实时计算，只下发等级展示名（不下发倍数/权重，守脱敏红线）；前端渲染专享角标 + 置灰按钮。橱窗效应：低等级用户看得见高等级专享商品，比运营文案更能拉动消费升级。

**工作量**：一列迁移 + 校验分支约十行 + 管理端字段校验 + `QueryService` 摘要字段 + 契约测试；全部收口 exchange 域现有文件，无新表无新链路。
