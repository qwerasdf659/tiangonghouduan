# 水晶奖品倍率活动设计方案

> 版本：v1.2　|　创建日期：2026-07-05（北京时间）　|　状态：待评审
> 适用项目：天工商户营销平台（Node.js + Express + Sequelize + MySQL 8 + Redis）
> 求证方式：本方案表结构/字段/代码落点均以 Node.js + Sequelize 实连 `restaurant_points_dev` 真实库
> 执行 `SHOW FULL COLUMNS` 核对，并对齐现网 `services/UnifiedLotteryEngine` 抽奖管线代码，未引用历史报告/备份。

## 0. 本文档解决什么问题（一句话导读）

**核心目标**：让运营能对"特定活动 + 特定人群"配置"抽中水晶类奖品数量翻倍"（如新用户/高等级 ×1.5~×2.5），且**倍率按活动隔离、成本可控、用户感知清晰**。

**本文档依次解决 4 个问题**：

| # | 问题 | 解决方案 | 章节 |
|---|---|---|---|
| P1 | 怎么给"某活动的某些人"翻倍水晶奖品，且运营可自助调控 | 独立倍率规则实体，绑定抽奖活动 + 复用现网分群引擎圈人群 | §1–§6 |
| P2 | 多个活动/多条规则都能翻倍时，倍率怎么算才不打架、用户不困惑 | 活动隔离 + 同活动内取最高(Max) + 用户只见单一倍率 | §2、§5 |
| P3 | 翻倍放大真实成本，怎么防止成本失控与薅羊毛 | 倍率硬上限 + 每规则额外成本封顶(`extra_cost_limit`) + 兜底不翻 | §7、§10 |
| P4 | 用户平时囤积分、专等翻倍日套现（囤积套利）怎么根治 | 活动专属预算(隐藏层) + 活动专属可见积分/代币(可见层) + 到期清零 | §12 |

**一句话方案**：翻倍规则绑定到具体抽奖活动、只作用"抽中水晶的发放数量"；短期翻倍活动用**独立的活动专属货币**（可见层入场 + 隐藏层成本双桶）承接，到期清零，从根上杜绝"平时囤、翻倍日套现"。全部复用现网技术栈（`segment_rule_configs`/`lottery_campaigns.allowed_campaign_ids`/`material_asset_types`/`SettleStage`），零新增技术栈。

**版本变更**：
- v1.1：人群定向改为复用现网 `segment_rule_configs`/`user_ad_tags`/`user_growth_levels`（§1.3、§3）。
- v1.2：新增 §12 防囤积套利（活动专属预算 + 双层货币模型 + 到期清零）。

## 1. 背景与业务目标

### 1.1 现有业务链路（实测确认）

平台现有资产与抽奖链路：

```
用户消费 → 审核通过 → 获得可见积分(points) + 0.08% 计入预算账户(预算积分/水晶碎片)
→ 预算积分抽奖 → 抽中材料类奖品(material，如红源晶碎片) → 按 material_amount 发放入账
```

关键实测事实：

| 环节 | 现网真实载体 |
|---|---|
| 预算账户 | `account_asset_balances`（`lottery_campaign_id` 区分预算积分归属活动） |
| 抽奖活动 | `lottery_campaigns`（`budget_mode`=user/pool/none） |
| 活动奖品 | `lottery_campaign_prizes`（`win_weight`/`stock_quantity`/`reward_tier`） |
| 奖品定义 | `prize_definitions`（`prize_type`=material/item/coupon/points） |
| 水晶类资产 | `material_asset_types`（如 `red_core_shard` 红源晶碎片、`red_core_gem` 红源晶，`group_code`=red/orange/...，`form`=shard/gem） |
| 发奖入账 | `SettleStage.js` material 分支 → `BalanceService.changeBalance(delta_amount = material_amount)` |
| 抽奖流水 | `lottery_draws`（含 `result_metadata` JSON、`material_amount` 展示字段） |
| 资产流水 | `asset_transactions`（双录记账 + `idempotency_key` 幂等 + `meta` JSON） |

### 1.2 业务目标

> **对特定用户 / 特定用户群体 / 特定活动，抽中"水晶类奖品"时，发放数量可 ×2 / ×3（运营后台可调控、可开关）。**

核心诉求：

| 目标 | 说明 |
|---|---|
| 运营可调控 | 后台配置倍率、作用人群、作用奖品范围、时间窗、开关，无需改代码 |
| 多活动不打架 | 用户同时命中多个倍率活动时，结果唯一、成本可控、用户无困惑 |
| 感知清晰 | 用户任何时候只看到"一个确定的倍率"，抽中后能看懂 `基础×倍率=最终` |
| 成本封顶 | 倍率直接放大真实价值奖品，必须防止成本失控与薅羊毛 |
| 与现有系统解耦 | 只作用于"抽中水晶类 → 发放数量"这一步，不动抽奖概率、不动预算发放 |

### 1.3 方案选型：两个轴 + 复用现网人群基础设施

需求"某些人的水晶奖品翻倍"其实涉及两个正交的轴，早期容易混淆，先厘清：

| 轴 | 选项 A | 选项 B | 本方案选择 |
|---|---|---|---|
| **作用目标（翻什么）** | 预算积分（抽奖的"钱"，input） | 水晶奖品数量（抽到的"货"，output） | **B 水晶奖品数量** |
| **作用对象（给谁）** | 绑用户属性 / 绑抽奖活动 | 独立规则实体，引用人群 | **独立规则 + 引用人群** |

**为什么作用目标选"水晶奖品"而非"预算积分"**：

- 翻预算积分 = 放大抽奖次数 → 连非水晶奖品（item/coupon/points）一起变多，成本不可控，且不符合"只翻水晶"的目标；还会牵动 `account_asset_balances` 整个预算体系。
- 翻水晶奖品 = 只在"抽中水晶"这一小概率事件放大产出，精准、成本可预测、改动面最小（只碰 `SettleStage` 发奖最后一步）。

**为什么作用对象用"独立规则实体 + 引用现网人群"，不绑用户属性、也不绑抽奖活动**：

| 建模流派 | 缺点 | 是否采用 |
|---|---|---|
| 倍率写死在用户/人群属性上 | 无法时间盒、无法多规则共存、无法审计、规则塞进主数据=技术债 | ❌ |
| 倍率绑在抽奖活动 `lottery_campaigns` 上 | 与"参加活动"耦合；抽奖活动管"能不能抽/概率"，塞进"发奖翻倍"职责混乱 | ❌ |
| **独立倍率规则实体 + 引用现网人群引擎** | 可时间盒/开关/审计、多规则 Max 合并、人群零重复造轮子、与抽奖/预算/主数据解耦 | ✅ |

**关键：人群定向复用现网已有基础设施（实测确认存在，无需新造人群表）**：

| 现网已有 | 用途 | 实测状态 |
|---|---|---|
| `segment_rule_configs` | 条件构建器**分群引擎**（field/operator/value → `segment_key`），已用于抽奖档位概率的"新老用户分层""VIP 分层" | 5 条规则，`is_system` 内置，活跃使用中 |
| `user_ad_tags` | 用户**标签**（`tag_key`/`tag_value`，凌晨 3 点定时计算） | 表已就位 |
| `user_growth_levels` | 用户**成长等级**（青铜/白银/黄金/钻石） | 已有阶梯 |
| `lottery_campaign_user_quota` + `quota_grants` | 按"用户×活动"定向发配额 | 已有定向发放机制 |

倍率规则的 target 直接引用上述 `segment_key` / `tag_key` / `growth_level_key`，"某些人"由现成引擎圈定，本方案不再自造人群表。

**对"活动 1/2/3"场景的落地示例**：

```text
活动1（全体可参加）        → 不配倍率规则，水晶正常 ×1
活动2（某些人可参加）      → 不改活动2本身；另配一条倍率规则，
                            target 引用 segment_key=vip_premium（或某标签/等级）→ 命中者抽中水晶 ×2
活动3（另一批人）          → 再配一条规则，引用另一 segment_key → ×3
用户同时命中活动2、活动3规则 → 取 max = ×3（不叠加）
```

倍率与抽奖活动彻底解耦：以后加活动、换人群，只增改倍率规则，**不动抽奖活动、不逐个调商品兑换条件**。

## 2. 核心设计原则

### 2.1 倍率作用位置（只作用一层）

倍率**只作用于**：抽中 `prize_type='material'` 且属于"水晶类"的奖品后，其 `material_amount` 发放数量的放大。

**不影响**：

- ❌ 不影响抽奖概率 / 选奖权重（`win_weight`），避免与抽奖引擎耦合。
- ❌ 不影响预算积分本身（`account_asset_balances` 预算余额）。
- ❌ 不影响非水晶类奖品（item/coupon/points，及非水晶 material）。

作用点唯一：`services/UnifiedLotteryEngine/pipeline/stages/SettleStage.js` 的 `material` 发放分支（现约 710–716 行），在 `delta_amount: prize.material_amount` 之前乘以倍率。

### 2.2 同活动内多规则合并策略：取最高（Max），不叠加、不叠乘

> **同一抽奖活动内，某用户同时命中多条倍率规则时，取"最高倍率"，不叠乘、不叠加。**
> （不同抽奖活动之间由 §2.4 的活动隔离天然互不影响，此处合并只针对"同一活动内命中的多条规则"。）

| 策略 | 示例（命中 ×2/×3/×2） | 结果 | 是否采用 |
|---|---|---|---|
| **取最大 Max** | max(2,3,2) | ×3 | ✅ 采用 |
| 叠乘 | 2×3×2 | ×12 | ❌ 成本爆炸 |
| 叠加 | 1+1+2+1 | ×5 | ❌ 易被薅 |
| 优先级取一 | 按 priority 取某个 | 视配置 | ❌ 用户难理解 |

选择 Max 的理由：

1. **成本封顶**：同活动内无论命中多少规则，最多是当前最高倍率，预算不失控。
2. **用户易懂**：用户永远只看到"本次最高 ×N"，不用自己算。
3. **运营易解释**：一句"倍率不叠加，取最高"即可公示（同大厂优惠券"不可叠加，取最优"）。
4. **防薅羊毛**：杜绝在同一活动内故意凑齐多条规则把倍率叠到天上。

### 2.3 "水晶类"如何界定

复用现网 `material_asset_types` 已有维度，不新造分类：

- 按 `group_code` 命中（如 red/orange/yellow/green/blue/purple 等水晶色系），或
- 按 `form in ('shard','gem')`（碎片/宝石形态），或
- 按显式资产码白名单（倍率活动可配 `reward_scope` 精确到某资产码）。

**拍板默认**：**`form in ('shard','gem')` 且非货币类（排除 points/star_stone/budget_points）即视为水晶类，碎片(shard)与宝石(gem)均参与翻倍**；运营可用 `reward_scope` 收窄到指定色系或资产码。新增水晶形态资产自动纳入，免维护清单。

### 2.4 活动隔离：倍率规则绑定到具体抽奖活动（目标效果）

> 倍率规则通过 `lottery_campaign_id` **绑定到具体抽奖活动**，只在该活动内生效；不同活动的倍率**天然互不影响**。

目标效果（与用户诉求对齐）：

| 场景 | 行为 |
|---|---|
| 只有一个活动、未配倍率规则 | 任何人参加都是**初始中奖数量（×1）** |
| 运营给活动内某人/某人群配倍率 | 满足条件者（新用户/高等级/指定人）享 1.5/1.75/2/2.5 等运营设定倍率，不满足者仍 ×1 |
| 多个活动 | 每条规则绑定各自活动，**互不影响**；用户在活动1 是 ×2、在活动2 可能是 ×1.5 或 ×1，各算各的 |
| 同一活动内某用户命中多条规则 | 取最高（Max），不叠加 |

举例（抽中 10 个红源晶碎片）：

```text
活动1：配"高等级会员 ×2"规则
  - 高等级会员小王抽中 10 个 → 发 20 个
  - 普通用户小李抽中 10 个 → 发 10 个（不满足条件，×1）
活动2：配"新用户 ×1.5"规则（与活动1 无关）
  - 新用户小王在活动2 抽中 10 个 → 发 15 个
  - 小王在活动1 的 ×2 不会带到活动2
活动1 同时又配了"新用户 ×1.75"：
  - 既是高等级又是新用户的小王在活动1 → max(2, 1.75)=×2 → 发 20 个
```

> 说明：`multiplier` 支持小数（1.50/1.75/2.50…），非整数结果按 `rounding_mode`（**已拍板默认向上 ceil**）取整，如 10×1.75=17.5→18、10×1.55=15.5→16。

## 3. 数据模型设计

新增两张表（主表 + 作用对象表），全部遵循现网规范：`utf8mb4_unicode_ci`、BIGINT 主键、`created_at/updated_at` 北京时间、Sequelize 迁移事务化。
作用对象表**不存人群定义**，只存"引用哪个现网人群标识"（segment_key / tag / growth_level），人群本身由 `segment_rule_configs` / `user_ad_tags` / `user_growth_levels` 负责。

### 3.1 倍率活动主表 `reward_multiplier_campaigns`

```sql
CREATE TABLE reward_multiplier_campaigns (
  multiplier_campaign_id BIGINT NOT NULL AUTO_INCREMENT COMMENT '倍率规则主键',
  lottery_campaign_id INT NOT NULL COMMENT '绑定的抽奖活动(FK→lottery_campaigns.lottery_campaign_id);该规则只在此活动内生效;强制必填,禁止全局规则(拍板:活动隔离)',
  campaign_name   VARCHAR(100) NOT NULL COMMENT '规则名(对内运营识别)',
  display_name    VARCHAR(100) NOT NULL COMMENT '对用户展示名(如"新春水晶翻倍")',
  multiplier      DECIMAL(4,2) NOT NULL COMMENT '倍率(支持小数,如1.50/1.75/2.00/2.50),>=1',
  reward_scope    ENUM('crystal_all','group','asset_codes') NOT NULL DEFAULT 'crystal_all'
                  COMMENT '作用奖品范围:crystal_all=全部水晶/group=按group_code/asset_codes=指定资产码',
  scope_values    JSON NULL COMMENT 'reward_scope=group时存group_code数组;=asset_codes时存asset_code数组;=crystal_all时NULL',
  target_type     ENUM('all','segment','tag','growth_level','user') NOT NULL DEFAULT 'all'
                  COMMENT '作用人群:all=全体/segment=分群(segment_rule_configs)/tag=标签(user_ad_tags)/growth_level=等级(user_growth_levels)/user=指定用户',
  rounding_mode   ENUM('round','floor','ceil') NOT NULL DEFAULT 'ceil'
                  COMMENT '小数倍率取整方式(拍板默认ceil向上,偏用户体感):ceil=向上(如10×1.75=17.5→18,10×1.55=15.5→16)/round=四舍五入/floor=向下',
  stack_strategy  ENUM('max') NOT NULL DEFAULT 'max' COMMENT '同活动内多规则命中合并策略(当前全局仅max,预留枚举)',
  priority        INT NOT NULL DEFAULT 0 COMMENT '优先级(并列同倍率时决胜与展示排序,越大越优先)',
  max_multiplier_cap DECIMAL(4,2) NOT NULL DEFAULT 3.00 COMMENT '倍率硬上限(拍板默认3.00,覆盖2.5需求且留余量;防误填,发放时二次夹紧)',
  extra_cost_limit   BIGINT NOT NULL COMMENT '因翻倍额外送出水晶的成本上限(按material_asset_types.budget_value_points折算累计;非用户账户,是本规则的成本刹车;拍板强制必填,达上限自动停翻回落×1)',
  extra_cost_used    BIGINT NOT NULL DEFAULT 0 COMMENT '本规则已累计的额外翻倍成本(实时累加,达 extra_cost_limit 自动停翻)',
  per_user_daily_limit  INT NULL COMMENT 'per-user 每日最多享受翻倍次数(拍板:运营可调,NULL=不限);防单人长期薅,复用 lottery_campaign_user_quota 计次范式',
  eligibility_days      INT NULL COMMENT '资格时间盒:仅在"进入命中人群后N天内"享翻倍(如新用户注册后30天),NULL=不限;过期实时判定自动失效',
  per_user_extra_cap    INT NULL COMMENT 'per-user 累计翻倍额外发放数量上限(单人最多多拿N个水晶),NULL=不限',
  start_at        DATETIME NULL COMMENT '生效开始(北京时间),NULL=不限',
  end_at          DATETIME NULL COMMENT '生效结束(北京时间),NULL=不限',
  status          ENUM('active','inactive') NOT NULL DEFAULT 'inactive' COMMENT '开关:active生效/inactive停用',
  remark          VARCHAR(500) NULL COMMENT '备注',
  created_at      DATETIME NOT NULL, updated_at DATETIME NOT NULL,
  PRIMARY KEY (multiplier_campaign_id),
  INDEX idx_rmc_scope_status (lottery_campaign_id, status, start_at, end_at),
  INDEX idx_rmc_target (target_type),
  CONSTRAINT fk_rmc_lottery_campaign FOREIGN KEY (lottery_campaign_id)
    REFERENCES lottery_campaigns(lottery_campaign_id) ON DELETE CASCADE
) COMMENT='水晶奖品倍率规则(可绑定到具体抽奖活动,活动间隔离)';
```

- **活动隔离（关键）**：`lottery_campaign_id` 把规则绑定到某个抽奖活动，该规则**只在此活动内生效**。用户 A 在活动1 配 ×2，只影响活动1 抽中的水晶，活动2/活动3 完全不受影响。**强制必填（已拍板），禁止全局规则**，杜绝"一条规则到处翻倍"的误伤。
- **倍率支持小数**：`1.50/1.75/2.00/2.50` 等运营任意设置；小数结果按 `rounding_mode` 取整（**已拍板默认向上 ceil，偏体感**）。
- **`lottery_campaign_id` 为 INT**：对齐现网 `lottery_campaigns.lottery_campaign_id`（实测为 INT，非 BIGINT）。
- **per-user 三重护栏（拍板·常驻翻倍防长期薅，均运营可调、NULL=不限）**：`per_user_daily_limit`（每日翻倍次数）、`eligibility_days`（资格时间盒，如新用户注册后 N 天）、`per_user_extra_cap`（单人累计额外数量上限）。配合"人群实时判定（掉出人群即失效）"与规则级 `extra_cost_limit`，形成"实时资格 + 单人次数/时长/数量 + 规则总成本"多层限制。

### 3.2 倍率活动作用对象表 `reward_multiplier_targets`（引用现网人群，不自造人群）

本表只登记"引用哪个现网人群标识"，人群定义由 `segment_rule_configs` / `user_ad_tags` / `user_growth_levels` 负责，本方案零重复造轮子。

```sql
CREATE TABLE reward_multiplier_targets (
  id BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键',
  multiplier_campaign_id BIGINT NOT NULL COMMENT '所属倍率活动',
  target_type ENUM('segment','tag','growth_level','user') NOT NULL COMMENT '对象类型',
  target_ref  VARCHAR(100) NOT NULL
    COMMENT '引用现网人群标识:segment→segment_rule_configs.rules[].segment_key / tag→user_ad_tags.tag_key / growth_level→user_growth_levels.level_key / user→users.user_id',
  target_value VARCHAR(100) NULL
    COMMENT '可选精确值:target_type=tag时匹配 user_ad_tags.tag_value(如 true);其它类型可空',
  created_at  DATETIME NOT NULL, updated_at DATETIME NOT NULL,
  PRIMARY KEY (id),
  UNIQUE INDEX uk_rmt (multiplier_campaign_id, target_type, target_ref),
  INDEX idx_rmt_ref (target_type, target_ref),
  CONSTRAINT fk_rmt_campaign FOREIGN KEY (multiplier_campaign_id)
    REFERENCES reward_multiplier_campaigns(multiplier_campaign_id) ON DELETE CASCADE
) COMMENT='倍率活动作用对象(仅引用现网人群标识);target_type=all时本表无记录';
```

- `target_type='all'`（全体生效）时本表无记录，主表 `target_type=all` 即可。
- 其它类型在本表登记引用，支持一个倍率活动绑定多个人群标识：
  - `segment`：`target_ref` = `segment_rule_configs` 规则里的 `segment_key`（如 `vip_premium`、`new_user`）。命中判定复用现网分群引擎对该用户求值。
  - `tag`：`target_ref` = `user_ad_tags.tag_key`（如 `lottery_active_7d`），`target_value` 可选精确匹配 `tag_value`。
  - `growth_level`：`target_ref` = `user_growth_levels.level_key`（如 `gold`/`diamond`）。
  - `user`：`target_ref` = 具体 `user_id`（用于"某个人"精确定向）。
- **不建外键到人群表**（`target_ref` 是弱引用软标识，人群配置可独立增删版本），仅在读取时由服务层解析命中；引用失效（如 segment_key 被删）视为"无人命中"，安全降级为 ×1。

### 3.3 发放倍率快照（复用现有 `lottery_draws.result_metadata`）

**不新增列**，把倍率来源快照写入现网已有的 `lottery_draws.result_metadata`（JSON），防纠纷、可审计：

```json
{
  "crystal_multiplier": {
    "base_quantity": 10,
    "applied_multiplier": 3.00,
    "final_quantity": 30,
    "multiplier_campaign_id": 12,
    "reason": "新春水晶翻倍 ×3",
    "candidates": [
      { "multiplier_campaign_id": 12, "multiplier": 3.00 },
      { "multiplier_campaign_id": 8, "multiplier": 2.00 }
    ]
  }
}
```

同时 `asset_transactions.meta`（材料入账流水）追加 `crystal_multiplier` 摘要，保证资产侧也可对账追溯。

## 4. 抽奖发放计算逻辑

### 4.1 计算流程（伪代码）

```text
入口：SettleStage 处理 prize_type='material' 且命中水晶类的奖品
1. base_quantity = prize.material_amount
2. 判断 prize 是否水晶类（form in shard/gem 且非货币，或命中 reward_scope）
   - 非水晶类 → multiplier = 1，正常发放，不写快照
   - 是水晶类但 prize.is_fallback=1（兜底/保底奖）→ multiplier = 1，兜底不翻倍（拍板）
3. 查询「当前对该用户、本次抽奖活动生效」的倍率规则：
     status = 'active'
     AND lottery_campaign_id = 本次 draw.lottery_campaign_id  // ★活动隔离(强制绑定,无全局规则)
     AND (start_at IS NULL OR now >= start_at)
     AND (end_at   IS NULL OR now <= end_at)
     AND reward_scope 覆盖该奖品
     AND 命中人群（复用现网人群引擎解析）：
         target_type='all'
         OR target_type='segment'      且该用户经 segment_rule_configs 求值命中 target_ref(segment_key)
         OR target_type='tag'          且 user_ad_tags 存在 tag_key=target_ref (且 tag_value 命中 target_value)
         OR target_type='growth_level' 且该用户当前成长等级(user_growth_levels 派生) = target_ref
         OR target_type='user'         且 target_ref = 本人 user_id
     AND extra_cost_used < extra_cost_limit   // 该规则额外成本未击穿(强制必填,无NULL)
4. 收集所有命中规则的 multiplier（因已按活动过滤，这些都是"本活动内"的规则）
5. 合并（同活动内取 max）：applied = max(candidates.multiplier, 1)
6. 二次夹紧：applied = min(applied, 命中规则的 max_multiplier_cap 最小值)   // 防误填
7. 取整（按 rounding_mode，已拍板默认 ceil 向上）：final_quantity = ceil(base_quantity × applied)
      // 例：10×2=20；10×1.5=15；10×1.75=17.5→18(ceil)；10×1.55=15.5→16；10×2.5=25
8. extra_cost = (final_quantity - base_quantity) × 该资产 budget_value_points
9. 事务内：
     - 累加中选规则 extra_cost_used += extra_cost（行锁，达上限则该规则本次不参与→回退到次高倍率）
     - 用 final_quantity 入账（BalanceService.changeBalance）
     - 写 lottery_draws.result_metadata.crystal_multiplier 快照
     - asset_transactions.meta 追加倍率摘要
```

> **活动隔离说明**：一次抽奖必然属于且仅属于一个抽奖活动（`draw.lottery_campaign_id`），第 3 步已按此过滤，因此**不同抽奖活动的倍率规则天然互不影响**；`max` 合并只发生在"同一活动内、同一用户命中的多条规则"之间（如该活动同时配了"新用户 ×1.5"和"高等级 ×2"，用户两者都命中则取 ×2）。

### 4.2 代码落点（现网真实文件）

| 步骤 | 落点 |
|---|---|
| 人群命中判定 | **复用现网分群/标签/等级**：分群走 `segment_rule_configs` 现有求值逻辑（与抽奖档位概率同源）、标签查 `user_ad_tags`、等级查 `user_growth_levels` 派生；不新写人群引擎 |
| 倍率查询与合并 | 新增 `services/lottery/CrystalMultiplierService.js`（读倍率规则 + 调人群判定 + max 合并 + cap 夹紧，纯计算，可单测） |
| 发放时应用倍率 | `services/UnifiedLotteryEngine/pipeline/stages/SettleStage.js` material 分支（约 710–716 行）：`delta_amount` 由 `prize.material_amount` 改为 `finalQuantity` |
| 快照写入 | 同 SettleStage，写 `lottery_draws.result_metadata` + `asset_transactions.meta` |
| 额外成本累加/击穿 | 事务内对 `reward_multiplier_campaigns.extra_cost_used` 行锁自增（复用现有 `TransactionManager`/分布式锁范式） |

> 关键：全程在**现有结算事务内**完成，复用现网幂等键（`${idempotency_key}:material`），杜绝重复发放翻倍奖品。

### 4.3 击穿回退（预算保护）

若最高倍率规则的额外成本已击穿（`extra_cost_used >= extra_cost_limit`），该规则本次不参与合并，自动回退到**次高**仍有额度的倍率；全部击穿则 ×1 正常发放。保证成本永不超上限。

## 5. 用户感知设计（避免多活动困惑）

这是本方案的核心，多个活动都能翻倍时用户最易困惑，靠"规则透明 + 展示唯一 + 结果可解释"解决。

### 5.1 全局公示一句话

抽奖页/活动页固定文案：

> **水晶奖品倍率活动不叠加，同一次抽奖按最高倍率计算。**

### 5.2 抽奖前：只显示一个"合并后倍率"

顶部只显示合并后的最终倍率，**不罗列多个活动让用户自己算**：

```
🎉 当前水晶奖品 ×3 倍（新春水晶翻倍）  有效期至 2 月 15 日
```

只给结果、不给过程。日常态（×1）不显示倍率，避免"×1"让用户觉得亏；仅在有加成时高亮。

### 5.3 抽中后：写清 `基础 × 倍率 = 最终`

```
恭喜获得 红源晶碎片 ×30
（基础 10 × 新春翻倍 3 倍）
```

数据来源即 `result_metadata.crystal_multiplier` 快照，用户一看就懂，客服也可据此答疑。

### 5.4 定向人群不做公开攀比

A 用户 ×3、B 用户 ×2（因人群不同）时，**各自只看到自己的倍率**：

- 定向活动 `display_name` 用中性词（如"专属加成"），不强调"因为你是 XX"。
- 用户只在自己的抽奖页看到自己的倍率，看不到他人倍率，杜绝攀比纠纷。

### 5.5 倍率回落有预告

活动结束前提示"活动即将结束"，避免用户从 ×3 突然回到 ×1 的落差感。

## 7. 成本与风控

| 风险 | 对策 |
|---|---|
| 倍率放大真实价值奖品 → 成本失控 | 同活动内 Max 不叠乘 + `max_multiplier_cap` 硬上限 + `extra_cost_limit` 成本封顶 + 击穿回退 |
| 多活动被薅 | Max 策略天然防叠加；定向人群精确控制 |
| 重复发放 | 复用结算事务 + 幂等键 `${idempotency_key}:material` |
| 并发重复累加成本 | `extra_cost_used` 行锁自增（`SELECT ... FOR UPDATE`）或分布式锁 |
| 成品奖品不可翻倍 | 倍率只作用"可计数材料数量"(shard/gem)；成品(item)不参与，奖品定义层区分 |
| 审计追溯 | `lottery_draws.result_metadata` + `asset_transactions.meta` 双端快照 |

## 9. 范围边界

- 本方案**只做"水晶类材料奖品发放数量倍率"**，不改抽奖概率、不改预算发放、不改非材料奖品。
- 倍率与抽奖引擎、预算系统**解耦**：仅在 SettleStage 发放材料的最后一步介入，改动面最小、最安全。
- 与既有 `lottery_campaigns.participation_conditions`（参与条件）互不冲突：参与条件决定"能不能抽"，本方案决定"抽中水晶发多少"，两层正交。

## 12. 防囤积套利：活动专属预算 + 到期清零/折算（2026-07-05 追加）

### 12.1 问题：翻倍活动的"囤积套利"

用户平时消费攒预算积分不抽，专等翻倍活动一次性抽走，等于用平时低成本预算套翻倍高价值奖品，平台资损。

**根因**：预算积分"赚"与"用"之间无活动隔离——翻倍活动若接受平时囤的全局预算，即被套利。

### 12.2 现网已有的钩子（实测确认，无需大改）

| 现网结构 | 现状 | 用途 |
|---|---|---|
| `services/consumption/CoreService.js` 预算发放 | 消费审核通过发 `BUDGET_POINTS`，打标 `lottery_campaign_id`（当前硬编码 `'CONSUMPTION_DEFAULT'` 全局桶） | 预算积分本就按活动分桶存储 |
| `account_asset_balances.lottery_campaign_id` / `lottery_campaign_key` | 预算积分按活动隔离存储 | 天然支持活动专属预算 |
| `lottery_campaigns.allowed_campaign_ids` (JSON) | 抽奖活动可指定"接受哪些来源活动的预算" | 决定翻倍活动吃不吃全局囤积预算 |
| `lottery_campaign_user_quota.expires_at` + status `expired` | 配额已支持到期 | 到期清零/折算范式可复用 |
| `lottery_campaign_user_quota` / `max_draws_per_user_*` | 每用户配额/次数上限 | 兜底护栏 |

### 12.3 方案：三层防护

**第 1 层（根治）活动专属预算，翻倍活动不吃全局囤积预算**

- 翻倍活动期间，消费审核发的预算积分**额外打标该活动的 `lottery_campaign_id`**（改 `CoreService` 那行硬编码 `CONSUMPTION_DEFAULT` 为"命中活动专属累积配置时用活动桶"）。
- 翻倍活动 `allowed_campaign_ids` **只允许本活动专属预算桶**，排除 `CONSUMPTION_DEFAULT`。
- 效果：享受翻倍必须活动期间消费赚活动专属预算；平时囤的全局预算进不了翻倍场 → 套利失效（游戏"活动限定代币"模型）。

**第 2 层（防长期囤积）活动专属预算到期清零/折算**

- 活动专属预算带 `expires_at`（复用 `lottery_campaign_user_quota.expires_at` 范式）。
- 活动结束由定时任务处理：**直接清零（已拍板 防2，不折算）**，与游戏活动代币一致。

**第 3 层（兜底护栏，已在本方案内）**

- 翻倍活动 per-user 抽奖次数/额度上限（`lottery_campaign_user_quota` / `max_draws_per_user_*`）。
- 翻倍规则 `extra_cost_limit` 成本封顶（§10 已定稿）——即便钻空子，平台总成本封死。

### 12.4 行业对照

| 主体 | 做法 | 归类 |
|---|---|---|
| 游戏公司(米哈游/腾讯/网易) | 活动限定代币，活动内赚活动内用、结束清零 | 活动专属货币(最强) |
| 大电商(阿里/京东) | 积分年底清零 + 膨胀券时效 + 抵扣上限 | 到期清零 + 时效 + 封顶 |
| 航空里程 | 里程有效期过期作废 | 到期清零 |
| 美团/本地生活 | 活动券短有效期 + 活动积分活动内有效 | 时效 + 活动隔离 |
| 活动策划公司 | 活动积分周期内有效，结束清零/折算，预算锁周期 | 活动专属 + 到期折算 |
| 小公司 | 无隔离无到期 → 被薅穿 | 反面教材 |
| 虚拟物品/二手平台 | 赛季/活动货币季末重置 | 活动专属 |

**共识**：成熟玩家均为「活动专属货币 + 时间盒 + 到期清零/折算」。本方案第 1+2 层与之一致，第 3 层为额外兜底。

### 12.6 两类翻倍活动的区分（常驻翻倍 vs 限时翻倍）

现实会同时存在两类翻倍活动，二者**是否清零由"翻倍是否稀缺限时"决定**，不是拍脑袋：

- **全局长期翻倍（如活动1）**：翻倍长期有效 → 用户随时抽都翻倍、囤积无额外收益 → **无囤积套利动机** → 用全局永久预算 `CONSUMPTION_DEFAULT`，**不清零**。典型：新用户/高等级常驻加成。
- **短期稀缺翻倍（如活动2）**：翻倍限时 → 有"平时囤、翻倍日套现"动机 → **必须活动专属预算 + 到期清零**（§12.3 第 1、2 层）。

用同一套机制、靠配置区分，无需两套代码：

| 维度 | 活动1 全局长期翻倍 | 活动2 短期稀缺翻倍 |
|---|---|---|
| `lottery_campaign_id` | 活动1 | 活动2 |
| 预算桶 | `CONSUMPTION_DEFAULT`（全局永久） | 活动2 专属桶 |
| 预算来源 | 平时消费常规归集 | 活动2 期间消费归集 / 活动任务 |
| `allowed_campaign_ids` | `[CONSUMPTION_DEFAULT]` | `[活动2专属桶]`（排除全局） |
| 到期 | 永不清零 | 活动 `end_time` 触发直接清零 |
| 倍率规则 | 绑活动1，target=新用户/高等级 | 绑活动2，target=符合条件用户 |
| 套利风险 | 无（常驻翻倍无囤积动机） | 有 → 专属预算+清零根治 |

两个关键设计点（降低耦合与技术债）：

1. **"到期清零"是预算桶级属性，不是倍率级属性**：倍率逻辑单一职责（只管发奖翻倍），预算生命周期由"桶 + 归集 + 清理任务"管理，两块解耦。
2. **到期用活动 `lottery_campaigns.end_time` 驱动，不给每条预算加 `expires_at`**：活动2 结束时清理任务扫其专属桶余额直接清零；活动1 无 end_time（常驻）则桶永不清理。"短期清零、长期不清零"是桶级配置的自然结果。

### 12.7 双层货币模型：可见积分也要分（防套利要"看得见"）

**实测确认的关键机制**：抽一次奖，扣的是 **`AssetCode.POINTS`（可见积分）**（`SettleStage.js` 第 206 行 `delta_amount: -draw_cost`）；**预算积分是后台的"奖品价值门槛 + 成本控制"，用户看不见**。

因此：若防套利只在"看不见的预算积分"层做，会出现**用户看到自己有一堆可见积分、却在活动2 抽不了/抽不到好东西**的困惑（后台预算桶不匹配，用户无法理解）。所以**可见积分层也要区分**，且要让用户看得懂。

**双层货币模型**：

| 层 | 货币 | 用户可见 | 作用 | 分桶策略 |
|---|---|---|---|---|
| **可见层（入场）** | 可见积分 `points` | ✅ 看得见 | 决定"能在哪个活动抽" | 全局可见积分（常驻）+ 活动专属可见积分/代币（限时，到期清零） |
| **隐藏层（成本）** | 预算积分 `budget_points` | ❌ 看不见 | 决定"能抽出多大价值 + 平台成本封顶" | 已分桶（§12.3），按活动隔离 |

可见层怎么分：
- **全局可见积分**：平时消费获得，供常驻活动（活动1），不清零。
- **活动专属可见积分/代币**：短期翻倍活动2 期间获得，**只能在活动2 抽**，活动结束清零 —— 等价游戏"活动限定代币"（可见、限时、到期清空），用户天然理解，无"有积分却抽不了"的困惑。

**拍板实现（方式 A：新增独立"活动代币"资产类型）**：

- **新增一个通用资产类型** `event_points`（`material_asset_types` 加一条），与全局 `points` 是两种不同的钱；全局 `points` 语义/代码**完全不动**。
- **按 `lottery_campaign_id` 分桶**：一种 `event_points` 服务所有活动，每个活动的余额独立（同预算积分分桶范式），不为每个活动新造货币类型，避免货币泛滥。
- **展示名运营每活动可配**：系统名 `event_points`，展示名如"新春活动积分""周年庆代币"（同倍率规则 display_name 套路），默认"活动积分"。
- **仅在活动场景露出**：主钱包用户只见"积分"；进入该活动才显示"本活动积分：N，可在本活动抽奖 + 到期倒计时"。等价游戏活动代币（活动外不可见），把用户认知负担降到最低。
- **到期清零**：活动 `end_time` 触发清理任务清零该活动的 `event_points`（与隐藏层预算桶同一任务处理）。

> 为何不选"给全局 `points` 分桶"（方式 B）：`points` 用途广、改动影响面大；且"同名货币不同可用范围"会让用户困惑（"我有 120 积分为何活动 2 只能用 20"）。新增独立 `event_points` 让用户从第一天就建立"活动积分分活动、限时、会清零"的认知，不违背既有习惯。

设计原则：**可见层（`event_points`）管"能不能进这个活动抽"，隐藏层（`budget_points` 分桶）管"抽出多大价值 + 平台成本"**，两层各司其职，共同根治套利。

### 12.9 活动预算账户模型（个人账户 + 池账户，复用现网预算架构）

用户诉求"个人活动账户 vs 全局(池)活动账户区分、池空回退个人、按活动隔离"——**实测确认 1:1 命中现网 `lottery_campaigns` 已有预算字段，零新增账户体系**。

| 概念 | 现网真实载体 | 说明 |
|---|---|---|
| 个人活动预算账户 | `account_asset_balances`（`asset=BUDGET_POINTS` + `lottery_campaign_id=活动ID`） | 每"用户×活动"一行，个人预算，按活动隔离 |
| 全局(池)活动预算账户 | `lottery_campaigns.pool_budget_total` / `pool_budget_remaining` | 运营注资的活动池，实时扣减 |
| 预算模式 | `lottery_campaigns.budget_mode`（user/pool/none） | 个人扣 / 池扣 / 不限 |
| 抽取顺序（池空回退个人） | `lottery_campaigns.preset_budget_policy = 'pool_first'` | 先扣池（运营补贴）→ 池空自动用个人账户 |
| 只吃本活动个人预算 | `lottery_campaigns.allowed_campaign_ids = [本活动]` | 排除其他来源，实现活动隔离 |
| 活动间隔离 | 全部按 `lottery_campaign_id` 分桶 | 活动1/活动2/常驻 天然互不干扰，与常驻活动同机制 |

归集与抽取流程：

```
活动期间消费审核通过：
  预算积分全额 → 该用户在该活动的【个人活动预算账户】(BUDGET_POINTS, lottery_campaign_id=活动ID)
  (不再进全局 CONSUMPTION_DEFAULT —— 防7 全量重定向)

活动内抽奖扣预算(pool_first)：
  先扣【活动池账户】pool_budget_remaining (运营补贴)
  池 = 0 → 回退扣用户【个人活动预算账户】(自己攒的)
  两账户均按 lottery_campaign_id 隔离，与常驻活动1、其他活动互不影响
```

设计要点：

- **与常驻活动一致的隔离**：常驻活动1 用全局永久桶（`CONSUMPTION_DEFAULT` + 其自身池），限时活动2 用活动专属桶（个人 + 池），全部 `lottery_campaign_id` 隔离，一套机制、配置区分。
- **池账户 = 运营补贴，个人账户 = 用户自攒**：`pool_first` 让活动初期用运营池给足体感，池耗尽后回落到用户自有预算，成本与体感平衡。
- **到期清零**（防2）作用于活动的个人账户 + `event_points`；活动池余额到期一并处理（清零或回收，运营配）。

### 12.10 预算归集入口设计（不加人工选择口，后端规则自动判定）

现网消费提交由**商家人员/管理员扫用户码提交审核**（非用户操作）。预算进"全局还是活动"**不做人工选择口**，改由后端归集规则自动判定：

**为何不加人工选择口**：一线手动选 = 舞弊/选错风险 + 培训与客诉成本 + 泄露内部机制 + 违背"规则化配置"设计哲学。

**归集规则（运营在 Web 后台配置）**声明：关联活动 `lottery_campaign_id`、生效时间窗（对齐活动 `end_time`）、命中条件（门店/商家/消费类型/用户人群，复用现网分群与门店维度）、归集去向（防7：全额进活动个人账户）。

**各端职责**：

| 端 | 是否加选择口 | 职责 |
|---|---|---|
| 用户微信小程序 | ❌ 不加 | 仅出示二维码；可选展示"本次获得 X 活动积分" |
| 商家端（提交消费） | ❌ 不加 | 照常提交消费；可选展示"当前有 XX 活动进行中" |
| Web 管理后台 | ✅ 在此设计 | 配置"活动预算归集规则"（时间/条件/去向），真正的入口设计在这里 |

**落地点**：`services/consumption/CoreService.js` 预算分配处，现硬编码 `lottery_campaign_id: 'CONSUMPTION_DEFAULT'` → 改为"先查命中的活动归集规则，命中则全额投该活动个人账户（`BUDGET_POINTS` + 活动ID）+ 发 `event_points`；未命中维持全局 `CONSUMPTION_DEFAULT`"。全自动，一线与用户零选择。
