## 二、P0 — 功能缺陷（全部确认存在，均为后端问题）

### 2.1 hourly-unlock-timeout-trade-orders.js — `_releaseTimeoutLockedItems()` 空壳

**文件**：`jobs/hourly-unlock-timeout-trade-orders.js`
**归属**：后端

| 子项 | 验证结果 |
|---|---|
| `_releaseTimeoutLockedItems()` 是空壳 | **确认**。方法体全部注释为 TODO，始终返回 `released_count: 0` |
| `LOCK_TIMEOUT_MINUTES` 未定义 | **已修复**。第81行已改为 `this.LOCK_TIMEOUT_MINUTES = await this.getLockTimeoutMinutes()` |
| `_cancelTimeoutOrders()` 逻辑 | **正常**。已正确使用 `this.LOCK_TIMEOUT_MINUTES`、TradeOrder/MarketListing/BalanceService.unfreeze 完整链路 |

**实际影响**：订单超时取消（`_cancelTimeoutOrders`）已能正常工作；但物品hold释放（`_releaseTimeoutLockedItems`）完全失效，trade类型的item_holds记录（252条）中超时的永远不会被释放。

**修复方案**（后端）：

| 步骤 | 操作 | 涉及文件 | 可复用 |
|---|---|---|---|
| 1 | 重写 `_releaseTimeoutLockedItems()` | `jobs/hourly-unlock-timeout-trade-orders.js` | 复用已有的 `ItemHold` 模型（`models/ItemHold.js`），字段 `hold_type`, `status`, `expires_at` 已在数据库中就绪 |
| 2 | 查询条件 | `ItemHold.findAll({ where: { hold_type: 'trade', status: 'active', expires_at: { [Op.lt]: new Date() } } })` | 数据库 `item_holds` 表已有 `idx_item_holds_status` 索引 |
| 3 | 释放逻辑 | 更新 `status='expired'`, `released_at=NOW()`；同时更新关联 `items.status` 从 `held` → `available` | 复用 `Item` 模型，`items` 表的 `status` enum 已包含 `held`→`available` |
| 4 | 记账 | 通过 `ItemLedger` 记录释放事件 | 复用 `ItemLedger` 模型（`item_ledger` 表，event_type 字段支持扩展） |

**前端影响**：无，这是纯后端定时任务。

### 2.2 AuditRollbackService.js — 双重错误

**文件**：`services/AuditRollbackService.js`
**归属**：后端

| 子项 | 验证结果 |
|---|---|
| `getService('asset')` 不存在 | **确认**。ServiceManager 注册的是 `asset_balance`，调用 `getService('asset')` 直接抛异常 |
| 调用 `adjustPoints()` 不存在 | **确认**。BalanceService 暴露的是 `changeBalance()`，`adjustPoints` 全项目唯一出现就在此处 |

**修复方案**（后端，约15分钟）：

| 步骤 | 操作 |
|---|---|
| 1 | `getService('asset')` → `getService('asset_balance')` |
| 2 | `adjustPoints({...})` → `changeBalance({...})`，对齐参数签名：`{ user_id, asset_code, amount, business_type, idempotency_key, meta }` |

**可复用**：`BalanceService.changeBalance()` 是资产体系的统一余额变更接口，已被 20+ 个调用者使用，参数签名稳定。

**前端影响**：无，审计回滚是后端管理功能。

### 2.3 UserRoleService.getUserList 分页元数据 NaN

**文件**：`services/UserRoleService.js`
**归属**：后端

| 子项 | 验证结果 |
|---|---|
| SQL 查询层 | **安全**。`finalLimit = Math.min(parseInt(limit || page_size || 20), 100)` 有三层回退 |
| 响应 pagination 对象 | **确认 bug**。`per_page: parseInt(limit)` 和 `total_pages: Math.ceil(count / parseInt(limit))` 当 limit 为空时返回 NaN |

**修复方案**（后端，约5分钟）：响应 pagination 中 `parseInt(limit)` → `finalLimit`（复用已计算的安全值）

**Web管理后台前端影响**：修复后分页控件自动恢复正常（前端 data-table 组件读取 `pagination.per_page` 和 `pagination.total_pages`）。
