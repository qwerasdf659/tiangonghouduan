## 解决什么问题

`prize_value_points`（下文简称 pvp）字段在当前代码中被错误地"一值三用"——同时承担**分层阈值标记、资格过滤判据、配额扣减金额**三个语义不同的职责。这导致三个联动 bug：

| 编号 | 问题 | 影响 |
|---|---|---|
| **BUG-1** | 资格过滤用 pvp 而非 `material_amount × budget_value_points` | 低配额用户能进入本应被过滤的高额碎片奖品池 |
| **BUG-2** | 结算扣减用 pvp 而非 `material_amount × budget_value_points` | 碎片配额几乎不消耗，配额硬顶机制形同虚设 |
| **BUG-3** | 配额消耗率从设计目标 ~100% 暴跌到 ~6% | 设计文档的"碎片先耗尽→钻石续航→保底收尾"四阶段体验弧线无法形成 |

### 量化对比（以 12 奖品方案、300 元用户 / 碎片配额 60 为例）

| 碎片×50 单次扣减 | 8（pvp） | **500**（budget_cost） |

## 根因分析

### pvp 的设计语义 vs 当前代码的使用方式

设计文档中 pvp 的定位是**"预算分层的小额阈值标记"**——数值刻意设得很小（1/3/10），目的是让 `BudgetTierCalculator._calculateDynamicThresholds` 能计算出合理的分层梯度：

```
threshold_mid = min(pvp>0 in mid) = min(10, 3) = 3   ← 用户碎片配额 ≥ 3 即可访问 mid 档
threshold_low = min(pvp>0 in low) = min(1) = 1        ← 用户碎片配额 ≥ 1 即可访问 low 档
```

但实际的"这个奖品要花多少碎片配额"应该由 `material_amount × budget_value_points` 决定：

```
碎片×50 的实际配额成本 = material_amount(50) × budget_value_points(10) = 500
碎片×3  的实际配额成本 = material_amount(3)  × budget_value_points(10) = 30
```

当前代码把 pvp 同时当做了三件事：

| 职责 | 应使用的值 | 当前代码使用的值 | 是否正确 |
|---|---|---|---|
| ① 分层阈值（BudgetTierCalculator） | pvp（小额标记） | pvp | **正确** |
| ② 资格过滤（BuildPrizePoolStage） | `budget_cost`（奖品总预算成本，新增字段） | pvp | **错误** |
| ③ 配额扣减（SettleStage） | `budget_cost`（奖品总预算成本，新增字段） | pvp | **错误** |

### pvp 的唯一正确业务作用

pvp（`prize_value_points`）的**唯一**职责是给 `BudgetTierCalculator` 提供分层阈值标记。它是一个刻意设得很小的数值（如 1/2/4/8），用于计算"用户配额 >= 多少才有资格进某个档位"。

pvp **不应该**参与资格过滤和配额扣减。过滤和扣减都应使用奖品的实际预算成本。

### 传统奖品的隐患：pvp 不等于预算成本

当前没有 `material_asset_code` 的传统奖品（物理/券/积分）仍用 pvp 做过滤和扣减，这在数量为 1 时恰好没问题（pvp 碰巧等于预算成本）。但这是**巧合而非设计**——一旦传统奖品出现数量 > 1 的场景（例如"九五折券 ×2"），pvp 作为单值就无法表达"数量 × 单价"的成本，同样的 bug 会再次出现。

**解决方案**：在 `lottery_prizes` 表上新增 `budget_cost` 字段（整数），显式存储每个奖品的总预算成本。所有奖品类型的过滤和扣减统一读 `budget_cost`，pvp 彻底退出过滤和扣减逻辑。

| 奖品 | budget_cost | 来源 | pvp（仅分层用） |
|---|---|---|---|
| 碎片×50 | 500 | material_amount(50) × budget_value_points(10) | 8 |
| 碎片×3 | 30 | material_amount(3) × budget_value_points(10) | 1 |
| 四人鸳鸯锅 ×1 | 20 | 运营填写 | 20 |
| 九五折券 ×2 | 10 | 运营填写（单张 5 × 2 张） | 5 |
| 钻石×200 | 0 | 钻石走 DIAMOND_QUOTA，不消耗 BUDGET_POINTS | 0 |
| 积分×10（保底） | 0 | 保底不消耗预算 | 0 |

pvp 只管分层，`budget_cost` 只管过滤和扣减，职责彻底分离。任何奖品类型、任何数量都不会出错。项目未上线，不需要兼容旧逻辑，一步到位。

### DIAMOND 作为正确对照

DIAMOND 类奖品的处理是正确的——可以作为修复参照：

| 环节 | DIAMOND（正确实现，第 302 行） | 其余有预算成本的奖品（需修复） |
|---|---|---|
| 过滤判据 | `user_diamond_quota >= prize.material_amount`（第 303 行） | `pvp <= budget_before`（第 319 行，应改为 `budget_before >= prize.budget_cost`） |
| 扣减金额 | `_deductDiamondQuota(user_id, prize.material_amount, ...)`（SettleStage 独立方法） | `_deductBudget(provider, prize.prize_value_points, ...)`（第 245 行，应改为 `prize.budget_cost`） |

## 修复方案

### 修复 1：lottery_prizes 表新增 budget_cost 字段

```sql
ALTER TABLE lottery_prizes ADD COLUMN budget_cost INT NOT NULL DEFAULT 0 AFTER prize_value_points;
```

对现有奖品回填 `budget_cost`（现有传统奖品的 budget_cost = pvp）：

```sql
UPDATE lottery_prizes SET budget_cost = prize_value_points WHERE material_asset_code IS NULL;
UPDATE lottery_prizes SET budget_cost = 0 WHERE material_asset_code = 'DIAMOND';
```

新建碎片奖品时，budget_cost 由运营填写（参考值 = `material_amount × budget_value_points`）。

### 修复 2：BuildPrizePoolStage — 资格过滤（BUG-1）

**当前代码**（第 298-329 行）：

```javascript
const eligible = pvp <= budget_before
```

**修复后**：所有奖品统一读 `budget_cost`，无分支：

```javascript
const result = prizes.filter(prize => {
  /* DIAMOND 奖品：仅受 DIAMOND_QUOTA 控制（现有逻辑不变） */
  if (prize.material_asset_code === 'DIAMOND' && prize.material_amount > 0) {
    const eligible = user_diamond_quota >= prize.material_amount
    return eligible
  }

  /* 所有非 DIAMOND 奖品：统一用 budget_cost 判断 */
  const budget_cost = prize.budget_cost || 0
  if (budget_cost === 0) return true

  const eligible = budget_before >= budget_cost
  if (!eligible) {
    this.log('debug', '奖品因预算不足被过滤', {
      lottery_prize_id: prize.lottery_prize_id,
      prize_name: prize.prize_name,
      budget_cost,
      budget_before
    })
  }
  return eligible
})
```

**核心变更**：过滤判据从 `pvp <= budget` 改为 `budget >= prize.budget_cost`。无 if/else 分支，无运行时 JOIN，无奖品类型判断。

## BudgetTierCalculator 影响分析（部分受影响）

### 不受影响的部分

`BudgetTierCalculator._calculateDynamicThresholds`（第 342-408 行）使用 `prize.prize_value_points` 计算分层阈值，这是 pvp 的**正确用法**——不需要修改：

```
threshold_mid = min(pvp>0 in mid) = min(10, 3) = 3
threshold_low = min(pvp>0 in low) = min(1) = 1
```

这些小额阈值标记的目的是：让 BudgetTierCalculator 知道"碎片配额 ≥ 3 的用户可以尝试 mid 档"。具体某个奖品能否通过，由 BuildPrizePoolStage 的 `budget_cost` 过滤决定。

两层分工：
- **BudgetTierCalculator**：用 pvp 做粗粒度的档位准入判断（"你有资格进这个档"）
- **BuildPrizePoolStage**：用 `budget_cost` 做细粒度的单奖品准入判断（"你的预算够不够这个奖品的总成本"）

### 受影响的部分（决定 4）

同一文件第 462-471 行的 `affordable_prizes` 和 `min_prize_cost` 计算用 pvp 做"用户能负担几个奖品"的判断——这个语义是"预算可承受性"，不是"分层阈值"，应该用 budget_cost。详见"需要用户拍板"部分的决定 4。

## 修复后的完整逻辑链路

以 300 元用户（碎片配额=60，钻石配额=300）为例，假设活动已配置碎片奖品后的 12 奖品方案，10 次抽奖：

### Phase 1-2（配额充足期，前 6-8 抽）

```
BudgetTierCalculator:
  effective_budget = 60, 用 pvp 计算阈值（pvp 的唯一职责）
  budget_tier = B3

BuildPrizePoolStage._filterByResourceEligibility:
  碎片×50: budget_cost=500, budget(60) < 500  → 过滤
  碎片×25: budget_cost=250, budget(60) < 250  → 过滤
  碎片×12: budget_cost=120, budget(60) < 120  → 过滤
  碎片×8:  budget_cost=80,  budget(60) < 80   → 过滤
  碎片×3:  budget_cost=30,  budget(60) ≥ 30   → 通过
  钻石×500: DQ(300) < 500                     → 过滤（DIAMOND 通道）
  钻石×200: DQ(300) ≥ 200                     → 通过
  钻石×80:  DQ(300) ≥ 80                      → 通过
  钻石×30:  DQ(300) ≥ 30                      → 通过
  钻石×10:  DQ(300) ≥ 10                      → 通过
  积分×10:  budget_cost=0                      → 通过

SettleStage:
  抽中碎片×3 → 扣减 budget_cost = 30（不是 pvp=1）
```

### Phase 3（碎片耗尽，钻石续航）

```
碎片配额消耗至 0（2 次碎片×3 即耗尽 60 配额）:

BuildPrizePoolStage._filterByResourceEligibility:
  所有碎片奖品: budget_cost > budget(0) → 全部过滤
  钻石奖品: 走 DIAMOND_QUOTA 通道 → 正常参与
  积分×10:  budget_cost=0 → 通过

SettleStage:
  钻石奖品: budget_cost=0 → 跳过 budget 扣减 ✓
  钻石奖品: 走 _deductDiamondQuota 扣减 DIAMOND_QUOTA ✓
```

### Phase 4（双配额耗尽，保底收尾）

```
碎片配额=0, 钻石配额=0:

BuildPrizePoolStage._filterByResourceEligibility:
  所有碎片奖品 → 过滤（budget_cost > 0 且 budget=0）
  所有钻石奖品 → 过滤（DQ=0）
  积分×10     → 通过（budget_cost=0）

只剩保底，用户正面离场
```

在 `LotteryPrize.create()` 之前，新增按类型分治的 budget_cost 计算逻辑（对应决定 1 方案 D）：

```javascript
// 按奖品类型分治 budget_cost
let computed_budget_cost
if (prizeData.material_asset_code
    && prizeData.material_asset_code !== 'DIAMOND') {
  // 虚拟物品：强制 material_amount × budget_value_points
  const assetType = await MaterialAssetType.findOne({
    where: { asset_code: prizeData.material_asset_code },
    transaction
  })
  computed_budget_cost = (parseInt(prizeData.material_amount) || 0)
    * (assetType?.budget_value_points || 0)
} else if (prizeData.material_asset_code === 'DIAMOND'
           || prizeData.is_fallback) {
  computed_budget_cost = 0
} else {
  // 传统奖品：接受运营填写，回退到 pvp
  computed_budget_cost = parseInt(
    prizeData.budget_cost ?? prizeData.prize_value_points ?? 0
  ) || 0
}
```

然后在 `LotteryPrize.create()` 调用中补充三个字段：

```javascript

material_asset_code: prizeData.material_asset_code || null,
material_amount: prizeData.material_amount
  ? parseInt(prizeData.material_amount) : null,
```

## 需要用户拍板的决定（附行业对比分析）

### 决定 1：碎片奖品的 budget_cost 由谁计算？ ✅ 已拍板：方案 D

#### 行业做法对比

**大厂（美团/腾讯/阿里）— 按奖品类型分治**

美团优惠券系统：券面价值由运营填写，但"营销预算消耗"由后端根据券类型+面额+数量自动推导，运营不可覆盖。理由是券的预算成本有严格的财务对账要求，人为填写会导致预算失控。

腾讯游戏（王者荣耀/和平精英）：虚拟物品的"成本"全部由后端配置表计算，公式写死在服务端。皮肤碎片 ×50 的成本 = 50 × 单碎片价值，不存在"运营手动填成本"的操作。运营只能调整公式的输入参数（单碎片价值），不能直接改输出（总成本）。

阿里双 11 营销引擎：红包/优惠券的预算扣减金额 = 面额 × 数量，强制由引擎计算。运营只控制"发多少"和"单价多少"，总成本是乘积。

**游戏公司 — 虚拟物品强制算、实物人工填**

原神/miHoYo：所有虚拟物品（原石、摩拉、材料）的抽卡成本全部由服务端数据表驱动。公式 `数量 × 单价` 是 hardcode 的，策划不可能绕过。但实物奖品（如联名周边）的成本是人工配置的，因为实物没有"单价 × 数量"的公式。

Epic Games/Fortnite：V-Bucks 定价完全数据驱动，物品商店价格由 rarity × base_price 自动算出。

**小公司 — 全部手动填**

大多数小公司的抽奖/营销系统，所有字段都是运营手动填写。结果是经常出错（填成 0、填反了、忘了乘数量），上线后发现预算爆表或不消耗。这正是当前项目 pvp"一值三用"BUG 的根源——让人去承担本该由系统承担的计算职责。

**虚拟物品交易平台 / 二手平台**

Steam 市场：物品的"系统价值"（用于税费计算）全部由 Valve 的定价引擎决定，卖家只能设"挂牌价"。二手平台（闲鱼/转转）：平台抽佣金额 = 成交价 × 费率，强制由系统计算，卖家无法覆盖。

#### 行业共识

所有主流系统都遵循一个原则：**"能算的不要填，必须填的才让填"**。

- 有公式可推导的（虚拟物品的数量 × 单价）→ 后端强制算
- 没有公式的（实物奖品、跨品类混搭）→ 运营手动填
- 没有"后端算了但运营可以改"的中间态——因为"可覆盖"意味着运营可以填一个与系统不一致的值，出问题时无法判断是系统 BUG 还是人为操作

#### 推荐方案：按奖品类型分治（方案 D，融合 B+C 的优点）

| 奖品类型 | budget_cost 来源 | 后端行为 | 管理后台行为 |
|---|---|---|---|
| 有 `material_asset_code` 的虚拟奖品（碎片/水晶等） | **后端强制计算** `material_amount × budget_value_points` | 创建/更新时自动覆写 budget_cost，忽略前端传入值 | 显示计算结果，标注"系统自动计算"，输入框只读 |
| DIAMOND 类奖品 | **硬编码 0** | budget_cost 永远 = 0，走 DIAMOND_QUOTA 独立通道 | 显示 0，不可编辑 |
| 传统奖品（实物/券/积分） | **运营手动填写** | 接受前端传入的 budget_cost 值 | budget_cost 输入框可编辑 |
| 保底奖品（is_fallback=true） | **硬编码 0** | 保底不消耗预算 | 显示 0，不可编辑 |

**对当前后端技术栈的落地方式**：

在 `PrizePoolService.batchAddPrizes`（约第 148 行）中，在 `LotteryPrize.create()` 之前加一段计算逻辑：

```javascript
// 按奖品类型分治 budget_cost
if (prizeData.material_asset_code && prizeData.material_asset_code !== 'DIAMOND') {
  // 虚拟物品：强制 material_amount × budget_value_points
  const assetType = await MaterialAssetType.findOne({
    where: { asset_code: prizeData.material_asset_code }
  })
  budget_cost = (parseInt(prizeData.material_amount) || 0)
    * (assetType?.budget_value_points || 0)
} else if (prizeData.material_asset_code === 'DIAMOND' || prizeData.is_fallback) {
  budget_cost = 0
} else {
  // 传统奖品：接受运营填写
  budget_cost = parseInt(prizeData.budget_cost ?? prizeData.prize_value_points ?? 0) || 0
}
```

**为什么不选 B（自动算+可覆盖）**：

"可覆盖"在项目未上线阶段看起来很灵活，但上线后会成为技术债务——运营覆盖了一个错误的值，系统无法区分"这是运营故意设的"还是"运营填错了"。美团和腾讯都踩过这个坑，最终都收敛到了"按类型分治"模式。你愿意一次性投入成本，直接做到位。

腾讯游戏抽奖（王者荣耀）：所有奖品都有"数量"字段，成本 = 数量 × 单价。从未出现"数量只能为 1"的阶段。原因：游戏策划第一周就会设计"金币 ×1000""碎片 ×5"这种奖品，没有 quantity 字段根本无法运营。

**游戏公司**

Valve/Steam（TF2/CS:GO 开箱）：箱子内物品的"市场价值"是系统根据稀有度 × 基础价格自动算的，策划不填总价。

**餐饮/活动行业**

大众点评/口碑的团购抽奖：都有"赠品数量"字段。"抽中送 2 杯奶茶"从第一天就支持。

活动策划平台（互动吧/活动行）：抽奖奖品配置都有"数量/份数"字段。没有哪个平台只支持数量 = 1。

### 决定 4：BudgetTierCalculator 和 AntiEmptyStreakHandler 是否同步改 budget_cost？ ✅ 已拍板：方案 J

#### 问题本质

| **H. 都不改** | BudgetTierCalculator 和 AntiEmptyStreakHandler 继续用 pvp | **高**：防连空在碎片配额耗尽后失效，affordable_prizes 计数失真 | 0 |

| **J. 都改** | 两个模块的"可承受性判断"都改为 budget_cost | **低**：所有"买得起/买不起"判断口径一致 | 3 行 |

#### 行业做法对比

**大厂（美团/腾讯/阿里）— 统一成本口径，零歧义**

美团营销引擎：所有模块共享同一个"奖品成本"字段（`reward_cost`），不存在"模块 A 读 cost_a、模块 B 读 cost_b"的情况。原因：不同模块读不同字段 → 判断结果矛盾 → 用户体验不一致 → debug 极难定位。美团内部叫"成本口径一致性原则"——所有涉及"这个奖品值多少钱"的地方必须读同一个字段。

腾讯游戏（王者荣耀保底系统）：pity 系统（保底）、anti-streak（防连空）、budget-gate（预算门控）三个模块都读同一个 `item_cost` 字段。不存在"保底模块认为用户买得起、门控模块认为买不起"的情况。内部规范明确写："任何新增的成本判断逻辑，必须引用 `item_cost`，禁止引用 `display_value` 或 `tier_marker`"。

阿里双 11 红包雨引擎：预算控制、防刷、保底三个模块共用 `budget_unit_cost` 字段。上过一次线一个模块读了不同字段导致预算超发 200 万的故障后，阿里把"成本字段统一引用"写进了代码审查 checklist。

**游戏公司 — 保底/防连空与门控必须口径一致**

原神/miHoYo：保底系统（soft pity）计算"用户还抽得起几次"时，用的成本字段和抽卡扣费用的是同一个。不存在"保底认为用户还有钱，实际扣费时发现没钱"的情况。内部把这种 bug 叫"幽灵保底"——保底触发了，但奖品发不出来，用户看到"恭喜中奖"然后什么都没收到。

Epic Games/Fortnite：物品商店的"你能买得起"标签和实际购买扣费读同一个 `price` 字段。从未出现"标签说买得起、点击购买时提示余额不足"的问题。

Steam 市场：物品列表的"可购买"标记和实际交易扣费读同一个 `listing_price`。Valve 的规范是："任何显示给用户的价格必须等于实际扣费价格"。

**活动策划行业**

互动吧/活动行的抽奖系统：奖品成本就一个字段，所有逻辑都读它。没有"分层标记"和"实际成本"分离的设计，因为它们没有你这种复杂的分层机制。

**小公司的教训**

典型 bug 场景：保底模块用字段 A 判断"用户还有预算"，发奖模块用字段 B 扣费。A=1（pvp，很小），B=30（实际成本）。结果：用户预算只剩 5，保底模块认为 5 ≥ 1 → 触发保底 → 分配碎片×3 → 扣费时发现 5 < 30 → 事务回滚 → 用户看到"中奖"动画但什么都没拿到。这就是"成本口径不一致"的典型后果。

**虚拟物品交易平台**

Steam/CSGO 开箱：从未出现"开箱界面显示买得起，点击开箱时提示余额不足"的问题。所有判断统一读 `key_price`。

### 抽奖 API 返回数据变化

**API**：`POST /api/v4/lottery/draw`

**🔒 安全决策（2026-03-06 拍板）**：所有敏感字段一律不对微信小程序普通用户开放。

> 路由层（`routes/v4/lottery/draw.js`）对抽奖结果进行了脱敏处理，以下字段**不在响应中**：
> - `budget_cost`：暴露内部预算成本结构
> - `material_asset_code`：暴露内部资产编码体系
> - `material_amount`：暴露虚拟物品发放明细
> - `prize_value_points`：后端分层阈值标记
> - `is_fallback`：后端保底标记
>
> 这些字段仅保留在引擎内部（settle_result）用于监控、日志和管理后台。

**实际返回字段**：
