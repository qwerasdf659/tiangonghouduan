# 消费审核「同一二维码 5 分钟内无法再次提交」机制说明与优化方案（模式①：一码一用 + 提交后自动刷新）

> 文档类型：后端数据库项目机制说明 + 三端对接方案（决策权在后端，前端配合）
> 日期：2026-06-25（北京时间）
> 归属：后端数据库项目（机制权威）；微信小程序前端（落地改造）；Web 管理后台前端（无需改）
> 核查方式：直读后端真实代码（`utils/QRCodeValidator.js`、`routes/v4/shop/consumption/submit.js`、`routes/v4/shop/consumption/qrcode.js`、`routes/v4/user/consumption-qrcode.js`、`services/consumption/QueryService.js`、`services/consumption/CoreService.js`）+ 直连真实库 `restaurant_points_dev` 机制核对，未引用历史报告
> 一句话结论：**这不是 bug，是 V2 动态身份码「一码一用（nonce 一次性）」的安全设计。推荐模式①——nonce 机制完全不动，由小程序在提交成功后自动刷新出新码，每笔消费各用一张新码。后端零改动，安全性零削弱，仅需小程序前端配合。**

---

## 一、现象

管理员（商家端）扫描用户微信小程序的身份二维码并提交审核后，**5 分钟内拿同一张二维码再次提交，会被拒绝**（提示"二维码已使用，请刷新后重试"或"二维码已过期"）；等约 5 分钟后用户刷新出新二维码，即可再次提交。

## 二、根因（直读真实代码，非推测）

提交审核（`POST /api/v4/shop/consumption/submit`）对二维码有两道关卡，**第 2 道是直接原因**：

### 2.1 二维码 5 分钟过期（`exp`）

用户端生成二维码时写入 `exp = 生成时刻 + 5 分钟`：

```80:81:utils/QRCodeValidator.js
    // 默认有效期（毫秒）：5分钟
    this.default_expiry_ms = 5 * 60 * 1000
```

提交时校验 `now > exp` 即拒绝：

```315:324:utils/QRCodeValidator.js
      // 9. 验证过期时间
      const now = Date.now()
      if (now > payload.exp) {
        const expired_ago = Math.floor((now - payload.exp) / 1000)
        logger.warn(`⏰ [QRCodeValidator] 二维码已过期: ${expired_ago}秒前`)
        return {
          valid: false,
          error: '二维码已过期，请刷新后重试',
          error_code: 'QRCODE_EXPIRED'
        }
      }
```

### 2.2 一次性 nonce（核心原因）

每张二维码携带一个随机 `nonce`。**提交审核**时通过 Redis `SET NX EX` 原子操作把该 nonce 标记为"已用"，同一 nonce 第二次提交即失败：

```449:466:utils/QRCodeValidator.js
      const nonce_key = `consumption:nonce:${payload.nonce}`
      const remaining_ms = payload.exp - Date.now()
      const ttl_seconds = Math.max(Math.ceil(remaining_ms / 1000) + 10, 60) // 至少60秒

      // SET NX EX 原子操作：只有 nonce 不存在时才能成功
      const set_result = await redis.set(nonce_key, '1', 'EX', ttl_seconds, 'NX')

      if (!set_result) {
        // nonce 已存在，检测到重放攻击
        logger.warn(
          `🚫 [QRCodeValidator] 检测到重放攻击: nonce=${payload.nonce.substring(0, 16)}..., user_uuid=${payload.user_uuid.substring(0, 8)}...`
        )
        return {
          valid: false,
          error: '二维码已使用，请刷新后重试',
          error_code: 'REPLAY_DETECTED',
          http_status: 409 // Conflict
        }
      }
```

- Redis key `consumption:nonce:{nonce}` 的 TTL = 二维码剩余有效期 + 10 秒（`:451`），所以约 5 分钟后该 nonce 自动从 Redis 清除——这就是"等 5 分钟后又能提交"的原因（但此时旧码也已过期，实际仍需新码）。
- **刷新二维码会生成全新 nonce + 新 exp**，新 nonce 在 Redis 中不存在，`SET NX` 成功 → 立即可再次提交——这就是"刷新后无需等待即可再次提交"的原因。

商家扫码看用户信息（`GET /api/v4/shop/consumption/user-info`）走的是**预览校验，不消耗 nonce**：

### 2.4 这是有意的安全设计，不是 bug

代码注释记录了升级动机——防止一张码被反复拍照/重复提交刷消费：

```27:30:utils/QRCodeValidator.js

```

消费记录的业务唯一键 `business_id` 自带毫秒时间戳，天然区分同一用户的多笔消费：

> 结论：**卡住"同一用户多次提交"的不是数据库、不是 business_id、不是幂等键，而纯粹是二维码的一次性 nonce。** 因此正确的放宽方式不是去掉 nonce（那会让"一张码刷 9 次"复活），而是**让用户快速拿到新码**，每笔消费各用一张新码提交——即模式①。

## 四、推荐方案（模式①：一码一用 + 提交成功后自动刷新出新码）

### 4.1 设计

- **nonce / exp 机制完全不动**（防重放、防刷 N 次能力全部保留）。
- **微信小程序前端**在每次「提交审核成功」后（或商家端提示提交成功后），自动调用用户端刷新接口拿一张全新二维码并重新渲染，使下一笔消费用新 nonce 提交。
- 行业对标：微信支付付款码、美团核销码均为「动态码 + 用后即刷」，这是主流做法。

### 4.2 涉及的真实接口（均已存在，后端零改动）

| 端 | 接口 | 作用 | 是否消耗 nonce |
| --- | --- | --- | --- |
| 用户端（小程序） | `GET /api/v4/user/consumption/qrcode` | 生成/刷新当前用户动态二维码 | 否（仅生成） |
| 商家端 | `GET /api/v4/shop/consumption/user-info?qr_code=...&store_id=...` | 扫码预览用户信息 | 否（预览） |
| 商家端 | `POST /api/v4/shop/consumption/submit` | 提交消费审核 | **是（一次性消耗）** |

用户端刷新接口真实返回（`routes/v4/user/consumption-qrcode.js:74-88`）：

```jsonc

    "nonce": "a1b2c3...",            // 每次刷新都是全新 nonce
```

### 4.3 微信小程序前端改造清单（交前端同事，唯一改动方）

1. **自动刷新触发点**：在身份二维码展示页，监听到「本张码已被成功提交一笔消费」后，立即重新调用 `GET /api/v4/user/consumption/qrcode` 获取新码并刷新展示。
   - 触发方式二选一（前端按现有架构择优）：
     - (a) 轮询/WebSocket 收到"该用户有新消费记录待审核/已提交"信号后刷新；
     - (b) 简化版：二维码展示页加「定时刷新」（如每 4 分钟，早于 5 分钟过期）+「手动刷新」按钮，保证用户随时能出示有效新码。
   - 推荐 (b) 作为最小改动起步：定时刷新 + 手动刷新按钮，零后端依赖、体验稳定。
2. **过期前预刷新**：二维码展示页用 `data.expires_at` 做倒计时，临近过期（如剩 30 秒）自动刷新，避免商家扫到过期码。
3. **错误兜底**：商家端若收到 `QRCODE_EXPIRED` / `REPLAY_DETECTED`，提示"请让顾客刷新二维码"，顾客点刷新即可。

### 4.4 Web 管理后台前端 / 后端：无需改动

- 后端：nonce/exp/submit/刷新接口均已就绪，**零改动**。
- Web 管理后台：不涉及（扫码与提交在商家端/小程序）。

## 六、为什么不选其它模式（备查）

| 模式 | 做法 | 为何不推荐 |
| --- | --- | --- |
| ② 会话式身份码（一码 N 分钟可多次提交，靠 business_id 去重） | 放宽 nonce 一次性 | **直接削弱防"一张码刷 9 次"能力**，与 V2 升级初衷相悖；需改 nonce 逻辑 + 补新去重维度，改动与风险都大 |
| ③ 商家端一次扫码连续录多笔 | 扫码建身份会话 + 凭会话提交 | 体验最好（收银台场景），但需后端新增"身份会话 + 凭会话提交"接口，工程量最大，三端都要配合 |

> 模式①在「解决诉求（同一用户可连续多笔提交）+ 安全零削弱 + 后端零改动 + 最贴合现有 V2 动态码设计」上最优，长期维护成本最低。仅当业务确为"收银台一次扫码、连续开多单"的高频场景时，才升级到模式③。

## 七、五类公司视角（线下身份码/核销码怎么做，供决策参考）

| 公司类型 | 线下身份码/核销码设计 | 对本项目的启示 |
| --- | --- | --- |
| 大厂（微信支付/支付宝/美团） | **动态码 + 用后即刷**（付款码每分钟刷新、核销码一次性） | 模式①正是此主流：一码一用 + 自动刷新 |
| 中小公司/SaaS | 动态码 + 一次性 nonce，前端定时刷新 | 与本项目现状一致，补"自动刷新"即可 |
| 游戏公司 | 一次性兑换码/CDKey，用后失效 | 一次性是常态，复用靠重新发码 |
| 活动策划公司 | 短时效核销码，过期重领 | 过期重领=刷新，体验同模式① |
| 游戏虚拟物品/二手平台 | 交易凭证码一次性 + 可追溯 | 一次性保证不可重放，符合本项目风控诉求 |

> 结论：**模式①（动态码一码一用 + 提交后/过期前自动刷新）就是线下身份码/核销码的行业主流**，与本项目 V2 动态码（5 分钟过期 + 一次性 nonce）的设计完全契合，无需改后端、不引入新债。

### 拍板项 1：二维码刷新触发方式（定时刷新 / 提交成功推送刷新）

## 九、后端权威要求（小程序前端必须遵守，以后端为准）

> 本节是后端数据库项目对微信小程序前端的**强制约束**。一切以后端为准：后端二维码有效期是 **5 分钟**（`utils/QRCodeValidator.js:81 default_expiry_ms = 5*60*1000`），提交时 `now > exp` 即返回 `QRCODE_EXPIRED`（`:317`）。前端的刷新策略必须围绕这条硬约束设计。

### 9.1 核心要求：刷新周期必须**早于**有效期，绝不能"正好 5 分钟"

| 要求 | 后端依据 | 说明 |
| --- | --- | --- |
| **定时刷新周期 = 4 分钟**（不是 5 分钟、不能 ≥5 分钟） | 有效期 5 分钟 | 必须留出"商家看码→输金额→提交"的操作窗口；若 5 分钟才刷新，会与过期时机重合，出现"用户出示时还没过期、商家提交时已过期"的失败窗口 |
| **临近过期预刷新** | 后端返回 `expires_at` | 前端用 `data.expires_at` 倒计时，剩约 30 秒自动刷新，避免商家扫到即将过期的码 |
| **手动刷新按钮** | — | 兜底：用户可随时主动换新码 |
| **刷新即换新 nonce** | 后端每次生成新 nonce/exp | 必须调 `GET /api/v4/user/consumption/qrcode` 重新获取（不能本地复用旧 payload） |

> ⚠️ 重点：**"每 5 分钟刷新"是错的**。后端有效期就是 5 分钟，刷新必须比它早（4 分钟），否则边界处必然偶发"扫了却提交不了（QRCODE_EXPIRED）"。一切以后端 5 分钟有效期为基准倒推前端刷新节奏。

### 9.2 小程序前端自检清单（交前端同事逐条核对）

- [ ] 二维码展示页定时刷新周期 **= 4 分钟**（早于后端 5 分钟有效期，非 5 分钟、非更长）。
- [ ] 使用后端返回的 `data.expires_at` 做倒计时，**剩约 30 秒时自动预刷新**。
- [ ] 提供**手动刷新按钮**。
- [ ] 刷新调用的是 `GET /api/v4/user/consumption/qrcode`，每次取返回里的**全新 `qr_code` + `nonce`**，不复用旧码。
- [ ] 商家端收到 `QRCODE_EXPIRED`（400）/ `REPLAY_DETECTED`（409）时，提示"请让顾客刷新二维码后重试"并高亮刷新入口（拍板项 3）。
- [ ] 不依赖任何"提交结果推送"才刷新（拍板项 1：纯定时 + 预刷新即可，不引入实时推送）。

### 9.3 验收口径（以后端行为为准）

- 正常路径：用户出示 4 分钟内的新码 → 商家扫码预览（不消耗 nonce）→ 输金额提交（消耗 nonce）→ 成功；用户页已自动刷新出下一张新码 → 可立即提交下一笔。
- 边界验证：在码生成第 4 分钟后、第 5 分钟前提交应成功；超过 5 分钟提交应返回 `QRCODE_EXPIRED`（证明刷新周期 4 分钟是必要的）。
- 防重放验证：同一张码提交两次，第二次返回 `REPLAY_DETECTED`（409）——此为后端正确行为，前端引导刷新即可。

> 本节为后端对前端的权威要求，前端实现细节可自主，但**刷新周期早于 5 分钟有效期**这条不可违背。后端二维码机制保持不动、零改动。

### 10.3 ⚠️ 发现的真实后端 Bug：刷新接口 `validity` 字段恒为 `undefined`（必须后端修，不能让前端兜底）

这是本次实测**唯一发现的真实问题**，且**推翻了"后端零改动"的部分前提**：刷新二维码接口要想给前端提供倒计时所需的"有效期秒数"，后端当前是坏的。

根因（直读真实代码）：

- 路由 `routes/v4/user/consumption-qrcode.js:82` 把 `data.validity` 透传给前端：

```82:82:routes/v4/user/consumption-qrcode.js
        validity: qrCodeInfo.validity,
```

- 但 `generateQRCodeInfo()` 真实返回的是 `validity_seconds`，**根本没有 `validity` 字段**：

```505:517:utils/QRCodeValidator.js
    return {
      qr_code: result.qr_code,
      user_uuid: result.user_uuid,
      nonce: result.nonce,
      exp: result.exp,
      expires_at: result.expires_at,
      validity_seconds: result.validity_seconds,
      generated_at: result.generated_at,
      algorithm: this.algorithm,
      version: 'v2',
      note: result.note,
      usage: '商家扫描此二维码后可录入消费金额'
    }
```

> 后果：前端实际拿到的刷新响应里 `validity` 永远是 `undefined`。文档 §4.2 示例里写的 `"validity": "..."` 与真实返回不符，是个"看起来有、实则没有"的字段。这违背了 §9.2 自检清单里"用 `expires_at` 倒计时"的隐含前提——好在 `expires_at` 是真实存在的（`:80`），倒计时仍可用，但 `validity` 字段是死的。

**修复方案（以后端为准、长期债最小，二选一，推荐 A）：**

- **方案 A（推荐，最小且语义最清晰）**：删掉路由层那行坏的 `validity`，改为透传后端真实已有的 `validity_seconds`，前端用 `validity_seconds` 直接做倒计时秒数（数值型，比字符串更好用）。后端改 1 行 + `generateQRCodeInfo` 返回里已有 `validity_seconds`，前端直接消费同名字段、零映射。
- **方案 B**：在 `generateQRCodeInfo()` 里补一个 `validity` 文案字段（如 `'5分钟'`）。但这是给前端"造文案"，把展示逻辑塞进后端，长期看是债，不推荐。

> 结论：按"以后端为准、字段名前端直用"，采用**方案 A**——后端暴露数值型 `validity_seconds`（语义准确、可直接计算），小程序前端用 `data.validity_seconds` 与 `data.expires_at` 配合做 4 分钟定时刷新 + 临近过期预刷新，**不做任何字段映射**。这条修复需要后端动 1 行代码，因此前八节"后端零改动"的表述应修正为"后端仅需修复此 `validity` bug（1 行），其余零改动"。
