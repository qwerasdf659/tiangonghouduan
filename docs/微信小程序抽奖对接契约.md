# 微信小程序抽奖对接契约（合规改造后）

> **性质**：后端→前端 API 对接契约（供微信小程序前端开发对照实现）
> **创建**：2026-06-05 北京时间
> **后端真相源**：`restaurant_points_dev`（dbconn.sealosbja.site:42569）+ 后端实际路由代码
> **背景**：抽奖 per-user 暗箱干预（强制中奖/强制不中/概率调整/用户队列）已整体下线（合规）。本文件描述对小程序生效的契约变化与字段口径。

---

## 一、最重要的两条变化（前端必读）

### 1. 奖品标识字段：对外统一为 `id`

后端内部主键叫 `lottery_campaign_prize_id`，但**发给小程序的所有奖品对象一律用通用字段 `id`**（防抓包暴露内部表结构，符合数据脱敏规范）。

- ❌ 不要再读 `prize.lottery_prize_id`（旧假名，已废弃）
- ❌ 不要读 `prize.lottery_campaign_prize_id`（内部字段，脱敏层已删除，不会下发）
- ✅ 统一读 `prize.id`

```js
// 解构注意：后端发的是 id
const { id, prize_name, prize_type, prize_value } = prize  // ✅
// const { lottery_prize_id } = prize                       // ❌ undefined
```

### 2. 抽奖结果只有"中奖档位"，没有"中没中"

每次抽奖 **100% 从奖品池获得一个奖品**（只是价值不同），用 `reward_tier`（low/mid/high/fallback）区分价值档位，**不存在"未中奖"语义**。前端不要写"很遗憾未中奖"分支。

## 二、对接 API 一览（对小程序生效）

> 统一响应格式：`{ success, code, message, data, timestamp, version, request_id }`。前端只读 `data`。错误时 `success=false`，按 `code`/`message` 提示。

### 2.1 查看活动奖品列表

```
GET /api/v4/lottery/campaigns/:code/prizes
```

`data` 为奖品数组，每个奖品对象（普通用户脱敏后）字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | number | 奖品ID（对外通用标识，抽奖/展示用它） |
| `prize_name` | string | 奖品名称 |
| `prize_type` | string | 类型：points/coupon/physical/virtual/service/product/special |
| `prize_value` | number | 展示价值（已转数字） |
| `rarity_code` | string | 稀有度代码（common 等，前端按此显示颜色光效） |
| `image` | object | `{ url, thumbnail_url, ... }` 奖品图片 |
| `sort_order` | number | 九宫格/转盘位置编号 |

> ⚠️ 脱敏层已删除以下敏感字段，小程序拿不到也不应依赖：`win_probability`（中奖概率）、`win_weight`（权重）、`stock_quantity`（库存）、`cost_points`、`prize_value_points`、`max_daily_wins`、`reward_tier`（奖品定义级的档位）、`reserved_for_vip` 等商业机密。

### 2.2 执行抽奖

```
POST /api/v4/lottery/draws        （以后端实际路由为准，body 含活动码/抽奖次数/幂等键）
```

`data` 结构：

```jsonc
{
  "campaign_code": "xxx",
  "lottery_session_id": "...",
  "prizes": [                       // 单抽 1 个，连抽多个
    {
      "reward_tier": "mid",         // 价值档位 low/mid/high/fallback
      "reward_tier_text": "中档",
      "id": 9,                      // ✅ 奖品ID（通用字段）
      "prize_name": "8星玉石",
      "prize_type": "product",
      "prize_value": 50,
      "sort_order": 3,
      "rarity_code": "epic",
      "material_asset_code": "star_stone",
      "image": { "url": "...", "thumbnail_url": "..." }
    }
  ],
  "total_points_cost": 90,          // 实际消耗积分（折后）
  "original_cost": 100,             // 原价
  "discount": 0.9,                  // 折扣率（1.0=无折扣）
  "saved_points": 10,
  "remaining_balance": 1234,        // 剩余积分余额
  "draw_count": 1,
  "draw_type": "single"             // single / multi
}
```

### 2.3 抽奖历史 / 用户积分

```
GET /api/v4/lottery/history/...     抽奖记录（奖品对象同样用 id）
GET /api/v4/lottery/user-points/... 用户积分概况
```

记录里的奖品对象统一为 `{ id, name, type, value }` 口径。

## 三、被删除的能力（小程序如有引用需移除）

以下 per-user 暗箱干预**后端已删除**，小程序端如有任何直接/间接引用，请移除（这些本就不该出现在 C 端）：

- 强制中奖 / 强制不中 / 概率调整 / 用户专属队列
- 相关后端路径（`/console/lottery-management/force-win` 等）均为**管理后台**路径，C 端本不应调用。

**个人发奖**（给某用户补一个奖品）现统一走**客服补偿（cs_compensate）**流程：用户侧表现为"收到一笔客服补偿"，而非"抽奖中奖"。如小程序有"申请补偿/客服"入口，结果通过既有的库存/通知渠道下发，无需新增抽奖侧逻辑。

## 四、新增的能力：成长等级公示分级概率（可选展示）

为合规替代暗箱调赔率，新增**按成长等级的公示分级概率**机制：

- 用户有**成长等级**（由累计消费积分 `history_total_points` 实时派生，如 青铜/白银/黄金/钻石）。
- 不同成长等级在某活动下的**高档中奖率倍数**可由运营公示配置（如白银 1.3、黄金 1.6），受"高档 ≤ 8%"硬上限约束。
- 这是**公示规则**（明示、合规），不是暗箱。

**对小程序的影响**：
- 若产品希望展示"我的成长等级 / 升级享更高中奖率"，需要后端提供 C 端公示接口。**当前后端仅实现了管理后台侧的成长等级/倍数配置接口（`/console/lottery-management/growth-levels`、`/level-probability/:id`，需管理员权限），尚未开放 C 端公示只读接口。**
- ⚠️ **需你确认产品需求**：是否需要在小程序展示成长等级与各档公示倍数？如需要，请告知，我会评估在后端新增一个**面向 C 端的只读公示接口**（注意：对外是否暴露"倍数"等需走数据脱敏评估，避免泄露运营策略细节）。

## 五、字段命名约定（前后端一致）

- 后端→小程序数据统一 **snake_case**（如 `prize_name`、`reward_tier`、`remaining_balance`、`total_points_cost`）。
- 奖品主键对外是通用 `id`（唯一例外，出于防抓包脱敏）。
- 解构务必用后端真实字段名，避免 camelCase 误写导致 `undefined`。
