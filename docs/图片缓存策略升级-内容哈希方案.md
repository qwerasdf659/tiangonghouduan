# 图片缓存策略升级 — 内容哈希方案实施记录

> 日期：2026-05-24
> 范围：后端数据库项目全量图片 URL 生成逻辑
> 状态：已完成

---

## 一、问题背景

微信小程序"我的仓库"页面资产文字前面没有图标。排查发现：

1. `BackpackService` 和 `/api/v4/assets/balances` 路由之前不返回 `icon_url` / `image_url` 字段
2. 修复后图标 URL 正确返回，但小程序仍报 404 — 原因是客户端缓存了之前的 404 响应
3. 图片代理路由之前对 404 响应没有设置 `Cache-Control: no-store`，导致客户端缓存了错误响应

---

## 二、方案选型

### 业界方案对比

| 方案 | 代表公司 | 原理 | 长期维护成本 | 实时性 |
|------|----------|------|-------------|--------|
| **A. 内容哈希 URL** | 美团/阿里/腾讯 | URL 含文件哈希，文件变→URL变→自动失效 | ⭐最低 | 即时 |
| B. 版本号 URL | 小公司/独立开发 | `?v=2` 或 `?t=时间戳` | 低 | 即时 |
| C. ETag 验证 | 通用 | 每次请求带 If-None-Match 验证 | 中 | 每次回源 |
| D. 短 TTL | — | 缓存时间短，频繁回源 | 高 | 延迟 |
| E. CDN 主动清除 | 大公司 | 后端更新时调 CDN API 清除 | 高 | 即时 |
| F. 启动版本号 | 临时方案 | 服务重启生成新版本号 | 中 | 重启后 |

### 选定方案：A（内容哈希 URL）

理由：
- 项目 `media_files` 表已有 `content_hash` 字段（SHA-256）
- 项目架构已是"数据库存 object_key，API 层拼 URL"
- 项目未上线，可一次性改到位
- 无 CDN，不需要考虑 CDN 清除
- 微信小程序按完整 URL 做图片缓存 key
- 长期零维护：图片换了→hash 变→URL 变→前端自动拿新图

## 三、技术方案

### 3.1 URL 格式

```
图片存在 + 有 content_hash：
  https://domain/api/v4/images/{objectKey}?h={content_hash前8位}
  → Cache-Control: public, max-age=31536000, immutable（永久缓存）

图片存在 + 无 content_hash（降级）：
  https://domain/api/v4/images/{objectKey}?v={启动版本号}
  → Cache-Control: public, max-age=86400, must-revalidate（24小时缓存）

图片不存在（404）：
  → Cache-Control: no-store（禁止缓存）
```

### 3.2 工作原理

```
┌──────────────┐     上传新图片     ┌──────────────┐
│ 运营后台      │ ──────────────→  │ media_files  │
│ 更换图标      │                   │ content_hash │ ← SHA-256 自动计算
└──────────────┘                   │ = "a1b2c3d4"  │
                                   └──────┬───────┘
                                          │
                                          ▼
┌──────────────┐     API 返回       ┌──────────────────────────────────────┐
│ 客户端        │ ←───────────────  │ icon_url: .../icon.png?h=a1b2c3d4   │
│ (小程序/管理台)│                   └──────────────────────────────────────┘
└──────┬───────┘                            ↑ hash 变了 = URL 变了
       │                                    │ = 客户端认为是新图片
       │ 之前缓存的是 ?h=旧hash              │ = 重新请求
       │ 新 URL 是 ?h=新hash                 │
       │ 两个不同 URL → 不会命中旧缓存        │
       └────────────────────────────────────┘
```

### 3.3 与其他方案的区别

- 不需要 CDN 主动清除（没有 CDN）
- 不需要短 TTL 频繁回源（浪费带宽）
- 不需要前端手动清缓存（URL 自动变）
- 不需要重启服务来刷新版本号（hash 跟着文件走）

## 六、后续维护说明

### 6.1 运营更换图片时

无需任何额外操作。流程：
1. 运营通过管理后台上传新图片
2. 系统自动计算新文件的 `content_hash`
3. API 返回的 URL 自动带新的 `?h=` 参数
4. 客户端发现 URL 变了，重新请求新图片

### 6.2 新增 getImageUrl 调用时

开发规范：所有新增的 `getImageUrl()` 调用必须传入第二个参数 `contentHash`：

```javascript
// ✅ 正确：传入 content_hash
const url = getImageUrl(media.object_key, media.content_hash)

// ⚠️ 降级：没有 content_hash 时自动使用启动版本号
const url = getImageUrl(objectKey)
```
