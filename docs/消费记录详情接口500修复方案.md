## 一、问题确认（真实存在的后端 bug）

接口 `GET /api/v4/shop/consumption/detail/:id` **当前直接 500 崩溃**，任何消费记录详情都打不开。

### 实测证据

```
GET /api/v4/shop/consumption/detail/3058  （真实账号 13612227930）
→ HTTP 500
{
  "success": false,
  "code": "INTERNAL_SERVER_ERROR",
  "message": "Association with alias \"points_transaction\" does not exist on ConsumptionRecord"
}
```

### 根因（纯文本数据流图）

```
路由 routes/v4/shop/consumption/query.js  GET /detail/:id
   → getConsumptionDetailWithAuth(recordId, viewerId, isAdmin,
        { include_review_records:true, include_points_transaction:true })   ← 路由写死传 true
   → getConsumptionRecordDetail()：
        if (options.include_points_transaction)
            include.push({ association: 'points_transaction', required:false })   ← 引用了不存在的关联
   → ConsumptionRecord.findByPk(recordId, { include })
   ❌ ConsumptionRecord.associate() 只定义了 user / merchant / reviewer / store / review_records
      并无 points_transaction 关联
   → Sequelize 抛 "Association with alias points_transaction does not exist"
   → 500，整个详情接口不可用（连 store_name 等其它字段一并取不到）
```

> 说明：原「问题1：未 JOIN stores 返回 store_name」表述不准确——Service 其实已 JOIN `store` 且 `toAPIResponse` 已输出 `store_name`（直连 Service 实测返回门店"是是是"）。真正的故障是 `points_transaction` 不存在的关联导致**整个接口 500**，所以 store_name 也一起取不到。

## 二、关联键核实（避免再次引入错误关联）

| 项目 | 真实值（已实测） |
| --- | --- |
| 源模型 | `ConsumptionRecord`（表 `consumption_records`） |
| 源外键字段 | `reward_transaction_id`（BIGINT，允许 NULL；注释「关联奖励积分流水ID（逻辑外键，用于对账，审核通过后填充）」；pending/拒绝时为 NULL） |
| 目标模型 | `AssetTransaction`（表 `asset_transactions`，modelName 实测 `AssetTransaction`） |
| 目标主键 | `asset_transaction_id`（BIGINT，PK） |
| 关联类型 | `belongsTo`（一条消费记录对应一条奖励积分流水） |

### 闭环验证（真实数据）

```
consumption_records.reward_transaction_id = 11047
        │  reward_transaction_id → asset_transactions.asset_transaction_id
        ▼
asset_transactions(11047)：asset_code='points', delta_amount=20, business_type='consumption_reward'
```

即 `reward_transaction_id` 就是奖励积分流水主键，关联关系真实成立、键明确。

## 三、方案B 实施方案（只描述，不含落地代码）

### 3.1 在 `models/ConsumptionRecord.js` 的 `associate()` 补关联

新增一个 `belongsTo(AssetTransaction)` 关联，别名 **`points_transaction`**（与 `getConsumptionRecordDetail` 里 `include` 用的别名对齐）：

- 关系：`ConsumptionRecord.belongsTo(models.AssetTransaction, { foreignKey: 'reward_transaction_id', targetKey: 'asset_transaction_id', as: 'points_transaction' })`
- `required:false`（左连接）：因为 pending/拒绝记录 `reward_transaction_id` 为 NULL，必须允许为空，否则这些记录会被 INNER JOIN 过滤掉。
- 这是**纯应用层 ORM 关联**（逻辑外键），与「材料资产互锁 / 物品三表互锁」无关，不触碰余额/持有/锁定，安全。

### 3.2 `getConsumptionRecordDetail` 的 include 无需改

现有代码 `include.push({ association: 'points_transaction', required:false })` 在关联补好后即可正常工作；别名已一致，**不需要改 Service**。

### 3.3 详情响应里如何输出积分流水（需配套确认）

补关联后 `record.points_transaction` 可加载，但 `toAPIResponse()` 目前**没有**把它拼进响应。需在 `toAPIResponse()` 末尾按现有 store/reviewer 的同款写法追加（仅当 `this.points_transaction` 存在时），例如输出：
- `reward_points`（= `points_transaction.delta_amount`，奖励到账积分数）
- `reward_transaction_no`（= `points_transaction.transaction_no`，流水单号，便于对账展示）

> ⚠️ 数据敏感性提醒：消费详情接口面向「用户本人 / 商家 / 管理员」。给微信小程序用户端展示积分流水时，**只下发 `delta_amount`（到账积分）/ `transaction_no`（单号）等非敏感字段**，不要把 `balance_after`/`balance_before`（账户余额快照）、`counterpart_account_id` 等内部记账字段透传给小程序前端（防抓包泄露）。web 管理端可不限。具体输出哪些字段请你拍板。

### 3.4 字段命名

遵循项目规范：模型类 PascalCase（`AssetTransaction`）、别名 snake_case（`points_transaction`）、响应字段 snake_case（`reward_points` / `reward_transaction_no`），前端零映射直读。
