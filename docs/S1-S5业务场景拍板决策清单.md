# S1–S5 业务场景拍板决策清单

> 版本：v1.0　|　日期：2026-07-06（北京时间）
> 配套文档：`docs/商品编码体系设计方案.md` §13/§14/§17
> 用途：S1–S5 五个后续业务场景（采购单/批次/组合/分销/寄卖）**开工前需你拍板的全部设计决策**。
> 现状：10 张表 + 外键 + 索引 + 模型已建好（实连真实库确认），本清单定完即可按 §17 分期施工。
>
> **用法**：每条带【建议】是我基于你现有技术栈+业务给的默认解，你在每条后面写「按建议」或你的选择即可。
> **标记说明**：⚠️ = 硬阻断项（不定则对应期无法开工）；其余为一般设计项（可先用建议、后续微调）。

## 一、S1 采购单 / 进货管理（第 1 期）

| # | 决策点 | 我的建议 | 你的决定 |
|---|---|---|---|
| 1 | 采购单号 `order_no` 怎么来 | 系统自动生成 `PO+日期+序号`（如 `PO20260706001`），复用现网 `TrackingCodeGenerator` 范式，不手填 | |
| 2 | 是否要审批环节 | 本期不做审批，草稿→下单→到货直接走 | |
| 3 | ⚠️ 收货是否自动生成实物 `items` | 收货只记采购数量、**不自动铸造** `items` 实物（实物仍走现有商品/发货链路），避免两套库存打架 | |
| 4 | 进货价 `purchase_price` 单位 | DECIMAL(10,2) 按「元」存（如 12.50） | |
| 5 | 进货价回写 `exchange_item_suppliers.purchase_price` 规则 | 回写「最近一次进货价」（简单直观），不做加权平均成本 | |
| 6 | 供货质量评分 `quality_score`（0.0~10.0）谁打、何时打 | 本期先留空不启用，S1 稳定后再做人工评分入口 | |
| 7 | 是否支持部分收货（一单多次到货） | 本期不支持，一次性全收 | |
| 8 | 采购退货/退单 | 本期不做，只做正向进货 | |
| 9 | `received`（已收货）后单据锁定程度 | 收货后行不可改，只能整单作废（cancelled） | |

## 二、S2 批次管理 / 一批一码（第 2 期）

| # | 决策点 | 我的建议 | 你的决定 |
|---|---|---|---|
| 10 | 批次码 `batch_code` 怎么来 | 系统自动 `BC+日期+序号`，运营可读，非随机码；不允许手填 | |
| 11 | ⚠️ 批次挂 SPU 还是 SKU | 允许两级：整款进货挂 SPU、分规格进货挂 SKU（表字段两者都可空） | |
| 12 | 批次成本 `batch_cost` 来源 | 有 S1 时从采购单自动带入；无 S1 时手填 | |
| 13 | 一个实物 `items` 只属一个批次（`batch_id` 单值） | 单值够用，无需一物多批 | |
| 14 | ⚠️ 批次召回是什么动作 | 召回=把该批次实物标记状态（可追溯/暂停流通），不物理删除 | |
| 15 | 防伪连号 `anti_fake_code` 本期做不做 | 本期不做（§3.7 预留字段留空），验真是独立子系统 | |
| 16 | 批次 `status`（active/inactive）业务含义 | active=在流通、inactive=已售罄/停用 | |

## 三、S4 组合商品 / 套装 / 赠品（第 3 期）

| # | 决策点 | 我的建议 | 你的决定 |
|---|---|---|---|
| 17 | ⚠️ 组合品定价 | 组合品独立定价（运营给套装单独定价），不自动汇总子项 | |
| 18 | ⚠️ 下单时库存怎么扣 | 按 BOM 拆解扣子项库存，组合本身不备货 | |
| 19 | 赠品 `is_gift` 价格如何计 | 赠品计价为 0，只占库存不计价 | |
| 20 | 子项能否是另一个组合（嵌套） | 不允许嵌套，子项只能是普通 SPU/SKU（避免循环依赖） | |
| 21 | 组合类型 `bundle_type`（suit 套装 / gift 赠品搭售）够不够 | 两种够用；如需别的类型请补充 | |
| 22 | 履约方式 | 组合整单一起发货，不支持子项分开发 | |
| 23 | 子项下架/删除时组合怎么办 | 外键 SET NULL；子项失效则组合自动置为不可售并提示运营 | |

## 四、S5 第三方平台分销（淘宝/抖音，第 4 期）

## 五、S3 二手回流 / 寄卖 / 转赠（第 5 期，最后做）

| # | 决策点 | 我的建议 | 你的决定 |
|---|---|---|---|
| 30 | ⚠️ 风控/合规规则谁定、什么标准 | 硬前置：反向资产流（用户→平台）的方向等价/向下、真实销毁校验规则，须你或风控先定标准 | |
| 31 | 寄卖定价 `list_price` 谁定 | 寄卖人自报价 + 平台审核核价 | |
| 32 | ⚠️ 寄卖计价资产 `list_asset_code` 用什么 | 需你指定真实资产码（积分/某种星石/现金券） | |
| 33 | 平台是否抽佣、比例 | 本期先不抽佣 | |
| 34 | 回流后再上架 `relist_item_id` | 复用原 SPU 上架为二手（同款），不新建二手专属 SPU | |
| 35 | 转赠是否需审核、有无次数/额度限制 | 走现有 `item_ledger` 流转 + 幂等键；是否审核和限额待你定 | |
| 36 | 寄卖物来源限制 | 只能是平台发出的、带 `tracking_code` 的正品实物，杜绝外部来路不明物 | |
| 37 | 寄卖状态机 pending→listed 谁审核 | 需指定审核角色 | |
| 38 | 二手商品是否给 C 端单独标识/空间展示 | 打「二手」标签，混在现有空间，不建独立专区 | |
