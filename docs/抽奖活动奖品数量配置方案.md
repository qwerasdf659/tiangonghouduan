## 一、先搞懂三个概念

配置奖品前，你需要理解下面这些概念：

### 1. 三个档位

系统把所有奖品分成三个档位，每次用户抽奖时，系统先随机选中一个档位，再从该档位里随机选一个奖品发给用户。

| 档位 | 含义 | 你可以理解为 |
|---|---|---|
| 大奖档（high） | 最值钱的奖品 | 一等奖 |
| 中奖档（mid） | 中等价值的奖品 | 二等奖 |
| 小奖档（low） | 小奖和安慰奖 | 三等奖 + 参与奖 |

### 2. 权重 = 概率

系统用「权重」来控制概率，不用百分比。规则很简单：

- 每个档位的概率权重之和必须 = **1,000,000**
- 同一个档位内，所有奖品的权重之和也必须 = **1,000,000**

换算方法：**想要 X% 的概率，就填 X × 10,000 的权重**

| 你想要的概率 | 填多少权重 |
|---|---|
| 5% | 50,000 |
| 10% | 100,000 |
| 15% | 150,000 |
| 20% | 200,000 |
| 50% | 500,000 |
| 80% | 800,000 |

### 3. 格位布局

奖品在前端界面上怎么展示，取决于你选的玩法。**关键点：抽奖按钮占据正中间的位置，奖品排列在按钮周围一圈。**

**3×3 九宫格**（最常用）

9 个格子，中心 1 格是按钮，剩下 8 格放奖品：

```
┌────┬────┬────┐
│ ①  │ ②  │ ③  │
├────┼────┼────┤
│ ⑧  │抽奖│ ④  │
│    │按钮│    │
├────┼────┼────┤
│ ⑦  │ ⑥  │ ⑤  │
└────┴────┴────┘

总格数 = 3×3 = 9
按钮占 = 1 格（正中间）
奖品位 = 8 个
```

**4×4 宫格**

16 个格子，中心 2×2 共 4 格是按钮区，剩下 12 格放奖品：

```
┌────┬────┬────┬────┐
│ ①  │ ②  │ ③  │ ④  │
├────┼────┴────┼────┤
│ ⑫  │          │ ⑤  │
│    │ 抽奖按钮 │    │
│ ⑪  │ （2×2）  │ ⑥  │
├────┼────┬────┼────┤
│ ⑩  │ ⑨  │ ⑧  │ ⑦  │
└────┴────┴────┴────┘

总格数 = 4×4 = 16
按钮占 = 4 格（中心 2×2）
奖品位 = 12 个
```

**为什么是 8 和 12？** 因为只有这两种布局，按钮刚好能完美居中：
- 3×3 奇数格 → 正中间恰好 1 格 → 周围 8 格放奖品
- 4×4 偶数格 → 正中间没有单独 1 格，所以按钮用 2×2 占 4 格 → 周围 12 格放奖品

**转盘**

扇区数量自由，常见 6 扇区或 8 扇区，指针在圆心：

```
      ①
  ⑥      ②
     指针
  ⑤      ③
      ④

→ 可放 6 或 8 个奖品
```

### 4. 用户消费后会获得三样东西

用户在餐厅消费后，管理员审核通过时，系统会**同时**发放三样东西。理解这个流程才能正确设置可见价值和预算价值。

**以用户消费 100 元为例：**

```
用户到店消费 100 元
       │
       ▼ 管理员审核通过，同时发放 ↓↓↓
       │
       ├─→ ① 100 积分（POINTS）
       │      用途：付抽奖门票，每次抽奖扣 100 积分
       │      比例：消费金额 × 1 = 积分
       │      100元 → 100积分 → 够抽 1 次
       │
       ├─→ ② 22 预算积分（BUDGET_POINTS）
       │      用途：决定你能中多贵的奖品
       │      比例：消费金额 × 0.22 = 预算积分
       │      100元 → 22预算积分
       │
       └─→ ③ 100 钻石配额（DIAMOND_QUOTA）
              用途：限制你能赢多少钻石
              比例：消费金额 × 1 = 钻石配额
              100元 → 100配额
```

**三个比例速查：**

| 消费金额 | 积分（抽奖门票） | 预算积分（能中多贵） | 钻石配额（能赢多少钻石） |
|---|---|---|---|
| 50 元 | 50 积分（抽 0.5 次） | 11 预算 | 50 配额 |
| 100 元 | 100 积分（抽 1 次） | 22 预算 | 100 配额 |
| 200 元 | 200 积分（抽 2 次） | 44 预算 | 200 配额 |
| 500 元 | 500 积分（抽 5 次） | 110 预算 | 500 配额 |

### 5. 每个奖品有两个「价值」，一个给用户看，一个给系统用

这是设置奖品时最容易搞混的地方。每个奖品要填两个不同的价值字段：

| 字段 | 用户能看到吗 | 作用 | 怎么填 |
|---|---|---|---|
| **可见价值**（prize_value） | **能看到** | 显示在奖品卡片上，告诉用户这个奖品值多少 | 填奖品的实际价值，比如 158（元）、100（积分） |
| **预算价值**（prize_value_points） | **看不到** | 后台预算控制，决定用户的预算够不够「解锁」这个奖品 | 填一个控制门槛的数字 |

**它们是怎么配合工作的？**

```
奖品「生腌拼盘」的两个值：
  ┌─────────────────────────────────────────────────┐
  │  可见价值 = 158      ← 用户在界面上看到「价值158元」  │
  │  预算价值 = 40        ← 用户看不到这个数字            │
  │                                                    │
  │  系统逻辑：                                         │
  │  用户预算 ≥ 40 → 这个奖品出现在他的奖品池里          │
  │  用户预算 < 40 → 这个奖品不会出现，他抽不到           │
  └─────────────────────────────────────────────────┘
```

**举个完整的例子：**

```
用户小王消费了 200 元 → 获得 44 预算积分

他去抽奖时，系统在后台悄悄过滤：

  奖品        可见价值    预算价值    小王预算(44)够吗？
  ─────────   ────────   ────────   ──────────────
  鸳鸯锅套餐   388元       60        60 > 44  ❌ 抽不到
  八折券       -           30        30 ≤ 44  ✅ 能抽到
  虾滑1份      68元        15        15 ≤ 44  ✅ 能抽到
  鸭血1份      18元         5         5 ≤ 44  ✅ 能抽到
  好运加持      0元         0         0 ≤ 44  ✅ 能抽到

  → 小王这次在 八折券/虾滑/鸭血/好运加持 四个里面抽
  → 鸳鸯锅套餐对他「隐形」了，等他再消费攒够预算才会出现
```

**预算价值（prize_value_points）设多少合适？**

| 预算价值 | 用户需要累计消费多少才能「解锁」 |
|---|---|
| 10 | 约 46 元（10 ÷ 0.22） |
| 22 | 100 元 |
| 44 | 200 元 |
| 60 | 约 273 元 |
| 100 | 约 455 元 |
| 150 | 约 682 元 |

**设置建议：**

1. **大部分奖品的预算价值应该 ≤ 22**：这样消费 100 元的用户就能参与，不会觉得「都抽不到什么好东西」
2. **只有 1-2 个大奖设到 40-60**：作为消费 200-300 元用户的回馈
3. **不建议设超过 100**：意味着用户要消费 455 元以上才解锁，门槛太高
4. **空奖/微回馈设 0**：确保所有人都能抽到保底
5. **可见价值随便写多高都行**，比如套餐标价 388 元，只要预算价值控制合理就行

**根据客单价来设预算价值的参考表：**

| 你的餐厅客单价 | 大奖预算价值 | 中奖预算价值 | 小奖预算价值 | 空奖 |
|---|---|---|---|---|
| 50 元左右 | 20-30 | 10-15 | 5-8 | 0 |
| 100 元左右 | 30-50 | 15-25 | 8-12 | 0 |
| 200 元左右 | 50-80 | 25-40 | 10-20 | 0 |
| 300 元以上 | 80-120 | 40-60 | 15-30 | 0 |

## 二、第一次配置：照着做就行

### 第 1 步：选一个玩法

| 如果你… | 推荐选 | 要准备几个奖品 |
|---|---|---|
| 第一次配，想简单点 | 九宫格 grid_3x3 | 8 个 |
| 想放更多奖品 | 4×4 宫格 grid_4x4 | 12 个 |
| 想做转盘效果 | 转盘 wheel | 6 或 8 个 |
| 想做翻牌效果 | 卡牌翻转 card_flip | 8 个 |
| 拿不定主意 | **九宫格 grid_3x3** | **8 个** |

### 奖品类型速查：系统能发什么

在准备奖品清单之前，先搞清楚系统实际能发放哪几种奖品。模型里定义了 7 种类型，但**真正能用的只有 4 种**：

| 你要发什么 | 选哪个类型 | 中奖后系统做什么 | 举例 |
|---|---|---|---|
| 给积分 | **points** | 直接往用户积分账户加 `可见价值` 数量的积分 | 幸运5积分、100积分奖励 |
| 给优惠券 | **coupon** | 铸造一个 voucher 物品，进用户背包，到店出示核销码使用 | 八折券、满200减50 |
| 给实物/菜品 | **physical** | 铸造一个 product 物品，进用户背包，到店出示核销码领取 | 虾滑1份、鸭血1份 |
| 给钻石/材料/纯空奖 | **virtual** | 配了 material_asset_code → 发放对应材料（如钻石）；没配 → 什么都不发，纯空奖 | 钻石×1、好运加持 |

**以下 3 种类型不要用**（结算引擎没写处理逻辑，配了也发不出东西）：

| 类型 | 代码值 | 为什么不能用 |
|---|---|---|
| 服务 | service | 引擎未实现，中奖后不会发放任何东西 |
| 商品 | product | 引擎未实现（注意：和 physical 不同，physical 能用） |
| 特殊 | special | 引擎未实现 |

### 第 2 步：准备你的奖品清单

按下面五层结构准备奖品。**每一层的作用不同，缺一不可。**

---

**第一层：镇场大奖（放 1 个）**

这是全场最值钱的奖品。它的作用不是让很多人中，而是让所有人都「惦记着它」，愿意一次又一次来抽。

> 怎么选：问自己「这个奖品发到朋友圈，别人会不会羡慕？」
>
> 如果会 → 就是它了。

举例：

| 餐厅类型 | 可以放什么 |
|---|---|
| 火锅店 | 四人免单套餐 / 锅底全年免费卡 |
| 奶茶店 | 全年奶茶畅饮卡 / 联名周边礼盒 |
| 日料店 | 双人omakase体验 / 和牛拼盘 |
| 通用 | 500元储值卡 / iPhone手机壳+200元券 |

设置要点：
- 档位：大奖档（high）
- 稀有度：**legendary**（金色光效，全活动唯一一个金色）
- 库存：少一点（20-100），物以稀为贵
- 每日上限：1-3 个，不要太多

---

**第二层：惊喜奖（放 1-2 个）**

中了会开心、会跟朋友提一嘴的奖品。

举例：

| 类型 | 示例 |
|---|---|
| 优惠券 | 八折券、满200减50 |
| 实物 | 招牌甜品1份、特色凉菜1份 |
| 积分 | 500积分（可用于下次消费） |

设置要点：
- 档位：大奖档（high）或中奖档（mid）
- 稀有度：**epic**（紫色）或 **rare**（蓝色）
- 库存：适量（200-2000）

---

**第三层：实惠小奖（放 2-3 个）**

不算大奖但用户觉得「不亏」的奖品。这一层是留住用户的关键——大部分人中不了大奖，但如果经常能中个小实惠，他就会持续来抽。

举例：

| 类型 | 示例 |
|---|---|
| 小实物 | 米饭1碗、小菜1份、加料免费 |
| 小优惠 | 九五折券、满100减5 |
| 小积分 | 50-100积分 |

设置要点：
- 档位：中奖档（mid）或小奖档（low）
- 稀有度：**uncommon**（绿色）或 **common**（灰色）
- 库存：大方一点（2000-10000）

---

**第四层：微回馈（放 1-2 个）**

用户看到会觉得「虽然没中大的，但起码得了个东西」。这一层的关键作用是**避免用户觉得白花了积分**。

举例：

| 类型 | 示例 | 说明 |
|---|---|---|
| 钻石×1 | virtual 类型，配 material_asset_code=DIAMOND | 可交易、可投广告，比纯积分更有玩法（但需要用户有钻石配额） |
| 微积分 | points 类型，可见价值=5 | 所有用户都能拿到，无任何限制 |
| 微体验 | physical 类型 | 下次到店送纸巾1包等 |

设置要点：
- 档位：小奖档（low）
- 稀有度：common（灰色）
- 库存：可以设很多或不限
- 建议放 1 个钻石微回馈 + 1 个积分微回馈，让有配额和没配额的用户都有东西拿

---

**第五层：保底 fallback（放 1 个）**

这是系统的**最后一道安全网**。不管发生什么情况，用户抽奖都不能出现「系统错误，没选出奖品」的结果。fallback 就是保证这一点的兜底奖品。

**fallback 什么时候出场：**

| 场景 | 发生了什么 | fallback 的作用 |
|---|---|---|
| 用户预算为 0 | high/mid/low 所有有价值奖品全被过滤掉 | fallback 是唯一剩下的，保证用户拿到一个结果 |
| 所有奖品库存都见底 | high/mid/low 的奖品全缺货 | fallback 兜底 |
| 极端边界情况 | 某些配置异常导致三个档位都空了 | fallback 防止系统崩溃 |

代码里的逻辑是：**fallback 始终保留，确保 100% 中奖**——无论预算怎么过滤、库存怎么扣减，fallback 永远不会被移出奖品池。

**和普通 low 档奖品的区别：** low 档的奖品会被预算系统过滤（如果预算价值 > 0 且用户预算不够），但 fallback 的预算价值 = 0，永远不会被过滤。

**fallback 只能用 points 或 virtual（不配材料），不能用钻石。** 因为钻石奖品受 DIAMOND_QUOTA 配额检查，没配额的用户抽不到：

| 用户类型 | 有钻石配额吗 | 钻石做 fallback 会怎样 |
|---|---|---|
| 到店消费过的用户 | 有 | 正常 |
| 新注册送了初始积分的用户 | **没有** | fallback 被过滤，系统可能报错 |
| 管理员手动充积分的用户 | **没有** | 同上 |
| 配额刚好用完的用户 | **没有** | 同上 |

设置要点：
- 档位：小奖档（low）
- 类型：**points**（最安全，无任何限制）
- 可见价值（prize_value）：**5**（用户抽中后积分 +5，不会空手）
- 预算价值（prize_value_points）：**0**（满足系统强制约束，且永远不会被预算过滤）
- is_fallback：**必须设为 true**
- 稀有度：common（灰色）
- 名称：**幸运5积分**（正面词+具体数字）
- 名称别用：「谢谢参与」「下次好运」「未中奖」（让人不爽）

### 关于钻石奖品：额外要知道的事

你的系统里有一种特殊奖品——**钻石**。它和普通奖品不同，受「钻石配额」限制：

**钻石配额的规则：**

- 用户到店消费多少元，就获得多少钻石配额（1:1 关系）
- 消费 50 元 → 获得 50 钻石配额
- 消费 100 元 → 获得 100 钻石配额
- 抽中钻石奖品时，系统先检查用户的钻石配额够不够
- 配额不够 → 这个钻石奖品直接不会出现在用户的奖品池里

**这意味着什么？**

钻石奖品不是想设多大就多大。你需要考虑用户实际消费水平：

| 用户消费水平 | 钻石配额 | 钻石奖品该怎么设 |
|---|---|---|
| 人均消费 50 元左右 | 约 50 配额 | 钻石奖品设 5-10 个钻石最合适 |
| 人均消费 100 元左右 | 约 100 配额 | 钻石奖品可以设 10-20 个钻石 |
| 人均消费 200 元以上 | 约 200 配额 | 钻石奖品可以设 20-50 个钻石 |

**设置钻石奖品时的注意事项：**

1. **钻石数量不要超过人均消费的一半**：如果人均消费 50 元（配额 50），钻石奖品设 5-10 个比较合理。设 50 个意味着用户只要中一次就把配额用完了，之后所有钻石奖品对他「消失」
2. **可以设多个不同数量的钻石奖品**：比如「钻石×5」放小奖档、「钻石×20」放中奖档、「钻石×50」放大奖档，让不同消费水平的用户都有东西可抽
3. **钻石奖品天然有成本封顶**：用户最多赢走和消费额等量的钻石，不用担心亏损失控

**带钻石奖品的配置示例（九宫格 8 个奖品）：**

| 格位 | 奖品 | 档位 | 说明 |
|---|---|---|---|
| 1 | 钻石×30 | high | 大奖，需要配额≥30（消费≥30元的用户才看得到） |
| 2 | 八折优惠券 | high | 非钻石大奖，所有人都能抽到 |
| 3 | 钻石×10 | mid | 中等钻石奖，大部分用户配额够 |
| 4 | 甜品1份 | mid | 非钻石中奖 |
| 5 | 九五折券 | mid | 非钻石中奖 |
| 6 | 钻石×3 | low | 小钻石奖，几乎所有人配额都够 |
| 7 | 幸运5积分 | low | 微回馈 |
| 8 | 好运加持 | low | 保底空奖 |

这样配的好处：消费多的用户有机会抽到大钻石奖，消费少的用户抽不到大的但能抽到小钻石奖，不会出现「全场钻石奖品都对我隐形」的尴尬。

### 预算价值的隐藏陷阱：low 档不能全填 0

这是一个非常重要的技术细节。系统会根据你配的奖品的预算价值**自动计算**预算分层阈值，决定不同预算的用户能解锁哪些档位：

```
阈值计算规则：
  threshold_high = high 档奖品中预算价值最小的正数
  threshold_mid  = mid  档奖品中预算价值最小的正数
  threshold_low  = low  档奖品中预算价值最小的正数

然后系统保序：threshold_high ≥ threshold_mid ≥ threshold_low

用户预算 ≥ threshold_high → B3（全部档位解锁）
用户预算 ≥ threshold_mid  → B2（mid + low + fallback）
用户预算 ≥ threshold_low  → B1（low + fallback）
用户预算 < threshold_low  → B0（只有 fallback）
```

**如果 low 档所有奖品的预算价值都是 0，会出大问题：**

```
你这样配 low 档：
  钻石×1  预算价值=0
  幸运5积分 预算价值=0
  fallback 预算价值=0

系统计算：
  threshold_low = 没有正数 → 回退默认值 100
  保序后：threshold_high、threshold_mid 也被拉到 100

结果：用户需要 100 预算积分才能进 B1
  消费 100 元只有 22 预算 → 22 < 100 → B0
  → 所有人永远只能抽到 fallback！大奖中奖全废了！
```

**运营层面的规避方法：low 档非 fallback 的奖品，预算价值至少设 1（不是 0）。**

只有 fallback 那一个才填 0。这样阈值就能正确计算。

### 根治方案：三层代码修复（A + B + C）

上面的「至少填 1」是运营层面的规避手段。但根本问题出在代码里，需要三层修复彻底根治。三层各管一件事，缺一不可：

```
┌──────────────────────────────────────────────────────────────┐
│  三层修复关系图                                              │
│                                                              │
│  A 逻辑层 ─→ 全免费档位不该有门槛（null → 0，不是 100）     │
│       ↓                                                      │
│  B 防御层 ─→ 阈值只向下收敛，不向上污染                      │
│       ↓                                                      │
│  C 参数层 ─→ 默认值和经济体系匹配（跟着 ratio 走）          │
│                                                              │
│  优先级：A（必须）> B（必须）>> C（推荐）                    │
└──────────────────────────────────────────────────────────────┘
```

**方案 A：null 回退到 0，不是 100（修逻辑）**

| 项目 | 内容 |
|---|---|
| 问题 | `getMinPositiveCost` 在某档全是 0 值奖品时返回 null，代码用 `?? this.thresholds.low`（默认 100）兜底 |
| 修复 | `?? 0`（三个档位都改） |
| 文件 | `services/UnifiedLotteryEngine/compute/calculators/BudgetTierCalculator.js` 的 `_calculateDynamicThresholds` |
| 改动量 | 3 处，每处改 1 个数字 |

修复逻辑：如果一个档位里所有奖品都是免费的（预算价值全是 0），说明这个档位本来就不需要预算门槛，阈值应该是 0（任何人都能进），而不是 100（几乎没人能进）。

```
修复前：
  low 档全是 0 值 → getMinPositiveCost 返回 null → 回退 100
  用户需要 ≥100 预算才能进 B1 → 消费 455 元以上才行 ❌

修复后：
  low 档全是 0 值 → getMinPositiveCost 返回 null → 回退 0
  用户需要 ≥0 预算就能进 B1 → 所有人都能进 low 档 ✅
```

**方案 B：保序逻辑改成向下收敛（防扩散）**

| 项目 | 内容 |
|---|---|
| 问题 | 保序逻辑 `if (low > mid) { mid = low }`，把 mid 拉高了（向上污染） |
| 修复 | 改成 `if (low > mid) { low = mid }`，把 low 拉低（向下收敛） |
| 文件 | 同上 |
| 改动量 | 2 行赋值方向反转 |

修复逻辑：阈值的保序应该是「偏高的往下拉」而不是「偏低的往上推」。即使某个档位因为意外原因阈值偏高，也只影响它自己，不会感染其他档位。

```
修复前（向上污染）：
  threshold_low=100, threshold_mid=5
  → 100 > 5，所以 mid 被拉高到 100
  → mid 档也要 100 预算才能进 ❌

修复后（向下收敛）：
  threshold_low=100, threshold_mid=5
  → 100 > 5，所以 low 被拉低到 5
  → low 档只要 5 预算就能进 ✅
  → mid 档不受影响，还是 5
```

**方案 C：默认值跟着 budget_allocation_ratio 走（防参数脱节）**

| 项目 | 内容 |
|---|---|
| 问题 | 默认阈值 100/500/1000 是硬编码的，和实际的 0.22 系数完全脱节 |
| 修复 | 构造函数读取 `budget_allocation_ratio`，动态计算合理的默认值 |
| 文件 | 同上，构造函数部分 |
| 改动量 | 中等（需要读取系统配置） |

修复逻辑：默认值应该和经济体系匹配。当 ratio = 0.22 时，合理的默认值应该是：

```
修复前（硬编码，和 0.22 脱节）：
  threshold_low 默认 = 100  → 需要消费 455 元
  threshold_mid 默认 = 500  → 需要消费 2273 元
  threshold_high 默认 = 1000 → 需要消费 4545 元
  （对一家客单价 100 元的店，这些数字完全不现实）

修复后（动态计算，跟着 ratio 走）：
  threshold_low 默认 = round(100 × ratio × 1) = 22   → 对应消费 100 元
  threshold_mid 默认 = round(100 × ratio × 2) = 44   → 对应消费 200 元
  threshold_high 默认 = round(100 × ratio × 5) = 110 → 对应消费 500 元
  （如果 ratio 从 0.22 改成 0.5，默认值自动跟着变，不用改代码）
```

**三层协作关系：**

| 场景 | 谁生效 | 结果 |
|---|---|---|
| 某档有正价奖品 | 都不需要（阈值由实际奖品数据决定） | 正常 |
| 某档全是 0 值奖品 | **A 生效**：阈值 = 0 | 任何人都能进该档 |
| 某档没配任何奖品 | **C 生效**：回退到动态默认值 | 默认值和经济体系匹配 |
| 某档阈值意外偏高 | **B 生效**：只向下收敛 | 不会污染其他档位 |
| 上述场景组合 | 三层依次兜底 | 全部覆盖 |

**修复后，运营配奖品时 low 档填 0 还是 1 都不会出问题。** 但在代码修复落地之前，仍建议遵守「low 档非 fallback 至少填 1」的规则作为安全保障。

### 配套清理：删除 .env 中的静态门槛配置（D7 已确认）

三层修复落地后，`.env` 中以下三行应当删除：

```
BUDGET_TIER_THRESHOLD_HIGH=1000    ← 删除
BUDGET_TIER_THRESHOLD_MID=500      ← 删除
BUDGET_TIER_THRESHOLD_LOW=100      ← 删除
```

同时清除 `BudgetTierCalculator` 构造函数中读取这三个值的代码。

**为什么删：** 这三个值在正常运转时不生效（门槛由奖品数据动态算出），只在动态计算失败时当备胎登场——而三层修复的目的就是消灭所有「算不出来」的场景。修复后它们没有任何代码路径会读取，留着只会误导维护者。

**行业做法对比：**

| 做法 | 代表 | 门槛谁定 | 有没有 .env 静态门槛 | 维护成本 |
|---|---|---|---|---|
| 全手动 | 游戏公司（原神、王者） | 数值策划逐个填写 | 没有（都在配置表） | 高：每次改奖品要同步改门槛 |
| **全自动** | **美团、拼多多、阿里** | **系统从奖品数据推导** | **没有** | **低：改奖品就行，门槛自动变** |
| 混合兜底 | 中小公司小程序 | 系统算，算不出用默认值 | 有（兜底用） | 中：兜底值容易和奖品脱节，成为 bug 源 |

本项目选择**全自动**（和大厂一致），理由：

1. `BudgetTierCalculator` 的动态计算逻辑已经写好，三层修复补上边界后就是完整的全自动方案
2. 没有专职数值策划，全手动维护门槛迟早出错
3. 项目未上线，删除不影响任何线上用户
4. ratio 已在 `system_settings` 表中，奖品数据在 `lottery_prizes` 表中，不需要引入新依赖
5. 运营只管配奖品，门槛永远自动对齐，不存在「配置脱节」的技术债

## 五、奖品命名怎么起

名字直接影响用户看到结果时的心情。

### 好名字

| 示例 | 好在哪 |
|---|---|
| 四人鸳鸯锅套餐 | 一看就知道值多少，有画面感 |
| 八折优惠券 | 折扣明确 |
| 招牌虾滑1份 | 具体菜品，有食欲 |
| 幸运5积分 | 虽然少但「幸运」是正面词 |
| 好运加持 | 中性偏积极，不说「没中」 |

### 坏名字

| 示例 | 问题 |
|---|---|
| 谢谢参与 | 直接告诉用户「你没中」，很扫兴 |
| 下次好运 | 暗示这次运气不好 |
| 神秘彩蛋（但价值是0） | 「神秘」制造期待，结果是空的，落差更大 |
| 特等奖 / 一等奖 | 太generic，没记忆点 |

### 起名公式

- 有价值的奖品 → **写清楚是什么、值多少**：八折券、虾滑1份、200积分
- 微回馈奖品 → **正面词 + 具体数字**：幸运5积分、好运+10积分
- 空奖 → **中性积极词，不提「没中」**：好运加持、幸运祝福、福气满满

## 附录：行业调研与决策依据

以下是 D4/D5/D6 三项拍板决定的行业调研依据。

### A. 空奖数量：行业怎么做的

| 类型 | 代表 | 纯空奖数量 | 空手概率 | 核心逻辑 |
|---|---|---|---|---|
| 大厂抽奖 | 美团天天领红包、淘宝双11、拼多多转盘 | 0 个 | 0% | 最差也给 0.01 元红包/几个金币，绝不让用户空手 |
| 头部手游 | 原神祈愿、崩铁跃迁、王者荣耀夺宝 | 0 个 | 0% | 每次必出物品（最差 3 星武器/碎片），没有「空」的概念 |
| 中小手游 | 各种卡牌/MMO 抽卡 | 0 个 | 0% | 跟头部一样，最差出 N 碎片/铜币 |
| 游戏道具交易 | Steam 开箱、CS2 武器箱 | 0 个 | 0% | 每个箱子必出一个皮肤，只是稀有度不同 |
| 小餐饮/零售 | 各种小程序转盘 | 3-6 个 | 40-70% | 成本考虑，但这恰恰是用户流失最快的做法 |
| 活动策划公司 | 线下大转盘/抽奖箱 | 0-1 个 | 10-20% | 专业策划一般保证 80%+ 中奖率，最差给个小礼品 |

**行业共识：越成功的产品，空奖越少。** 原因不是有钱任性，而是算过账——一个用户流失的成本远大于一份微奖品的成本。

**为什么我们选 1 个且给 5 积分：**

- 本项目已有 BUDGET_POINTS 预算系统控制成本，用户预算不够时自动只能抽到低价值奖品。空奖作为成本控制手段是多余的，预算系统已经干了这件事。
- 代码强制校验（`validateEmptyPrizeConstraint`）要求至少 1 个 `prize_value_points=0` 的奖品，所以最少保留 1 个。
- 这 1 个保底奖品的预算价值设 0（满足系统约束），但可见价值设 5 积分、类型选 points、名字叫「幸运5积分」。用户感知上是「得了 5 积分」而非「空手」，对预算系统来说它是零成本。

### B. 连续空奖保护阈值：行业怎么做的

| 类型 | 代表 | 保底机制 | 阈值 |
|---|---|---|---|
| 原神/崩铁 | 5星保底 | 硬保底 90 抽必出，73 抽起「软保底」概率逐步提升 | 73-90 |
| 原神/崩铁 | 4星保底 | 10 抽必出 | 10 |
| 王者荣耀 | 夺宝 | 积分保底，361 次必得 | 361 |
| 美团/饿了么 | 红包抽奖 | 通常 3-5 次内必给一个有感知的券 | 3-5 |
| 拼多多 | 砍价/转盘 | 几乎每次都给东西，没有连续空奖的概念 | 不需要 |
| 线下活动策划 | 抽奖箱 | 一般不做保底，靠奖品比例控制 | 无 |

**关键区别：** 游戏里保底阈值高（10-90）是因为每次抽奖都出物品、只是稀有度低。它保底的是「高稀有度」而不是「非空」。本项目的 ANTI_EMPTY_THRESHOLD 保底的是「非空」，逻辑不同。

**为什么我们选 5：**

- 按 D4 决定（空奖只有 1 个且给微积分），真正的「空手体验」已经几乎不存在，这个阈值变成了保底「中档以上奖品」的安全网。
- 设太低（3）会导致中档奖品发放量明显增加，抬高成本。
- 设太高（8）用户遇不到这个机制，等于白设。
- 5 是安全网的平衡点：大部分用户永远不会触发，但极端倒霉时能兜住。

### C. 新用户大奖概率：行业怎么做的

| 类型 | 代表 | 新用户 vs 老用户 | 倍率 |
|---|---|---|---|
| 拼多多 | 新人专享红包 | 新人中大额概率极高，老用户基本抽不到 | 5-10x |
| 美团 | 新人首单红包 | 新用户首单必给大额券（8-15元） | 3-5x |
| 原神 | 新手祈愿池 | 50 抽保底 5 星（常规池 90 抽），打了 8 折 | 1.8x |
| 大部分手游 | 新手十连 | 首次十连保底 SSR | 2-3x |
| 淘宝/京东 | 新人专区 | 大额券 + 免邮 + 专属价 | 2-3x |
| 小程序抽奖 | 中小商家 | 大部分不区分新老用户（错失机会） | 1x |
| 游戏道具交易 | Steam/二手平台 | 不区分 | 1x |

**行业共识：** 几乎所有用户增长型产品都给新用户更好的概率。区别只在于倍率：保守 1.5x，标准 2x，激进 3-5x（拼多多模式）。

**为什么我们选 10%（2x）：**

- 这是行业标准倍率，不激进不保守。
- 本项目是餐厅小程序，用户来源靠到店消费，获客成本已经通过线下场景分摊。新用户首次体验好 → 下次还来消费 → 正循环。
- 预算系统已兜底成本：新用户预算低，就算大奖概率翻倍，也只能中预算价值低的奖品，实际发放成本有限。
- 数据库中已有此配置（new_user segment：high=100,000），无需改动。

#### D11：fallback 数量规则——严格 1 个还是允许多个

**行业深度调研：保底/兜底机制的设计**

| 公司/类型 | 保底机制 | 保底奖品几个 | 为什么这么设计 |
|---|---|---|---|
| **原神/崩铁（米哈游）** | 每次祈愿必出 1 个物品，最低 3 星武器/圣遗物。3 星是一个"类别"，不是具体某个固定物品 | **1 种保底类别**，但该类别下有多个物品随机 | 保底负责兜底，具体给哪个由该类别内的权重决定 |
| **王者荣耀夺宝** | 每次夺宝必得 1 个碎片/皮肤碎片 | **1 种保底机制**（碎片） | 碎片可累积合成，不是无价值的空奖 |
| **美团天天领红包** | 每次转盘必得金额（最低 0.01 元） | **1 个最低金额兜底** | 金额由系统动态决定，不是多个"保底奖品"并存 |
| **拼多多转盘** | 每次必得优惠券/金币/现金 | **1 种保底**（最低现金/金币） | 同上 |
| **CS2 开箱/Steam** | 每个箱子必出 1 个皮肤，最低军规级（Mil-Spec） | **1 个最低品质保底** | 开箱不存在"空"的概念 |
| **线下大转盘** | 通常设"谢谢参与"格子 | 1-2 个 | 线下无法动态调整，多设 1 个是物理限制（格子够多） |
| **中小小程序** | 经常设 3-6 个"谢谢参与"/"下次好运" | 3-6 个 | 想靠空奖比例控成本——这恰恰是**用户流失最快的做法** |

**多 fallback vs 单 fallback 的技术分析**：

| 维度 | 多个 fallback（当前状态：5 个） | 严格 1 个 fallback |
|---|---|---|
| **权重分配** | 5 个 fallback 分走 low 档 630,000/1,000,000 权重（63%），正常 low 奖品只剩 37% | 1 个 fallback 按需分配权重，正常奖品有更多空间 |
| **运营语义** | 运营要管 5 个名字不同但功能完全一样的奖品（"好运加持""美食推荐""厨师祝福"…），多余的认知负担 | 1 个 fallback 职责清晰：它就是系统的安全网 |
| **用户体验** | 用户反复抽到不同名字但一样空的奖品，会觉得"这 5 个东西有什么区别？都是骗人的" | 1 个 fallback 配 points 类型（给 5 积分），用户感知是"得了积分"而非"抽了个空" |
| **代码简洁性** | `is_fallback=true` 出现 5 次，`validateEmptyPrizeConstraint` 要处理多个 fallback 的逻辑 | 校验简单：`COUNT(is_fallback=true) === 1` |
| **预算系统** | 5 个 fallback 的 `prize_value_points` 全是 0，全部绕过预算过滤——对预算系统来说没有区别 | 完全相同 |

**为什么大厂都是 1 个保底而不是多个**：

核心原因：**保底的本质是"系统安全网"，不是"奖品池的一部分"**。安全网只需要 1 个就够了——你不会在高空走钢丝时铺 5 张安全网，1 张就够接住你了。多张网除了增加成本，不提供额外保护。

**本项目后端 `validateEmptyPrizeConstraint` 当前校验逻辑**：只检查"至少 1 个 `prize_value_points=0`"，没有上限约束。需要加上限。

| 选项 | 建议 |
|---|---|
| **A) 严格 1 个，添加后端校验** | **推荐**：与大厂一致，职责清晰，减少运营认知负担。在 `validateForLaunch()` 加 `fallbackCount === 1` 校验 |
| B) 允许多个但至少 1 个 | 无行业先例支持多 fallback 有正面价值。多个只增加混乱，不增加安全性 |

#### D12：ANTI_EMPTY_THRESHOLD 选 5 还是 8

**行业深度调研：连续"差结果"保护阈值**

先区分两种保底：

| 类型 | 保底什么 | 代表产品 | 阈值 |
|---|---|---|---|
| **非空保底** | 保证不连续抽到空/最低奖 | 本项目的 ANTI_EMPTY_THRESHOLD | 本决定讨论的 |
| **高稀有度保底** | 保证 N 抽内必出高级奖品 | 原神 5 星保底（90抽） | 不同概念 |

本项目的 ANTI_EMPTY_THRESHOLD 属于**非空保底**——连续 N 次抽到 fallback/空奖后，强制发放 low 档以上奖品。

| 公司/类型 | 非空保底阈值 | 场景说明 | 为什么选这个值 |
|---|---|---|---|
| **美团红包** | **3-5 次** | 连续 3-5 次只得到 0.01 元后，系统会给一张有感知的券（2-5 元） | 用户容忍 0.01 元红包的极限大概 3-5 次，超过就不再点了 |
| **饿了么吃货卡抽奖** | **3 次** | 连续 3 次低价值后保证一张实际可用券 | 比美团更激进，因为吃货卡是付费产品，用户期望更高 |
| **拼多多** | **几乎 1 次** | 几乎每次都给有感知的东西（大额券/金币），不存在"空"的概念 | 拼多多的核心策略是让用户"上头"，任何空手体验都会打断循环 |
| **原神 4 星保底** | **10 次** | 10 抽内必出 4 星（但每次都出 3 星，不是空） | 10 次是因为每次都有 3 星保底，用户不会觉得"空手" |
| **王者荣耀夺宝** | **无非空保底** | 因为每次都给碎片（最差也给碎片），不存在空 | 不需要非空保底 |
| **线下抽奖/活动策划** | **无** | 靠物理比例控制，没有连续空奖检测 | 技术限制 |
| **中小游戏** | **5-10** | 通常 5-10 次区间，根据用户付费意愿调节 | 平衡成本与留存 |

**5 vs 8 对本项目的实际影响**：

按 D4 落地后（只有 1 个 fallback），以 low 档 80% 概率估算：

| 阈值 | 连续空奖概率 | 对应抽奖频次 | 用户感受 |
|---|---|---|---|
| **5** | 约 0.03%（每 3000 次出现 1 次） | 餐厅用户月均 4 次抽奖 → 约 60 年才遇到 1 次 | 理论上几乎不触发，但在极端倒霉时能兜住 |
| **8** | 约 0.0001%（几乎不可能） | 几乎永远不会触发 | 等于没有这个机制 |

**关键分析**：

- 本项目是**餐厅小程序**，用户黏性靠到店消费，不是游戏那种高频行为
- 餐厅用户消费 100 元 → 100 积分 → 抽 1 次。一个月来 4 次 → 抽 4 次
- 如果阈值 8，用户要连续空手 **2 个月**（每周 1 次）才触发保底——他早就不来了
- 如果阈值 5，连续空手约 **5 周**触发——仍然很长，但至少在"用户快要失去耐心"的边界上
- 参照美团/饿了么的 3-5 次标准，5 是这个范围的上限，已经偏保守

**实际代码中的配置路径**：

`ComputeConfig.js` 第 253 行：`force_threshold: parseInt(process.env.ANTI_EMPTY_THRESHOLD) || 8`

`.env` 第 147 行：`# ANTI_EMPTY_THRESHOLD=8`（已注释）

| 选项 | 建议 |
|---|---|
| **A) 5** | **推荐**：与美团/饿了么的 3-5 范围一致，对餐厅低频场景是合理上限 |
| B) 8 | 对低频餐厅用户来说太高，用户可能连续 2 个月空手。仅适合高频游戏场景 |

#### D13：奖品数量与 display_mode 是否后端强制校验

**行业深度调研：奖品数量与前端展示的校验方式**

| 公司/类型 | 校验在哪层 | 怎么做 | 为什么 |
|---|---|---|---|
| **美团/阿里** | **后端强制校验** | 活动配置保存时校验奖品数量与模板匹配，不匹配则拒绝保存 | 防止运营配错导致前端崩溃 |
| **原神/米哈游** | **后端** | 卡池配置有严格的数量和概率校验，CMS 提交时后端拒绝不合规配置 | 概率合规性要求 |
| **拼多多** | **后端** | 转盘/红包活动有固定模板，后端校验奖品数量是否匹配模板 | 活动量大，必须自动化校验 |
| **中小公司** | **仅前端** | 前端表单做简单校验，后端不管 | 常见结果：换个管理员用 API 直接提交就绕过了 |
| **线下活动策划** | **物理限制** | 转盘几个格子就放几个奖品，天然匹配 | — |
| **虚拟物品交易平台** | **后端** | 盲盒/开箱系统在后端校验奖品池配置合法性 | 防止概率和展示不一致 |

**固定格位 vs 弹性格位的区别**：

| display_mode | 格位是否固定 | 后端是否需要校验数量 |
|---|---|---|
| `grid_3x3` | **固定 8 格**（9-1 按钮） | **必须校验**：多了放不下少了有空格 |
| `grid_4x4` | **固定 12 格**（16-4 按钮） | **必须校验** |
| `wheel` | **弹性**（6/8/10/12 扇区均可） | 校验上下限（4-16），不强制固定值 |
| `card_flip` | **弹性**（可自适应排列） | 校验上下限（4-20），不强制固定值 |
| `scratch_card` | **弹性** | 同上 |
| `blind_box` / `gashapon` | **弹性** | 同上 |
| 其余 | 视情况 | 建议设上下限 |

**本项目后端 `validateForLaunch()` 当前校验内容**：

1. `validatePrizeWeights` — 每个档位权重之和 = 1,000,000
2. `validateEmptyPrizeConstraint` — 至少 1 个 `prize_value_points=0`
3. `validateCampaignBudgetConfig` — 预算配置校验

**当前缺失**：没有奖品数量与 `display_mode` 的匹配校验。

| 选项 | 建议 |
|---|---|
| **A) 分类校验** | **推荐**：grid 类强制精确匹配，弹性类校验上下限 |
| B) 后端不校验，前端自适应 | 反面教材做法，运营配错后端无感知，前端崩溃 |

**推荐的校验规则**（加入 `validateForLaunch()`）：

```
固定格位校验：
  grid_3x3 → 必须恰好 8 个 active 奖品
  grid_4x4 → 必须恰好 12 个 active 奖品

弹性格位校验（上下限）：
  wheel → 4-12 个
  card_flip → 4-20 个
  scratch_card → 4-16 个
  其余 → 4-20 个（通用范围）
```
