## 一、为什么要改（问题根因，已实证）

### 1.1 现状架构

- 会话表：MySQL `authentication_sessions`（主键 `authentication_session_id`，字段 `session_token`、`user_type`、`user_id`、`login_platform`、`is_active`、`last_activity`、`expires_at`）。
- 隔离粒度：**`(user_type + login_platform)` 双维度互斥**。同一 `user_type` + 同一 `login_platform` 只允许一个活跃会话，新登录踢旧会话。
- 校验：`middleware/auth.js` 每个请求都查一次 MySQL（`AuthenticationSession.findValidByToken`）。
- 撤销：改 DB `is_active=0` + Socket.io 断连（`ChatWebSocketService.disconnectUser`）。
- Token：JWT `access_token`（payload 带 `session_token`）+ `refresh_token`（HttpOnly Cookie）。

### 1.2 实证到的结构性问题（2026-06-01 真实库抽样）

```
问题1：同账号"互踢"——隔离 key 用 (user_type+login_platform)，两个真实同类端被判同一会话
  现状隔离规则（实测代码 routes/v4/auth/login.js / console/config/auth.js）：
    登录时 deactivateUserSessions(user_type, user_id, null, platform) → 同 user_type+platform 旧会话全部 is_active=0
  → 两个 Web 浏览器都被 detectLoginPlatform 判为 'web' → 抢同一个 (admin/user + web) 槽位 → 互踢
  → 被踢方下次请求(dashboard 轮询) → middleware/auth.js 查 findValidByToken 失败 → 401 → 跳登录页 → 全页面无数据

问题2：僵尸会话——login_platform='' 的非法 ENUM 值（实测仍在库里）
  实测：authentication_sessions 中 login_platform='' 共 3 条（均 is_active=0，排查时已置失效）
  库里 ENUM 定义为 ('web','wechat_mp','douyin_mp','alipay_mp','app','unknown') 默认 'unknown'，空串不在白名单
  根因：createSession 落库前未做 ENUM 白名单兜底，仍可能再次写入非法值

问题3：每请求查库——middleware/auth.js 每个带 session_token 的请求都 findValidByToken 查 MySQL
  PM2 cluster 多进程 + 万级并发下是热点（实测连接池每进程 max=20）

问题4（实测新增，初稿未提）：互斥并非真正"强一致单点"——刷新链路会绕过互斥造出重复活跃会话
  实测真实库当前活跃会话分组：
    user_id=10845 user+web → 2 条同时 is_active=1（ids=18821,18896）
    user_id=12600 user+web → 2 条同时 is_active=1（ids=18773,18774）
  原因：routes/v4/auth/token.js 的 POST /refresh 在"旧 session 不存在/无 session_token"时
        直接 createSession 新建，且不调用 deactivateUserSessions → 同 user+platform 出现并存活跃会话。
  → 说明现状既"该并存的被踢"（问题1），又"不该并存的却并存"（问题4），隔离语义已自相矛盾，进一步证明 platform 不该当隔离 key。
```

### 1.3 定性结论（实测校准）

- 后端业务数据健康（用户/抽奖/各域报表均有真实数据），**"数据为空"是 401 被踢下线后的下游表现，不是数据问题。**
- 前端（admin）拿到 401 → 清 token → 跳登录页的行为是**正确**的，无前端 bug（实测 `admin/src/api/base.js` 的 `request()` 与 `handleResponse()` 均已按 401 细分 code 跳登录页）。
- 根因是**会话隔离粒度选错了**：用 `login_platform` 当隔离 key，导致"同一类端的多个真实设备"被强制互斥；同时刷新链路又破坏了单点一致性（问题4）。这是小公司常见的实现副作用，**安全收益低、维护成本高**。

## 二、行业做法对比与选型依据

### 2.1 各类公司怎么设计（公开可观察的通用模式）

| 类型 | 隔离粒度 | 多设备 | 单点(顶号) | 会话存储 | 撤销 | 敏感操作 |
|---|---|---|---|---|---|---|
| 大厂(美团/腾讯/阿里) | **device_id 设备级** | 允许并存 | 仅同类端互斥 | Redis 注册表 + DB 审计 | Redis 删/标记 + 网关校验 | 风控 + 二次验证 |
| 微信(可直接观察) | 端类型 | 手机1 + PC/Pad 并存 | 新手机踢旧手机 | 服务端会话 | 长连接实时推下线 | 支付独立密码 |
| 小公司 | user 或 token | 看实现(常碰巧) | 常碰巧单点 | 纯JWT / DB单token | 难撤销(等过期) | 多数没有 |
| 游戏公司 | account 账号级 | **强制单设备** | **强单点**(防多开/盗号) | 网关长连接 + 状态服 | 长连接实时踢 | 二次密码/动态令牌 |
| 活动策划/H5抽奖 | openid | 容忍多端 | 基本不踢 | 短时token/无状态 | 靠过期 | 靠风控防刷 |
| 虚拟物品交易/小众二手 | **device_id + 风控** | 允许并存 | 一般不踢 | Redis + DB | 设备列表手动撤销 | **交易/提现强制二次验证** |

### 2.2 差异只在 4 个轴上

1. **隔离粒度**：用什么 key 判定"两个会话是不是同一个"。本项目现状 `(user_type+platform)` 太粗 → 两个浏览器都算 `admin+web` 必然互踢。大厂用 `device_id` → 两浏览器=两设备=两会话，天然并存。
2. **会话存储**：JWT 无状态 / DB / Redis。现状每请求查 MySQL；大厂用 Redis 毫秒级 + DB 仅审计。
3. **撤销机制**：现状改 DB + Socket.io；大厂 Redis 删 key + 长连接推送（同思路、更快）。
4. **敏感操作**：现状"会话有效就放行"；交易平台"会话有效 ≠ 能交易"，交易/提现要二次验证。

### 2.3 本项目的真实诉求

```
系统组成              对会话策略的真实诉求
─────────────────────────────────────────────
抽奖(游戏属性)     → 不需要单点(非实时对战,无"一角色双客户端"问题)
积分/资产余额(钱)  → 需要"能撤销 + 敏感操作二次验证"(交易平台属性)
C2C 市场交易      → 需要风控 + 交易二次验证(交易平台属性)
微信小程序(用户端) → 手机+Pad 同时在线正常 → 要多设备并存
Web 管理后台      → 管理员多电脑/多标签 → 要多设备并存(当前被坑点)
```

**选型结论**：本项目**没有"游戏强单点"硬需求**，但**有"交易平台"的资产安全需求**。因此走**交易平台/大厂路线：设备级多会话并存**，而非用"单点互斥"卡用户和自己。资产安全本方案采用"会话在线校验 + 高危操作二次确认 + 全程审计 + 设备可随时踢下线"（已有能力组合），按决策不引入短信二次验证。该路线与现有技术栈高度契合（见第三节），不引入新依赖。

## 四、目标架构（纯文本数据流图）

```
【登录】
客户端(小程序/Web) ── 生成稳定 device_id(存 localStorage / 小程序 storage) ──┐
                                                       Header: X-Device-Id   │
                                                                             ▼
后端登录路由(auth/login.js, console/config/auth.js)
   │  隔离 key: (user_id, user_type) + device_id     ← platform 不再当隔离 key
   │  同 device_id 再次登录 = 替换"自己这台设备"的旧会话(token 轮换)   ← 正常
   │  不同 device_id     = 新会话并存,不踢                            ← 不再误伤
   ▼
写 Redis: session:{user_id}:{device_id} = {session_token, user_type, platform, ip, exp}  (热路径,带 TTL)
        + 集合 user_sessions:{user_id} 记录该用户所有 device_id(用于"我的设备列表"/批量撤销)
写 MySQL authentication_sessions (审计/持久, 加 device_id 列, 可异步落库)
   ▼
签发 JWT(payload 带 session_token + device_id)

【每次请求校验】 middleware/auth.js
   │  解析 JWT → 取 user_id + device_id + session_token
   │  查 Redis session:{user_id}:{device_id}
   │     ── 命中且 session_token 一致且未过期 → 放行(毫秒级)
   │     ── 未命中 → 回查 MySQL(降级容错) → 命中则回填 Redis → 放行
   ▼  查不到/已撤销/token 不一致 → 401 (区分 SESSION_REVOKED / SESSION_EXPIRED / SESSION_NOT_FOUND)

【单点(可选,仅按业务需要开启,默认关闭)】
某业务要求单点/限设备数时 → 在该业务策略内显式调用"踢同 user 其它 device 会话"
默认不踢 → "单点"成为可选配置开关,不再是结构性默认事故

【撤销 / 设备管理】 用户端新增 /auth/sessions；管理端复用 /console/sessions
   GET    /api/v4/auth/sessions                          → 我的登录设备列表(authentication_session_id, device_id, platform标签, ip, 最后活动时间)
   DELETE /api/v4/auth/sessions/:authentication_session_id → 踢自己的某台设备
   撤销动作 = 删 Redis key + MySQL is_active=0 + Socket.io 推送该 device 下线

【高危操作防护(不引入短信二次验证)】
会话有效仅是"入场券" → 高危写操作叠加已有防护:
   L1 requireValidSession(会话在线) + L2 highRiskOperationMiddleware(二次确认 _confirmed / 审批 _approved)
   + 全程 AuditLogService 审计 + 设备管理可随时踢下线
   (本方案按决策不新增短信二次验证 L3)
```

### 关键设计点

- **platform 降级为展示标签**：仅用于"设备列表"展示"微信小程序 / Web 后台 / 抖音小程序"等，不再参与隔离判定。`detectLoginPlatform` 保留。
- **device_id 由前端生成并持久化**：Web 用 `crypto.randomUUID()` 存 `localStorage`；小程序用 `wx.getStorageSync` 持久化一个 UUID。后端只信任并记录，不依赖它做安全决策（安全靠 session_token + Redis 校验 + 设备可随时踢下线 + 高危操作二次确认/审计）。
- **Redis 为主、MySQL 为辅**：热路径走 Redis（对齐《万级并发优化方案》）；MySQL 保留为审计与降级数据源，Redis 不可用时自动回退查库，不影响可用性。
- **僵尸会话根治**：`createSession` 落库前校验 `login_platform` ∈ ENUM 白名单，非法值兜底 `unknown`；Redis key 带 TTL 自动过期，从机制上杜绝"永不清理"。

## 六、Redis 键设计

> 复用 `utils/UnifiedRedisClient.js`（`getRedisClient()` 封装客户端 / `getRawClient()` 原生 ioredis），不自带连接池配置。实测 `REDIS_URL=redis://localhost:6379` 在线（PING=PONG，7.0.15），方案可直接落地。
> 实测注意：`getRawClient()` 返回原生 ioredis 实例，用其 `set(key, val, 'EX', ttl)`、`sadd`、`srem`、`smembers`、`del` 即可；封装层 `set(key,val,ttl)` 也可用。`middleware/auth.js` 已 `require('../utils/UnifiedRedisClient').getRawClient()`，会话校验热路径在此扩展。

| Key | 类型 | 值 | TTL | 用途 |
|---|---|---|---|---|
| `session:{user_id}:{device_id}` | String(JSON) | `{session_token, user_type, login_platform, login_ip, expires_at}` | 与会话有效期一致（7天，登录续期） | 请求热路径校验 |
| `user_sessions:{user_id}` | Set | 成员为 `device_id` | 随会话维护 | "我的设备列表" / 批量撤销 |

写入/失效时机：
- 登录成功：写 `session:{uid}:{did}`，并 `SADD user_sessions:{uid} {did}`。
- 撤销/登出：`DEL session:{uid}:{did}`，`SREM user_sessions:{uid} {did}`，MySQL `is_active=0`，Socket.io 推送下线。
- 校验未命中：回查 MySQL，命中则回填 Redis（带剩余 TTL）。

降级策略（与现有限流降级一致的思路）：Redis 不可用时，会话校验自动回退到 MySQL `findValidByToken`，保证可用性；恢复后自动回填，无需人工干预。

### 7.4 高危操作防护（不引入短信二次验证 —— 已拍板）

> 决策：**不新增短信二次验证（L3）**。高危写操作仅复用已有两层 + 审计 + 设备可踢，避免高频体验受损与额外短信成本。

```
分层（复用已有机制，不新造轮子，不加短信码）：
  L1 会话在线   requireValidSession            已存在  缺/失效 → 401 SESSION_REQUIRED/SESSION_INVALID
  L2 二次确认   highRiskOperationMiddleware    已存在  高风险写 → 400 CONFIRMATION_REQUIRED(_confirmed) / 403 APPROVAL_REQUIRED(_approved)
  L3 短信二次验证  ❌ 不采用（本次决策明确不加）
  审计          AuditLogService                已存在  全程记录高危操作，可追溯/可回滚
  兜底          设备管理(/auth/sessions、/console/sessions) 可随时踢下线异常设备
```

> 说明：交易/挂单/提现/积分资产调整等高危操作，靠"会话在线(L1) + 二次确认(L2) + 审计 + 可踢设备"组合防护即可；如未来确需短信验证，再单独评估开启，不在本方案范围。

### 7.5 401 错误码语义（前端按此分流，遵循 ApiResponse）

| code | HTTP | 含义 | 前端处理 |
|---|---|---|---|
| `SESSION_REVOKED` | 401 | 会话被显式撤销（设备管理踢下线 / 登出） | 清 token，跳登录页，提示"已在设备管理中退出" |
| `SESSION_EXPIRED` | 401 | 会话超时过期 | 清 token，跳登录页，提示"登录已过期" |
| `SESSION_NOT_FOUND` | 401 | 记录被清理 | 清 token，跳登录页，提示"登录状态失效" |

> 高危操作的 `CONFIRMATION_REQUIRED`(400) / `APPROVAL_REQUIRED`(403) 由已有 `highRiskOperationMiddleware` 返回，前端弹二次确认框即可（非会话失效，不跳登录页）。
> 注：旧的 `SESSION_REPLACED` 在默认多设备并存下不再产生；仅当某业务显式开启"单点/限设备数"开关时才可能出现，届时复用 `SESSION_REVOKED`。错误码统一经 `res.apiUnauthorized(message, code)` 返回。

## 十三、三端问题归属（哪些是谁的问题）

> 基于真实代码/数据库核实的责任划分。原则：**以后端数据库提供的能力为权威，前端适配后端。**

### 13.1 后端数据库项目的问题（根因在此）

1. **隔离粒度选错**（核心根因）：`(user_type+login_platform)` 当隔离 key，导致两个真实同类端设备被判同一会话而互踢。实测 `routes/v4/auth/login.js`、`quick-login`、`console/config/auth.js` 均按 `deactivateUserSessions(user_type,user_id,null,platform)` 互斥。→ 改 device_id 隔离。
2. **僵尸会话落库口子**：`createSession`（`models/AuthenticationSession.js`）未校验 `login_platform` 白名单，实测库里仍有 3 条 `login_platform=''`。→ 加白名单校验 + 全表兜底清理。
3. **模型与真实库不一致**：`models/AuthenticationSession.js` 主键写 `INTEGER`，真实库是 `BIGINT`。→ 修正模型定义（库主键带 FK，不 ALTER 库）。
4. **管理后台登录互斥**：`console/config/auth.js` 固定 `user_type='admin'` 后按 platform 互斥，多电脑/多标签同被判 `admin+web` 互踢。→ 切 device_id 隔离后自然消失。
5. **会话校验走 MySQL 热路径**：`middleware/auth.js` 每请求 `findValidByToken` 查库，cluster + 高并发下是瓶颈。→ 迁 Redis 优先 + MySQL 降级。
6. **刷新链路破坏单点一致性（实测新增）**：`routes/v4/auth/token.js` `POST /refresh` 用 `role_level>=100?'admin':'user'` 反推 user_type 并在缺会话时直接 `createSession` 不互斥，造出同 user+platform 重复活跃会话（实测 user_id=10845/12600 各 2 条）。→ refresh 沿用旧会话 user_type + device_id 维持，不再反推。

### 13.2 Web 管理后台前端（admin/）的问题

- **核心无 bug**：拿到 401 → 清 token → 跳登录页的行为正确（实测 `admin/src/api/base.js` `request()`/`handleResponse()` 已实现），"数据为空"是被踢下线的下游表现。
- **实测发现 1 个真实前端 Bug**：`admin/src/modules/system/pages/sessions.js` `init()` 读 `payload.session_id`，但后端 JWT 字段是 `session_token`（无 `session_id`）→ `currentSessionId` 恒 undefined，"当前会话不可撤销"保护失效。→ 改按后端 `is_current` 字段判定。
- 待配合后端的小改：注入 `X-Device-Id`、401 错误码补全（`SESSION_REVOKED/EXPIRED/NOT_FOUND`，现状只特判 `SESSION_REPLACED`）、设备信息展示。**不新建页面**（会话管理页已存在）。

### 13.3 微信小程序前端的问题

- 与 admin 同类：需生成并持久化 `device_id` 并在请求头携带；适配新错误码（`SESSION_REVOKED/SESSION_EXPIRED/SESSION_NOT_FOUND`）。
- 高危操作沿用已有 L1/L2 防护，**不接入短信二次验证**（按本次决策）。
- 这是**求证项**：小程序当前是否已有稳定 device_id 持久化、是否已处理 401 细分，需要小程序团队确认（本仓库不含小程序代码）。

### 13.4 一句话

> 根因 100% 在后端会话策略；两个前端都只是"被动受害者 + 需做适配"。**修后端为主，改前端适配为辅。**

### 14.2 六项待拍板的「全行业横向对比 + 落到本项目」

> 你的诉求：项目未上线、愿一次性投入、不兼容旧接口、长期维护成本低 / 技术债最低、必须贴合后端现有技术栈（Node+Express / Sequelize+MySQL / ioredis / Socket.io / JWT）。
> 下面每项先给「大厂 / 小公司 / 游戏公司 / 活动策划H5 / 虚拟物品交易·小众二手」五类做法，再给"最适合你"的结论与理由。**结论已同步更新 14.1 表。**

#### A. device_id 缺失时是否拒登

| 公司类型 | 典型做法 |
|---|---|
| 大厂(美团/腾讯/阿里) | 不拒登。设备指纹缺失→标记"弱信任设备"+风控加权，照常放行，靠后置风控而非前置硬卡 |
| 小公司 | 多数没有 device 概念，自然不拒 |
| 游戏公司 | 端游/手游强制设备标识（反多开/防盗号），缺失可能拒登或要求重新鉴权 |
| 活动策划/H5抽奖 | 完全不在意 device，openid/无状态即可 |
| 虚拟物品交易/小众二手 | 不拒登，但"无设备指纹"会限制提现/大额交易，引导补全 |

**最适合你：(b) 缺失兜底 `legacy` 放行 + 记告警。** 理由：①你存量会话 device_id 全是 NULL，硬拒会把现有在线用户全部锁死，与"平滑切换"冲突；②device_id 由前端生成、本就不可信，安全不该依赖它（决策4已定），真正的安全闸门是 session_token+Redis 校验+可踢设备+高危二次确认；③走交易平台路线即可——"缺指纹不拒登，但敏感操作叠加已有 L1/L2"。**技术债更低**：避免为一个不可信字段写一堆拒登/豁免分支。

#### B. 单用户最大设备数

| 公司类型 | 典型做法 |
|---|---|
| 大厂 | 一般不限设备总数，但限"同类端并发数"（如 1 手机+1 Pad+多浏览器），异常多设备触发风控 |
| 小公司 | 多不限（也没能力统计） |
| 游戏公司 | **强限**：常 1 设备/账号（防多开、防工作室），新登录踢旧 |
| 活动策划/H5 | 不限（拉新优先，多端越多越好） |
| 虚拟物品交易/小众二手 | 不硬限设备数，但"活跃设备过多"作为风控信号，提现时要求收敛设备 |

**最适合你：默认不限 + 保留"限设备数 N"可选开关（默认关），并加一条软上限告警。** 理由：①你是"抽奖皮+交易核"，非游戏强单点（决策2已定）；②不限最贴合 C 端体验与拉新；③但建议给 `user_sessions:{user_id}` 这个 Redis Set 加一个**软上限（如 >10 设备记审计/告警）**，既不踢人又能发现盗号/工作室异常——这正是交易平台的折中。**技术债更低**：上限只是一行 `SCARD` 判断 + 告警，不写复杂的"踢最旧"调度（要踢再开开关）。

#### C. `idx_user_device` 索引是否带 `is_active`

| 公司类型 | 典型做法 |
|---|---|
| 大厂 | 热路径几乎不查 MySQL（走 Redis/KV），DB 索引面向"审计/排查"，按高频过滤列组合建复合索引 |
| 小公司 | 常只建单列索引，慢查询频发 |
| 其余类型 | 与隔离策略强相关，无统一范式 |

**最适合你：(b) `(user_type, user_id, device_id, is_active)` 复合索引。** 理由：①与现有 `idx_user_sessions_platform(user_type,user_id,login_platform,is_active)` **完全同款式**，团队心智一致、维护成本最低；②你的高频查询正是"某用户某设备的活跃会话"（登录替换、踢设备、降级回查），`is_active` 在最后一列能让这些查询走覆盖过滤；③MySQL 此处退居"降级/审计"，索引为这两类查询服务最划算。**技术债更低**：照搬已验证的索引范式，不另立一套。

#### D. Redis 命中后是否还比对 MySQL 的 expires

| 公司类型 | 典型做法 |
|---|---|
| 大厂 | **Redis/KV 即权威**，命中即放行，毫秒级；DB 仅审计。撤销靠"删 key + 长连接推下线"，TTL 兜底过期 |
| 小公司 | 多为纯 DB 或纯 JWT，无双层 |
| 游戏公司 | 网关/状态服内存即权威，长连接实时踢 |
| 活动策划/H5 | 无状态 JWT，靠过期 |
| 虚拟物品交易/小众二手 | Redis 权威 + DB 审计，与大厂同 |

**最适合你：Redis 命中即放行（Redis 为热路径权威），不再回比 MySQL。** 理由：①这正是《万级并发优化方案》的方向，也是大厂/交易平台标准；②你已实测 Redis 在线，`session:{uid}:{did}` 带 TTL，撤销时**双写**（DEL key + MySQL `is_active=0` + Socket.io 推下线）能保证一致；③回比 MySQL 会让 Redis 失去意义（又退化成每请求查库）。Redis 未命中才降级查 MySQL 并回填——可用性与性能兼得。**技术债更低**：单一权威源，不写"双源比对"的复杂一致性逻辑。

#### E. 用户端设备列表是否展示 IP 归属地（结论已据你新诉求改为「是」）

| 公司类型 | 典型做法 |
|---|---|
| 大厂 | **必有**"登录设备/登录地"列表（微信、支付宝、淘宝账号安全页），用自建/离线 IP 库，异地登录提醒 |
| 小公司 | 多数没有，或只显 IP |
| 游戏公司 | 有"登录地/常用设备"，异地触发二次验证 |
| 活动策划/H5 | 没有 |
| 虚拟物品交易/小众二手 | **强需**：登录地是核心风控与用户自查信号，交易/提现页突出展示 |

**最适合你：是，用离线 IP 库 `ip2region`（纯本地、零运行成本）。** 理由：①你定位"交易核"，"登录设备+登录地"是交易平台账号安全页标配，对用户自查盗号价值高；②你已明确"愿一次性投入、对标交易平台"，离线库正契合；③**关键技术选型**：用 `ip2region`（单个 ~5MB 离线文件，纯内存查询、毫秒级、**不调任何第三方 API、无网络依赖、无月费、无隐私外发**），而**不用**在线 IP API（那才是长期成本与依赖债）。这与你"长期低成本"完全一致。**唯一代价**：新增 1 个本地依赖——因为前文承诺"无需新增依赖"，所以此项**单独拎出来请你点头**（见下方"需你最终确认"）。若你不想加任何依赖，退回"只显 IP+平台+时间"也完全可用。

#### F. admin 端"当前会话"如何判定

| 公司类型 | 典型做法 |
|---|---|
| 大厂 | 设备列表每条由**服务端**打 `is_current` 标记（服务端最清楚哪条是本次请求的会话），前端只渲染 |
| 小公司 | 常前端自己解析 token，易错（你现状的 `session_id` Bug 就是此类） |
| 其余 | 同大厂：服务端判定更可靠 |

**最适合你：后端返回 `is_current`。** 理由：①后端在 `authenticateToken` 里天然拿到本次请求的 `session_token`，判定 100% 准确；②能**直接修掉** 13.2 实测 Bug（前端读不存在的 `payload.session_id`）；③前端从此不解析 JWT，逻辑更薄、跨端一致。**技术债更低**：消灭"前端猜当前会话"这一类长期易错点。

### 14.3 一句话总览：哪种"流派"最适合你，为什么

> 你的系统 = **抽奖(游戏皮) + 真实积分资产 / C2C 交易(交易平台核)**。
> 安全等级**对标交易平台（美团/支付宝账号安全那一档），而非游戏强单点、也非 H5 拉新裸奔**。
> 因此 A~F 的统一基调是：**多设备并存、设备级会话、Redis 热路径权威、服务端打标、缺指纹不拒登靠后置风控、给软上限与登录地等"交易平台标配"的自查能力**——
