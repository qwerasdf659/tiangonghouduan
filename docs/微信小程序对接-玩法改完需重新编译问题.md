# 微信小程序前端对接文档：玩法改完为何"必须重新编译才生效"

> 适用范围：微信小程序前端项目
> 文档性质：后端数据库项目对外提供的对接说明（后端为权威源，前端适配后端）
> 生成时间：2026-06-15（北京时间）
> 核对方式：已对真实库 `restaurant_points_dev`、运行中的 C 端 API、后端缓存 TTL 配置实测核对，非历史文档推断

---

## 一、问题现象

运营在 web 管理后台改完活动「回馈方式（玩法）」后，微信小程序端：

- 玩法没有变化，仍是旧玩法（例如一直是砸金蛋）
- 用户退出、重新登录，玩法仍是旧的
- 只有把微信小程序重新编译（重新构建 + 发布）后，玩法才会变成新的

## 二、结论（定性）

这是**微信小程序前端项目的问题**，不是 web 管理端、也不是后端数据库的问题。

更精确地说：**玩法值被写死/打包进了小程序的编译产物里（编译期常量），前端没有在运行时从 API 读取 `display.mode`。**

因此玩法只能靠"改源码 + 重新编译 + 重新发布"来改变，运营在后台的修改对已发布的小程序无效。

## 三、为什么能这样判定（两个现象分别排除了什么）

两个现象本身就是定位根因的关键证据：

```
现象1：退出重新登录，玩法还是旧的
  → 排除了"用户会话/登录态缓存"。
    登录态都换了玩法还没变，说明玩法不跟着用户走，与登录无关。

现象2：必须重新编译小程序，玩法才变（决定性证据）
  → 后端是运行时服务，没有"编译"这个概念；
    重新编译小程序这个动作，根本不会触碰后端数据库，也不会清后端 Redis 缓存。
    既然"只有重新编译能改变玩法"，那玩法值只可能来自
    小程序自己的代码 / 编译产物（编译期常量），
    而不是运行时拿到的 API 响应。
```

一句话：能被"重新编译"改变、却不能被"后台修改 + 重登 + 等待"改变的东西，只能是编译进前端包里的常量。

## 四、纯文本数据流图（正确做法 vs 当前小程序的错误做法）

```
✅ 正确做法（玩法应是"运行时数据"）：

运营改玩法 → DB.display_mode = card_collect → API 返回 display.mode = card_collect
                                                        │
                                            小程序每次进抽奖页读取该值
                                                        │
                                            按值动态渲染对应玩法组件 → 玩法实时生效
   （不需要重新编译，不需要重登；最多等后端缓存 30~60 秒自动刷新）


❌ 当前小程序的错误做法（玩法被当成"编译期常量"）：

运营改玩法 → DB 变了 → API 也返回新值 card_collect
                                │
                                ✗ 小程序忽略 API 的 display.mode
                                ✗ 玩法写死在小程序源码里（如 const GAME = 'golden_egg'）
                                │
                        该常量被打包进编译产物
                                │
            只有改源码 + 重新编译 + 重新发布，常量才会变
            → 退出重登无效（常量不跟登录态走）
            → 必须重新编译才生效（常量固化在编译产物里）
```

## 五、后端已被彻底排除（实测证据）

后端这条链路上不存在"几小时/几天不更新、必须重新编译才变"的可能：

后端缓存 TTL（来自 `services/lottery/QueryService.js`，实测配置）：

| 查询方法 | 缓存 TTL | 说明 |
|----------|---------|------|
| `getActiveCampaigns`（活动列表） | 30 秒 | 无 user_id 时缓存，有 user_id 时不缓存 |
| `getCampaignConfig`（活动配置） | 60 秒 | |
| `getCampaignByCode`（按 code 查活动） | 60 秒 | |
| `getCampaignWithPrizes`（活动+奖品） | 60 秒 | |

推理：

- 运营改完玩法后，后端最多 30~60 秒后必定下发新值，绝不可能长期不变。
- 真实库实测：活动 `CAMP20250901001` 的 `display_mode = card_collect`（新值已落库）。
- C 端 API 实测：`GET /api/v4/lottery/campaigns/CAMP20250901001/config` 返回 `display.mode = card_collect`（新值已下发）。
- 重新编译小程序这个动作，既不连后端数据库、也不清后端 Redis，与后端缓存毫无关系。

→ 后端不是原因，web 管理端也不是原因（玩法已正确保存）。问题在小程序端没有消费 API 返回的 `display.mode`。

## 六、小程序前端需要做的修改

核心一句话：**把玩法从"编译期常量"改成"运行时从 API 读取"。**

可能的病因（三选一，都属于小程序前端）：

1. 玩法值硬编码为代码常量（最符合现象）
   - 小程序里写死了类似 `const playType = 'golden_egg'`，或页面模板固定引用砸金蛋组件，根本没用 API 返回的 `display.mode`。这类常量只能靠重新编译改变。

2. 构建期注入
   - 玩法值在打包/构建配置里被当成环境常量注入，运行时不读 API。

3. 持久化本地缓存（可能性较低）
   - 把玩法存进小程序 Storage 且从不更新。但若是这种，清缓存/重装即可解决，不至于"非要重新编译"，所以可能性低于第 1 种。

修复步骤：

- 进入抽奖页时调用 `GET /api/v4/lottery/campaigns/:code/config`，读取 `res.data.display.mode`。
- 按 `display.mode` 的值动态加载对应玩法组件（16 种玩法枚举与组件映射，见配套文档
  《微信小程序对接-活动玩法display_mode渲染.md》第四、六节）。
- 删除任何写死玩法的常量 / 固定组件引用 / 构建期注入。
- 字段解构用 snake_case：`const { mode, grid_cols } = res.data.display`，不要写成 `gridCols`。
- 不要把玩法长期存进本地 Storage；若要缓存，活动页 onShow 时重新拉取覆盖。

改完后效果：运营在后台改玩法 → 小程序下次进活动页自动变成新玩法，**无需重新编译、无需用户重登**（最多等后端缓存 30~60 秒）。

## 七、验证方法（改完后如何确认修复成功）

不重新编译、不重登，按以下步骤端到端验证：

1. 运营在 web 管理后台把活动玩法从 A 改成 B 并保存。
2. 等待约 60 秒（让后端缓存自然过期），或确认已超过缓存 TTL。
3. 小程序退到首页再重新进入该活动抽奖页（触发重新拉取 config）。
4. 预期：玩法变成 B，全程没有重新编译、没有重新发布、没有让用户重登。

如果上述步骤玩法能跟着变 → 修复成功（玩法已是运行时数据）。
如果仍需重新编译才变 → 说明仍有写死的玩法常量未清除，继续排查第六节第 1、2 点。

辅助核对（确认后端确实在下发新值，排除前端"怀疑后端"）：

```
GET /api/v4/lottery/campaigns/active
GET /api/v4/lottery/campaigns/{活动code}/config
```

查看返回里的 `display.mode` 是否已是新玩法值；若已是新值，则问题 100% 在小程序端的消费逻辑。

## 八、责任边界说明

- 后端数据库项目：无需改动。API 实时下发新玩法值，缓存仅 30~60 秒。重新编译小程序与后端无关。
- web 管理后台前端项目：无需改动。玩法已正确保存到数据库。
- 微信小程序前端项目：需修复——玩法当前被固化在编译产物里，要改为运行时消费 API 的 `display.mode`，删除写死的玩法常量。
