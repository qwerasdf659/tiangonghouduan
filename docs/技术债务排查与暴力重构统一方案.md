# 技术债务排查与暴力重构统一方案

- 排查日期：2026-07-11（北京时间）；同日深夜完成三端二次求证（见第十三章）
- 数据来源：当前工作区真实代码（commit 工作态）+ 真实数据库 `restaurant_points_dev`（Node.js mysql2 直连核对，非备份文件）
- 原则：不引用历史报告；不兼容旧路由/旧接口；不留过渡方案；只保留新方案；接口路径/字段/响应格式以后端数据库项目为权威，前端直接改用后端字段名，不做映射层
- 排查范围：后端全部（app.js / routes / services / models / middleware / utils / migrations / scripts / tests / docs / jobs / config / constants）+ admin 前端（admin/src 逐项求证，见 12.2/十三）；微信小程序仓库不在本工作区，按契约级求证（见 12.1）

## 一、总体结论

项目整体较干净：无 v1/v2/v3 路由；无孤儿路由文件；135 个模型与数据库 136 张表（含 sequelizemeta）一一对应；108 条迁移文件与 sequelizemeta 执行记录完全一致；旧 AssetService / PointsService / ExchangeRateService / C2C 市场服务与表均已物理删除。

剩余债务集中四类：
1. "拆分了一半"的服务：新服务已注册但路由仍走旧服务，真双轨并存（最高风险）
2. 数据迁移完成后未拆除的兼容分支（PII 明文兜底、DIY 旧格式、旧枚举值）
3. 死代码 / 死注册 / 一次性脚本 / 与项目无关文件
4. 超大文件（服务层、路由层、模型层各一批 1000+ 行）

## 二、边界划清：合法双轨（禁动）vs 兼容残留（必清）

### 2.1 合法业务双轨——保留，不是统一对象

| 双轨 | 内容 | 判定依据 |
|---|---|---|
| 背包双轨 | assets（可叠加：account_asset_balances + AssetTransaction 账本）↔ items（不可叠加：items + item_ledger 双录 + item_holds 锁） | 两种资产形态各有完整账本闭环，BackpackService 文件头明确声明 |
| 兑换频道双轨 | item_type：prop（道具商城/星石轨）↔ product/voucher（商品兑换/源晶轨） | 产品频道设计，exchange/QueryService 有白名单校验 |
| 积分货币双轨 | points 永续积分 ↔ event_points 活动积分（独立过期 job） | 双货币经济设计 |
| 商品编码双轨 | ProductSeries 可读系列号（SLNB-001）↔ ProductBatch 批次码 | 两个正交编码维度 |
| 用户等级双轨 | users.user_level（身份角色）↔ user_growth_levels（消费成长等级） | 语义不同 |
| 兑换订单双链路 | exchange_records（B2C 商城履约）↔ redemption_orders（核销码履约） | 不同履约链 |
| 基础设施降级 | Redis 限流退化内存限流、Sealos 内网转公网、分群规则回退 default | 容错降级，非新旧并存 |
| 统计分级查询 | 今日查 lottery_draws / 7-90 天查 hourly / 90 天以上查 daily | 性能分层 |

注意误判项：`old_status`（状态机审计字段）、`old_item_ids`（以物易物"投入的旧物"参数）、抽奖 `fallback`（保底档位）均为业务词，不是旧代码。

## 十一、风险评估与护栏对照（2026-07-11）

### 11.1 删除侧：风险低，分级如下

- 代码文件/注册键/枚举值删除：完全可逆（git 历史即归档），且 36 项批准项均有零引用二次亲核证据（含排除动态 getService(变量) 调用，见 3.4 复核记录）

### 11.2 拆分侧：五类经典事故与护栏对照

| 风险 | 场景 | 护栏 |
|---|---|---|
| 循环依赖 | 拆出的子模块反向 require 原模块，Node 循环 require 导致不完整导出，运行时才炸 | madge 门禁，每拆一个跑 check:circular（基线：真循环 0 处），不干净不进下一个 |
| 单例状态分裂 | 子模块各自 new 实例，内存状态（缓存/计数器）分裂；DrawOrchestrator 三实例为前车之鉴 | Facade 保键、子模块不独立注册，单一实例经 Facade 持有 |
| this 绑定断裂 | 类方法搬走后内部 this.xxx() 找不到兄弟方法 | 纯搬移规则 + 拆完立跑该域测试 |
| jest.mock 路径静默失效 | 测试 mock 指向旧路径，mock 不生效但测试可能碰巧仍绿 | 每次拆分后 grep 原路径在 tests/ 的全部 mock 引用同步更新 |
| 事务上下文断链 | TransactionContext（ALS）跨层传事务，改调用链层级可能丢上下文 | 纯搬移不改调用链是硬规则，行为修改单独提交单独测 |

### 11.3 已识别软肋

scheduled_tasks.js（3169 行）是最需要拆的文件，同时是测试覆盖最弱区域（15/22 job 无测试），出问题测试接不住——已在 7.5 设强制前置：拆前补冒烟测试或拆后手动逐任务触发核对。

### 11.4 结论

删除 = 已逐项验证 + 可回滚；拆分 = 五类已知坑 × 五道护栏 + 一个已识别盲区。配合未上线窗口、已完成的数据库与 git 备份、按逻辑单元切分的 commit（便于 bisect 定位），最坏情况为回滚到备份点重来，不存在既不可逆又无备份的操作路径。
