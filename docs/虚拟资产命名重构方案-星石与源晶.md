## 一、行业方案对比总览

### 1.1 大厂方案

| 公司 | 资产体系 | 表结构策略 | code 与 display_name | 扩展方式 | 平台名称 |
|------|---------|-----------|---------------------|---------|---------|
| **美团** | 积分 / 红包 / 优惠券统一管理 | **单表 + asset_type_code** | 分离，code 不可变 | 配置驱动，加行不改表 | 万象平台 |
| **腾讯** | Q币 → Q点 → 游戏货币，多层级 | 联邦式单表（每应用一套，API 层统一） | app_id + currency_code 复合键 | Midas SDK 框架扩展 | 米大师 |
| **阿里** | 淘金币 / 天猫积分 / 88VIP / 蚂蚁积分 | **单表 + asset_code** | 分离，code 不可变，域名前缀 | 配置驱动，资产中台 | 资产中台 |
| **京东** | 京豆 / 京券 / E卡 / 红包 | **单表 + asset_type_code** | 分离，UPPER_SNAKE_CASE | 配置驱动，幂等事务 | 虚拟资产中台 |
| **拼多多** | 多多币 / 现金余额 / 实物权益 | 混合（可替代资产单表，不可替代资产独立表） | 规则引擎驱动 | 配置 + 规则引擎 | — |

**大厂共识：**
- code 是不可变契约，display_name 随时可改
- 单表 `(user_id, asset_type_code)` 是压倒性主流
- 新增资产类型 = 插入配置行，零 schema 变更

### 1.2 游戏公司方案

| 公司/游戏 | 货币层级 | code 设计 | 关键模式 |

- **三层标识**是黄金标准：数据库数字PK → 字符串code → display_name
- 重命名只改 display_name，**从不改 code**
- 货币废弃时：改名为"Legacy XXX"，缩小使用范围，另建新 code
- 软/硬货币严格分离，转换单向不可逆

### 1.4 小公司 / 创业公司方案

| 方案 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| 单表 `user_balances(user_id, asset_type_code, balance)` | < 10 种资产，< 10万用户 | 简单，加类型无需改表 | 高并发下热表 |
| 多表（每种资产一张表） | 资产间业务逻辑差异极大 | 性能隔离 | 迁移多，代码重复 |
| **开源方案 medici** (Node.js) | 需要审计级别记账 | 真双重记账，层级账户 | MongoDB 依赖 |
| **开源方案 bavix/laravel-wallet** | PHP 生态 | 多态钱包，转账=提现+存入 | PHP 生态 |

**小公司共识：**
- 先用单表，遇到瓶颈再拆
- 乐观锁(version 字段)是标配
- 配置驱动优于硬编码

## 二、关键设计决策对比

### 2.1 asset_code 是否要改？

| 做法 | 谁这么干 | 理由 |
|------|---------|------|
| **只改 display_name，code 不动** | 美团、阿里、京东、米哈游、暴雪、Steam — **所有大厂和游戏公司** | code 是系统契约，改 code = 改接口，风险极高 |
| **code 也一起改** | 几乎没有公司这样做 | — |
| **废弃旧 code + 新建 code** | 暴雪(OW Credits→Legacy Credits + 新建 OW Coins)、Roblox(砍 Tix) | 仅限货币体系重构，不是"改名" |

> **行业结论：改 display_name 是唯一主流做法。但你的项目未上线，没有历史数据包袱。**

### 2.2 code 命名风格（大写 vs 小写）

#### 行业调研

| 阵营 | 代表 | DB 存储值示例 | 技术栈背景 |
|------|------|-------------|-----------|
| **UPPER_SNAKE_CASE** | Google Protobuf、PayPal、阿里Java开发手册、京东 | `GOLD_COIN`, `STAR_STONE` | 主要是 **Java 生态** |
| **lower_snake_case** | Stripe、GitHub API、Node.js/Sequelize 社区 | `gold_coin`, `star_stone` | 主要是 **Node.js/Ruby 生态** |
| **数字 ID** | 米哈游(201/202)、腾讯米大师、Steam | `201`, `202` | 游戏引擎 |
| **PascalCase** | Epic(Fortnite)、medici | `Currency:MtxPurchased` | 特殊场景 |

#### 关键发现：大写 vs 小写本质是技术栈之争

- **Java 世界**（阿里/京东/Google）：常量 = UPPER_SNAKE_CASE，这是 Alibaba Java 开发手册的强制规范，被国内大厂广泛采用
- **Node.js 世界**（Stripe/GitHub/Sequelize）：DB 值 = lower_snake_case，JS 代码中的常量键用 UPPER 但值用 lower
- **ISO 4217 货币码**：大写（`USD`, `CNY`），但 Stripe 故意用小写 `usd` 保持 JSON 风格一致性

#### 本项目现状（Node.js + Sequelize + MySQL）

扫描全部 50+ 个 Sequelize 模型，**所有 ENUM 值无一例外全是小写**：

```js
// 状态值
'active', 'inactive', 'pending', 'completed', 'cancelled', 'draft'
// 多词值
'pending_review', 'on_break', 'on_sale', 'auto_approve'
// 资产形态（form ENUM 同步改为 gem）
'shard', 'gem', 'currency', 'quota'
// 系统配置键
'basic/system_name', 'points/lottery_cost_points'
```

代码中的常量已经遵循 **键大写、值小写** 的 Node.js 惯例：

```js
const PRIORITY = { LOW: 'low', MEDIUM: 'medium', HIGH: 'high' }
const AGENT_STATUS = { ACTIVE: 'active', INACTIVE: 'inactive', ON_BREAK: 'on_break' }
```

唯独 `DIAMOND`、`POINTS`、`BUDGET_POINTS`、`DIAMOND_QUOTA` 这 4 个资产 code 是大写 — 这本身就是不一致。

#### 选型结论：lower_snake_case

| 维度 | 选大写 | 选小写 |
|------|-------|-------|
| 和项目现有 50+ 模型 ENUM 一致 | 要改 50+ 个模型 | **完全一致，零改动** |
| 和现有 12 个水晶 code 一致 | 要改 12 个 | **完全一致** |
| 和 Node.js/Stripe 生态一致 | 不一致 | **一致** |
| 和阿里/京东一致 | 一致 | 不一致（但他们是 Java） |
| 需要额外改的文件数 | 50+ 模型 + 全部服务 | **只改 4 个系统资产 code** |

**决定：全部统一为 lower_snake_case，跟着自己的技术栈走。**

### 2.3 code 中"源晶"的英文词选择

| 候选词 | code 示例 | 长度 | 评估 |
|--------|----------|------|------|
| `source` | `red_source_shard` | 17 | 易与 source code 混淆 |
| `origin` | `red_origin_shard` | 17 | 易与 git origin 混淆 |
| `core` | `red_core_shard` | 15 | **最短，无技术术语歧义** |
| `prime` | `red_prime_shard` | 16 | 游戏感强，但语义稍远 |

**决定：使用 `core`（核心/本源），最短且无歧义。**

### 2.4 完整形态用词：`gem` 替代 `crystal`

#### 行业调研

| 用词 | 谁在用 | 和电商撞车风险 |
|------|--------|--------------|
| **gem** | Steam（交易卡→宝石）、Diablo 全系列、Clash of Clans、WoW、Dota 2 — **游戏行业最广泛** | **低** — 电商商品 code 不会用 gem |
| crystal | Summoners War、王者荣耀 | **高** — 水晶饰品/水晶灯/施华洛世奇，电商大品类 |
| orb | Path of Exile、火焰纹章 | 低 — 但不够主流 |
| stone | Pokemon 进化石 | 中 — 且已用于 `star_stone` |

#### 关键发现

- 米哈游内部 code 叫 `Hcoin`/`Scoin`/`Mcoin`，玩家看到的"创世结晶"(crystal) 只是 display_name
- Diablo 的碎片→完整体层级：Chipped → Flawed → Normal → Flawless → **Perfect Gem**
- `shard → gem` 语义配对最自然：碎片(shard) → 完整宝石(gem)
- `gem` 3 个字母，比 `crystal` 短 4 个字符
- 电商商品 code 用具体矿物名（`amethyst`、`topaz`），不会用笼统的 `gem`

**决定：完整形态用 `gem` 替代 `crystal`。form ENUM 同步改为 `'shard'`, `'gem'`, `'currency'`, `'quota'`。**

### 2.4 表结构策略

| 策略 | 使用方 | 适用条件 |
|------|-------|---------|
| **单表 + code 区分** | 美团、阿里、京东 — **压倒性主流** | 资产类型 < 100，业务逻辑统一 |
| 联邦式 | 腾讯(每游戏一套) | 多应用多团队 |
| 混合 | 拼多多 | 可替代和不可替代资产差异大 |

## 四、已确认决策

### 前提条件

- 项目未上线，无历史数据包袱
- 愿意一次性投入成本
- 不需要兼容旧接口
- 目标：长期维护成本最低，技术债务最少

### 4.1 Q1: code + display_name 全改

**决定：code + display_name 全改，彻底改到位。使用 `core` 作为"源晶"的英文前缀。**

选择 `core` 而非 `source`/`origin` 的理由：
- `core` 最短（`red_core_shard` 15字符 vs `red_source_shard` 17字符）
- 语义清晰：core = 核心，和"源晶"的"本源"含义接近
- 不与技术术语撞车（`source` 易混淆 source code，`origin` 易混淆 git origin）

| 考量 | 分析 |
|------|------|
| 行业惯例 | 大厂不改 code，因为有海量历史数据。**但你没上线，这个约束不存在** |
| 长期维护 | 开发者看到 `DIAMOND` 会想到"钻石"而不是"星石"，心智负担永远在 |
| 技术债务 | 现在 code 和名字不一致 = 每个新人都要被教育一次"DIAMOND 其实是星石" |
| 成本 | 现在改：约 6-8 个文件，无数据迁移。上线后改：所有余额表、流水表、转换规则都要迁移 |

### 4.2 Q2: 前端图标改

**决定：改。**

- 图标文件名从 `diamond.png` 改为 `star-stone.png`，水晶类从 `red-crystal.png` 改为 `red-core-gem.png`
- `_mapping.js` 跟着改
- 项目未上线，没有用户认知包袱
- 图标风格可以暂时复用，后续让设计师出新图

### 4.3 Q3: 历史数据不需要迁移

**决定：不需要。**

- 项目未上线，没有真实用户数据
- 种子数据 (seeder) 跟着改就行
- 如果有测试数据在 dev 库里，跑一个迁移脚本更新即可
- **重要原则**：流水记录应该存 asset_code 而不是 display_name（这是所有大厂的共识），这样改名永远不需要迁移历史数据

### 4.4 Q4: DIY 材料不动

**决定：不需要。**

- DIY 材料表 (`diy_materials`) 用的是真实矿物名（巴西黄水晶、粉水晶等），这些就是国标珠宝名称
- 虚拟资产改名后已经完全区分开了："紫源晶" vs "紫水晶"，不会再混淆
- 改真实矿物名反而不专业

---

### 4.5 顺带修复的技术债务

既然要动 asset_code，顺手解决现有的硬编码问题：

| 问题 | 修复方案 | 理由 |
|------|---------|------|
| TradeOrderService 白名单硬编码 | 改为从 `material_asset_types` 表查询，或配置到 `system_settings` | 消除最大的硬编码点 |
| PrizePoolService 对 DIAMOND 的特殊分支 | 在 asset_type 表加字段标识（如 `is_primary_currency`），逻辑改为查表 | 去除业务逻辑对特定 code 的依赖 |
| 大小写不一致 | 统一为 lower_snake_case（详见 2.2 节选型分析） | 和项目整体风格、Node.js 生态一致 |
| 流水表缺 balance_before/after | 交易日志加上余额快照字段 | 符合行业审计标准 |

| 黄色资源 | **黄源晶** | 巴西黄水晶（天然矿石） |

| 语境 | "我账户里有 3 颗紫源晶" | "这条手链用了紫水晶珠" |

**混淆问题彻底消除。**

### 方案对比总结

| 维度 | 方案A: 只改 display_name | 方案B: code + display_name 全改 (已选) |
|------|------------------------|---------------------------------------|
| 改动量 | 1 个种子文件 | ~8 个文件 + 种子 + 迁移脚本 |
| 一次性成本 | 极低 | 中等 |
| 长期心智负担 | **永远存在** — 每个开发者都要记"DIAMOND=星石" | **零** — 代码自解释 |
| 长期维护成本 | 每次涉及资产的代码都需要注释/文档 | 无额外成本 |
| 上线后再改的成本 | 极高（数据迁移、API 兼容、灰度发布） | — |
| 行业对标 | 大厂做法（因为他们已上线） | 大厂如果从零开始也会这样做 |

**结论：项目未上线是最大的优势。一次性彻底改到位，省掉未来无限期的心智税和技术债。**
