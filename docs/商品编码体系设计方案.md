# 商品编码体系设计方案

> 版本：v1.0　|　创建日期：2026-06-29（北京时间）　|　状态：已定稿，待实施
> 适用项目：天工商户营销平台（Node.js + Express + Sequelize + MySQL 8 + Redis）

## 1. 背景与目标

平台商品中心采用 EAV 三层模型（SPU `exchange_items` / SKU `exchange_item_skus` / 属性），目前 SPU 层缺少对外业务编码，仅有自增主键；SKU 层 `sku_code` 现有规则将属性含义编入编码，存在耦合与语义漂移风险。

核心业务诉求：**用户需根据产品手册上印刷的商品编码，到系统中手工搜索定位商品。**

| 目标 | 落地方式 |
|---|---|
| 长期维护成本低、零技术债 | 编码无意义，业务含义全部落在 EAV 属性表，编码永不"说谎" |
| 不耦合类目/属性 | 编码不含任何业务语义，类目或属性变更不影响编码 |
| 防枚举、不泄露经营数据 | 随机生成，竞品无法顺序爬取全量目录或推算上架速度 |
| 适合手册查找（人工读/输） | 全大写、去易混淆字符、分组展示、搜索容错归一化 |
| 零新增技术栈 | 复用现有 `crypto` 随机 + 唯一索引重试范式，不引入雪花/Redis 发号器 |

**核心铁律**：内部关联只认 BIGINT 自增主键；对外标识用随机无意义码；业务含义一律落在属性表，不编进编码本身。

## 2. 方案选型依据（行业对照）

| 标识类型 | 代表 | 形态 | 是否适合本项目 |
|---|---|---|---|
| 内部主键（机器流转） | 阿里 item_id、京东 SKU、腾讯雪花 | 12–19 位长数字 | 否，人无法手输 |
| 含义编码（人工编） | 小公司货号 `CB4385`、活动码 | 4–15 位含义串 | 否，类目变更即语义错误 |
| **对外随机短码（人读/输）** | **亚马逊 ASIN `B0xxxxxxxx`**、YouTube、Stripe | **前缀 + 随机无意义码** | **是，本项目采用** |

结论：本项目商品形态为半标品（手串等），编码需对外展示且供人工手册查找，同时是交易平台需防枚举。唯一同时满足的方案是"前缀 + 随机无意义码 + 全大写无歧义字符集"，与亚马逊 ASIN 范式一致。

## 3. 编码规范定义

### 3.1 字符集（全大写无歧义，30 字符）

```
23456789ABCDEFGHJKMNPQRSTUVWXYZ
```

剔除易混淆字符 `0 O I L 1`，全大写、不区分大小写，与项目现有 `RedemptionCodeGenerator` 字符集完全统一。

### 3.2 编码格式

| 用途 | 字段 | 存储规范形 | 展示形（分组） | 示例 |
|---|---|---|---|---|
| SPU 平台展示码 | `exchange_items.item_code` | `SP` + 12 位随机 | `SP-XXXX-XXXX-XXXX` | `SP-7K9M-Q3RW-X2NV` |
| SKU 平台展示码 | `exchange_item_skus.sku_code` | `SK` + 12 位随机 | `SK-XXXX-XXXX-XXXX` | `SK-4N8P-TX2H-9QYR` |
| 供应商货号 | `exchange_items.supplier_item_code` | 原样存储 | 原样 | `CB4385`、`F-090` |

- 前缀 `SP`/`SK` 仅标识层级（SPU/SKU），层级永不变，不携带业务含义，不违反无意义原则。
- 随机体 12 位 @ 30 字符集 = 30¹² ≈ 5.3×10¹⁷ 空间，撞码概率极低，唯一索引兜底 + 重试。
- 12 位可被 4 整除，分组工整为 4+4+4。

### 3.3 存储 / 展示 / 搜索的三态处理

| 形态 | 用途 | 例子 |
|---|---|---|
| 规范形（存储） | 数据库存储、唯一索引、内部匹配 | `SP7K9MQ3RWX2NV` |
| 展示形（输出） | 前端、产品手册、对账单 | `SP-7K9M-Q3RW-X2NV` |
| 输入归一化（搜索） | 用户输入 → 转大写 + 去分隔符/空格后匹配规范形 | 输入 `sp-7k9m...` / `sp 7k9m...` 均命中 |

### 3.5 多端场景与展示边界

本编码同一套贯穿线下手册与线上系统，但**各端展示策略不同**：

| 端 | 编码展示 | 编码搜索入口 | 说明 |
|---|---|---|---|
| 线下产品手册 | 印刷展示 `SP-XXXX-XXXX-XXXX` | —— | C 端顾客 + 内部人员均使用 |
| 微信小程序（C 端） | **不展示编码字段** | **保留：输编码搜商品** | 编码"只进不出"，作搜索输入但不作展示字段露出（同淘宝隐藏 item_id） |
| 管理后台 | 完整展示 SP/SK 码 | 支持 | 运营管理、导出手册清单 |

落地约束：
1. **小程序 C 端**：商品列表/详情**不渲染** `item_code` 字段；但搜索接口必须接受编码输入（经容错归一化后匹配），支持用户照手册输码定位商品。API 可返回 `item_code` 供搜索回显，但前端不作为商品信息字段展示。
2. **统一展示形**：凡展示编码处（手册、管理后台、对账单）一律输出带连字符展示形 `SP-XXXX-XXXX-XXXX`；数据库统一存无连字符规范形。禁止部分带横线部分不带。
3. **容错归一化**：用户输入无论大小写/横线/空格，搜索接口先归一化再匹配。
4. **手册编码由系统导出**：手册编码由管理后台自动生成并导出（商品名 + SP 码 + 图片清单），杜绝人工誊抄出错。
5. **可选增强**：手册编码旁可附二维码/条形码，支持扫码跳转与手工输入并存。

### 3.4 供应商货号说明

截图中的"商品端码"（`CB4385`/`F-090`）是供货方自有货号，属外部来源：**可重复、可空、不建唯一索引、不参与任何内部关联**，仅作采购对账参考字段。严禁用于系统内部关联或作为主键。

#### 为什么必须把它当"脏数据参考字段"而非标识

供应商货号天然**杂乱、格式不统一、且可能重复**，这正是本方案坚持"内部只认 BIGINT 主键、对外用自生成随机码"的核心理由：

| 现实问题 | 举例 | 本方案的处理 |
|---|---|---|
| 各供应商编码规则各异 | A 厂 `CB4385`、B 厂 `F-090`、C 厂 `2024-春-001` | 原样存 `VARCHAR(100)`，**不强制格式**、不归一化校验 |
| 不同供应商货号可能撞号 | A 厂和 B 厂都叫 `001` | **不建唯一索引**，允许重复；靠内部 `exchange_item_id` 区分 |
| 同一货号被复用/改版 | 旧款下架后新款沿用 `F-090` | 我方 `item_code` 不变不复用，与供应商改版解耦 |
| 可能为空/缺失 | 小作坊无正式货号 | `allowNull`，留空不影响任何流程 |
| 含中文/空格/特殊符号 | `天然-巴西·A级` | 当纯文本存储，不参与匹配/排序键 |

落地约束：
- **只读参考**：仅用于采购对账、与供货方沟通时人工比对，**不参与** SPU/SKU 关联、库存、定价、搜索主链路。
- **不做唯一约束、不做去重**：重复是正常现象，系统不报错、不合并。
- **辅助查询（本次做）**：运营可"按供应商货号找货"，做成**模糊查询且明确提示可能多条命中**（因可重复），结果仍以我方 `item_code`/`item_name` 为准定位。详见 §3.8。
- **供应商维度（本次做，多供应商）**：新增独立 `suppliers` 主数据表 + `exchange_item_suppliers` 关联表（一个 SPU 可挂多个供应商，多源采购）；货号 `supplier_item_code` 落在关联表上（每个供应商对同一 SPU 有各自货号）。详见 §3.8 与 §4.5。

一句话：**供应商货号是"他们的语言"，我方系统只把它当便签贴着看；所有内部逻辑只认我方自己生成的、干净唯一的 `item_code`/`sku_code`。**

### 3.8 供应商维度 + 货号辅助查询（本次新增，多供应商模型）

> 实测：现网 `merchants` 表是"核销商家"（带 `commission_rate`/结算账户，用于券核销），**与"供货商"语义完全不同**，不可复用。故新增独立 `suppliers` 表承载供货维度。
>
> 拍板结论：**一个商品可对应多个供应商**（多源采购），故采用「`suppliers` 主数据 + `exchange_item_suppliers` 多对多关联」，货号挂在关联行上。

数据模型（多对多）：

| 表 | 关键字段 | 说明 |
|---|---|---|
| `suppliers` | `supplier_id` PK、`supplier_name` UNIQUE | 供应商主数据；名称**加唯一约束**防重复建档 |
| `exchange_item_suppliers` | `exchange_item_id` + `supplier_id` + `supplier_item_code` | SPU↔供应商关联行；每行存该供应商对此 SPU 的货号 |

字段约束：
- `exchange_item_suppliers.supplier_item_code`：NULL、**可重复**、不建唯一约束（脏数据如实存），**加普通索引**加速辅助查询。
- `(exchange_item_id, supplier_id)`：建唯一约束（同一 SPU 同一供应商只挂一行，避免重复关联）。
- SPU 不强制挂供应商：自营/未知来源可不建任何关联行（等价"供应商可空"）。
- 可选标记 `is_primary`：多供应商时标主供货商（展示/默认对账用），非必需。

辅助查询能力（管理端）：
1. **按货号模糊查**：`exchange_item_suppliers.supplier_item_code LIKE %kw%`，结果列表标注"共 N 条命中"，每条带 `供应商名 + 我方 item_code + 商品名`，让运营据我方码精确定位。
2. **按供应商筛货**：选定 `supplier_id` 列出其全部供货商品。
3. **组合定位**：`(supplier_id + 货号)` 组合查询，把"不同供应商撞号"区分开。
4. **冲突提示**：当同一供应商对**多个 SPU** 用了相同 `supplier_item_code` 时，列表给非阻断的"⚠️ 该供应商存在重复货号"提示，供人工核对（系统不自动合并/拒绝）。

### 3.6 系列连号编码（双轨制）

业务需求：同一产品系列需要一组**可读、连续**的编码，用于（B）人能看出"某系列第几个"、（C）手册/包装印连号成套、（D）实物批次/防伪连号。

连号与随机互斥（连号必然有序可枚举），故采用**双轨制**：一个商品同时拥有两个编码，各司其职。

| 轨道 | 字段 | 形态 | 是否连号 | 用途 |
|---|---|---|---|---|
| 随机主码 | `item_code` | `SP-XXXX-XXXX-XXXX` 随机 | 否 | 对外标识、系统关联、小程序搜索、防枚举 |
| 可读系列号 | `series_code` + `series_seq` | `SLNB-001` 系列码+连续序号 | 是 | 系列归类、手册/包装成套印刷、实物批次 |

示例：

```
玫瑰石英手串  item_code=SP-7K9M-Q3RW-X2NV  系列号=SLNB-001
黑曜石手串    item_code=SP-4N8P-TX2H-9QYR  系列号=SLNB-002   ← 主码无关联，系列号连续
紫水晶手串    item_code=SP-A3CD-FH5J-K6MN  系列号=SLNB-003
```

设计要点：
- **随机主码不变**：`item_code` 仍随机，防枚举能力不受影响。
- **系列号是含义码 + 连号**：会暴露单系列规模（如 `SLNB-001~050` 即 50 款），但这正是主动印给用户看的成套信息，非机密；主码仍保密。
- **序号在系列内连续**：`series_seq` 按系列独立递增，展示为 `series_code` + 补零（默认 3 位）。
- **序号空洞**：商品下架/删除后序号不回收（保持连续可追溯），允许出现空号。
- 系列归属可空：未归入任何系列的商品 `series_id` 为 NULL，仅有随机主码。

#### 数据模型

新增 `product_series` 表 + `exchange_items` 两个字段：

| 表/字段 | 类型 | 说明 |
|---|---|---|
| `product_series.series_id` | BIGINT PK AI | 系列主键 |
| `product_series.series_code` | VARCHAR(20) UNIQUE | 可读系列码（如 `SLNB`），全大写 |
| `product_series.series_name` | VARCHAR(100) | 系列名称 |
| `product_series.next_seq` | INT | 系列内下一个序号（发号用，事务内自增） |
| `product_series.status` | ENUM(active/inactive) | 状态 |
| `exchange_items.series_id` | BIGINT NULL FK | 所属系列，可空 |
| `exchange_items.series_seq` | INT NULL | 系列内连续序号 |

系列号展示形 = `series_code` + `-` + `series_seq` 补零，如 `SLNB-001`。`(series_id, series_seq)` 建唯一索引保证系列内不重号。

### 3.7 前瞻性扩展字段（现在预留，将来启用）

为避免将来加列改动迁移，本次预留两类外部/实物标识字段，**当前留空不强制使用**：

| 字段 | 所在层 | 类型 | 说明 |
|---|---|---|---|
| `barcode` | SKU `exchange_item_skus` | VARCHAR(20) NULL | 国际标准条码（UPC/EAN-13/GTIN），对接商超/海外/扫码枪。全球唯一标准码，区别于供应商货号 |
| `nfc_uid` | 实物 `items` | VARCHAR(64) NULL | NFC 芯片硬件 UID，实物防伪载体 |
| `anti_fake_code` | 实物 `items` | VARCHAR(32) NULL | 防伪码（可独立于 tracking_code），用于扫码验真 |
| `verify_count` | 实物 `items` | INT NOT NULL DEFAULT 0 | 被验真次数（判断盗刷） |
| `first_verified_at` | 实物 `items` | DATETIME NULL | 首次验真时间 |

设计约束：
- **条码挂 SKU 层**：条码对应具体规格，与国际惯例一致；区别于 `supplier_item_code`（供应商内部货号）。
- **防伪挂实物层**：防伪是"每一件实物"的事，依托已有 `items.tracking_code`（一物一码）。
- **仅预留字段，验真流程为后续路线图**：扫码验真接口、防盗刷规则、NFC 读写为独立子系统，本次不实现，待业务启动时再设计。
- 预留字段全部 NULL/默认值，不影响现有逻辑。

## 4. 数据库变更（Sequelize Migration）

### 4.1 `exchange_items` 新增 `item_code` 列

```sql
ALTER TABLE exchange_items
  ADD COLUMN item_code VARCHAR(14) NULL COMMENT '平台商品展示码(SPU,无意义随机码 SP+12位规范形)' AFTER exchange_item_id,
  ADD UNIQUE INDEX uk_item_code (item_code);
```

- `item_code` 先建为 `NULL`（兼容存量数据未回填），回填完成后可在后续迁移收紧为 `NOT NULL`。
- **供应商货号不在本表加列**：因拍板「多供应商」，货号 `supplier_item_code` 落在 `exchange_item_suppliers` 关联表（见 §4.5），`exchange_items` 不再新增 `supplier_item_code`/`supplier_id`。

### 4.2 `exchange_item_skus.sku_code` 重构

保持 `UNIQUE` 约束不变，列宽由 `VARCHAR(100)` 收窄至 `VARCHAR(14)`，生成规则改为 `SK` + 12 位随机规范形。

同时预留国际条码字段：

```sql
ALTER TABLE exchange_item_skus
  ADD COLUMN barcode VARCHAR(20) NULL COMMENT '国际标准条码(UPC/EAN/GTIN,预留,可空)' AFTER sku_code;
```

### 4.4 实物防伪预留字段 + 批次列（`items`）

```sql
ALTER TABLE items
  ADD COLUMN nfc_uid          VARCHAR(64) NULL COMMENT 'NFC芯片UID(防伪,预留)',
  ADD COLUMN anti_fake_code   VARCHAR(32) NULL COMMENT '防伪码(扫码验真,预留)',
  ADD COLUMN verify_count     INT NOT NULL DEFAULT 0 COMMENT '被验真次数',
  ADD COLUMN first_verified_at DATETIME NULL COMMENT '首次验真时间',
  ADD COLUMN batch_id         BIGINT NULL COMMENT '所属批次(product_batches.batch_id,预留;S2批次管理启用时补FK)';
ALTER TABLE items ADD INDEX idx_items_batch (batch_id);
```

- 防伪字段当前不参与业务逻辑，验真流程为后续路线图（S2/防伪子系统）。
- `batch_id` 本次**一并加列 + 普通索引 + 外键**：因 §14 本次同时建 `product_batches` 表，外键可直接建（`FK→product_batches.batch_id`，`ON DELETE SET NULL`）。当前全部为 NULL，不影响任何逻辑。

### 4.3 系列连号（新增表 + 字段）

```sql
CREATE TABLE product_series (
  series_id   BIGINT NOT NULL AUTO_INCREMENT COMMENT '系列主键',
  series_code VARCHAR(20) NOT NULL COMMENT '可读系列码(全大写,如 SLNB)',
  series_name VARCHAR(100) NOT NULL COMMENT '系列名称',
  next_seq    INT NOT NULL DEFAULT 1 COMMENT '系列内下一个序号(发号用)',
  status      ENUM('active','inactive') NOT NULL DEFAULT 'active',
  created_at  DATETIME NOT NULL, updated_at DATETIME NOT NULL,
  PRIMARY KEY (series_id),
  UNIQUE INDEX uk_series_code (series_code)
) COMMENT='产品系列(连号系列号)';

ALTER TABLE exchange_items
  ADD COLUMN series_id  BIGINT NULL COMMENT '所属系列(product_series.series_id,可空)' AFTER item_code,
  ADD COLUMN series_seq INT NULL COMMENT '系列内连续序号' AFTER series_id,
  ADD UNIQUE INDEX uk_series_seq (series_id, series_seq);
```

- 发号在事务内：读 `product_series.next_seq` → 赋给 `series_seq` → `next_seq+1`，用行锁/分布式锁防并发重号。
- 序号不回收，允许空号，保证系列内连续可追溯。
- `product_series` 增 `seq_pad INT NOT NULL DEFAULT 3`：序号展示补零位数（默认 3 → `001`），个别超 999 的系列可单独调大，无需全局改。

### 4.5 供应商主数据 + 商品多对多关联（本次新增，多供应商）

```sql
CREATE TABLE suppliers (
  supplier_id   BIGINT NOT NULL AUTO_INCREMENT COMMENT '供应商主键',
  supplier_name VARCHAR(100) NOT NULL COMMENT '供应商名称',
  contact_name  VARCHAR(50)  NULL COMMENT '联系人',
  contact_phone VARCHAR(20)  NULL COMMENT '联系电话',
  status        ENUM('active','inactive') NOT NULL DEFAULT 'active' COMMENT '状态',
  notes         VARCHAR(500) NULL COMMENT '备注',
  created_at    DATETIME NOT NULL, updated_at DATETIME NOT NULL,
  PRIMARY KEY (supplier_id),
  UNIQUE INDEX uk_suppliers_name (supplier_name)
) COMMENT='供货商主数据(区别于核销 merchants)';

CREATE TABLE exchange_item_suppliers (
  id                 BIGINT NOT NULL AUTO_INCREMENT COMMENT '关联主键',
  exchange_item_id   BIGINT NOT NULL COMMENT '商品SPU(exchange_items)',
  supplier_id        BIGINT NOT NULL COMMENT '供应商(suppliers)',
  supplier_item_code VARCHAR(100) NULL COMMENT '该供应商对此SPU的原始货号(可空可重复,仅参考)',
  is_primary         TINYINT(1) NOT NULL DEFAULT 0 COMMENT '是否主供货商(展示/默认对账)',
  purchase_price     DECIMAL(10,2) NULL COMMENT '最近进货价(预留,S1采购单启用时维护)',
  quality_score      DECIMAL(3,1)  NULL COMMENT '供货质量评分(预留,0.0~10.0,S1采购评估用)',
  created_at         DATETIME NOT NULL, updated_at DATETIME NOT NULL,
  PRIMARY KEY (id),
  UNIQUE INDEX uk_item_supplier (exchange_item_id, supplier_id),
  INDEX idx_eis_supplier (supplier_id),
  INDEX idx_eis_supplier_code (supplier_item_code)
) COMMENT='商品-供应商多对多关联(货号挂关联行)';
```

- `suppliers` 与现网 `merchants`（核销商家）**互不复用**：语义不同（供货 vs 核销结算）；`supplier_name` **加唯一约束**防重复建档。
- 货号 `supplier_item_code` 落在**关联行**上：同一 SPU 多个供应商各有各的货号；**加普通索引**加速 §3.8 辅助查询；仍非唯一、允许重复、可空。
- `purchase_price`/`quality_score` 本次**一并加列**（按用户要求一次性到位），当前留空不参与逻辑；S1 采购单/对账（§13.2）启用时再写入维护。
- `(exchange_item_id, supplier_id)` 唯一：同一 SPU 同一供应商只挂一行。
- SPU 不强制挂供应商（自营/未知留空，无关联行即可），等价"供应商可空"。
- **不在 `exchange_items` 上加 `supplier_id` 列**（多供应商场景下单列表达不了多对多），归属全部走关联表。

> 说明（与早前单列方案的差异）：因拍板「一个商品可多供应商」，原 §3.4/§4.5 设想的 `exchange_items.supplier_id + supplier_item_code` 单列方案作废，改为上面的关联表模型。`exchange_items` 本表**不再新增** `supplier_id`/`supplier_item_code` 两列。

## 5. 生成器 `utils/ProductCodeGenerator.js`

与 `RedemptionCodeGenerator` / `TrackingCodeGenerator` 同范式（静态类 + 唯一性重试），职责单一：

| 方法 | 作用 |
|---|---|
| `generate(prefix)` | 生成 `prefix` + 12 位随机规范形 |
| `format(code)` | 规范形 → 4+4+4 分组展示形 |
| `normalize(input)` | 用户输入 → 转大写 + 去分隔符/空格的规范形 |
| `validate(code, prefix)` | 校验格式 `^PREFIX[CHARS]{12}$` |
| `generateUnique(prefix, checkUnique, maxRetries=5)` | 带唯一性回调重试，5 次冲突抛错（显式失败，不静默兜底） |

要点：
- 使用 `crypto.randomInt`（非 `Math.random`），密码学随机不可预测。
- 唯一性校验在事务内执行 `findOne`，撞码重试；超限抛错，符合"显式失败"原则。

### 5.1 系列号发号 `utils/SeriesSeqAllocator.js`

| 方法 | 作用 |
|---|---|
| `allocate(seriesId, transaction)` | 事务内分配系列内下一个连续序号（行锁读 `next_seq` → 返回 → `next_seq+1`） |
| `format(seriesCode, seq, pad=3)` | 组装展示形系列号，如 `SLNB-001` |

并发安全：发号必须在事务内对 `product_series` 行加锁（`SELECT ... FOR UPDATE`）或配合现有 `UnifiedDistributedLock`，避免并发重号；`(series_id, series_seq)` 唯一索引兜底。

## 11. 行业对照与拍板建议（含未答的 4 项 + 系列号 2 项 + 5 项决策）

> 实测前置事实（决定方案选型）：后端 `node_modules` **无任何二维码库**；admin **无二维码库**，但已有 `jspdf` / `html2canvas` / `xlsx`；后端已有 `exceljs`。
> 更关键：项目**已有成熟的二维码范式**——消费/核销二维码全部是「后端只产出**签名字符串 payload**（如 `QRV2_{base64}_{signature}`，见 `routes/v4/user/consumption-qrcode.js`、`utils/QRCodeValidator.js`），前端负责把字符串渲染成图」。本方案的商品二维码**必须沿用同一范式**，不另起炉灶。

### 11.1 各类公司怎么做（横向对照）

| 主体 | 商品/物品标识 | 二维码 | 系列/连号 | 对本项目的借鉴 |
|---|---|---|---|---|
| 阿里/京东（大电商） | 内部雪花/自增 item_id（不对外手输）+ 对外 SKU 数字 | 后端出 payload，端侧渲染；跳转走短链 | 无"系列连号"，靠类目+属性聚合 | 主键内部化思路一致；不照搬雪花（你单库无需） |
| 美团/腾讯（券码/核销） | 一物一码、人类可读券码（你 `tracking_code` 即此范式） | **后端签名串 + 端侧渲染**（你现网就是这套） | 批次号连号印刷 | 二维码范式**直接复用现网** |
| 亚马逊（ASIN） | 前缀 `B0`+随机无意义码，对外展示+可搜 | 条码/二维码贴在商品 | 无连号 | **你 `item_code` 的直接母版**（§2 已采纳） |
| 小公司/传统货号 | 人工含义码 `CB4385` | 多为第三方草料等手动生成 | 货号即连号（语义耦合） | 反面教材：含义码会"说谎"，仅留作 `supplier_item_code` 参考 |
| 游戏公司（装备/道具） | 内部 instance_id + 模板 template_id（你 `items`/`item_templates` 同构） | 少用 | **限量编号 `#42/100`**（你 `serial_number/edition_total` 已有！） | 系列连号 = 游戏"限量编号"思路，落 `series_seq` |
| 活动策划/盲盒 | 系列名 + 套内序号（第几款/共几款） | 包装印二维码扫码验真 | **强连号**（成套收集核心卖点） | 你的 `series_code+series_seq` 正是此场景 |
| 游戏虚拟物品交易/小众二手 | 平台短码 + 防伪/溯源码 | 扫码验真 + 转移记录 | 批次/版次连号 | 对应你预留的 `nfc_uid`/`anti_fake_code`（路线图） |

结论：你的项目是「**半标品手串 + 线下手册查找 + 成套系列 + 交易防枚举**」的复合体，**没有任何单一大厂方案能整套照搬**。最贴合的是「**亚马逊 ASIN 随机主码（防枚举+可搜）＋ 游戏限量编号/盲盒连号（成套印刷）双轨制**」——这正是文档 §3.6 已定的方向，本节只是给出每个细节该取哪种流派。

### 11.5 一句话总览：哪种最适合你

**「亚马逊式随机主码（`item_code`/`sku_code` 防枚举可搜）＋ 盲盒/游戏式语义连号（`series_code`+`series_seq` 成套印刷）双轨」＋「美团式后端签名码、端侧渲染的二维码」＋「全部复用现网 `crypto` 生成器/`exceljs`/admin `jspdf` 范式，零新增技术栈」**。
这套在「长期低维护、零技术债、贴合你 Node.js+Sequelize+MySQL+Redis 现状」三个维度上同时最优——主码不含义永不说谎、连号满足成套卖点、二维码不引入新库、所有改动落在你已有的代码范式里。

## 13. 前瞻性业务场景与编码体系演进（2026-06-29 追加，已确认会做）

> 用户确认下列场景**均会落地**。本节明确：每个场景与现有编码体系的关系、需要的新表/字段、以及"编码本身是否要改"。核心结论：**编码体系（`item_code`/`sku_code`/`series_code`/`tracking_code`）全部无需改动，扩展只加表/加字段**——这正是"无意义码 + 含义落表"设计的长期回报。

### 13.1 场景全景与编码影响评估

| # | 未来场景 | 是否动编码 | 落地方式（新表/字段） | 与现有结构的挂接点 |
|---|---|---|---|---|
| S1 | 采购单 / 进货管理 | 否 | 新增 `purchase_orders` + `purchase_order_items`；`exchange_item_suppliers.purchase_price`/`quality_score` 本次已预留 | 挂 `supplier_id` + `exchange_item_id` |
| S2 | 批次管理 / 一批一码 | 否 | 新增 `product_batches`；`items.batch_id` 本次已加列（FK 待 S2 补） | 批次↔SKU/系列号/防伪连号 |
| S3 | 二手回流 / 寄卖 / 转赠 | 否 | 复用 `items` + `item_ledger` 流转；新增寄卖/转赠单据表 | `tracking_code` 一物一码做载体 |
| S4 | 组合商品 / 套装 / 赠品 | 否 | 新增 `product_bundles`（BOM 式组合关系表） | SPU↔SPU/SKU 组合，`item_code` 不变 |
| S5 | 第三方平台分销（淘宝/抖音） | 否 | 新增 `external_channel_mappings`（外部平台 ID↔我方 `item_code`） | `item_code` 作对接唯一键 |

### 13.2 S1 采购单 / 进货单（确认会做）

业务：向多个供应商按批进货，记录进货价、批次、到货时间，支撑按供应商结算对账与供货质量评分。

| 表/字段 | 说明 |
|---|---|
| `purchase_orders` | 采购单头：`supplier_id`、单号、下单/到货时间、总额、状态 |
| `purchase_order_items` | 采购单行：`exchange_item_id`/`sku_id`、数量、`purchase_price`（进货价）、`batch_no` |
| `exchange_item_suppliers`（已预留列） | `purchase_price`、`quality_score` 已在 §4.5 本次一并加列，S1 启用时直接写入维护 |

编码影响：**无**。进货价/批次属"采购域"数据，不进商品对外编码，符合"编码无意义"铁律。
### 13.3 S2 批次管理 / 一批一码（确认会做）

业务：半标品天然"这批料 vs 那批料"，需按批次管成本、按批次召回、按批次做防伪连号。

| 表/字段 | 说明 |
|---|---|
| `product_batches` | 批次主数据：`batch_code`（批次码，可读连号或日期+序号风格）、关联 `exchange_item_id`/`sku_id`、进货来源 `supplier_id`、批次成本、生产/入库日期、数量、状态 |
| `items.batch_id`（本次已加列） | 实物实例归属批次，支撑"按批召回/溯源"；§4.4 本次已加列+索引，S2 建 `product_batches` 时补 FK |
| 与防伪连号挂接 | 批次内可发连号防伪码（复用 §3.7 `anti_fake_code` + 批次前缀），实现"一批一码段" |

编码影响：**无主码改动**。批次码是**采购/生产域的独立编码**，与对外 `item_code` 解耦；可借鉴现有 `tracking_code` 的"前缀+日期+序号"风格（运营可读），不纳入随机码体系。

> 设计提醒：`product_batches` 与 §4.3 `product_series`（系列连号）是**两个正交维度**——系列=营销成套，批次=进货生产。一个商品可同时属于某系列、又分多个批次。不要合并两表。

### 13.4 S3 二手回流 / 寄卖 / 转赠（确认会做）

业务：用户把旧物寄卖回平台再售、实物转赠他人；延续现有竞价（`BidService`）与以物易物（`barter`）的资产流转。

| 机制 | 落地方式 |
|---|---|
| 所有权流转 | 复用现有 `items` + `item_ledger`（一物一码 + 流水账），转赠/寄卖即新增 ledger 流转记录，`owner_account_id` 变更 |
| 寄卖单 | 新增 `consignment_orders`：寄卖物 `item_id`、寄卖人、定价、回流后再上架的目标 SPU |
| 转赠 | 新增轻量转赠记录表，或复用现有资产转移链路 + 幂等键 |

编码影响：**无**。`tracking_code`（一物一码）正是二手流转的天然载体；二手物仍指向原 `item_template_id`/`item_code`，流转只改归属不改身份。

> 风控提醒：二手回流涉及"用户→平台"的反向资产流，需复用现有交易合规校验（方向等价/向下、真实销毁等），上线前单独评估，不在本编码方案范围。

### 13.5 S4 组合商品 / 套装 / 赠品（确认会做）

业务："手串 + 礼盒 + 保养油"打包成套售卖。

| 表/字段 | 说明 |
|---|---|
| `product_bundles` | 组合主体：组合自身也是一个 SPU（有自己的 `item_code`），`bundle_type`（套装/赠品搭售） |
| `product_bundle_items` | 组合明细（BOM）：父 `exchange_item_id` ↔ 子 `exchange_item_id`/`sku_id` + 数量 + 是否赠品 |

编码影响：**无**。组合品本身用标准 `item_code`（`SP` 前缀），明细走关系表；子项仍是独立 SPU/SKU，库存/履约按明细拆解。符合电商主流"虚拟组合 SPU + BOM"做法。

### 13.6 S5 第三方平台分销（淘宝/抖音，确认会做）

业务：把商品铺到外部平台，需要把外部平台商品 ID 映射回我方商品。

| 表/字段 | 说明 |
|---|---|
| `external_channel_mappings` | `channel`（taobao/douyin/...）、`external_item_id`、我方 `exchange_item_id`/`item_code`、同步状态、价格/库存同步策略 |

编码影响：**无，且正是 `item_code` 的用武之地**。`item_code` 作为对外稳定唯一键，天然适合当对接锚点；外部平台各自的 ID 只存映射表，不污染主数据。

### 13.7 总结：编码体系的前瞻性结论

- 上述 5 个确认会做的场景，**没有一个需要修改 `item_code`/`sku_code`/`series_code`/`tracking_code` 的生成规则或格式**。
- 扩展全部是"加表 + 加少量外键列 + 填已预留字段"，与"无意义码 + 含义落 EAV/关联表"的设计哲学完全一致，验证了本方案的**长期低维护、零技术债**目标。
- **拍板更新（2026-06-29）**：用户决定 **S1–S5 所有新表的表结构本次一次性建出来**（空表 + 完整外键），业务逻辑/接口/前端仍分阶段实现。表结构 DDL 见 §14。这样 `items.batch_id` 等外键可一次建全，免将来补 FK 迁移。
- **预留字段本次一并加**：`exchange_item_suppliers.purchase_price`/`quality_score`（§4.5）、`items.batch_id`（§4.4，本次因 §14 同时建 `product_batches`，可**直接建 FK**）。
- 落地顺序（业务逻辑层）建议：S1 采购单 → S2 批次 → S4 组合 → S5 分销 → S3 二手（风控最重，最后做）。表结构虽一次性建好，但各场景的 Service/接口/前端按此顺序逐步填充。

## 14. S1–S5 表结构（本次一次性建空表，2026-06-29 拍板）

> 拍板：S1–S5 所有新表**本次随迁移一次性建出**（结构 + 完整外键），仅建表不写业务逻辑；各场景的 Service/接口/前端按 §13.7 顺序后续填充。
> 外键主键口径以实测真实库为准：`accounts.account_id`、`item_ledger.ledger_entry_id`、`exchange_records.exchange_record_id`、`exchange_items.exchange_item_id`、`exchange_item_skus.sku_id`、`suppliers.supplier_id`。
> 所有表统一 `utf8mb4_unicode_ci`、`created_at/updated_at` 北京时间、BIGINT 主键，与现网风格一致。

### 14.1 S2 批次 `product_batches`（先建，供 items.batch_id 外键引用）

```sql
CREATE TABLE product_batches (
  batch_id         BIGINT NOT NULL AUTO_INCREMENT COMMENT '批次主键',
  batch_code       VARCHAR(32) NOT NULL COMMENT '批次码(可读,前缀+日期+序号风格,运营可读)',
  exchange_item_id BIGINT NULL COMMENT '关联SPU(可空,批次可跨SKU)',
  sku_id           BIGINT NULL COMMENT '关联SKU(可空)',
  supplier_id      BIGINT NULL COMMENT '进货来源供应商',
  batch_cost       DECIMAL(10,2) NULL COMMENT '批次成本(进货价)',
  quantity         INT NOT NULL DEFAULT 0 COMMENT '批次数量',
  produced_at      DATETIME NULL COMMENT '生产/入库日期',
  status           ENUM('active','inactive') NOT NULL DEFAULT 'active',
  created_at       DATETIME NOT NULL, updated_at DATETIME NOT NULL,
  PRIMARY KEY (batch_id),
  UNIQUE INDEX uk_product_batches_code (batch_code),
  INDEX idx_pb_item (exchange_item_id),
  INDEX idx_pb_supplier (supplier_id),
  CONSTRAINT fk_pb_item    FOREIGN KEY (exchange_item_id) REFERENCES exchange_items(exchange_item_id) ON DELETE SET NULL,
  CONSTRAINT fk_pb_sku     FOREIGN KEY (sku_id)           REFERENCES exchange_item_skus(sku_id)       ON DELETE SET NULL,
  CONSTRAINT fk_pb_supplier FOREIGN KEY (supplier_id)     REFERENCES suppliers(supplier_id)            ON DELETE SET NULL
) COMMENT='产品批次(一批一码;批次与系列号正交)';
```

> 迁移顺序：`suppliers` → `product_batches` → `ALTER items ADD batch_id + FK`，保证外键目标已存在。
### 14.2 S1 采购单 `purchase_orders` + `purchase_order_items`
