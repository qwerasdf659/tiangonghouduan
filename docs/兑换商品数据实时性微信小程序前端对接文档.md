## 一、要解决的问题

运营在 Web 管理端下架/删除商品后，小程序里**已经打开的列表页**仍显示旧商品，要等用户「点进去」才发现商品已不存在。

**这不是 bug，也不需要重新编译/发版小程序**。商品数据是后端动态下发的，与小程序代码包无关；问题只是小程序**拉取数据的时机** —— 列表数据是「进页面时拉一次」，停留期间不会自动刷新。

**后端现状（已实测确认，无需改造）**：

- 兑换商品列表 Redis 缓存 TTL 仅 **60 秒**（`BusinessCacheHelper.DEFAULT_TTL.EXCHANGE = 60`）；
- 管理端增删改商品/SKU 时**主动失效缓存**（`invalidateExchangeItems`，全链路 20+ 处调用）。
- 即：数据一旦在管理端变更，后端最迟 60 秒内、通常即时就是最新。**实时性瓶颈在前端拉取时机，不在后端**。

本文档落地两个方案（你已确认采用）：

- **方案一**：列表页 `onShow` + 下拉刷新 重新拉取（解决「回到列表看到最新」）。
- **方案二**：进详情页时校验，商品已下架/不存在则提示并从列表移除（解决「点进去优雅提示而非报错」）。

> ✅ **最终决策（2026-06-22 已拍板）：两个方案一起做，不是二选一。**
> 二者解决的是**不同时刻**的问题、互为补充，缺一会留漏洞：
> - 方案一管「回到列表」时整体刷新（覆盖面广，批量失效商品自动消失）；
> - 方案二管「正好点到某个刚失效商品」时的优雅兜底（避免报错/进到无法兑换的商品）。
> - 若只做方案一：用户停在列表不动、直接点了旧商品，仍会进到失效商品。
> - 若只做方案二：列表上失效商品一直挂着，要逐个点过才消失。
> 两个都做才是完整方案，且均为纯前端逻辑、后端零改动、成本低，契合「长期维护成本低、降技术债」目标。
> **执行优先级**：方案一优先（收益最大），方案二紧接着补齐（安全网）；本次交付要求**两节都实现**。

## 二、涉及的后端接口（真实契约，前端零映射直读）

### 2.1 统一响应格式（所有接口一致）

成功（HTTP 200）：

```json
{
  "success": true,
  "code": "SUCCESS",
  "message": "获取商品列表成功",
  "data": { },
  "timestamp": "2026-06-22 06:00:00",
  "version": "v4.0",
  "request_id": "req_xxx"
}
```

失败（HTTP 4xx/5xx）：

```json
{
  "success": false,
  "code": "EXCHANGE_ITEM_NOT_FOUND",
  "message": "商品不存在",
  "data": {},
  "timestamp": "2026-06-22 06:00:00",
  "version": "v4.0",
  "request_id": "req_xxx"
}
```

> 前端判断成功与否**以 `success` 字段为准**，业务错误类型**以 `code` 字段为准**（不要用 message 文案做逻辑判断）。

### 2.2 商品列表接口（方案一用）

| 项 | 内容 |
| --- | --- |
| 方法/路径 | `GET /api/v4/exchange/items` |
| 鉴权 | optionalAuth（未登录可浏览；登录态带 Access Token 可见会员价/权限数据） |
| 关键 Query | `status`(默认 active) / `space`(lucky/premium) / `item_type`(prop/product/voucher/virtual) / `keyword` / `category_id` / `page`(默认1) / `page_size`(默认20，最大50) / `sort_by`(默认 sort_order) / `sort_order`(ASC/DESC) |
| 强制刷新 | `refresh=true`（跳过后端 60s 缓存，强制读库。**下拉刷新时建议带上**，确保拿到最新） |

响应 `data` 结构：

```json
{
  "items": [
    {
      "exchange_item_id": 123,
      "item_name": "示例商品",
      "status": "active",
      "default_sku_id": 456,
      "primary_image": { "url": "https://.../api/v4/images/products/xxx.jpg" }
    }
  ],
  "pagination": { "total": 50, "page": 1, "page_size": 20 },
  "summary": null
}
```

> 列表只返回 `status=active` 的在售商品（默认）。运营下架/删除的商品，**下一次拉取列表时自然就不在 `items` 里了** —— 这正是方案一生效的原理。

### 2.3 商品详情接口（方案二用）

| 项 | 内容 |
| --- | --- |
| 方法/路径 | `GET /api/v4/exchange/items/:exchange_item_id` |
| 鉴权 | optionalAuth |
| 路径参数 | `exchange_item_id`（商品 ID，正整数） |

响应：成功返回 `{ "item": { ... } }`；商品不存在/已删除时返回**错误响应**：

| 场景 | HTTP | `code` | `message` |
| --- | --- | --- | --- |
| 商品不存在 / 已被删除 | 404 | `EXCHANGE_ITEM_NOT_FOUND` | 商品不存在 |
| 商品 ID 非法（非正整数） | 400 | `BAD_REQUEST` | 无效的商品ID |

> ⚠️ 注意「下架(inactive)」与「删除」的区别：详情接口按 ID 查，**只要记录还在库里就能查到**（含 inactive，返回 `item.status='inactive'`，前端可据此提示「已下架」）；只有记录被**物理删除**才返回 404 `EXCHANGE_ITEM_NOT_FOUND`。前端两种都要处理（见第四节）。

## 三、方案一 — 列表页 onShow + 下拉刷新重新拉取

### 3.1 目标

用户每次「回到列表页」或「下拉」时，重新向后端拉取最新列表，已下架/删除的商品自动消失。

### 3.2 实现要点（小程序端）

1. **页面 `onShow` 触发刷新**：用户从详情页返回、切 Tab 回来、息屏后回到页面，都会触发 `onShow`，在此重新拉第 1 页。
2. **下拉刷新**：`onPullDownRefresh` 中带 `refresh=true` 强制跳过后端缓存，拉最新后 `wx.stopPullDownRefresh()`。
3. **页面 json 开启下拉**：`"enablePullDownRefresh": true`。
4. **节流（可选优化）**：`onShow` 距上次刷新很近（如 < 10 秒）可跳过，避免频繁请求；下拉刷新不节流（用户主动意图）。

### 3.3 参考代码（小程序，仅示意，前端按本项目封装调整）

```javascript
// pages/exchange/exchange.js
Page({
  data: { items: [], page: 1, pageSize: 20, hasMore: true },

  onLoad() { this.refreshList() },

  // 关键①：每次回到页面自动拉最新（解决"返回列表看到最新"）
  onShow() {
    const now = Date.now()
    if (this._lastRefreshAt && now - this._lastRefreshAt < 10000) return // 10s 内不重复
    this.refreshList()
  },

  // 关键②：下拉刷新带 refresh=true 跳过后端 60s 缓存
  onPullDownRefresh() {
    this.refreshList({ refresh: true }).finally(() => wx.stopPullDownRefresh())
  },

  async refreshList(opts = {}) {
    const res = await this.fetchItems({ page: 1, refresh: opts.refresh })
    if (res.success) {
      this.setData({ items: res.data.items, page: 1, hasMore: res.data.items.length >= this.data.pageSize })
      this._lastRefreshAt = Date.now()
    }
  },

  fetchItems({ page = 1, refresh = false }) {
    return new Promise((resolve) => {
      wx.request({
        url: `${getApp().globalData.apiBase}/api/v4/exchange/items`,
        method: 'GET',
        data: { status: 'active', page, page_size: this.data.pageSize, ...(refresh ? { refresh: 'true' } : {}) },
        header: { Authorization: `Bearer ${wx.getStorageSync('accessToken') || ''}` },
        success: (r) => resolve(r.data),
        fail: () => resolve({ success: false })
      })
    })
  }
})
```

```json
// pages/exchange/exchange.json
{ "enablePullDownRefresh": true, "navigationBarTitleText": "兑换商城" }
```

> 说明：`onShow` 不带 `refresh`（走后端 60s 缓存即可，足够新且省请求）；下拉刷新带 `refresh=true`（用户主动要最新，强制读库）。

## 四、方案二 — 进详情页校验，已下架/删除则提示并从列表移除

### 4.1 目标

用户从列表点进某商品时，若该商品已被下架或删除，**优雅提示「该商品已下架」并把它从当前列表里移除**，而不是停留在报错页或展示一个无法兑换的商品。

### 4.2 判定规则（按后端真实返回）

| 详情接口返回 | 含义 | 前端处理 |
| --- | --- | --- |
| `success=true` 且 `data.item.status === 'active'` | 商品正常在售 | 正常进详情页 |
| `success=true` 且 `data.item.status === 'inactive'` | 商品已下架（记录还在） | 提示「该商品已下架」，返回列表并移除该项 |
| `success=false` 且 `code === 'EXCHANGE_ITEM_NOT_FOUND'`（HTTP 404） | 商品已被删除 | 提示「该商品已下架」，返回列表并移除该项 |
| `success=false` 且 `code === 'BAD_REQUEST'`（HTTP 400） | 商品 ID 非法 | 提示「商品信息有误」，不进详情 |

### 4.3 参考代码（小程序，仅示意）

```javascript
// 列表项点击 → 先校验再决定是否进详情
async onTapItem(e) {
  const itemId = e.currentTarget.dataset.id
  const res = await this.fetchItemDetail(itemId)

  // 已删除（404）或已下架（status=inactive）→ 提示 + 从列表移除
  const notFound = !res.success && res.code === 'EXCHANGE_ITEM_NOT_FOUND'
  const inactive = res.success && res.data?.item?.status === 'inactive'
  if (notFound || inactive) {
    wx.showToast({ title: '该商品已下架', icon: 'none' })
    this.removeItemFromList(itemId)   // 本地移除，无需整页刷新
    return
  }
  if (!res.success) {
    wx.showToast({ title: '商品信息有误', icon: 'none' })
    return
  }
  // 正常 → 跳详情页
  wx.navigateTo({ url: `/pages/exchange-detail/exchange-detail?id=${itemId}` })
},

removeItemFromList(itemId) {
  this.setData({ items: this.data.items.filter(it => it.exchange_item_id !== itemId) })
},

fetchItemDetail(itemId) {
  return new Promise((resolve) => {
    wx.request({
      url: `${getApp().globalData.apiBase}/api/v4/exchange/items/${itemId}`,
      method: 'GET',
      header: { Authorization: `Bearer ${wx.getStorageSync('accessToken') || ''}` },
      success: (r) => resolve(r.data),   // r.data 即统一响应体 {success, code, data, ...}
      fail: () => resolve({ success: false, code: 'NETWORK_ERROR' })
    })
  })
}
```

> 详情页 `onLoad` 内同样应做一次校验：若直接通过分享链接/历史记录进入一个已删除商品，按 4.2 同规则提示并 `wx.navigateBack()`。

## 六、联调验证清单（前端自测）

1. 列表页停留 → 管理端下架某商品 → 小程序**下拉刷新** → 该商品消失（带 `refresh=true` 时立即生效，不带时 ≤60s）。
2. 列表页停留 → 进任意页再**返回列表**（触发 `onShow`） → 列表为最新。
3. 列表里点一个「已被管理端删除」的商品 → 提示「该商品已下架」+ 该项从列表移除，不跳详情、不报错。
4. 列表里点一个「已下架(inactive)」的商品 → 同上提示并移除。
5. 通过旧分享链接直接进一个已删除商品详情 → 详情页 `onLoad` 校验 → 提示后自动返回。

## 七、常见疑问

- **Q：以后运营改商品，小程序要重新编译吗？** 不用。商品是后端动态数据，与小程序代码包无关；只有改小程序页面/逻辑才需发版。
- **Q：为什么不用 WebSocket 实时推送商品变化？** 对商品列表属过度设计：列表「拉取式刷新」体验已足够，且后端缓存仅 60s。WebSocket 推送维护成本高、收益低，不推荐为此场景引入。
- **Q：`refresh=true` 会不会增加后端压力？** 仅在用户「主动下拉」时带，频率低；`onShow` 不带 refresh 走 60s 缓存，绝大多数请求命中缓存，压力可控。

## 八、商品多图与轮播能力核查（2026-06-22，待后端+后台修复后小程序对接）

> 追加背景：是否支持类似淘宝天猫——**主图多张轮播**、点进**子规格(SKU)多图轮播**、运营可**调轮播速度**。下述为连真实库 `restaurant_points_dev` + 直读代码（`models/ExchangeItem.js`、`models/ExchangeItemSku.js`、`models/MediaAttachment.js`、`services/exchange/QueryService.js`、`utils/mediaAttachmentGallery.js`、后台 `exchange-items.js`）的核查结论。

### 8.1 核查结论（能力现状）

| 能力 | 现状 | 小程序能否直读 |
| --- | --- | --- |
| 商品单张主图 `primary_image` | ✅ 通（`exchange_items.primary_media_id`） | ✅ 现状可读 |
| 商品主图多张轮播 `images[]` | 🔸 后端详情接口已预留字段，但**写入链断裂**，真实库 0 条商品图 | ⏳ 需修复后才有数据 |
| 详情长图 `detail_images[]` / 展示图 `showcase_images[]` | 🔸 同上，字段已下发但无数据 | ⏳ 同上 |
| 商品视频 `video_url` | ✅ 通（单个视频） | ✅ 现状可读 |
| SKU 子图多张轮播 | ❌ 不支持（`exchange_item_skus` 只有单张 `image_id`） | ❌ 需中等改造 |
| 运营调轮播速度 | ❌ 当前无配置 | ❌ 需新增配置 |

### 8.2 实测发现的 4 处断点（商品多图为何"名存实亡"）

1. **读写 `attachable_type` 不一致**：读图工具 `mediaAttachmentGallery.js` 查 `attachable_type='product'`，后台查图用 `'exchange_item'`，两者对不上 → 存的图前端读不到。
2. **后台上传未落关联**：`uploadDetailImage/uploadShowcaseImage` 只上传到媒体库、**没调 attach 落 `media_attachments`**，图仅存前端本地数组，刷新即丢。真实库 `media_attachments` 中 product/exchange_item 记录 **0 条**，印证此点。
3. **`role` 语义未对齐**：读图按 `role IN ('gallery','detail','showcase')` 分组，后台传的是 `category`，概念未统一。
4. **模型字段名疑误**：真实库列为 `role`，`MediaAttachment.js` init 处注释含 `usage`，需核对纠正避免 ORM 写错列。

> 多态附件表 `media_attachments`（`attachable_type`+`attachable_id`+`role`+`sort_order`）结构本身**支持多图**，无需建新表；断的是"上传→落库→读取"这条链。

### 8.3 修复方案（后端+后台，小程序零改动或按新字段展示）

**方案一：打通商品多图通道（建议优先，修复存量坏链，无表结构变更）**
- 统一 `attachable_type='exchange_item'`（以后端模型注册为准，改读图工具的 `'product'`）。
- 统一 `role`：`gallery`=主图轮播组、`detail`=详情长图、`showcase`=展示图。
- 后台上传成功后补调 attach 接口落 `media_attachments`（含 `sort_order` 排序），排序拖动改写 `sort_order`。
- 纠正 `MediaAttachment` 模型字段名与真实库列 `role` 一致。
- 涉及：后端 `utils/mediaAttachmentGallery.js`、`models/MediaAttachment.js`；后台 `exchange-items.js`。

**方案二：SKU 子图多图轮播（中等改造，复用多态表）**
- `MediaService` 实体注册加 `exchange_item_sku`；SKU 图改挂 `media_attachments`（`role='gallery'`），`image_id` 保留作首图兼容。
- 详情接口为每个 SKU 组装 `images[]`（复用 `fetchProductMediaGallery` 抽出的通用函数）。
- 涉及：后端 `MediaService.js`、`QueryService.js`、`mediaAttachmentGallery.js`；后台 SKU 编辑区。

**方案三：运营调轮播速度（建议选 A 全局配置）**
- 选 A（推荐）：存 `system_settings`（如 `exchange.gallery_autoplay_interval_ms`），后台一个输入框全站生效，复用现有 settings 机制、零表变更。
- 选 B：`exchange_items` 加列 `gallery_interval_ms`（走 migration），按商品配，粒度细但运营逐个配。

### 8.4 修复后小程序对接契约（前端零映射直读，待后端落地后生效）

商品**详情接口** `GET /api/v4/exchange/items/:exchange_item_id` 的 `data.item` 将稳定下发以下多图字段（修复后有数据；无图时为空数组，前端做空兜底）：

{
  "item": {

    "primary_image": { "url": "https://.../xxx.jpg" },

- 主图轮播读 `images[]`（已按 `sort_order` 排序）；详情长图读 `detail_images[]`；视频读 `video_url`。
- SKU 子图轮播（方案二落地后）：每个 `sku` 增加 `images[]`，结构同上，按 SKU 切换轮播。
