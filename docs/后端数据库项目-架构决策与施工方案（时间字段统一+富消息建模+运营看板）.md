## 一、背景与问题

后端时间字段对外（API 响应）存在**两种并存的序列化形态**：

- **形态 A：对象**（`BeijingTimeHelper.formatForAPI()` 产出）
  ```json
  "created_at": { "iso": "2026-06-22T12:38:39.000Z", "beijing": "2026/06/22 20:38:39", "timestamp": 1782131919000, "relative": "2天前" }
  ```
- **形态 B：字符串**（Sequelize Date 原样透传，JSON 序列化为 ISO 串）
  ```json
  "created_at": "2026-06-22T12:38:39.000Z"
  ```

两种形态并存导致：前端每个页面要分别判断"这个接口的时间是对象还是字符串"，读法不一致（对象读 `.beijing`、字符串直接用），是潜在的"时间显示"bug 温床（本次「时间未知」即形态 A 被当字符串读所致）。

## 二、真实现状盘点（连真实代码，量化）

### 2.1 形态 A（对象 formatForAPI）— 少数

实测使用 `formatForAPI()` 包装时间字段对外下发的，集中在：

| 模块 | 文件 | 说明 |
| --- | --- | --- |
| 消费记录 | `models/ConsumptionRecord.js`（`toAPIResponse`）、`routes/v4/shop/consumption/submit.js` | `created_at`/`reviewed_at`/`updated_at` 为对象 |
| 商家风控告警 | `services/MerchantRiskControlService.js`、`services/MerchantRiskAlertService.js`、`routes/v4/console/risk/risk-alerts.js` | 告警 `created_at` 为对象 |
| 部分审核链/门店模型 | `models/ApprovalChain*.js`、`models/StoreStaff.js`、`models/MerchantOperationLog.js` | 局部 `toAPIResponse` 内使用 |

> 注：`utils/timeHelper.js` 的 `formatForAPI(date)` 已做判空（null/空/无效 → 返回 `null`，不再落到 1970）。

### 2.2 形态 B（字符串原样透传）— 绝大多数

实测约 **90+ 个 service/route 文件**直接 `created_at: x.created_at` 原样透传（字符串），覆盖几乎所有业务域，典型且与三端相关的：

| 模块 | 文件 | 时间字段形态 |
| --- | --- | --- |
| **反馈 feedback** | `routes/v4/system/feedback.js:225`、`services/FeedbackService.js` | 字符串（无 toAPIResponse、无 formatForAPI） |
| **申诉 disputes** | `services/TradeDisputeService.js` | 字符串（列表仅 `attributes:['created_at','resolved_at']` 原样返回） |
| **客服会话 chat** | `services/ChatWebSocketService.js:469/1620`、`routes/v4/system/chat.js` | 字符串（`created_at: message.created_at`） |
| 兑换/核销 | `routes/v4/shop/redemption/*.js`、`services/exchange/*` | 字符串 |
| 抽奖/活动/字典/资产/媒体等 | 大量 service/route | 字符串 |

### 2.3 关键：数据脱敏中枢按"字符串"假设处理时间

`services/DataSanitizer.js`（中间件+路由层调用的三端数据脱敏统一出口）对时间字段是**按字符串处理**的，例如：

```javascript
// services/DataSanitizer.js（节选）
sanitized.member_since = sanitized.created_at
  ? (typeof sanitized.created_at === 'string' ? sanitized.created_at.split('T')[0] : ...)
  : null
```

即：**经 DataSanitizer 下发给微信小程序的主链路，时间字段当前以字符串为默认假设。** 这说明形态 B（字符串）才是当前事实上的"主流约定"，形态 A（对象）是局部特例。

### 2.4 结论

**当前后端时间字段未全局统一**：字符串为主（90+ 文件 + DataSanitizer 主链路），对象为少数特例（消费记录、风控告警等）。

| 类别 | 形态 | 实际下发给前端 | 是否处理 | 处理动作 |
| --- | --- | --- | --- | --- |
| **第①类** | 完整对象 `{iso,beijing,timestamp,relative}` | **对象** | ✅ **要改** | 对象 → 字符串 ISO（~17 个文件） |
| **第②类** | `formatForAPI(x).iso` | **ISO 字符串**（已取 `.iso`） | ➖ **不用改** | 已是方案B；如愿可顺手简化为直接传值（纯风格，不影响前端、不改形态） |
| **第③类** | `xxx`(.iso) + `xxx_beijing`(.beijing) 双字段 | 字符串，但**一个时间发两个字段** | ✅ **要改** | 客服会话：归一为单一 ISO 字符串，去掉 `_beijing` 伴随字段 |

> 为什么第②类不用改：它虽然调用了 `formatForAPI(x)`，但**紧跟着 `.iso`**，最终下发的就是一个 ISO 字符串（如 `"2026-06-22T12:38:39.000Z"`），**已经就是方案B的目标形态**，无需任何改动。
>
> 为什么第③类要改：它把一个时间拆成 `created_at`(.iso) + `created_at_beijing`(.beijing) 两个字段下发，是客服会话独有的自定义双字段约定，与全局单一约定不一致；方案B下应归一为单一 ISO 字符串（北京时间展示交前端算）。

**综上，方案B 真正要改的 = 第①类（~17 文件，对象→字符串）+ 第③类（客服会话，去双字段）；第②类不动；DataSanitizer 主链路 + 90+ 普通接口已是字符串，不动。**

> ⚠️ 顺带发现一处潜在 bug（非本决策范围，单独记）：`services/reporting/StatsService.js:345` 用 `formatForAPI(nowBeijing).formatted` —— 但 `formatForAPI` 产出对象**没有 `formatted` 键**（只有 `iso/beijing/timestamp/relative`），该处实际取到 `undefined`。建议改 `.beijing` 或 `.iso`。

## 三、对前端的直接影响（回答小程序前端的求证）

- **不能**按"所有接口时间都是对象"去统一收口 feedback/disputes/chat —— 它们当前是**字符串**，若按对象读 `.beijing` 会得到 `undefined`，反而制造新的"时间未知"。
- 当前正确做法：**消费记录接口**按对象读 `.beijing`/`.relative`；**feedback/disputes/chat 等其余接口**按字符串处理（保持现状）。
- 是否值得让前端"一套助手收口"取决于后端是否决定全局统一（见 §四）。

## 四、三种方案对比（五类公司视角 + 本项目技术栈适配）

> 取向：未上线、可一次性投入、不兼容旧形态、长期维护成本最低、技术债最小、基于现有技术栈（Express + Sequelize + DataSanitizer 脱敏中枢 + BeijingTimeHelper）。

### 方案对比：外部大厂/行业怎么做

| 公司类型 | 时间字段 API 形态 | 特征 |
| --- | --- | --- |
| 大厂（阿里/腾讯/美团） | **统一单一形态**：多为 ISO8601 字符串（UTC）或 epoch 毫秒，**展示格式化交给前端**；后端不下发"已格式化的本地时间串" | 单一、最简、时区由前端按用户本地化 |
| 中小公司/SaaS | 多为字符串 ISO；少数图省事下发后端格式化串 | 字符串为主 |
| 游戏公司 | epoch 时间戳（number）为主，跨时区/客户端本地化 | 时间戳 |
| 活动策划公司 | 后端直接下发展示用字符串（"2026-06-22 20:38"） | 展示串，省前端事 |
| 电商/二手平台 | ISO 字符串 + 前端统一时间组件格式化 | 字符串 + 前端组件 |

> 行业主流 = **单一形态、字符串(ISO)或时间戳**，把"格式化/相对时间/本地化"交给前端组件。**很少有大厂把 `{iso,beijing,timestamp,relative}` 四合一对象作为全局标准**（冗余、放大 payload、本地化写死在后端）。

### 方案 A：全局统一为「对象」`{iso,beijing,timestamp,relative}`

- 做法：所有 API 时间字段经 `formatForAPI()`；最佳落点是 DataSanitizer 脱敏中枢统一拦截转换 + 各 `toAPIResponse`。
- 优点：前端一套读法（`.beijing`/`.relative`），无需各页面判断；相对时间后端算好。
- 缺点：① 改动面巨大（90+ 文件 + DataSanitizer 主链路 + 所有前端页面同步改字符串→对象，含 Web 后台）；② payload 放大（每个时间字段 4 字段）；③ 本地化（北京时间）写死后端，违背"时区交前端"的行业主流；④ 与大厂主流相反。

### 方案 B：全局统一为「字符串」ISO8601（推荐）

- 做法：把现有少数"对象"特例（消费记录、风控告警等）**改回字符串 ISO**，与 90+ 主流 + DataSanitizer 主链路一致；前端用统一时间组件/helper 做展示格式化与相对时间。
- 优点：① 对齐行业主流；② payload 最小；③ 前端一套时间组件收口，本次「时间未知」根治。
- 缺点：相对时间（"2天前"）需前端算（成本极低，前端时间库标配）。
- ⚠️ **重要纠正（见验·二/验·九）**：本节原写"改动面最小、裸传即字符串"是基于 Sequelize 默认行为；但本项目连接配置 `dateStrings:true`，最终定稿的 **B-2（UTC ISO `Z`）实现需改 `dateStrings:false`（全局连接开关）**，从而**改动面并非最小**——要叠加 100+ 时间字段接口全量回归。该段"改动面最小"的旧表述**以验·九为准修正**。

### 方案 C：维持现状（不统一）

- 做法：不动。消费记录对象、其余字符串。
- 优点：零改动。
- 缺点：双形态长期并存，前端每个页面要分辨形态，持续是"时间显示"bug 温床（本次即其一）；违背"代码即文档、单一约定"。

## 五、推荐与定稿

**定稿方案 B（全局统一为字符串 ISO8601，把少数对象特例改回字符串）**，理由：对齐行业主流、payload 最小、DataSanitizer 主链路已是字符串、技术债最低；前端一套时间组件即可收口 feedback/disputes/chat 及所有页面。

> ⚠️ 修正（见验·九）：本段原列"改动面最小"为理由，但最终落地形态为 **B-2（UTC ISO `Z`）**，实现需改全局连接开关 `dateStrings:false`，**改动面并非最小**（叠加 100+ 接口全量回归）。"改动面最小"这一理由不再适用，其余理由仍成立。

> 若你更看重"后端算好相对时间、前端零计算"，可选方案 A，但需接受巨大改动面 + 与主流相反 + payload 放大 —— 不推荐。

### 5.1 三个拍板项 × 五类公司设计逐项对比（结合"愿重投入 + 长期债最小"取向）

#### 拍板项 1：全局时间字段统一形态（对象 / 字符串ISO / 维持现状）

| 公司类型 | 时间字段对外形态 | 设计理由 |
| --- | --- | --- |
| 大厂（美团/腾讯/阿里） | **ISO8601 字符串（UTC）为主**，部分内部系统用 epoch 毫秒；展示与时区本地化全部交前端 | 后端无状态、不耦合展示；同一接口全球客户端按各自时区渲染；payload 最小 |
| 中小公司/SaaS | ISO 字符串为主；少数图省事直接下发"YYYY-MM-DD HH:mm"展示串 | 简单；但下发展示串会埋时区/格式耦合债 |
| 游戏公司 | **epoch 时间戳（number）为主** | 跨区服/客户端本地化、倒计时/排行榜计算最方便 |
| 活动策划公司 | 后端直接下发展示用字符串 | 页面简单、活动短命，省前端格式化 |
| 游戏虚拟物品交易/二手平台 | ISO 字符串 + 前端统一时间组件 | 交易凭证需精确可追溯（ISO 标准），展示交前端组件 |
| **本项目最适合** | **B 字符串 ISO8601**（把少数对象特例改回字符串） | 与大厂/二手主流一致；后端 90+ 接口 + DataSanitizer 中枢已是字符串，改动面最小；payload 最小、不把"北京时间本地化"写死进每个字段；前端一套时间组件收口全部页面，根治双形态 bug |

> 为什么不选"对象 A"：把 `{iso,beijing,timestamp,relative}` 四合一当全局标准是**反主流**做法——它把展示格式（beijing 串）和派生计算（relative "2天前"）固化进后端每个时间字段，payload 放大 4 倍、本地化写死、且要改 90+ 文件 + 全部前端。即便你"愿重投入"，这笔投入买到的是**更高的长期维护成本**（与你的目标相反），故不推荐。

#### 拍板项 2：相对时间（"2天前"）归属（前端算 / 后端额外提供）

| 公司类型 | "X天前"等相对时间在哪算 | 理由 |
| --- | --- | --- |
| 大厂 | **前端算**（day.js/moment/Intl.RelativeTimeFormat） | 相对时间依赖"当前时刻"，后端算的值一离开响应就过时；前端实时算才准 |
| 中小公司 | 前端算 | 同上，前端时间库标配，零额外成本 |
| 游戏公司 | 前端算（基于下发的 timestamp） | 客户端本地实时计算 |
| 活动策划 | 偶有后端算（活动列表"剩余X天"） | 倒计时类才后端算，普通"X天前"仍前端 |
| 二手平台 | 前端算 | "发布于X天前"前端组件统一渲染 |
| **本项目最适合** | **前端算** | 行业惯例；后端算的 relative 一旦缓存/延迟就不准；前端基于 ISO 串用 day.js 实时算，成本极低、最准确 |

> 关键：相对时间是"易腐"的派生值，后端下发后随时间推移就失真。这也是"对象 A"方案里 `relative` 字段的硬伤——它在响应那一刻就开始过时。

#### 拍板项 3：施工范围（仅改对象特例→字符串 / 全量改对象）

| 公司类型 | 统一时间格式时的施工策略 | 理由 |
| --- | --- | --- |
| 大厂 | 向"已是主流的那一种"收敛，改少数离群者 | 改动面最小、回归风险最低 |
| 中小公司 | 同上，挑离群接口改 | 省成本 |
| 游戏/活动/二手 | 统一到既有主流形态，特例归一 | 一致性优先、不为统一而推翻多数 |
| **本项目最适合** | **仅改对象特例→字符串**（消费记录、风控告警等少数处） | 主流（90+ 接口 + DataSanitizer）已是字符串，只需把少数对象特例归一到字符串，相对"全量改对象（方案A）"改动面更小；反向"全量改对象"要动 90+ 文件 + 全部前端，纯属放大技术债 |

> ⚠️ 注（见验·九）：此处"改动面/回归面最小"是**相对方案A（全量改对象）**而言；最终落地的 **B-2** 因需改全局 `dateStrings:false`，**绝对回归面并不小**（100+ 接口全量回归），勿与验·九的施工范围混淆。

> 即便"愿一次性投入大量成本"，正确的成本投向是"把少数离群特例归一到主流"，而非"推翻主流去迁就少数特例"。前者长期债最小，后者是为统一而统一的反模式。

#### 拍板项 4：字符串时区表达（UTC ISO `...Z` / 北京时间串无时区）

| 公司类型 | 时间字符串怎么表达时区 | 理由 |
| --- | --- | --- |
| 大厂（阿里/腾讯/美团/Stripe/GitHub） | **UTC ISO8601 带 `Z`**（`2026-06-22T12:38:39.000Z`） | 无歧义、全球客户端各自本地化；数据库存 UTC、展示本地化是教科书标准 |
| 中小公司 | 多为 UTC ISO；少数图省事发本地串 | 发本地串无时区标识，易被前端当 UTC 再转一次→差8小时 |
| 游戏公司 | epoch 时间戳（隐含 UTC） | 时间戳本身无时区歧义 |
| 活动策划 | 直接发本地展示串 | 页面简单、单一时区，省事但埋时区耦合债 |
| 二手/虚拟物品交易 | UTC ISO + 前端本地化 | 交易凭证需跨端无歧义可追溯 |
| **本项目最适合** | **(a) UTC ISO 带 Z** | 标准、无歧义、对齐大厂；与你"数据库 UTC 存储 + 全链路展示北京时间"规则**完全一致**（存 UTC、前端展示转北京，本就是标准做法）；避免"无时区本地串被前端当 UTC 再转"的差 8 小时坑 |

> 澄清你"全链路北京时间"规则与 (a) 不冲突：标准做法 = **存储/传输用 UTC（无歧义），展示层转北京时间**。`...Z` 的 ISO 串前端 `new Date(str)` 自动按设备时区渲染，配合统一时间组件强制按 `Asia/Shanghai` 展示，即"全链路展示北京时间"。后端发本地串(b)反而违背"传输无歧义"原则。

#### 拍板项 5：第①类对象字段改造手法（裸传模型字段 / 仍包一层 helper）

| 公司类型 | 模型时间字段对外怎么给 | 理由 |
| --- | --- | --- |
| 大厂 | ORM/序列化层自动输出 ISO，**不手动包装** | 框架默认即标准，零封装、零维护 |
| 中小公司 | 多数裸传 ORM 字段 | 省事 |
| 游戏/活动/二手 | 序列化层统一输出，不在业务代码逐字段包 | 一致、不散落 |
| **本项目最适合** | **改 `dateStrings:false` 使裸传产出 UTC ISO + 删第①类 `formatForAPI` 包装** | ⚠️ 纠正：本项目连接配置 `dateStrings:true`，"裸传"实际得到的是**北京墙钟串、非 UTC ISO**（实测见验·二）。B-2 正确手法是先改 `dateStrings:false` 让 mysql2 返回真 `Date`，Sequelize JSON 才输出 UTC ISO `Z`，再删第①类包装裸传；不引入新 helper，但需接受全局连接开关带来的全量回归（见验·九） |

#### 拍板项 6：第③类客服会话去 `_beijing` 双字段（去掉 / 保留）

| 公司类型 | 同一时间会不会发"原始值+已格式化串"两个字段 | 理由 |
| --- | --- | --- |
| 大厂 | **不会**，一个时间一个字段 | 展示格式化是前端职责，后端不发冗余展示串 |
| 中小公司 | 极少数会（图前端省事），属反模式 | 双字段=数据冗余+口径分裂 |
| 游戏/活动/二手 | 不会 | 单一字段 + 前端组件 |
| **本项目最适合** | **去掉 `_beijing` 伴随字段，归一为单一 `created_at`(UTC ISO)** | 双字段是客服会话独有的自定义约定，与全局单一约定冲突；去掉后与全后端一致，前端统一组件展示北京时间。**需同步通知小程序客服页改读 `created_at`（不再读 `created_at_beijing`）** |

#### 拍板项 7：`StatsService.js:345` 的 `.formatted`（取到 undefined）bug（顺手修 / 单独处理）

> 这不是设计选型，是一个确凿的代码 bug（`formatForAPI` 对象无 `formatted` 键，该行 `date` 字段实际下发 `undefined`）。无公司对比。**建议顺手修**（改 `.beijing` 或 `.iso`），与本次时间统一施工一起做掉，避免遗留。

#### 5.1 小结（七项一致取向）

七个拍板项指向同一个最优解：**统一为 UTC ISO8601 字符串（方案 B）+ 裸传模型字段 + 相对时间与北京展示前端算 + 去客服会话双字段 + 顺手修 `.formatted` bug**。它同时满足你"不兼容旧形态、长期维护成本最低、技术债最小、基于现有技术栈（DataSanitizer 中枢与 90+ 接口已是字符串、Sequelize 默认输出 ISO）"的全部取向，且与大厂/二手交易平台主流一致。**这是唯一既"彻底统一"又"改动面与长期债都最小"的方案。**

## 客·一、问题A：文件名显示成一串编码（`tmp_xxx.pdf`）

### 客·1.1 现象

用户在小程序发文件，客服端/小程序端文件卡片显示 `tmp_d6b86e3e70e26de737f81408fd437405dd5d0b466c8c3e21.pdf`，而不是真实文件名 `dzfp_…浙江盛田机械…20260327035147.pdf`。

### 客·1.2 根因（证据来自前端日志 + 数据库真实记录 #22816）

```
微信小程序 wx.chooseMessageFile 选文件
  返回 { name: "dzfp_…真实名.pdf",  path: "…/tmp_xxx.pdf"(临时路径) }
  │  前端日志：拿到了真实名
  ▼
前端发送 POST /api/v4/system/chat/sessions/2197/messages
  │  ❌ file_name 传了临时路径名 tmp_xxx.pdf（应传 name）
  ▼
后端 routes/v4/system/chat.js → CustomerServiceSessionService.sendUserMessage()
  │  忠实存储前端传来的 file_name（后端不加工，正确行为）
  ▼
数据库 chat_messages #22816：file_name="tmp_xxx.pdf"、file_size=106119（同一文件）
```

### 客·1.3 责任归属与修复点

| 项目 | 是否涉及 | 说明 |
| --- | --- | --- |
| **微信小程序前端** | **是（唯一修复点）** | `file_name` 必须取 `wx.chooseMessageFile` 的 `name`（真实名），不要取临时 `path` 的 basename |
| 后端数据库项目 | 否 | `file_name` 透传存储是正确行为；上传接口 `req.file.originalname` 也是 multipart 临时名，不应后端兜底改名 |
| Web 管理后台前端 | 否 | 已按 `message_type` 渲染 `[文件] 文件名`；前端传什么显示什么 |

### 客·1.4 小程序前端具体改法（交前端同事）

- 选文件回调里把 `name`（真实名）与 `path`（临时路径，仅上传用）分开保存。
- 上传用 `path`；消息体 `file_name` 填 `name`。
- 自检：`file_name` 不应以 `tmp_` 开头。

## 客·二、问题B：位置消息显示小空白 — 已定型为「富消息建模重构」

### 客·2.1 现象与根因

用户点「位置」发送后气泡是小空白。直连库核查：`chat_messages.message_type` 实际枚举 = `enum('text','image','system','file')`，**不含 location**。后端不认识位置消息类型，故显示空白。

### 客·2.2 架构决策（已拍板，2026-06-25）

位置消息暴露了「这张聊天表今后如何承载持续增长的富媒体消息类型」的建模问题。业界对比：

| 阵营 | 代表 | 建模方式 |
| --- | --- | --- |
| 大厂 IM 中台 | 腾讯 IM、微信 | type 判别（Text/Image/Location/**Custom**Elem）+ 结构化 payload |
| 平台开放协议 | 微信小程序客服消息 | `msgtype`=text/image/link/**location**，每类配同名对象 |
| 二手/电商社交 | 闲鱼、美团 | 自定义消息 + JSON schema，引用存 id |
| 协作/社区 | Slack、Discord | Block Kit / embeds，消息体即 JSON blocks |
| 游戏频道聊天 | 各游戏公司 | type + extra JSON，物品存 item_id 引用 |
| 小公司/创业 | 通用 | 一个 type + 一个 JSON 列 |

结论：从腾讯到创业公司**无一例外都是「类型判别字段 + 结构化 JSON 负载」**。本项目采用同一范式，结合自身技术栈（Sequelize + MySQL 8 + `system_dictionaries` 字典 + `metadata` JSON 列）落地。

**第一批拍板点（已定）**

| 拍板点 | 结论 | 理由 |
| --- | --- | --- |
| 富内容存储 | 统一 JSON 负载全进 `metadata`；`file_name`/`file_size` 迁入 metadata 后**删列** | 表结构定型，今后加类型零改表 |
| `message_type` 类型 | **`VARCHAR(32)` + 复用 `system_dictionaries` 字典**（替换 ENUM） | 今后加类型=字典插一行，零 DDL |
| 坐标精度 | **原样存经纬度** | 与微信一致，不降精度 |

> `sender_type`/`status`/`message_source` 仍保留 ENUM（封闭稳定小集合）；唯 `message_type` 是开放增长集合，故字典化。

**第二批拍板点（已定）**

| 拍板点 | 结论 | 理由 |
| --- | --- | --- |
| 4. image/location 的 `content` | 保持 NOT NULL，填可读占位（图片 `[图片]`、位置填地址） | 搜索/预览友好 |
| 5. 系统消息归属 | **彻底重构**：`message_type` 只表内容类型（text/image/file/location）；系统消息改由 `message_source='system'` 表达，迁移修正存量 | 修正语义错位，消除冗余双枚举 |
| 6. `metadata` 结构治理 | Service 层按 type 校验固定字段，不引 JSON Schema 框架 | 轻量够用（YAGNI） |
| 7. 消息类型字典 `dict_type` | **沿用 `message_type`**（校准：原写 chat_message_type，真库已有 message_type 字典，见复·七A） | 据实沿用，零改名迁移 |
| 8. 图片消息 URL | 统一进 `metadata.image_url`，`content='[图片]'` | 全富媒体同一套规则 |

> ⚠️ 拍板点5 连带影响：`message_type` 合法值最终为 text/image/file/location（不含 system）；存量 `message_type='system'` 行迁移为 `message_type='text'`+`message_source='system'`；所有"判断系统消息"代码改判 `message_source==='system'`；校验复用 `SystemDictionary.findOne({ dict_type:'message_type', dict_code, is_enabled:true })`（字段名 `dict_name`）。

### 客·2.3 数据契约（前后端必须一致）

```jsonc

  "content": "浙江省杭州市西湖区xx路1号",        // 人类可读地址

// 文件消息（重构后：file_name/file_size 进 metadata，无专属列）

  "content": "[图片]",
```

- **客·4.1 现状**：`CustomerServiceSessionService.searchMessages` 已有——按 user_id 圈定会话隔离，对 content 做 `LIKE` 模糊匹配分页。搜索框架已存在。

- **客·4.3 演进路径**（基于现有技术栈，不引新债）：

| 阶段 | 数据量 | 方案 |

## 验·二、⚠️ 重大纠偏：拍板项 5「裸传即 UTC ISO」与真实运行结果相反（必须重新拍板）

文档 §五 拍板项 5 + 复·八 的核心假设是：**"删掉 `formatForAPI()` 包装，让 Sequelize 字段裸传，JSON 默认即 UTC ISO `...Z`"**。这是整个"方案 B = UTC ISO"成立的技术前提。**实测证明该前提不成立。**

### 真实证据（直连库 + 项目 Sequelize 实例实测）

`config/database.js` 的连接配置里有两项决定性设置：

```81:110:config/database.js
  timezone: '+08:00', // 全系统统一使用北京时间
  ...
  dialectOptions: {
    ...
    dateStrings: true,   // ← 关键
    typeCast: true,
    ...
    timezone: '+08:00'   // ← 关键
  }
```

在 `dateStrings: true` + `timezone:'+08:00'` 下，**Sequelize 对 DATETIME 列裸传出来的根本不是 `Date` 对象、也不是 UTC ISO，而是「北京墙钟字符串、且不带任何时区后缀」**：

| 取值路径 | 实测真实返回值 |
| --- | --- |
| `sequelize.query('SELECT created_at ...')` 原生查询 | 字符串 **`"2026-06-25 00:56:58"`**（北京墙钟，无 Z/无 +08:00） |
| `ChatMessage.findOne().toJSON().createdAt` | 字符串 **`"2026-06-25 00:56:58"`** |
| `ConsumptionRecord.getDataValue('created_at')` | 字符串 **`"2026-06-25 03:14:09"`** |
| 对照：`new Date().toISOString()` | `"2026-06-24T20:06:16.720Z"`（这才是 UTC ISO） |

也就是说：**"裸传 Sequelize 字段"得到的是形态 (b)「北京时间串、无时区」，恰恰是拍板项 4 明确否决的那一种**，而不是拍板项 4/5 选定的 (a)「UTC ISO `...Z`」。文档 §五拍板项 5 表格里写的"Sequelize 的 DATE 字段 JSON 序列化默认就是 UTC ISO 串"——这条结论在**默认配置**下成立，但在**本项目实际配置（`dateStrings:true`）**下不成立。

### 连带纠正：本次「审批管理页/审核记录时间差 8 小时」的真正根因

复·五把"差 8 小时"归因为"小程序读了 `.iso`(UTC) 而非 `.beijing`"。这只是表象之一。更底层的根因是：**全库时间以"北京墙钟"存储（非 UTC 存储），而 `formatForAPI().iso` 用 `new Date(beijingWallClock).toISOString()` 把这串北京墙钟又当成本地时区解析了一次**——

- 进程 `process.env.TZ='Asia/Shanghai'`（`app.js:12` 强制设定，已实测确认），
- `new Date("2026-06-25 03:14:09")` 被当作"北京时间 03:14"解析 → `.toISOString()` 减 8 小时 → `"2026-06-24T19:14:09.000Z"`，
- 前端拿到这个 `.iso` 再 `new Date()` 按本地（北京）渲染 → 显示 **19:14**，与真实的 03:14 **差了 8 小时（少 8 小时）**。

> 实测对照：对同一个库里的字符串 `"2026-06-25 03:14:09"`，`formatForAPI().iso` 在 TZ=Asia/Shanghai 下产出 `2026-06-24T19:14:09.000Z`（错，少 8h），`.beijing` 产出 `2026/06/25 03:14:09`（对）。这解释了为何"读 `.beijing` 是对的、读 `.iso` 就差 8 小时"。

**这意味着：问题的本质不是"对象 vs 字符串"，而是"存储用了北京墙钟 + `.iso` 推导错误"。** 文档把焦点放在"统一成 UTC ISO 字符串"，但**只要底层仍是"北京墙钟存储"，无论裸传还是包 helper，要么拿到带歧义的无时区北京串，要么拿到被 `.iso` 错误推导的 UTC 串。**

## 验·8.1 三个候选形态先讲清区别（同一时刻 `2026-06-25 03:14:09` 北京）

| 候选 | API 下发的字符串 | 是否带时区标识 | 前端 `new Date()` 解析 | 会不会差 8 小时 | 要不要动连接配置 |
| --- | --- | --- | --- | --- | --- |
| **B-1** 北京墙钟串（裸传现状） | `"2026-06-25 03:14:09"` | ❌ 无 | 各端按"本地时区"猜，iOS/部分浏览器会判 NaN 或当 UTC | ⚠️ 跨端有风险（无时区标识是隐患） | 否（裸传即得） |
| **B-2** UTC ISO | `"2026-06-24T19:14:09.000Z"` | ✅ UTC `Z` | 全端一致、自动转本地 | 否（标准做法） | **是**（须改 `dateStrings:false`，全局回归） |
| **B-0** 北京 ISO 带偏移 | `"2026-06-25T03:14:09.000+08:00"` | ✅ `+08:00` | 全端一致、自动转本地 | 否 | 否（用现成 `apiTimestamp()`） |

## 验·8.2 五类公司怎么设计「时间字段对外形态」

| 公司类型 | 主流做法 | 设计理由 | 映射到本项目的候选 |
| --- | --- | --- | --- |
| **大厂（美团/腾讯/阿里/字节）** | **UTC ISO8601 带 `Z`** 为主（部分内部系统用 epoch 毫秒）；展示与时区本地化全部交前端 | 后端无状态、不耦合展示；同一接口全球客户端各自渲染；存 UTC 是数据层铁律 | **B-2** |
| **小公司 / SaaS** | 多为 UTC ISO；少数图省事直接发"YYYY-MM-DD HH:mm"本地串 | ISO 简单通用；发无时区本地串会埋"被前端当 UTC 再转一次→差 8h"的债 | B-2 或（偷懒的）B-1 |
| **游戏公司** | **epoch 时间戳（number）** 为主 | 跨区服/客户端本地化、倒计时/排行榜计算最方便；时间戳天然无时区歧义 | （本项目时间字段非倒计时型，时间戳收益低，不单列） |

## 验·8.4 我的建议：首选 B-0，其次 B-2，不建议 B-1

**给你的项目最适合的是 B-0（北京 ISO 带 `+08:00`）**，理由：

1. **无歧义、根治差 8h**：显式 `+08:00`，任何端 `new Date()` 都得到正确绝对时刻，本次"审批页差 8h"这类问题从形态上根除。
2. **改动面可控、不碰全局连接开关**：只在业务响应层把时间统一过 `timeHelper.apiTimestamp()/_toBeijingISO()`（已存在，零新依赖），不动 `dateStrings`，**不会触发"全库时间字段形态突变 + 100+ 接口回归"的 B-2 级风暴**。
3. **与你"全链路北京时间"规则最贴合**：你的存储就是北京时刻，B-0 原样带偏移下发，语义最直；B-2 要在"概念上"把全链路改成"存传 UTC、展示转北京"，理念正确但与现状落差更大。
4. **前端可瘦身**：admin `formatDate` 可删对象/中文/纯DB三个兼容分支，只留 ISO 一条路；小程序直接 `new Date(带+08:00串)` 渲染即对。
5. **长期债低、对齐"带时区标识"的行业正道**（二手/交易平台路线），不落入 B-1 的无时区隐患。

**什么情况下改选 B-2**：如果你的硬指标是"必须和大厂完全一样、对外契约就是 UTC `Z`"（比如未来要对接第三方、开放 API、或多时区出海），那就选 B-2，并接受"改 `dateStrings:false` + 全量时间字段回归"这笔一次性重投入——你说了愿意一次性投入，这笔投入买到的是"最标准、最不用解释"的对外契约。

**为什么不建议 B-1**：它唯一优点是"裸传零改动"，但"无时区标识"是长期隐患（与你"技术债最小"取向直接冲突），且前端兼容分支永远删不掉——属于"省了眼前、欠了长期"，与你的取向相反。
