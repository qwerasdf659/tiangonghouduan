# 奖品管理架构决策 — 集中奖品目录方案分析

> 日期：2026-05-24
> 状态：待拍板
> 决策人：产品负责人

---

## 一、问题描述

当前抽奖系统的奖品配置是**每活动独立配置**（模式 A），每个 `lottery_campaigns` 下的 `lottery_prizes` 都是独立的行，没有集中的奖品目录。

这导致：
1. 同一个奖品在不同活动中价值可能不一致（运营手动填写，容易出错）
2. 新建活动要重新填写所有奖品信息（重复劳动）
3. 修改奖品图片/名称要改每个活动（维护成本高）
4. 无法统计"某个奖品总共发了多少"（数据分析困难）

**提议**：建立集中奖品目录（prize_definitions），活动只引用奖品 + 配置概率/库存。

## 三、当前架构问题分析

```
当前架构（模式 A）：

lottery_campaigns (活动1)
  └── lottery_prizes (奖品A: 星石500, budget_cost=100)
  └── lottery_prizes (奖品B: 红源晶碎片50, budget_cost=30)

lottery_campaigns (活动2)
  └── lottery_prizes (奖品A: 星石500, budget_cost=120)  ← 同一个奖品，不同价值！
  └── lottery_prizes (奖品B: 红源晶碎片50, budget_cost=25)  ← 又不一样！

问题：
1. 同一个奖品在不同活动中价值不一致 → 用户困惑、运营混乱
2. 新建活动要重新填写所有奖品信息 → 重复劳动、容易出错
3. 修改奖品图片/名称要改每个活动 → 维护成本高
4. 无法统计"某个奖品总共发了多少" → 数据分析困难
```

## 四、推荐方案：模式 B（集中奖品目录 + 活动引用）

```
推荐架构：

prize_definitions (奖品目录 — 全局唯一真相源)
  ├── prize_definition_id (PK)
  ├── prize_code (业务码，如 "star_stone_500")
  ├── display_name ("星石 ×500")
  ├── prize_type (material / item / coupon)
  ├── material_asset_code ("star_stone")
  ├── material_amount (500)
  ├── budget_cost (100)          ← 统一预算成本
  ├── visible_value_points (50)  ← 统一展示价值
  ├── rarity_code ("rare")
  ├── primary_media_id (图片)
  ├── is_enabled
  └── ...

lottery_campaigns (活动)
  └── lottery_campaign_prizes (活动-奖品关联表)
        ├── lottery_campaign_id (FK → 活动)
        ├── prize_definition_id (FK → 奖品目录)  ← 引用，不复制
        ├── weight (本活动中的权重/概率)
        ├── stock (本活动中的库存)
        ├── tier (本活动中的档位)
        └── sort_order
```

### 好处

1. 同一个奖品全局价值统一 → 不会出现"同奖品不同价"
2. 新建活动只需选择奖品 + 配置概率/库存 → 效率提升
3. 修改奖品信息一处改全局生效 → 维护成本降到最低
4. 可以统计"某奖品跨活动总发放量" → 数据分析能力

## 六、方案区别总结

| 维度 | 模式 A（当前） | 模式 B（推荐） |
|------|---------------|---------------|
| 奖品定义 | 每活动重复填写 | 全局一次定义 |
| 价值一致性 | ❌ 无法保证 | ✅ 强制统一 |
| 新建活动效率 | 慢（重新填写） | 快（选择+配概率） |
| 修改奖品信息 | 改N个活动 | 改1处全局生效 |
| 跨活动统计 | 困难 | 天然支持 |
| 技术债务 | 高（数据冗余） | 低（单一真相源） |
| 实现成本 | 已有 | 中等（新增1张表 + 改关联） |
| 长期维护成本 | 高 | ⭐最低 |

## 七、建议结论

**建立奖品目录（prize_definitions）是正确的决策。** 理由：

1. 项目未上线，没有历史包袱
2. 兑换系统已经是集中目录模式，抽奖系统应该对齐
3. 已经有 `material_asset_types` 和 `item_templates` 作为底层目录
4. 所有成熟的抽奖/游戏系统都是这么做的
5. 长期维护成本最低，运营效率最高

## 八、决策结果（已拍板）

### 决策点 1：奖品目录是否允许"活动级覆盖"？

**选定：A. 严格模式**

活动只能引用奖品目录，不能覆盖价值。保证全局一致性。

活动级别只能配置：
- `weight`（本活动中的权重/概率）
- `stock`（本活动中的库存）
- `tier`（本活动中的档位）
- `sort_order`（排序）

活动级别**不能**配置：
- 奖品名称、图片、稀有度、价值、预算成本（全部从 prize_definitions 读取）

### 决策点 2：budget_cost 如何确定？

**选定：A. 自动计算**

```
budget_cost = material_amount × material_asset_types.budget_value_points
```

奖品目录中不存储 `budget_cost` 字段，而是在抽奖引擎运行时实时计算。这样当 `material_asset_types.budget_value_points` 调整时，所有引用该材料的奖品自动同步。

### 决策点 3：现有 lottery_prizes 表的数据如何处理？

**选定：A. 迁移为 prize_definitions + 关联表**

一次性重构，将现有 `lottery_prizes` 数据迁移为：
- 去重后的奖品定义 → `prize_definitions` 表
- 活动与奖品的关联 → `lottery_campaign_prizes` 关联表

旧 `lottery_prizes` 表在迁移完成后删除，不保留兼容代码。

### 决策点 4：`reward_tier` 是否允许活动级覆盖？

**选定：A. 允许覆盖**

`prize_definitions` 存默认档位，`lottery_campaign_prizes` 可覆盖。同一奖品在不同活动可以放不同档位。

理由：与现有 `tier_first` + `lottery_tier_rules` 设计一致，零额外成本。美团/腾讯/米哈游全部是允许覆盖模式。

### 决策点 5：旧 `lottery_prizes` 表何时删除？

**选定：B. 迁移完立即删除**

项目未上线，没有线上流量。保留旧表 = 维护两套查询逻辑 = 技术债务。旧表只有 12 条数据，迁移后人工核对即可确认正确性。

### 决策点 6：`prize_code` 命名规范？

**选定：A. 结构化命名 `{material_asset_code}_{amount}`**

示例：`star_stone_500`、`red_core_shard_50`、`points_10`

与现有 `material_asset_types.asset_code` 风格一致（snake_case），迁移脚本可自动生成。物品类奖品使用 `item_{template_code}` 格式。

### 决策点 7：物品类奖品（item_template）是否纳入本次？

**选定：A. 本次一起做**

理由：
- 项目愿意一次性投入，不需要分期
- `item_templates` 表已有 20+ 条数据，业务上已有物品类奖品需求
- 增量代码 < 40 行，避免未来破坏性变更
- 表结构已预留 `item_template_id` 字段

## 十、实施方案（基于后端数据库项目实际技术体系）

### 10.1 数据库层变更

#### 新增表：`prize_definitions`（奖品目录）

```sql
CREATE TABLE prize_definitions (
  prize_definition_id BIGINT AUTO_INCREMENT PRIMARY KEY,
  prize_code VARCHAR(80) NOT NULL UNIQUE COMMENT '业务码，如 star_stone_500',
  display_name VARCHAR(100) NOT NULL COMMENT '展示名称，如 星石 ×500',
  prize_type ENUM('material','item','coupon','points') NOT NULL COMMENT '奖品类型',
  material_asset_code VARCHAR(32) DEFAULT NULL COMMENT 'FK→material_asset_types.asset_code',
  material_amount BIGINT DEFAULT NULL COMMENT '材料数量',
  item_template_id BIGINT DEFAULT NULL COMMENT 'FK→item_templates.item_template_id（物品类奖品）',
  rarity_code VARCHAR(50) NOT NULL DEFAULT 'common' COMMENT 'FK→rarity_defs.rarity_code',
  primary_media_id BIGINT UNSIGNED DEFAULT NULL COMMENT 'FK→media_files',
  reward_tier ENUM('high','mid','low') NOT NULL DEFAULT 'low' COMMENT '默认档位',
  is_enabled TINYINT(1) NOT NULL DEFAULT 1,
  description TEXT DEFAULT NULL,
  merchant_id INT DEFAULT NULL COMMENT 'FK→merchants（多商家隔离）',
  meta JSON DEFAULT NULL COMMENT '扩展字段',
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  deleted_at DATETIME DEFAULT NULL,
  INDEX idx_prize_type (prize_type),
  INDEX idx_rarity_code (rarity_code),
  INDEX idx_material_asset_code (material_asset_code),
  INDEX idx_is_enabled (is_enabled),
  INDEX idx_merchant_id (merchant_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### 新增表：`lottery_campaign_prizes`（活动-奖品关联表）

```sql
CREATE TABLE lottery_campaign_prizes (
  lottery_campaign_prize_id BIGINT AUTO_INCREMENT PRIMARY KEY,
  lottery_campaign_id INT NOT NULL COMMENT 'FK→lottery_campaigns',
  prize_definition_id BIGINT NOT NULL COMMENT 'FK→prize_definitions',
  win_weight INT UNSIGNED NOT NULL DEFAULT 0 COMMENT '本活动中的权重',
  stock_quantity INT NOT NULL DEFAULT 999999 COMMENT '本活动中的库存',
  reward_tier ENUM('high','mid','low') NOT NULL DEFAULT 'low' COMMENT '本活动中的档位（可覆盖默认）',
  is_fallback TINYINT(1) NOT NULL DEFAULT 0 COMMENT '是否兜底奖品',
  sort_order INT NOT NULL DEFAULT 100,
  status ENUM('active','inactive') NOT NULL DEFAULT 'active',
  max_daily_wins INT DEFAULT NULL,
  max_user_wins INT DEFAULT NULL,
  total_win_count INT NOT NULL DEFAULT 0,
  daily_win_count INT NOT NULL DEFAULT 0,
  created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  UNIQUE KEY uk_campaign_prize (lottery_campaign_id, prize_definition_id),
  INDEX idx_campaign_status (lottery_campaign_id, status),
  INDEX idx_prize_definition (prize_definition_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### 设计说明

- `budget_cost` **不存储**在 `prize_definitions` 中，运行时通过 `material_asset_types.budget_value_points × material_amount` 实时计算（已拍板决策点 2）
- `visible_value_points` 同理，通过 `material_asset_types.visible_value_points × material_amount` 实时计算
- `reward_tier` 在 `prize_definitions` 中是默认值，`lottery_campaign_prizes` 中可覆盖（活动可调整档位）
- `lottery_campaign_prizes` 保留 `total_win_count` / `daily_win_count` 用于活动级统计（与现有 `lottery_prizes` 行为一致）

| `BuildPrizePoolStage` | `budget_cost` 改为实时计算（已有逻辑，只需改数据源） |
| `PrizePoolFacade` | 废弃，改为调用 `lottery_campaign_prizes` 关联表操作 |

#### 抽奖引擎改造要点

`LoadCampaignStage._loadPrizes()` 当前直接查 `LotteryPrize`：

```javascript
// 当前

// 改为
await LotteryCampaignPrize.findAll({
  where: { lottery_campaign_id, status: 'active' },
  include: [{
    model: PrizeDefinition,
    include: [{ model: MaterialAssetType }]
  }]
})
```

返回的数据结构保持兼容（通过 Sequelize getter 或 toJSON 转换），`BuildPrizePoolStage` 读取的字段名不变。

## 十四、需要拍板的决策点（含业界对比分析）

### 决策点 4：`reward_tier` 是否允许活动级覆盖？

#### 业界做法

| 公司/场景 | 做法 | 理由 |
|-----------|------|------|
| 美团（红包雨/抽奖） | 允许覆盖 | 同一张 50 元券，日常活动是 high 档（稀有），618 大促是 mid 档（常见） |
| 腾讯游戏（王者荣耀） | 允许覆盖 | 同一个皮肤，限定池是 UP（高概率），常驻池是低概率 |
| 米哈游（原神祈愿） | 允许覆盖 | 5 星角色在 UP 池是"限定 UP"，在常驻池是普通 5 星 |
| 网易（阴阳师） | 允许覆盖 | SSR 式神在限定召唤中概率 UP，普通召唤中正常概率 |
| 阿里（支付宝集五福） | 不允许覆盖 | 福卡稀有度全局固定（敬业福永远最稀有） |
| 小公司/一次性活动 | 无此概念 | 每活动独立配置 |

#### 区别本质

- 允许覆盖 = "奖品价值固定，但在不同活动中的定位/稀缺度可以不同"
- 不允许覆盖 = "奖品定位全局统一，运营不能调"

#### 最终建议：选 A（允许覆盖）

理由：
1. 项目有 `tier_first` 选奖模式，档位直接决定概率分配。同一个"星石 ×200"在新手池可能是 high 档，在高级池是 mid 档
2. 已有 `lottery_tier_rules` 表按活动配置档位概率，系统设计就是"活动决定概率分配"
3. 允许覆盖不增加任何代码复杂度（只是关联表多一个字段），不允许覆盖未来要加回来就是破坏性变更
4. 零额外成本，最大灵活性

### 决策点 5：旧 `lottery_prizes` 表何时删除？

#### 业界做法

| 公司/场景 | 做法 | 理由 |
|-----------|------|------|
| 美团/阿里（大厂线上迁移） | 灰度切流 + 双写 + 验证 → 删旧表 | 线上流量大，不能出错 |
| 腾讯游戏（版本更新） | 停服维护 → 一次性迁移 → 删旧表 | 游戏可以停服 |
| 创业公司（未上线项目） | 迁移完直接删 | 没有线上流量，没有回滚需求 |
| 米哈游（原神版本更新） | 一次性迁移，不保留旧结构 | 客户端强制更新 |

#### 最终建议：选 B（迁移完立即删除）

理由：
1. **项目未上线**，没有线上流量，没有"回滚到旧版本"的场景
2. 保留旧表 = 代码中必须维护两套查询逻辑（旧 `LotteryPrize` model 和新 `LotteryCampaignPrize` model），这本身就是技术债务
3. 你明确说了"不要兼容旧接口，从长期维护成本低的角度出发"
4. 旧表只有 12 条数据（1 个活动），迁移验证 5 分钟就能确认正确性
5. 保留旧表的唯一价值是"万一迁移出错可以回滚"，但 12 条数据手动重建也就 2 分钟

### 决策点 6：`prize_code` 命名规范？

#### 业界做法

| 公司/场景 | 做法 | 示例 | 理由 |
|-----------|------|------|------|
| 美团（权益中心） | 结构化命名 | `coupon_50_off`、`redpacket_5_yuan` | 代码可解析，便于自动化 |
| 腾讯游戏（道具系统） | 结构化命名 | `skin_hero_001_s01`、`gem_blue_100` | 道具数量大，需要规则化管理 |
| 米哈游（原神） | 结构化 code | `avatar_ambor`、`weapon_bow_001` | 全局唯一，可读性强 |
| 阿里（商品 SPU） | 自由命名 | 商家自定义 SKU 编码 | C2C 场景，平台不强制 |
| Steam（虚拟物品交易） | 结构化 | `app_730_item_ak47_redline` | 需要跨系统引用 |
| 闲鱼/转转（二手平台） | 无强制 code | 用户自定义标题 | C2C 场景 |
| 小公司 | 无 code 或自增 ID | 只用数字 ID | 数据量小 |

#### 区别本质

- 结构化命名 = 代码可解析、可自动生成、可批量操作、可做唯一性校验
- 自由命名 = 灵活但容易重复、无法自动化、后期难以管理

#### 最终建议：选 A（`{material}_{amount}` 结构化命名）

理由：
1. 奖品 100% 是材料类，`material_asset_code` + `material_amount` 天然构成唯一标识
2. 已有 `material_asset_types.asset_code` 用 snake_case（`star_stone`、`red_core_shard`），风格统一
3. 迁移脚本可自动生成 `prize_code`，不需要人工填写
4. 未来物品类奖品可用 `item_{template_code}` 格式扩展
5. 长期维护：结构化命名支持 `WHERE prize_code LIKE 'star_stone_%'` 批量查询

### 决策点 7：物品类奖品（item_template）是否纳入本次？

#### 业界做法

| 公司/场景 | 做法 | 理由 |
|-----------|------|------|
| 美团（抽奖） | 一次性全做 | 奖品类型多（红包/券/实物/积分），必须统一 |
| 腾讯游戏 | 一次性全做 | 道具类型多（皮肤/英雄/铭文/钻石），目录必须覆盖全部 |
| 米哈游 | 一次性全做 | 角色/武器/材料全在同一个目录 |
| 创业公司（MVP 阶段） | 只做当前需要的 | 资源有限，先验证核心流程 |
| 阿里（权益中心） | 分期做 | 先做券类，再做红包，再做实物 |

#### 区别本质

- 一次性全做 = 架构完整，但开发量稍大
- 只做当前需要的 = 快速验证，但后续扩展时可能需要改表结构

#### 最终建议：选 A（本次一起做）

理由：
1. 你说了"愿意一次性投入大量成本" — 不需要分期
2. 表结构已预留 `item_template_id` 字段，代码层面多支持一个 `prize_type = 'item'` 分支只是多几行 if/else
3. `item_templates` 表已有 20+ 条数据（优惠券、实物产品），说明业务上已有物品类奖品需求
4. **长期维护成本角度**：现在不做，未来要做时需要改 ENUM 值、改 Service 逻辑、改前端表单、改迁移脚本 — 比现在一起做成本更高
5. 兑换系统已有完整的 `item_templates` 链路，抽奖系统引用只是加一个 FK

实际增量工作量：
- Model 加 `belongsTo(ItemTemplate)` — 1 行
- CrudService 校验 `item_template_id` — 5 行
- 前端表单多一个物品类选项 + 下拉 — 20 行
- 抽奖引擎发放多一个 `if (prize_type === 'item')` 分支 — 10 行
- 总增量 < 40 行代码，避免未来破坏性变更
