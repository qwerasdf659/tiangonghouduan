## 一、现象与根因

### 现象（真机 vConsole）
管理员在小程序审批页点「通过」：
```
POST /api/v4/console/consumption/approve/3054   ← 旧的"直接审核"接口
响应：success:true, message:"该记录使用审核链流程"
结果：反复点都只弹这句，记录停在待审核，没有"审核通过"提示，积分不发放
```

### 根因（纯文本数据流图）
```
管理员点「通过」
   │ 调旧接口 POST /console/consumption/approve/:id
   ▼
后端：这条消费记录有 in_progress 的审核链实例
   │ → 不能用"直接审核"接口，返回提示 + 正确下一步端点：
   │   data.redirect = true
   │   data.pending_step_id = <待处理步骤ID>
   │   data.approval_chain_url = /api/v4/console/approval-chain/steps/<step_id>/approve
   ▼
❌ 前端没读 data.redirect / approval_chain_url，没改调审核链步骤接口，只 toast 了 message
   ▼
结果：审核链一步都没推进 → 永远不会"审核通过" → 积分不发放
```

> 一句话：后端逻辑正确（已把这条记录纳入多级审核链，并告诉前端去哪审）；**问题是前端没对接审核链接口，仍在调不适用的旧接口**。

## 三、方案 B：审批页直接对接审核链

**核心思想：审批页面向"待办步骤（step）"，不再调消费的旧审核接口。**

- 列表数据源：`GET /api/v4/console/approval-chain/my-pending`（当前登录人的待审核步骤）。
- 审批动作：直接调 `POST /api/v4/console/approval-chain/steps/:step_id/approve|reject`。
- 不再使用 `POST /api/v4/console/consumption/approve/:id`（该接口对审核链记录只会返回 redirect，属过渡态，前端弃用）。

## 四、审核步骤流程图（纯文本）

```
用户提交消费记录
   │
   ▼
后端按金额等条件匹配审核链模板 → 创建审核链实例(instance, status=in_progress)
   │  生成有序步骤 step1, step2, ... stepN（每步指定"审核人"或"审核角色"）
   ▼
┌─────────────────────────────────────────────────────────┐
│ 逐步审核（按 step_number 顺序）：                          │
│                                                           │
│  当前 pending 步骤  ──审核人调 steps/:id/approve──►        │
│     │                                                     │
│     ├─ 不是最后一步(is_final=false) → 推进到下一步 pending  │
│     │     返回 is_chain_completed=false, next_step_number  │
│     │                                                     │
│     └─ 是最后一步(is_final=true) → 审核链 completed         │
│           final_result=approved                           │
│           → 触发 ContentAuditEngine.approve                │
│           → 回调 ConsumptionAuditCallback                  │
│           → 消费记录置为 approved + 自动发放积分 ✅          │
│                                                           │
│  任一步 reject → 审核链 rejected，整体拒绝，不发积分        │
└─────────────────────────────────────────────────────────┘
```

> 关键：**只有最后一步（is_final）通过，消费记录才真正 approved 并发积分**。中间步骤通过只是"推进到下一步"，前端要按 `is_chain_completed` 区分提示文案。

---

## 五、谁来审核（鉴权规则，后端 `_verifyOperatorPermission`）

每个步骤的审核人由该步骤配置决定，后端校验顺序：
1. `role_level >= 100`（超级管理员）：任意步骤都可审（兜底）。
2. **指定人模式**（步骤有 `assignee_user_id`）：必须是该 user 本人。
3. **角色池模式**（步骤有 `assignee_role_id`）：必须拥有该角色。

> 进入审批路由本身要求 `role_level >= 60`（business_manager 及以上）；具体能不能审某一步，由上面规则精确判定。审不了会返回 `APPROVAL_ERROR`（如"您不是当前步骤的指定审核人"）。

## 六、接口契约（前端按此对接，不做字段映射）

### 6.1 查我的待审核步骤 GET /api/v4/console/approval-chain/my-pending

```
Header: Authorization: Bearer <access_token>
Query : page=1&page_size=20（可选）
```

响应（分页，`data` 为步骤数组，`pagination` 为分页信息）：
```
{
  "success": true,
  "code": "SUCCESS",
  "data": [
    {
      "step_id": 88,                // ★ 审批动作要用它
      "instance_id": 50,            // 所属审核链实例
      "step_number": 1,            // 第几步
      "status": "pending",         // 待审核
      "is_final": false,           // 是否最后一步（true 时审过即终审）
      "assignee_user_id": 31,      // 指定审核人（角色池模式时为 null）
      "assignee_role_id": null,    // 指定审核角色（指定人模式时为 null）
      "instance": {
        "instance_id": 50,
        "auditable_type": "consumption",  // 业务类型
        "auditable_id": 3054,             // 对应消费记录 ID
        "status": "in_progress",
        "template": { ... }               // 命中的审核链模板
      },
      "node": { ... }              // 节点配置（节点名/审核人分配方式等）
    }
  ],
  "pagination": { "page": 1, "page_size": 20, "total": 2, "total_pages": 1 }
}
```

> 审批列表直接渲染这个数组：每条就是"待我审的一步"。消费金额等业务详情可用 `instance.auditable_id` 再查消费详情接口，或在 `instance` 中带出（按需）。

### 6.2 （可选）按业务记录查审核链进度 GET /api/v4/console/approval-chain/instances/by-auditable

用于"点开某条消费记录看审核进度/历史"。
```
Query: auditable_type=consumption&auditable_id=3054
```
返回该记录的审核链实例，含 `steps[]`（每步 `status`、`assignee`/`actor`、`actioned_at` 等）+ `template`。无实例时 `data=null`。

### 6.3 审核通过 POST /api/v4/console/approval-chain/steps/:id/approve

> ⚠️ 路径参数实际为 `:id`（不是 `:step_id`）。前端传 step_id 的值到 `:id` 位置即可。后端路由 `routes/v4/console/config/approval-chain.js` 实测就是 `/steps/:id/approve`。

```
Header: Authorization: Bearer <access_token>
Body  : { "reason": "核实无误，审核通过" }   // reason 可选
```
响应：
```
{
  "success": true,
  "code": "SUCCESS",
  "message": "审核通过，已推进到下一步",   // 或 "终审通过，审核链已完成"
  "data": {
    "step_id": 88,
    "action": "approved",
    "is_chain_completed": false,   // false=还有下一步；true=终审完成
    "final_result": null,          // 终审完成时为 "approved"
    "next_step_number": 2          // 下一步步号（终审完成时为 null）
  }
}
```

### 6.4 审核拒绝 POST /api/v4/console/approval-chain/steps/:id/reject

> ⚠️ 同 6.3，路径参数实际为 `:id`。后端对 reject 的 `reason` 有校验：必填且 `trim().length >= 5`，否则返回 400 `apiBadRequest('拒绝原因必须提供，且不少于5个字符')`。前端拒绝弹窗需强制用户填写 ≥5 字的原因。

```
Body: { "reason": "金额与凭证不符" }   // 拒绝建议必填 reason
```
响应：`data.is_chain_completed=true`、`data.final_result="rejected"`，整条链拒绝，不发积分。

### 6.5 常见失败 code

| HTTP | code | 场景 | 前端处理 |
|------|------|------|---------|
| 400 | APPROVAL_NOT_FOUND | step 不存在 | 刷新待办列表 |
| 400 | APPROVAL_ERROR | 步骤状态非 pending / 已结束 / 无权审该步 | 刷新列表或提示无权限 |
| 400 | APPROVAL_INVALID | action 非法 | 前端 bug，检查调用 |

### 6.6 批量审核 POST /api/v4/console/approval-chain/steps/batch（batch_keep 方案，命名已采纳）

> 路径命名决策（2026-06-12 已采纳，方案 A）：批量是"资源集合 `steps` 上的批量动作"，用固定子路径 `/batch`，**具体动作（approve/reject）放在 body 的 `action` 字段**，一个接口同时支持批量通过/拒绝。与单条接口 `steps/:id/approve` 同级同风格；不带单一 `:id`，不违背"事务实体 `:id`、配置 `:code`"的单资源定位规则，符合大厂/RESTful `/batch` 集合动作惯例。
>
> 批量审核改为收口到审核链：后端**逐条调 `processStep`**，每条仍走完整审核链流程（鉴权、状态机、终审发积分、留痕），不绕过流程。仅审"当前轮到当前登录人的步骤"。

```
Header: Authorization: Bearer <access_token>
Body  : {
  "step_ids": [88, 90, 92],          // 待审步骤ID数组（来自 my-pending）
  "action": "approve",               // approve | reject
  "reason": "批量审核通过"            // reject 时必填
}
```
响应（逐条结果 + 汇总）：
```
{
  "success": true,
  "code": "SUCCESS",
  "data": {
    "results": [
      { "step_id": 88, "success": true,  "is_chain_completed": true,  "final_result": "approved" },
      { "step_id": 90, "success": false, "error_code": "APPROVAL_ERROR", "message": "您不是当前步骤的指定审核人" }
    ],
    "stats": { "total": 3, "success_count": 2, "failed_count": 1 }
  }
}
```

> 说明：批量是"循环逐条 processStep"的便利封装，单条失败不影响其它条；前端按 `results[].success` 分别提示。多级链下批量只推进"我负责的那一步"，不会一键从头审到尾。

## 七、前端实施清单（小程序，前端开发负责）

1. **审批列表数据源**：从 `consumption/pending` 改为 `GET /approval-chain/my-pending`，渲染"待我审的步骤"；每条取 `step_id`、`instance.auditable_id`（消费记录ID，用于查/展示金额）。
2. **「通过」按钮**：调 `POST /approval-chain/steps/{step_id}/approve`，body `{ reason }`。
   - 按响应区分提示：`is_chain_completed=false` → 「已通过，进入下一步审核」；`is_chain_completed=true && final_result='approved'` → 「终审通过，积分已自动发放」。
3. **「拒绝」按钮**：调 `POST /approval-chain/steps/{step_id}/reject`，body `{ reason }`（reason 必填）；提示「已拒绝」。
4. **审核进度（可选）**：点详情用 `instances/by-auditable?auditable_type=consumption&auditable_id=<id>` 展示步骤进度（谁审过、当前到哪一步）。
5. **弃用旧接口**：不再调 `/console/consumption/approve/:id` 与 `/reject/:id`。
6. **不做兼容/映射**：直接用后端字段名（`step_id`、`is_chain_completed`、`final_result`、`auditable_id` 等），不起别名。
7. **写操作不自动重试**：审批是写操作，失败按 code 提示，不自动重发（与项目幂等规则一致）。
