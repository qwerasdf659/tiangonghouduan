> ## 议题 B（JWT 含明文手机号 + role_level）— 实测核对结论
>
> **真实代码核对：**
> - `middleware/auth.js:417-426` `generateTokens` payload 写入 `user_id/mobile/nickname/status/role_level/user_role/iat`（+可选 session_token/device_id）。
> - 校验中间件 `authenticateToken`（`auth.js:661-702`）**只用 `decoded.user_id`/`session_token`/`device_id`**，其余（mobile/nickname/status/role_level/permissions）**全部实时查库**注入 `req.user`。→ payload 里那些敏感字段属"塞了但不用来鉴权"的冗余。
>
> **⚠️ 实测新发现（前端文档说"只 1 处用到"不准确，落地必须一并处理）：**
> - `users` 表**已无 `mobile` 明文列**（迁移 `20260608005116` 已删，改为 `mobile_encrypted`/`mobile_hash`/`mobile_prefix_hash`/`mobile_suffix_hash`）；`models/User.js:43` `mobile` 是 **VIRTUAL 字段**（读时解密）。
>   → 即 `payload.mobile = user.mobile` 实际把**解密后的完整手机号**写进 Base64 payload，**泄密属实且更应清除**（明文不该出现在 Token）。
> - `services/ChatWebSocketService.js`：`socket.user = decoded`（整个 JWT payload），`role_level` 在 **234/264/359/1662/1667/1708 多处被读取**（不止 1 处）。核对后确认：这些**只用于日志**，连接路由用的是 `session_user_type` 而非 role_level，**无 `role_level >=/<` 鉴权判定**。
>   → 从 payload 移除 role_level 后，WS 这些日志会变成 0；需顺手改为查库或去掉该日志字段（不影响鉴权/路由）。
>
> **后端权威方案（精简 payload + 复用现有实时查库，零鉴权风险）：**
> - payload 仅保留 `user_id`（+`session_token`/`device_id`/`iat/exp`），移除 `mobile/nickname/status/role_level/user_role`。
> - 手机号：前端要展示 `136****7930` 由 `GET /api/v4/user/profile` 等接口经 DataSanitizer 返回脱敏值，Token 里压根不出现。
> - role_level：前端判管理员改调权限接口，不从 Token 解码（本就实时查库，移除零影响）。
> - 顺手修 `ChatWebSocketService` 几处 role_level 日志（查库或去字段）。不兼容旧 Token（未上线，24h 自然过期）。

## A1 · 限流配置怎么管（env 键名 / 配置源）

### 行业怎么做

| 主体 | 配置怎么管 | 本质 |
|------|-----------|------|
| 大厂（阿里/腾讯/字节） | 限流规则在**配置中心**（Nacos/Apollo/自研），运营可热更阈值不发版；代码只读 key，绝不硬编码 | 集中配置中心，单一真相源 |
| 美团/京东 | 同上，限流阈值挂在配置平台，按接口分组；**禁止**同一阈值在代码和配置里各存一份 | 配置平台 + 禁重复 |
| 小公司/创业 | 常见两种乱象：① 硬编码在代码里；② `.env` 里建了键但代码没读（孤儿键），后来又新建同义键 → **两套同名配置打架**（正是你现在 `MARKET_PUBLIC_RATE_LIMIT_PER_MIN` 的状态） | 易留孤儿键/双源 |
| 游戏公司 | 限流参数走服务端配置表 + 热加载（开服前调参频繁），键名规范统一前缀 | 配置表 + 统一前缀 |
| 活动策划平台 | 活动级限流参数挂活动配置，全局兜底走环境变量，键名按 `RATE_LIMIT_*` 规范前缀 | 环境变量 + 规范前缀 |
| 游戏虚拟物品/二手（BUFF/悠悠/闲鱼） | 行情读宽松、交易写严格，阈值配置化；强调"一个配置项只有一个来源"，避免线上调错 | 配置化 + 单一来源 |

### 核心规律与最适合你的选型

**核心规律**：成熟团队铁律是「**一个配置项只能有一个真相源**」。你没有配置中心（单体 + `.env` 唯一源），但同样要守住"不双源、不孤儿键、键名统一前缀"。

> **🎯 推荐 A1 = 方案①（收编复用 + 统一 `RATE_LIMIT_*` 前缀）**：
> - 复用已存在的 `ENABLE_RATE_LIMIT`（做总开关）；把孤儿键 `MARKET_PUBLIC_RATE_LIMIT_PER_MIN` **改名收编**为 `RATE_LIMIT_PUBLIC_READ_MAX`（语义更准、前缀统一），并在代码里真正读取；新增 `RATE_LIMIT_GLOBAL_MAX`、`RATE_LIMIT_WINDOW_MS`。
> - **为什么不选方案②（全建新键、留旧键不管）**：会让孤儿键继续躺在 `.env` 里，后人不知是否生效 → 正是大厂极力避免的"双源/僵尸配置"。你要"低技术债"，就该顺手清理，而不是绕开。
> - **为什么不上配置中心（Nacos/Apollo）**：那是为"多服务热更新"设计的，你是单体 + `.env` 唯一源，引入配置中心纯属为不存在的问题加运维。`.env` + `pm:restart --update-env` 已满足。
> - 贴合你技术栈：完全符合你"配置只从 `.env` 进、不重复、不硬编码"的铁律，零新依赖。

## A2 · 限流阈值定多少（全局 / 公开只读 / 写接口）

### 行业怎么做

| 主体 | 阈值与维度 | 读 vs 写分档 |
|------|-----------|-------------|
| 大厂（阿里/腾讯/字节） | 多维分层：网关按 IP 粗限（每 IP 每分钟数百~数千），应用层按 user/接口细限；读接口阈值常是写的 **5~10 倍** | 严格分档，读宽写严 |
| 美团/京东 | 登录按 user_id、未登录按 IP；热点读接口走缓存 + 宽松限流，下单/支付严格 | 按身份 + 读写分档 |
| 小公司/创业 | 常"一刀切"：全 `/api` 一个 IP 桶一个阈值（**你现在的 100/min/IP 就是这个**），正常用户易误伤 | 一刀切、易误伤 |
| 游戏公司 | 客户端令牌桶自限流 + 服务端按账号限；查询类宽松、登录/创角严格；开服洪峰错峰 | 客户端+服务端双限 |
| 活动策划平台 | 整点活动按 user+活动维度限，公开展示页极宽松（甚至 CDN 扛），活动写接口严防刷 | 活动维度 + 缓存扛读 |
| 游戏虚拟物品/二手 | 行情/列表高频读按 IP 宽松 + 短缓存；下单/出价按 user 严格 + 幂等 | 读宽松短缓存、写严格幂等 |

### 核心规律与最适合你的选型

**核心规律**：几乎没有成熟平台用"全 `/api` 一个 IP 桶"；普遍是 **登录按 user、未登录按 IP** + **读宽写严** 分档，且阈值配置化（按真实流量调）。

> **🎯 推荐 A2 = 采纳"全局 600/min/IP、public_read 300/min、写接口维持现状（lottery 20/login 10/chat 10）、窗口 60s"**：
> - **为什么是 600 而非更高/更低**：你是 Sealos devbox **单实例 + 公网网关**，多个测试设备/微信开发者工具常共用一个出口 IP，叠加冷启动十几个并发请求；600 给正常用户留足余量，又远低于脚本刷量级别（恶意脚本轻松上千/分钟仍会被挡）。
> - **为什么 public_read 单列 300**：`campaigns/active`/配置/版本闸门是 `needAuth:false` 公开只读，本就该比写接口宽松，单列档避免和写接口挤全局桶。
> - **为什么写接口不动**：抽奖/登录/聊天有经济价值或安全意义，维持严格 + 幂等键，防刷量薅羊毛。
> - **为什么不上客户端令牌桶/CDN 边缘限流**：那是给"开服百万并发/静态页海量读"设计的，你是餐饮积分小程序、动态数据，过度设计。
> - 贴合你技术栈：全部基于现有 `createLimiter` 调参 + 加 1 个 `public_read` 档，不改限流器内核，阈值进 `.env`。

## B1 · JWT payload 放什么（极简 vs 富 payload）

### 行业怎么做

| 主体 | payload 里放什么 | 敏感信息（手机号/角色）处理 | 本质 |
|------|-----------------|--------------------------|------|
| 大厂（阿里/腾讯/字节） | **极简**：只放 `sub(user_id)` + `jti/session` + `exp/iat`，权限/资料一律实时查或走独立权限服务/网关 | 手机号、角色**绝不进 Token** | 最小化 payload + 服务端权威 |
| 美团/京东 | Token 只认身份，权限走 RBAC 服务实时查；C 端资料走 `user/profile` 接口（脱敏） | 手机号脱敏只在展示接口返回 | 身份与权限分离 |
| 小公司/创业 | 图省事把 user_id+手机号+角色+昵称全塞进去（**你现在的状态**） | 常忽视 payload 是 Base64 明文，手机号/角色"半公开" | 方便优先、留隐患 |
| 游戏公司（腾讯游戏/米哈游/网易） | Token 放 `account_id` + 会话票据；角色/等级/权限在服务端会话内，不进 Token | 几乎不放可读个人信息（防作弊、防探测） | 票据化 + 服务端状态 |
| 活动策划平台 | 短时效 Token + user_id，权限活动级实时判 | 合规要求下不放明文手机号 | 短时效 + 实时判 |
| 游戏虚拟物品/二手（BUFF/悠悠/闲鱼） | 对"权限探测/身份枚举"敏感，Token 只放 user_id + 会话；**角色绝不放**（防识别管理员/特权号） | 手机号脱敏走接口；角色实时查 | 防探测优先 |

### 核心规律与最适合你的选型

**核心规律**：① JWT 只证明"你是谁"(user_id+会话票据)，不证明"你能干什么"(权限) 和"你的资料"(手机号/角色)；② payload 是 **Base64 明文**，放进去等于半公开；③ 角色放 Token 还会有"权限漂移"（改了权限旧 Token 仍生效）。

> **🎯 推荐 B1 = 极简 A 案（移除 mobile/nickname/status/role_level/user_role，只留 user_id + session_token + device_id + iat/exp）**：
> - **零鉴权风险**（实测已确认）：`authenticateToken` 本就只用 `decoded.user_id/session_token/device_id`，其余全实时查库；移除对鉴权零影响。
> - **泄密更该清**（实测发现）：`users` 表手机号已加密存储（`mobile_encrypted`），而 JWT 却把**解密后的明文**塞进 Base64 payload —— 等于把加密成果在 Token 里又泄出去，必须清。
> - **顺手清残留**：`ChatWebSocketService` 6 处读 `socket.user.role_level` 仅用于日志（不参与鉴权/路由，路由用 `session_user_type`），改查库或去字段即可。
> - **为什么不选"保留但手机号脱敏 + 留 role_level"**：仍把脱敏号和角色放 Token，role_level 仍可被解码探测管理员，且和"已实时查库"重复（双源技术债），是半吊子。
> - **为什么不上 opaque token / JWE 加密**：你已是 JWT + Redis 会话票据混合模式（可踢设备），安全足够；"敏感信息压根不进 Token"比加密更彻底、零开销。
> - 贴合你技术栈 + 你"不兼容旧"原则：未上线，直接精简，旧 Token 24h 自然过期，不做双读兼容。

## 二、根因数据流图（纯文本）

```
微信小程序冷启动首页（一次正常开屏，onShow 并行发起多个请求）
        │  GET /lottery/campaigns/active   （公开只读，needAuth:false）
        │  GET 配置缓存接口
        │  GET 版本闸门检查接口
        │   ⚠️ 用户未登录：storeHasToken:false / isAuthenticated:false
        ▼
后端限流中间件（按 IP / 设备聚合计数）
        │  判定"单位时间请求数超阈值" → 全部拦截
        ▼
后端统一返回 429 { success:false, code:"RATE_LIMIT_EXCEEDED" }
        ▼
前端 utils/api/client.ts:469 收到 429 → 如实抛错 + Toast "请求过于频繁"
        ▼
结果：campaigns/active、config-cache、版本闸门 同一秒集体 429 失败
```

**关键判断（已核对前端实际代码）：**
1. 前端 `APIClient` 对 429 **直接抛错、不重试、不轮询**（`client.ts:469`），**没有重试风暴**，不是把后端打到限流的肇因。
2. 触发限流的是**一次正常冷启动**的少量并行请求（版本检查 + 配置 + 活动列表），且用户**未登录**、请求的是 `needAuth:false` 公开接口。
3. 多个**不同**接口在同一秒**全部**一起 429 → 后端限流器按 **IP/设备聚合计数**，是后端策略，前端无法干预。

→ **治本在后端限流配置（阈值/窗口/计数维度）；前端只能做"治标 + 体验兜底 + 不自伤"。**

| 公司类型 | 代表 | 客户端请求治理做法 | 本质 |
|---------|------|-------------------|------|
| 大厂电商 | 美团/阿里/京东 | 统一网络层（Mtop/Shark）：请求去重 + 并发控制 + 指数退避+抖动重试 + 熔断降级 + 缓存兜底，业务层无感 | 重型统一网络中台 |

| 小公司/创业 | 多数 | 基本不治理：429 弹 toast，顶多防连点 | 几乎裸奔 |
| 游戏公司 | 腾讯游戏/米哈游/网易 | 请求队列 + 优先级 + 客户端令牌桶自限流 + 退避重试 + 弱网降级；开服洪峰用**随机错峰延迟**打散 | 客户端自限流 + 错峰 |
| 活动策划/运营平台 | 中小活动平台 | 整点洪峰用**随机抖动错峰** + 简单退避 | 错峰为主 |
| 游戏虚拟物品/小众二手 | BUFF/悠悠有品/交易猫/闲鱼 | 列表行情高频：请求节流 + 短 TTL 缓存 + 失败退避；交易类"宁可慢不可重复提交"（幂等+防连点） | 节流+短缓存+幂等 |

**核心规律（公司越成熟越遵守）：**
1. 请求治理**收口到统一网络层**，不散落到页面（本项目已有 `APIClient`，天然满足）。
2. 退避**必带抖动（jitter）+ 次数上限**——无抖动的重试会让一批客户端"同步重试"形成二次洪峰。
3. 首屏并发要么**聚合（BFF）**要么**错峰**，不让 N 个请求挤在同一毫秒。

## 七、前端实施清单（B/C/B 三项，待动手）

> 全部收口到 `utils/api/client.ts` 的 `APIClient`（治理项 1、4）+ 页面/缓存层（治理项 3）。

### 7.1 治理项1：指数退避 + 抖动 + 上限（`APIClient`，仅幂等 GET）

- 触发条件：响应 `statusCode === 429`（可选扩展到网络层 `fail`），**且** `method === 'GET'`（幂等）。
- 退避公式：`delay = baseMs * 2^attempt + random(0, jitterMs)`（如 base=500ms，jitter=300ms）。
- 上限：最多重试 **2 次**（共 3 次请求）；超出仍失败则抛原错误、走现有 Toast。
- 优先读后端 `Retry-After` 头或响应 `data.retry_after_seconds`（若后端提供），有则以其为准。
- 写操作（POST/PUT/DELETE）**绝不自动重试**（与项目"幂等键"规则一致，避免重复兑换/重复提交消费）。

### 7.2 治理项4：in-flight 请求去重（`APIClient`）

- 以 `method + url + 序列化 data` 为 key，维护"在途请求 Map"。
- 相同 key 的请求在途时，后来者直接复用同一个 Promise，不发第二次 wx.request。
- 请求 settle（成功/失败）后从 Map 删除 key。
- 仅对幂等 GET 去重；写操作不去重（每次都是独立业务意图）。

### 7.3 治理项3：onShow 时间窗节流（页面层）

- 在 `pages/lottery/lottery.ts` 的 `onShow` 对 `_loadCampaigns` / `loadHomeAnnouncements` / `_refreshBadgeCounts` 加"距上次刷新 < N 秒（建议 10~15s）则跳过"。
- 节流窗口用 UI 常量管理（前端可自主决定的延迟参数，非业务数据）。
- 仅控刷新频率；下拉刷新（`onPullDownRefresh`）作为用户显式意图，**不受节流限制**，保证用户能手动拿最新数据。

### 7.4 实施后质量门禁

- `npm run lint`（ESLint）+ `tsc --noEmit`（类型检查）+ `npm run format:check`（Prettier）全通过。
- 真机 vConsole 复验：冷启动不再出现"同一秒多接口集体 429"；偶发 429 时前端退避后自动恢复、无风暴。

### 8.1 限流策略排查/调整（必做，治本）—— 后端方案（贴合现有 `RateLimiterMiddleware`）

#### 行业怎么设计限流（按公司类型对比）

| 主体 | 限流维度与阈值 | 公开只读 vs 写接口 | 本质 |
|------|---------------|-------------------|------|
| 大厂（阿里/腾讯/字节） | **多维分层**：网关层按 IP 粗限、应用层按 user/接口细限；阈值按接口分级（读宽写严） | 严格分档：读接口阈值可达写接口的 5~10 倍 | 多层 + 分接口分档 |
| 美团/京东 | 登录用户按 user_id 限、未登录按 IP 限；热点读接口走缓存+宽松限流 | 公开读极宽松（甚至只防爬虫），下单/支付严格 | 按身份维度 + 读写分档 |
| 小公司/创业 | 常见"一刀切"：全 `/api` 一个 IP 桶、一个阈值（**正是你现在的状态**） | 不分：读写挤一个桶，正常用户易误伤 | 一刀切、易误伤 |
| 游戏公司 | 客户端令牌桶自限流 + 服务端按账号限；开服洪峰错峰 | 登录/创角严格，查询类宽松 | 客户端+服务端双限 |
| 活动平台 | 整点活动按 user + 活动维度限，公开页 CDN/缓存扛 | 活动写接口严格防刷，展示页极宽松 | 活动维度 + 缓存扛读 |
| 游戏虚拟物品/二手（BUFF/悠悠/闲鱼） | 行情/列表高频读：按 IP 宽松 + 短缓存；下单/出价按 user 严格 + 幂等 | 读宽松短缓存、写严格幂等 | 读写分治 |

**核心规律：** ① 几乎没有成熟平台用"全 `/api` 一个 IP 桶一个阈值"——这正是你当前的问题；② 普遍做法是 **登录按 user、未登录按 IP** + **读接口宽松、写接口严格** 分档；③ 阈值配置化，不硬编码（方便按线上真实流量调）。

#### 为什么不直接照搬其它公司的方案

| 没采用的方案 | 谁在用 | 不适合你的原因（基于你项目实际） |
|-------------|--------|--------------------------------|
| **网关层独立限流组件**（如 Sentinel、Kong、Nginx limit_req、APISIX 插件） | 阿里/字节等大厂多层架构 | 你是 **Sealos devbox 单实例 + Express 单体**，没有独立网关层。引入 Sentinel/Kong 要多部署一套中间件、多一层运维，**纯属为单体项目上重型基建**，违背"低维护、低技术债"。你已有的 Redis 滑动窗口限流器**够用且更轻**。 |

#### 具体改动

1. **提高全局兜底阈值**（`app.js:167`）：把全局 `100 次/分钟/IP` 提高到 **600 次/分钟/IP**（最终决策值）。理由：一个正常页面冷启动 + 角标 + 列表轻松十几个请求，单设备 100/分钟过严；提高后仍能挡住恶意脚本（脚本刷量远超此值）。
2. **给公开只读接口加"宽松档"预设**：在 `RateLimiterMiddleware.presets` 新增一个 `public_read` 档（如 `300 次/分钟/IP`），用于 `GET /api/v4/lottery/campaigns/active`、配置缓存、版本闸门等 `needAuth:false` 的只读接口。写接口/登录/抽奖维持现有更严的档。

4. **429 响应补标准 `Retry-After` 头**（`RateLimiterMiddleware.js` 返回 429 处）：`res.setHeader('Retry-After', retryAfter)`。`data.retry_after` 已有，前端可直接读，这步是兼容标准 HTTP 客户端。
5. **开发/测试环境放宽**：devbox 调试期可对开发出口 IP 加白名单（跳过限流），或临时设 `DISABLE_RATE_LIMITER=true`（已支持），避免反复刷新累积计数干扰联调。**注意：白名单/开关只在 `.env` 配置，不在代码里硬编码 IP。**

> ⚠️ 按你"配置只从 .env 进来"的铁律：上述阈值（全局 max、public_read max、窗口）建议**抽成 .env 变量**（如 `RATE_LIMIT_GLOBAL_MAX`、`RATE_LIMIT_PUBLIC_READ_MAX`），`app.js`/预设从 `process.env` 读，避免散落硬编码、方便不改代码调参。

## 十、重要边界声明

1. **前端治理是"治标"**：即便 B/C/B 三项全做，只要后端限流阈值过低，正常冷启动**仍可能 429**。真正修复必须后端调整限流（第 8.1 节）。前端治理的价值是：偶发限流时体验不崩 + 前端绝不自己制造洪峰。
2. **不碰业务数据真实性**：节流/去重只控"刷新频率与重复发送"，活动、积分等业务数据仍以后端实时返回为准，不用旧数据糊弄用户。
3. **退避只对幂等 GET**：写操作（兑换、提交消费）绝不自动重试，避免重复下单，与项目"幂等键（Idempotency-Key）"规则一致。
4. **BFF 聚合接口（治理项2/D）是后端的活**，前端无法代劳，仅作为削峰的长期建议提给后端。

### 🟢 后端核查结论（2026-06-12 核对真实代码后给出）

已核对 `middleware/auth.js` 的 `generateTokens()`（签发）和 `authenticateToken()`（校验）实际逻辑，确认前端判断正确，且**后端改造代价极低、几乎零风险**：

**核查证据：**
- 签发处 `middleware/auth.js:417-426`，JWT payload 当前写入：`user_id / mobile（完整）/ nickname / status / role_level / user_role / iat`，外加可选 `session_token / device_id`。**确实把完整手机号和 role_level 明文放进了 payload。**
- 关键发现：**后端自己根本不依赖 payload 里的这些敏感字段**。校验中间件 `authenticateToken()`（`auth.js:661-702`）只用 `decoded.user_id`、`decoded.session_token`、`decoded.device_id`，其余（mobile/nickname/status/role_level/permissions）**全部从数据库实时查询**（`auth.js:662-688`）。
- 全仓搜索 `decoded.mobile / decoded.role_level / decoded.nickname / decoded.status`：业务代码里**只有 1 处**用到——`services/ChatWebSocketService.js:234` 把 `decoded.role_level` 写进**日志**（不参与鉴权）。

**结论：** payload 里的 `mobile / nickname / status / role_level / user_role` 属于"塞进去了但没人真正用来做判定"的冗余敏感信息。移除它们对后端鉴权逻辑**零影响**，只需顺手修一处 WebSocket 日志。这正好契合你"删除无用代码、降低技术债"的原则，也符合"role_level 实时查库不进 Token"（payload 注释里 P1-2 修复已移除 is_admin，是同一思路的延续）。

### 后端方案（精简 payload + 实时查库，贴合现有架构）

JWT payload **只保留鉴权必需的最小字段**：

```
保留：user_id（身份）、session_token（会话校验）、device_id（多端）、iat/exp（时效）
移除：mobile、nickname、status、role_level、user_role
       → 这些一律由 authenticateToken 实时查库注入 req.user（现在已经是这么做的）
```

- 手机号：前端如需展示用 `136****7930`，应由**用户信息接口**（如 `GET /api/v4/user/profile`）返回脱敏值，**不放 Token**。比"在 Token 里脱敏"更彻底（Token 里压根不出现手机号）。
- `role_level`：本就实时查库，从 payload 移除后零影响；前端要判断是否管理员，应调用后端权限接口或由后端在受控接口按需返回，**不从 Token 解码**。

### 行业怎么设计 JWT payload（按公司类型对比）

| 主体 | JWT payload 里放什么 | 敏感信息处理 | 本质 |
|------|---------------------|-------------|------|
| 大厂（阿里/腾讯/字节） | **极简**：只放 `sub(user_id)`、`session/jti`、`exp/iat`，权限和资料一律实时查（或单独权限服务/网关鉴权） | 手机号、角色**绝不进 Token**；要么不放，要么放不可逆 ID | 最小化 payload + 服务端权威 |
| 美团/京东（电商） | Token 只认身份，权限走 RBAC 服务实时查；C 端资料走 `user/profile` 接口（脱敏） | 手机号脱敏只在展示接口返回，Token 不带 | 身份与权限分离 |
| 小公司/创业 | 图省事把 user_id、手机号、角色、昵称全塞进 payload（正是你现在的状态） | 常忽视：payload 是 Base64 明文，等于把手机号/角色"半公开" | 方便优先、留隐患 |
| 游戏公司（腾讯游戏/米哈游/网易） | Token 放 `account_id` + 会话票据；角色/等级/权限在游戏服务端会话内，**不进 Token** | 几乎不在 Token 放可读个人信息（防作弊、防探测） | 票据化 + 服务端状态 |
| 活动策划/运营平台 | 多为短时效 Token + user_id，权限活动级实时判 | 视合规要求，规范的也不放明文手机号 | 短时效 + 实时判 |
| 游戏虚拟物品/小众二手（BUFF/悠悠/闲鱼） | 交易平台对"权限探测/身份枚举"敏感，Token 通常只放 user_id + 会话；**角色绝不放**（避免被识别管理员/特权账号） | 手机号脱敏走接口；角色实时查 | 防探测优先 |

**核心规律（公司越成熟越遵守）：**
1. **JWT 只证明"你是谁"（user_id + 会话票据），不证明"你能干什么"（权限）和"你的资料"（手机号/角色）**——后两者服务端实时查才是权威，放 Token 里既泄密又有"权限漂移"（改了权限旧 Token 还生效）问题。
2. payload 是 **Base64 明文**，任何放进去的东西都等于半公开。手机号、role_level 这类**绝不放**。
3. 你项目其实**已经在往正确方向走**了：payload 注释里 P1-2 已移除 `is_admin`、`authenticateToken` 已全部实时查库——只差把残留的 `mobile/role_level` 也清掉，是"把没做完的事做完"，不是推倒重来。

| 没采用的方案 | 谁在用 | 不适合你的原因（基于你项目实际） |
|-------------|--------|--------------------------------|
| **独立鉴权中心 / OAuth2 授权服务器**（统一发 Token、网关校验） | 大厂微服务（阿里/字节） | 你是 **Express 单体**，鉴权就在 `middleware/auth.js` 一处。引入独立鉴权中心要多一套服务 + 服务间调用，**为单体上微服务基建**，纯增运维和技术债。现有 `generateTokens` + `authenticateToken` 一处签发一处校验，精简 payload 即可。 |
| **不透明 Token（opaque token）+ 服务端 session 全查** | 部分大厂/银行（每次请求查 session 存储） | 这是"Token 里啥都不放，全靠服务端存储查"。你已经是 **JWT + Redis 会话票据（session_token）** 的混合模式，安全性已足够（可踢设备、会话失效）。改纯 opaque 要重构会话体系，**收益小、改动大**。 |
| **Token 加密（JWE）而非签名（JWS）** | 强合规场景（金融） | 把 payload 加密让人无法解码。但这会增加加解密开销和密钥管理负担，而**根本解法是"敏感信息压根不进 Token"**——不放手机号/角色，就不需要加密。简单且零开销，比 JWE 更优。 |
| **角色/权限放 Token 里（带 scopes/claims）** | 很多中小公司、部分 OAuth 实现 | 把权限写进 Token 看似省一次查询，但有两大坑：① **权限漂移**（改了权限旧 Token 仍生效）；② **明文可被解码探测管理员**。你已实时查库，没有这两个坑，**不应该走回头路**。 |
| **完全保留现状（手机号+角色都放）** | 小公司图省事 | 就是当前状态，已被审计判定泄密。不可取。 |

> 一句话：大厂的鉴权中心/opaque token/JWE 都是为"多服务、强合规、海量并发"设计的重型方案，**你是单体 + 已有 Redis 会话票据**，照搬等于上一堆用不上的基建。**最适合你的是"精简 payload + 复用现有实时查库"**——不加任何依赖、不改会话体系、只删几个字段，安全性达标且技术债最低。

### 拍板项（JWT 部分）—— ✅ 已采纳推荐为最终决策（2026-06-12）

| # | 决策点 | 方案对比与区别 | ✅ 选定 |
|---|--------|---------------|------|
| J1 | payload 放什么 | **A 极简（只留 user_id+session_token+device_id+iat/exp）**：对齐大厂/游戏标准，零泄密、无权限漂移、技术债最低；后端本就不依赖被删字段，**零风险**。<br>**B 保留但手机号脱敏（`136****`）+ 留 role_level**：仍把脱敏手机号和角色放 Token，role_level 仍可被解码探测管理员，属"半吊子"，且和"已实时查库"重复（双源，技术债）。 | **A 极简** |
| J2 | 前端原先从 Token 解码 mobile/role_level 怎么办 | 改为调用户信息接口（`user/profile` 拿脱敏手机号）/ 权限接口（拿是否管理员）。区别：Token 不再"自解释"这些字段，前端多一次接口调用，但换来不泄密 + 权限实时准确。 | **改接口获取**（前端适配，不做映射兼容） |
| J3 | 是否兼容旧 Token | **不兼容**：项目未上线，直接精简，旧 Token 24h 自然过期。<br>兼容（双读）只会增加临时代码和技术债，与你"不兼容旧接口"原则相悖。 | **不兼容**（直接精简） |

> ✅ 决策已确认：**J1=A 极简、J2=改接口获取、J3=不兼容**。这套就是阿里/腾讯/游戏大厂的标准做法，最贴合你"长期低维护、低技术债、不兼容旧接口"的目标，且后端零风险（被删字段本就没人用来鉴权）。下方责任划分与执行步骤按此落地。

## 二、后端为什么返回这个错误（数据流图 · 纯文本）

```
微信小程序  GET /api/v4/shop/consumption/user-info?qr_code=QRV2_xxx   ← 请求体 {}，未带 store_id
        │
        ▼
[authenticateToken]  user_id=31, role_level=100 (admin/管理员)  → 通过
        │
        ▼
[resolveStoreContext({ storeIdParam:'query' })]   （middleware/resolveStoreContext.js）
        │  extractStoreId → 未取到 store_id（storeId = null）
        │  isAdmin = (role_level>=100) = true
        ├─ 进入"管理员分支" :96
        │     storeId 为空 且 required=true（取用户信息属门店相关操作）
        │     → :106 返回 400 ADMIN_STORE_ID_REQUIRED
        ▼
"管理员执行门店相关写操作必须指定 store_id（不允许无门店写入，以保证审计可定位）"
```

> 对比：上一轮错误是 `NO_STORE_BINDING`（旧缺陷，管理员被提前放行、门店上下文为空）。
> 错误码从 `NO_STORE_BINDING` → `ADMIN_STORE_ID_REQUIRED` 的变化，本身就证明后端方案丙已正确部署生效。

## 五、后端待办（属后端数据库项目，需拍板后我实现）

### 缺口

前端要做门店选择器，需要一个「查询我能选哪些门店」的接口。现状：

### 推荐方案（对标美团商家版「我的门店」/ 钉钉 `GET /tenants`）

新增 `GET /api/v4/shop/my-stores`：
- 普通员工（role_level<100）→ 复用已有 `getUserStores(user_id)`，返回其 active 在职门店。
- 管理员（role_level≥100）→ 通过 ServiceManager 取门店 QueryService，返回全部 active 门店（跨店特权，与 `resolveStoreContext` 管理员语义一致）。
- 薄路由 + 组合；读操作收口 QueryService（支持缓存），路由不直连 models。
- 返回：`res.apiSuccess({ stores:[{store_id, store_name, store_code, role_in_store}] })`
