## 二、为什么会"同一个星石、两个样子"（纯文本数据流图）

前端星石图标有三条彼此独立的数据来源，URL 全部由后端决定：

```
【抽奖页面】→ 显示新图 ✅
  后端 抽奖奖品接口
     └─ prize.image = { url, thumbnail_url, source: "material_icon" }
        （后端按 prize.material_asset_code = "star_stone" 拼接“材料图标服务”URL）
            │
            ▼  前端 packageLottery/lottery-activity/lottery-activity.ts → addPrizeIcon()
        prize_image_url = image.thumbnail_url || image.url
            ▼  <image src="{{item.prize_image_url}}" />
        界面：最新版星石图

【兑换商城 / 道具商城 顶部资产条】→ 显示旧图 ❌
  后端 GET /api/v4/assets/balances
     └─ balances[] = [{ asset_code: "star_stone", icon_url: <旧图地址>, ... }]
            │
            ▼  前端 pages/exchange/exchange.ts → _refreshAssetBalances()
        icon_url: asset.icon_url     （原样透传，不映射）
            ▼  exchange-shelf.wxml / props-mall.wxml
        <image src="{{item.icon_url}}" />
        界面：旧版星石图

【我的背包 资产条】→ 同源风险，会跟随 balances 一起表现
  后端 GET /api/v4/backpack
     └─ assets[] = [{ asset_code: "star_stone", icon_url: <旧图地址>, ... }]
            ▼  <image src="{{item.icon_url}}" />
```

**矛盾点**：抽奖用的 `prize.image.url`（材料图标服务，已是新图）与资产条用的 `assets/balances[].icon_url`（资产字典里存的图标地址，仍是旧图）**指向了不同的文件**。同一 `star_stone`，后端两套图片来源未对齐。

   - 兑换铸造会给新物品加一条「7天交易冷却」锁（前端弹窗也提示"7天冷却期后可上架交易"）。
   - ⚠️ 业务约定：交易冷却（`item_holds.hold_type='trade_cooldown'`）期间物品应**可见可用，仅不可上架交易**，因此 `items.status` 必须保持 `available`，冷却只应记录在 `item_holds` 表。**若铸造时把 `items.status` 直接改成了 `held`，就会被背包查询过滤掉**——这是最符合现象的解释。

2. **铸造与扣款未在同一事务，物品写入回滚 / 异常被吞**
   - 扣款提交了（前端余额变了），但 `INSERT items` 那一步失败/回滚，只返回了内存里的 `minted_item` 对象，没真正落库。

3. **物品落库了，但 `owner_user_id` 或背包查询条件不对**
   - 物品 INSERT 时归属用户写错，或背包查询的 `WHERE owner_user_id = ? AND status='available'` / JOIN 条件把它排除了。

## 议题一（抽奖星石未到账）— 实测根因：奖品类型词表不匹配，`material` 落入 default 被跳过

**真实证据（直连库 + 读码）：**
- `prize_definitions` 里星石奖品 `prize_type = 'material'`（6 条：星石×500/200/80/50/30/10，`material_asset_code='star_stone'`，`is_enabled=1`）。
- 但 `SettleStage._distributePrize`（`services/UnifiedLotteryEngine/pipeline/stages/SettleStage.js:629`）的 `switch(prize.prize_type)` **只有 `points/coupon/physical/virtual` 四个分支，没有 `material`** → 命中 `default:` →「未知奖品类型，跳过发放」，不入账。
- 真实库铁证：`lottery_draws` 的 `prize_type` 分布 = `points`(2756) / `virtual`(2160，最后一条 2026-05-25) / **`''` 空(26，含今天 user 31 的星石中奖)**；今天 user 31 的星石中奖记录 `prize_type=''`、`asset_transaction_id=0`，**13 点后该用户无任何 `star_stone` 入账流水**。
- 对比：历史 2221 笔 `star_stone` 的 `lottery_reward_material` 流水都是 `prize_type='virtual'` 时代产生的；定义改成 `material` 后，发奖静默断裂。

**结论修正前端猜测：** 不是"忘了调 changeBalance"，而是**奖品类型词表从 `virtual` 改成 `material` 后，结算 switch 没同步新增 `material` 分支**，导致 material 类奖品（star_stone、red_core_shard）全部漏发。这是后端 100% 的问题。
