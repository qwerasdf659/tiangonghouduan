### P0 — 定价方案核心机制，代码零实现

#### 3.1 钻石产出配额（DIAMOND_QUOTA）

**定价方案描述**：第十节"钻石产出配额控制"，双池隔离的第二条轨道。消费审核通过时按 1:1 发放钻石配额，抽中钻石奖品扣配额，耗尽则降档到小额钻石。

**实际状态**：全项目代码搜索 `DIAMOND_QUOTA` 和 `diamond_quota`，仅在定价方案文档中出现，无任何代码实现。

**具体缺失**：
- `account_asset_balances` 中不存在 `DIAMOND_QUOTA` 资产类型
- `material_asset_types` 中未定义 `DIAMOND_QUOTA`
- 消费审核通过逻辑中没有按比例发放配额的代码
- BuildPrizePoolStage 没有"钻石配额检查"步骤
- `lottery_strategy_config` 中不存在 `diamond_quota_enabled` / `diamond_quota_ratio` / `quota_exhausted_action` 配置项

**影响**：钻石奖品不受配额限制，可以无限产出。定价方案中"预算积分管实物、钻石配额管钻石"的双池隔离只有前半套（预算积分管实物）在运作，后半套（钻石配额管钻石）完全缺失。无法实现 1:1 锚定目标。

### P1 — 广告经济核心逻辑，框架在但关键机制缺失

#### 3.3 广告钻石销毁

**定价方案描述**：广告消耗的钻石直接从流通中永久移除（销毁），持续减少钻石总供给。

**实际状态**：AdBillingService 竞价日扣款（`processDailyBidding`）和固定包天扣款（`deductFrozenDiamonds`）的 `counterpart_account_id` 指向 `SYSTEM_PLATFORM_FEE`（平台手续费账户）。钻石扣减后进入平台账户，而非直接进入 `SYSTEM_BURN`（销毁黑洞账户）。0 条计费记录意味着此流程尚未在真实场景中验证过。

**业务决策（已确认）**：
- 广告扣除的钻石 **先归集到 `SYSTEM_PLATFORM_FEE`**（当前代码已满足）
- **不做自动销毁**，由运营在管理后台手动操作销毁
- 运营点击"销毁"按钮时，将指定数量的钻石从 `SYSTEM_PLATFORM_FEE` 转入 `SYSTEM_BURN`（永久不可逆）

**待实现**：
- 管理后台新增"平台钻石管理"页面，展示 `SYSTEM_PLATFORM_FEE` 账户的钻石余额
- 提供"销毁"操作按钮，运营输入销毁数量 → 二次确认 → 执行转账（`SYSTEM_PLATFORM_FEE` → `SYSTEM_BURN`）
- 销毁操作记录审计日志（操作人、时间、数量、销毁前后余额）
- 销毁动作不可撤回，`SYSTEM_BURN` 账户的钻石不会通过任何业务流程转出

#### 3.4 DAU 系数定价

**定价方案描述**：第六节核心公式 `日价 = 基础价 × DAU 系数`，DAU 区间 <500 / 500-2000 / 2000-10000 / >10000 对应系数 0.375 / 1.0 / 2.5 / 6.25。

**大白话**：用户少的时候便宜卖，用户多的时候贵卖，让广告价格跟着平台实际价值走。

**实际状态**：零实现。代码中无 DAU 跟踪、无系数计算、无动态定价逻辑。广告位价格是 `ad_slots` 表中的静态值。

**实施要求**：DAU 系数表必须由运营在后台动态配置，不允许硬编码。

**方案**：

1. DAU 统计：每日定时任务统计前一天的日活用户数（基于登录/访问记录），写入统计表
2. DAU 系数配置存储在 `system_configs` 中，运营可随时增删改档位：

```
配置键：ad_dau_coefficient_tiers
配置值示例：
[
  { "max_dau": 500,   "coefficient": 0.375, "label": "冷启动期" },
  { "max_dau": 2000,  "coefficient": 1.0,   "label": "成长期" },
  { "max_dau": 10000, "coefficient": 2.5,   "label": "规模期" },
  { "max_dau": null,  "coefficient": 6.25,  "label": "成熟期" }
]
```

3. 运营可调整的维度：
   - 档位数量（不限于 4 档）
   - 每档的 DAU 上限（max_dau，null 表示无上限）
   - 每档的系数值（coefficient）
   - 总开关：`ad_dau_pricing_enabled`（true/false），关闭则使用 `ad_slots` 表中的静态日价
4. 计算逻辑：
   - 包天模式：`实际日价 = ad_slots.daily_price_diamond × 当前 DAU 系数`
   - 竞价模式：DAU 系数影响竞价底价（见 3.5 动态底价）
5. 基础价（`ad_slots.daily_price_diamond`）由运营在广告位管理页面直接编辑

#### 3.5 动态竞价底价

**定价方案描述**：`底价 = 该广告位过去 7 天平均成交价 × 50%`，无成交时用最低保底价。

**大白话**：竞价广告的最低出价门槛自动跟着市场走。热门广告位底价自然上涨（防低价抢占），冷门广告位底价自然下降（鼓励参与），不需要运营手动调。

**实际状态**：零实现。`min_bid_diamond` 是数据库中的静态值，无自动调节逻辑。

**实施要求**：动态底价参数由运营配置，公式中的比例系数和保底价可调。

**方案**：

1. 底价公式配置存储在 `system_configs` 中：

```
配置键：ad_dynamic_floor_price_config
配置值示例：
{
  "enabled": true,
  "lookback_days": 7,
  "floor_ratio": 0.5,
  "fallback_prices": {
    "home_popup": 50,
    "home_carousel": 20,
    "lottery_popup": 30,
    "profile_popup": 20
  }
}
```

2. 运营可调整的维度：
   - 总开关（enabled）：关闭后竞价底价使用 `ad_slots.min_bid_diamond` 静态值
   - 回看天数（lookback_days）：计算平均成交价的时间窗口
   - 底价比例（floor_ratio）：乘以平均成交价的系数（0.5 即 50%）
   - 各广告位的保底价（fallback_prices）：无成交记录时的兜底底价
3. 计算逻辑（定时任务每日凌晨执行）：
   - 查询每个广告位过去 N 天的 `ad_billing_records`（billing_type = daily_deduct 或 deduct）
   - 有成交 → `动态底价 = 平均成交额 × floor_ratio`
   - 无成交 → `动态底价 = fallback_prices[slot_key]`
   - 计算结果写入 `ad_slots.min_bid_diamond`（覆盖旧值）
4. 运营也可以随时在广告位管理页面手动覆盖某个广告位的底价（手动值优先于自动计算）

#### 3.6 阶梯折扣（包天连投）

**定价方案描述**：连续投放天数越多折扣越大，鼓励长期投放锁定钻石消耗。

**实际状态**：零实现。无连续投放天数检测、无折扣计算逻辑。

**实施要求**：折扣档位和比例必须由运营通过后台动态配置，不允许硬编码。

**大白话**：买得久打折多，锁定长期消耗。鼓励广告主长期投放，一次性锁定更多钻石消耗，对平台来说钻石归集量更大更稳定。

**方案**：在 `system_configs` 中存储折扣规则，格式为 JSON 数组，运营可随时在管理后台增删改档位：

```
配置键：ad_consecutive_discount_tiers
配置值示例：
[
  { "min_days": 7,  "discount": 0.95, "label": "周投95折" },
  { "min_days": 14, "discount": 0.85, "label": "双周85折" },
  { "min_days": 30, "discount": 0.75, "label": "月投75折" }
]
```

运营可调整的维度：
- 档位数量（可加可删，不限于 3 档）
- 每档的最低连续天数门槛（min_days）
- 每档的折扣比例（discount，0-1 之间）
- 每档的展示标签（label，前端展示用）
- 可按广告位类型差异化配置（弹窗和轮播各一套折扣规则）
- 总开关：`ad_discount_enabled`（true/false），关闭则所有包天均原价

计算逻辑（伪码）：
```
用户选择连续投放 N 天
→ 从折扣规则中找到 min_days <= N 的最大档位
→ 总价 = 日价 × N × discount
→ 无匹配档位则原价（discount = 1.0）
```

管理后台界面需求：折扣规则的增删改表格 + 实时预览（输入天数即时显示折后价）。

#### 3.6.1 广告两种模式与定价机制的关系（业务决策已确认）

广告有两种计费模式，由运营决定每个广告位启用哪一种：

| 模式 | 说明 | 默认 | 定价机制 |
|------|------|------|----------|
| **固定包天**（fixed_daily） | 固定日价，先买先得 | ✅ 默认模式 | `实际日价 = 基础价 × DAU系数`，连投享阶梯折扣 |
| **竞价**（bidding） | 多广告主出价竞争，高价者优先展示 | — | 出价 ≥ 动态底价才能参与，每日按出价排序结算 |

运营可配置的控制点：
- **模式切换**：每个广告位可在管理后台切换 fixed_daily / bidding（当前 `ad_slots` 表已有 `billing_mode` 概念，需将其变为广告位级别的默认模式配置）
- **基础日价**：运营可随时在广告位管理页面调整 `daily_price_diamond`（按天模式的基础价）
- **每天最低钻石数**：运营可设置包天模式的最低日价下限，DAU 系数计算结果不得低于此值

三个定价机制各管一件事：
- **DAU 系数**管"广告位值多少钱"（按天和竞价都受影响）
- **动态底价**管"竞价至少出多少"（仅竞价模式）
- **阶梯折扣**管"买久了便宜多少"（仅按天模式）

广告主看到的最终价格：
```
按天模式：总价 = max(基础价 × DAU系数, 最低日价) × 天数 × 阶梯折扣
竞价模式：出价 ≥ 动态底价（动态底价本身受 DAU 系数和市场成交价影响）
```

### P2 — 规模期功能，当前冷启动阶段可暂缓

#### 3.9 商圈分区广告

**定价方案描述**：广告位按商圈分区投放，企业主投放所在商圈的广告位。100 个商圈 × 3 坑位/商圈 = 大量库存。

**大白话**：现在所有广告全站所有用户看到同一批。等将来有几十上百个商圈，广告位就按商圈拆分，比如"朝阳区弹窗位"和"海淀区弹窗位"是不同的广告位，企业主只投自己所在商圈。这样广告库存从全站 3 个坑变成 100 个商圈 × 3 个坑 = 300 个坑，能容纳更多广告主。

**实际状态**：Store 模型有行政区划（administrative_region）但无"商圈"概念。广告位无按商圈分区字段。当前全部为全站级别，符合文档中"冷启动期暂按全站统一"的描述。

**业务决策（已确认）**：现在实施。

**方案**：

**广告地域定向体系 — 两级分类：商圈 + 区域**

大白话：广告投放有两种地域维度。"商圈"是精准的，比如"望京商圈"就那几条街的用户能看到；"区域"是粗的，比如"朝阳区"整个区的用户都能看到。商圈优先级最高——如果用户同时属于望京商圈和朝阳区，先匹配商圈广告，商圈没有广告了再匹配区域广告。运营可以随时调优先级。

| 层级 | 说明 | 粒度 | 举例 | 默认优先级 |
|------|------|------|------|-----------|
| 商圈 | 精准投放，覆盖特定商圈的用户 | 小（几条街） | 望京商圈、五道口商圈 | 1（最高） |
| 区域 | 泛投放，覆盖整个行政区的用户 | 大（整个区/市） | 朝阳区、海淀区 | 2 |
| 全站 | 所有用户都能看到 | 全部 | — | 3（最低） |

实现方式：

1. 新建 `ad_target_zones` 表（统一地域定向表）：
   - `zone_id`：主键
   - `zone_type`：`district`（商圈）/ `region`（区域）
   - `zone_name`：名称（如"望京商圈"、"朝阳区"）
   - `priority`：优先级数字，越小越优先，运营可调
   - `parent_zone_id`：上级区域（商圈可挂在区域下面，如望京商圈 → 朝阳区）
   - `geo_scope`：覆盖范围描述（关联门店列表 / 行政区划 ID 等）
   - `status`：启用/停用

2. `ad_slots` 表新增 `zone_id` 字段（替代原 `business_district_id`，NULL 表示全站级别）

3. 广告分发优先级匹配逻辑：
```
用户打开首页 → 获取用户所在位置
→ 第 1 步：查用户属于哪个商圈（zone_type = district）→ 有商圈广告 → 展示
→ 第 2 步：商圈无广告 → 查用户属于哪个区域（zone_type = region）→ 有区域广告 → 展示
→ 第 3 步：区域也无广告 → 展示全站广告（zone_id = NULL）
```

4. 运营可调整的维度：
   - 商圈和区域的增删改（名称、覆盖范围、关联门店）
   - **优先级动态调整**：运营可在后台拖拽排序或直接修改 priority 数字，改变匹配顺序（比如某个区域搞大促时临时提升到商圈同级）
   - 每个商圈/区域的广告位数量和类型
   - 商圈与区域的父子关系（一个商圈属于哪个区域）
   - 全站级别广告位始终保留作为兜底

5. 冷启动期可先只建 1 个全站区域，后续按需拆分商圈和区域
6. **联合投放**：运营可将多个商圈和/或区域打包成一个"联合广告组"，广告主一次投放覆盖多个地域

**联合投放方案**：

大白话：比如望京商圈、五道口商圈、中关村商圈三个商圈单独卖广告位没人买。运营把这三个商圈打包成"海淀朝阳核心圈"联合广告组，广告主买一次就同时在三个商圈展示，价格比单独买三个便宜。也可以把商圈和区域混合打包（比如"望京商圈 + 整个丰台区"）。等某个商圈火了，运营随时可以把它拆出来单独卖。

实现方式：
- 新建 `ad_zone_groups` 表（联合广告组）：group_id、group_name、status、pricing_mode
- 新建 `ad_zone_group_members` 关联表：group_id、zone_id（关联 ad_target_zones，商圈和区域都可以加入）
- 运营在管理后台操作：
  - 创建联合广告组 → 选择要打包的商圈和/或区域 → 设置联合组名称
  - 随时增删成员（把某个商圈或区域加入或移出联合组）
  - 设置联合组的定价方式：
    - `sum`：各成员单价之和（无折扣）
    - `discount`：各成员单价之和 × 联合折扣（如 0.8 = 八折）
    - `fixed`：运营直接设定一个固定联合价
  - 联合组总开关（启用/停用）
- 广告主投放时：可以选单个商圈/区域投放，也可以选联合广告组一键投放多个地域
- 广告分发时：联合组内的广告在所有成员地域的对应广告位展示

#### 3.10 分阶段调价触发

#### 3.11 信息流广告 / CPM 计费

**定价方案描述**：C 级信息流广告按曝光计费（CPM = Cost Per Mille，每千次曝光的价格），不限量。

**大白话**：跟首页弹窗/轮播是两种完全不同的广告。弹窗/轮播是稀缺显眼的黄金位置，按天/竞价收费；信息流广告是夹在内容列表中间的小卡片广告（比如商家列表第 3 条和第 8 条之间插一条），不限量、不稀缺、容易划走，按"被多少人看到了"收费更合理。

两种广告的区别：

| 类型 | 位置 | 特点 | 收费 |
|------|------|------|------|
| 弹窗/轮播 | 首页一打开就看到 | 稀缺、显眼、强制曝光 | 按天/竞价（当前方案） |
| 信息流小广告 | 夹在列表内容中间 | 不限量、容易划走 | CPM 按曝光次数 |

CPM 举例：定价 10 钻石/千次曝光，被 3000 人看到 → 广告主付 30 钻石；没人看到 → 不付钱。风险在平台（没人看就没收入），适合日活高的时候。

**实际状态**：billing_mode 只有 fixed_daily / bidding / free，无 CPM 模式。小程序当前也没有内容列表场景可以插入信息流广告。

**业务决策（已确认）**：现在实施。先建好 CPM 计费能力，等有列表场景时可立即投入使用。

**方案**：

1. `AdCampaign.billing_mode` 新增 `cpm` 选项（现有：fixed_daily / bidding / free）
2. `ad_slots` 表新增 CPM 相关字段：
   - `cpm_price_diamond`：每千次曝光价格（钻石），运营可配
   - `slot_category`：广告位分类，新增 `feed`（信息流），区分于现有的 `popup` / `carousel` / `announcement`
3. 曝光计数服务：
   - 每次广告展示时记录一条曝光日志（ad_campaign_id、user_id、timestamp）
   - Redis 实时计数 + 每日定时任务归档到数据库
4. CPM 计费逻辑：
   - 每日凌晨定时任务：统计每个 CPM 广告计划的前日曝光次数
   - `扣费 = 曝光次数 / 1000 × cpm_price_diamond`（向上取整到最小单位）
   - 曝光为 0 → 不扣费
   - 通过 AdBillingService 创建 billing_type = `cpm_deduct` 的计费记录
5. 运营可调整的维度：
   - 每个信息流广告位的 CPM 价格
   - CPM 广告的每日预算上限（广告主设置，花完当天停止展示）
   - CPM 广告位的创建和管理（在广告位管理页面）
6. 前端对接（等有列表场景时）：
   - 内容列表中按间隔插入广告卡片（如每 5 条内容插 1 条广告）
   - 广告卡片样式与普通内容一致，标注"广告"标签
   - 插入位置和频率由运营配置

## 十、决策项行业对标分析与最终建议

> 以下对每个决策项进行大公司（美团/腾讯/阿里）、游戏公司（米哈游/腾讯游戏/网易）、小众平台、活动策划公司的横向对比，基于本项目"未上线、可一次性投入、长期低维护成本"的原则给出最终建议。

---

### D1：钻石配额发放比例

**选项**：A 固定 1:1 | B 可配比例（存 lottery_strategy_config）

**行业做法对比**：

| 类型 | 公司/平台 | 做法 | 比例是否可配 |
|------|-----------|------|------------|
| 大厂游戏 | 腾讯（王者荣耀） | 充值 → 点券比例 1:10 固定，但活动期间"首充双倍"（本质就是可配比例） | 可配 |
| 大厂游戏 | 米哈游（原神） | 充值 → 创世结晶基础比例固定，但"空月祝福"给额外每日产出 | 可配（活动叠加） |
| 电商大厂 | 阿里（淘金币） | 不同行为（签到/购物/评价）产出比例独立配置 | 可配 |
| 本地生活 | 美团（美团币） | 消费 → 美团币，比例按活动/品类/季节动态调整 | 可配 |
| 虚拟物品 | Steam/网易 BUFF | 充值比例固定，但手续费率可配 | 部分可配 |
| 小众平台 | 各类积分商城 | 消费 → 积分，比例按活动可调 | 可配 |

**行业共识**：所有平台的虚拟货币产出比例都是可配的。固定比例是经济系统设计反模式 — 经济失衡时固定比例 = 改代码发版，可配比例 = 后台改数字。

**本项目适配**：后端已有 lottery_strategy_config 表（config_key + config_value JSON + value_type，按 lottery_campaign_id 存储）。新增 diamond_quota_ratio 完全复用现有框架。

**最终建议：B（可配比例），初始值 1.0**。存 lottery_strategy_config，与 threshold_high 等现有配置同一管理界面。

---

### D2：配额耗尽行为

**选项**：A 过滤（踢掉钻石奖品） | B 降档（给最小额钻石） | C 可配

**行业做法对比**：

| 类型 | 公司/场景 | 做法 | 用户体验 |
|------|-----------|------|---------|
| 抽卡游戏 | 原神（保底） | 限定池资源耗尽 → 降档到标准池 | 好（永远有东西拿） |
| 抽卡游戏 | 王者荣耀（夺宝） | 高级奖池用完 → 降档到普通奖池 | 好 |
| 电商 | 美团（优惠券） | 预算耗尽 → 不展示该档优惠券 | 中性（用户无感知） |
| 电商 | 淘宝（双十一红包） | 红包池发完 → 停止发放 | 可接受 |
| 活动策划 | 线下活动公司 | 大奖发完 → 降档到安慰奖 | 好（参与感） |

**关键区别**：游戏行业偏"降档"（用户体验优先），电商偏"过滤"（预算控制优先）。本项目兼具抽奖（游戏属性）和广告经济（商业属性）。

**本项目适配**：lottery_strategy_config 已支持按活动存配置。可配的实现成本 ≈ 只做一种的成本（BuildPrizePoolStage 内根据值走一个 if 分支）。

**最终建议：C（可配），默认 'filter'**。config_key='quota_exhausted_action', config_value='filter'。上线初期用 filter（保守），验证后切 downgrade（提升体验）。不同活动可用不同策略。

### D3：DAU 统计数据源

**选项**：A users.last_login | B user_behavior_tracks | C 并集

**行业做法对比**：

| 类型 | 公司/平台 | DAU 定义 |
|------|-----------|---------|
| 社交 | 微信（小程序官方） | "访问过小程序的用户" = 打开即算 |
| 广告 | 腾讯广告 | 产生过 API 请求的唯一用户 |
| 本地生活 | 美团 | 当日有访问行为的唯一设备/用户 |
| 短视频 | 字节跳动 | 打开过 APP 或小程序的唯一用户 |
| 游戏 | 行业通用 | **当日登录过的唯一用户** |
| 电商 | 阿里（淘宝） | 产生有效行为（搜索/浏览/购买）的用户 |

**关键分析**：对于广告定价，DAU = "多少人可能看到广告"，**登录即算**是最合理口径（与游戏行业和腾讯广告一致）。精细行为跟踪对广告定价精度过剩。

**本项目适配**：users 表已有 `last_login`（注意：字段名是 `last_login` 而非 `last_login_at`，JWT 签发时更新），`COUNT(DISTINCT user_id) WHERE last_login >= 昨日` 即可。零额外表、零上报。另有 `last_active_at` 字段（登录、抽奖等操作时更新，用于用户分群），覆盖面更广但口径不同。还有 `authentication_sessions` 表（含 `login_platform` 可区分小程序/Web），以及 `user_behavior_tracks` 表（全行为跟踪）可作为未来升级数据源。

**最终建议：A（last_login）→ 已被 D11 修正为 B（last_active_at）**。原因：代码审查发现 `last_login` 仅在 JWT 签发时更新，JWT 有效期 7 天，导致 DAU 低估 70-85%。详见第十四节 D11 分析。落地方式：auth middleware 加 1 行异步更新 `last_active_at`，DAU 定时任务查询 `last_active_at` 替代 `last_login`。

### D4：用户位置获取（商圈匹配）

**选项**：A GPS 定位 | B 门店推断 | C 全站投放

**行业做法对比**：

| 类型 | 公司/平台 | 做法 | 用户授权率 |
|------|-----------|------|-----------|
| 本地生活 | 美团/大众点评 | GPS + 历史地址 + 搜索行为，多源融合 | 高（核心功能依赖定位） |
| 外卖 | 饿了么 | 配送地址为主，GPS 为辅 | 高 |
| 广告平台 | 腾讯广告 | IP 地理位置 + 设备定位 + 用户资料 | 被动获取 |
| 活动平台 | 大麦/秀动 | 用户选择的城市，非实时定位 | 100%（主动选择） |
| 小众 O2O | 各类到店平台 | **绑定门店推断**，不要求定位 | 100%（无需授权） |

**关键分析**：
- 微信小程序 GPS 授权率约 60-70%，拒绝的 30% 直接丢失地域数据
- 本项目是餐厅积分系统，用户到店消费天然绑定门店，消费记录 → 门店 → 行政区划链路已完整
- 门店推断覆盖率 100%，且反映**实际消费位置**（比用户当前物理位置更有广告价值）

**本项目适配**：Store 模型已有 administrative_region 关联。users → consumptions → stores → regions 链路完整，小程序零额外开发。

**最终建议：B（门店推断）**。100% 覆盖率、零额外开发、比 GPS 更准确（消费位置 > 物理位置）。

### D5：竞价排名坑位数

**选项**：A 每天 1 赢家 | B max_display_count 个

**行业做法**：Google Ads 搜索顶部 3-4 个位、百度竞价 3-5 个位、腾讯广告多坑位、美团搜索多推广位、淘宝直通车多位置 — **全行业多坑位，无一例外**。

**本项目适配**：ad_slots.max_display_count 字段已有（home_popup=3, home_carousel=3, lottery_popup=2）。

**最终建议：B（max_display_count）**。行业硬标准，无需讨论。

### D6：包天锁定时间颗粒度

**选项**：A 自然日 | B 滚动 24h

**行业做法**：Google Ads 按账户时区自然日、腾讯广告 CPT 自然日、百度自然日、美团自然日、所有游戏日任务 0 点重置 — **全球广告行业以自然日计费，从未有滚动 24h 的先例**。

**本项目适配**：BeijingTimeHelper 统一北京时间，ad-cron-jobs.js 凌晨 01:00 结算（隐含自然日周期）。

**最终建议：A（自然日）**。全球标准，无需讨论。

### D7：CPM 计费精度

**选项**：A 比例 ceiling | B 不足千免费 | C 上取整千次

**行业做法**：Google/腾讯/Facebook/百度全部按实际曝光比例计费。区别仅在取整方式。

| 曝光数 | 单价 10 钻/千次 | A: ceiling 比例 | B: 不足千免 | C: 上取整千次 |
|--------|---------------|----------------|------------|-------------|
| 500 | 10 | **5 钻** | 0 钻 | 10 钻 |
| 1500 | 10 | **15 钻** | 10 钻 | 20 钻 |
| 3001 | 10 | **31 钻** | 30 钻 | 40 钻 |

A 最公平（广告主多付最多 1 钻，平台不亏）。B 让小流量白嫖。C 对广告主不友好。

**最终建议：A（ceiling 比例）**。`Math.ceil(impressions * cpm_price_diamond / 1000)`。

### D8：竞价落选者是否扣费

**选项**：A 仅中标者 | B 所有参与者（当前代码）

**行业做法**：

| 公司/平台 | 做法 | 扣费机制 |
|-----------|------|---------|
| Google Ads | **仅中标者付费**，付第二名出价+1 | GSP 广义第二价格拍卖 |
| Facebook Ads | **仅中标者付费** | 修改版 Vickrey 拍卖 |
| 腾讯广告 | **仅展示的广告扣费** | CPC/CPM 实际产生扣 |
| 百度竞价 | **仅展示并点击的广告扣费** | CPC |
| 美团/淘宝 | **仅展示的广告扣费** | CPC/CPM |
| 所有拍卖平台 | **仅赢家付款**，落选者退保证金 | 标准拍卖 |

**行业共识：全球没有任何一个广告平台对落选者扣费**。对所有参与者扣费 = 参与竞价就亏钱 = 没人参与 = 竞价系统死亡。这是当前代码的**必修缺陷**。

**最终建议：A（仅中标者扣费）**。零争议。重构 processDailyBidding（B18）是不可跳过的必做项。

**进阶路径**：稳定后可升级到 GSP（中标者付第二名出价+1），激励真实出价。冷启动期按出价直接扣即可。

---

### D9：调价是否自动执行

**选项**：A 手动确认 | B 自动

**行业做法**：Google/腾讯的**实时竞价**完全自动，但**结构性调价**（基础价/系数变更）早期必须人工确认。游戏公司虚拟货币汇率调整**必须经产品经理确认**。美团涉及用户通知的调价需审批。

**本项目的"调价"是结构性的**（DAU 系数变 → 所有广告位基础价变），影响面大。

**最终建议：A（手动确认）**。auto_apply=false 存 system_configs，稳定 3 个月后改 true。

---

### D10：联合投放定价方式

**选项**：A 求和 | B 折扣 | C 固定价 | D 三种都支持

**行业做法**：腾讯广告支持多区域打包加权定价，美团支持城市联投+折扣，电商套餐三种模式并存，游戏礼包通常固定打包价，活动策划公司报价单三种模式全用。

**本项目适配**：ad_zone_groups 加 pricing_mode ENUM('sum','discount','fixed') + discount_rate + fixed_price 三个字段。三种模式的计算差异仅一个 switch-case，不到 10 行代码。

**最终建议：D（三种都支持）**。实现成本差异约等于零，运营灵活性极高。

### D11：DAU 统计字段选择

**背景**：D3 决策确定用 `users` 表统计 DAU，但实际代码审查发现关键问题 — `last_login` 仅在 JWT 签发时更新（`UserService.updateLoginStats`），而 JWT 有效期为 7 天。用户登录一次后连续 6 天活跃使用 app 都不会更新 `last_login`，DAU 会被严重低估。

**真实数据库验证（2026-03-01）**：

| 数据源 | 7 天内 | 1 天内 | 问题 |
|--------|--------|--------|------|
| `users.last_login` | 4 人 | 1 人 | 仅统计实际登录事件，JWT 7 天有效期内复访不计 |
| `users.last_active_at` | **0 人** | **0 人** | **字段从未被业务代码更新**，37/68 个用户值为 NULL（仅迁移脚本初始化过） |
| `authentication_sessions` | 175 条 / 9 人 | 4 条 | 每次登录创建会话，数据最完整 |
| `user_behavior_tracks` | **0 条** | **0 条** | **表完全为空**，行为追踪功能从未实际写入数据 |

**选项**：A `last_login` | B `last_active_at`（需先修复） | C `authentication_sessions`

**行业做法对比**：

| 类型 | 公司/平台 | DAU 统计方式 | 数据来源 |
|------|-----------|------------|---------|
| 社交 | 微信（小程序官方后台） | "访问过小程序的独立用户" | 请求日志（服务端无感采集） |
| 广告平台 | 腾讯广告 | 产生过 API 请求的唯一用户 | 请求日志 |
| 本地生活 | 美团/饿了么 | 当日有访问行为的唯一用户 | 请求日志 + 埋点 |
| 短视频 | 抖音/快手 | 打开过 APP 的唯一用户 | 客户端心跳 + 服务端请求日志 |
| 游戏（大厂） | 米哈游/腾讯游戏/网易 | **当日登录过的唯一用户** | 登录事件日志 |
| 游戏（中小） | 各类手游 | 当日有任何操作的唯一用户 | 操作事件日志 |
| 电商 | 淘宝/京东 | 有效行为（搜索/浏览/购买）的用户 | 行为埋点 |
| 小众交易平台 | Steam/网易BUFF | 登录或浏览的唯一用户 | 会话日志 |
| 活动策划公司 | 各类 | 扫码/签到/参与的唯一用户 | 活动事件日志 |

**行业共识**：广告定价的 DAU = "多少人可能看到广告"。大公司用请求日志（最准），游戏公司用登录事件（够用），关键是**不能有大量活跃用户被遗漏**。

**本项目的核心矛盾**：
- `last_login` 只在 JWT 签发时更新 → JWT 7 天有效 → 用户一周只计 1 天 → DAU 低估约 70-85%
- `last_active_at` 字段存在但**从未被业务代码写入** → 目前不可用
- `authentication_sessions` 统计的也是"登录事件"，与 `last_login` 同口径

**最适合本项目的方案**：

**B（last_active_at），但需先修一行代码**。

具体做法：在 `middleware/auth.js` 的 `authenticateToken` 中间件里，JWT 验证通过后追加一行异步更新 `last_active_at`。这样每个认证请求都会刷新用户活跃时间，DAU = "当日发过任何请求的唯一用户"。

实施成本：auth middleware 加 1 行代码（异步更新，不阻塞请求），零新增表、零新增依赖。

DAU 定时任务查询：`SELECT COUNT(DISTINCT user_id) FROM users WHERE last_active_at >= 昨日 00:00 AND last_active_at < 今日 00:00`

为什么不选 A（last_login）：JWT 7 天有效期导致严重低估，广告定价偏低损害平台收入。
为什么不选 C（authentication_sessions）：与 last_login 同口径（只统计登录事件），同样低估。

**升级路径**：`last_active_at` → 未来如需按平台（小程序/Web）分别统计 DAU，可改用 `authentication_sessions` + `login_platform` 字段。

**最终建议：B（last_active_at + 修复 auth 中间件），一次性修复后长期零维护成本**。

### D12：微信小程序项目仓库位置

**行业做法对比**：

| 类型 | 公司/平台 | 仓库策略 | 理由 |
|------|-----------|---------|------|
| 大厂 | 腾讯（微信团队） | **独立仓库** | 小程序和服务端完全不同的构建部署流程 |

**本项目实际情况分析**：

| 对比维度 | 本仓库现状 | 小程序项目 | 结论 |
|---------|-----------|-----------|------|
| 运行时 | Node.js | 微信开发者工具 / uni-app | **完全不同** |
| 构建工具 | Vite（admin）/ 无构建（backend） | 小程序编译器 / uni-app CLI | **完全不同** |

### D13：广告位 slot_category 分类粒度

**选项**：A display/feed 二分 | B popup/carousel/announcement/feed 细分

**行业做法对比**：

| 类型 | 公司/平台 | 分类维度 | 设计思路 |
|------|-----------|---------|---------|
| 广告平台 | 腾讯广告 | **二级分类**：大类（展示/信息流/视频）+ 子类（Banner/插屏/开屏等） | 大类决定计费模式，子类决定展示形态 |
| 广告平台 | Google Ads | **二级分类**：网络类型（搜索/展示/视频/购物）+ 格式（文字/图片/轮播等） | 网络类型决定计费，格式决定素材 |
| 广告平台 | 百度广告 | **二级分类**：产品线（搜索/信息流/开屏）+ 创意格式 | 产品线决定计费模式 |
| 电商 | 淘宝/京东 | **二级分类**：广告产品（直通车/钻展/超推）+ 资源位（搜索首位/猜你喜欢等） | 广告产品决定计费 |
| 游戏 | Unity Ads / IronSource | **二级分类**：广告类型（激励视频/插屏/Banner）+ 放置位 | 类型决定计费（CPM/CPI） |
| 小众平台 | 各类 | 通常不分类或仅一级 | 广告位少时不需要复杂分类 |

**行业共识**：几乎所有广告平台都用**两级分类**：

| 层级 | 用途 | 决定什么 | 举例 |
|------|------|---------|------|
| **大类（category）** | 区分计费模式 | 按天/竞价 vs CPM | display vs feed |
| **子类（type）** | 区分展示形态 | 素材尺寸、交互方式 | popup / carousel / banner |

本项目已有 `slot_type`（popup/carousel/announcement）作为"子类"。新增 `slot_category` 应作为"大类"使用。

**方案 A（display/feed 二分）对本项目的适配**：

```
slot_category = 'display' → 计费模式：fixed_daily / bidding（按天/竞价）
slot_category = 'feed'    → 计费模式：cpm（按曝光）

slot_type 保持原有：popup / carousel / announcement / feed（新增）
```

两个字段各管一件事，不重叠：`slot_category` 管计费逻辑，`slot_type` 管展示渲染。

**方案 B（细分）的问题**：`slot_category = popup` 与 `slot_type = popup` 完全重复，违反数据库设计的第三范式，查询时不知道该用哪个字段过滤。

**最终建议：A（display/feed 二分）**。全行业标准的两级分类架构，与现有 `slot_type` 互不冲突，长期零维护成本。
