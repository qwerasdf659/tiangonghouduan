## 一、背景：一次抽奖在管线里怎么跑

每次抽奖按顺序经过以下 stage（见 `services/UnifiedLotteryEngine/pipeline/NormalDrawPipeline.js`）：

```
LoadCampaignStage → EligibilityStage → LoadDecisionSourceStage → BudgetContextStage
→ PricingStage → BuildPrizePoolStage → GuaranteeStage → TierPickStage → PrizePickStage
→ DecisionSnapshotStage → SettleStage
```

关键决策点是 **`LoadDecisionSourceStage`**：它输出一个 `decision_source`（四选一，带优先级），决定本次抽奖走哪条路：

| 优先级 | decision_source | 含义 |
|--------|-----------------|------|
| 1（最高） | `preset` | 命中预设 |
| 2 | `override` | 管理干预（**仅 force_win / force_lose；本轮将整体删除，见第五节**） |
| 3 | `guarantee` | 触发保底 |
| 4（最低） | `normal` | 正常加权抽取 |

后续 `TierPickStage`（选档位）与 `PrizePickStage`（档位内选具体奖品）根据 `decision_source` 决定"按干预走"还是"正常抽"。

## 三、合规约束（决定架构走向的根本变量）

依据 `docs/legal/合规整改执行清单.md` §4.1：

> **抽奖须公示中奖概率，后台算法与公示数字严格一致**（《规范促销行为暂行规定》强制）。

本项目业务链为"消费（真钱给商家）→ 积分 → 抽奖"，属**有奖销售**，归市场监管，执法比游戏抽卡更严。由此判定：

- **"按 user_id 暗中改某人中奖概率/强制某人中奖"，且不公示、与规则页不一致 = 合规红线**（虚构中奖 / 内定中奖 / 虚假宣传）。`force_win`、`force_lose`、`probability_adjust` 的个人暗箱语义均落入此类。
- 涉及的法律面（非法律意见，最终以执业律师为准）：《反不正当竞争法》第十条（禁止虚构/内定中奖）、第八条（虚假宣传）；《规范促销行为暂行规定》（公示一致、禁操纵）；《消费者权益保护法》第八/二十/五十五条（知情权、欺诈退一赔三）；涉宣传加《广告法》；极端情形（金额大 + 隐匿证据）上探《刑法》第 266 条诈骗罪。
- **"虚构中奖"不以用户是否受损为要件**——即便是"白送好处"，只要伪装成抽奖结果即违法，因其破坏的是抽奖公平与市场秩序。

**取证暴露面**：`lottery_management_settings` 按 user_id 落库 + 时间戳 + 操作人，是结构化的"自认证据"。一旦因投诉/举报/诉讼被调取，无辩解空间。这是删除 per-user 暗箱干预、而非"想办法隐藏"的根本原因（隐藏=故意毁证，属加重情节）。

### 合规的两条正当路径（本方案采用）

1. **对个人发奖** → 走"明示赠送/补偿"：用户清楚知道是平台给的，不伪装成抽中。本项目已有 `CustomerServiceCompensateService`（`cs_compensate`）实现此能力。
2. **对群体调赔率** → 走"公示的分级概率（按用户等级）"：客观进入条件 + 概率公示，同级一视同仁；不可精确指定到个人。

### B 线概率与预算安全网（连库 + 读码确认）

- **任何方案都无法超出预算账户**：抽奖前 `BuildPrizePoolStage.js:319` 用 `budget_before >= budget_cost` 把付不起的奖品提前踢出池；中奖后 `SettleStage.js:246-257` 按 `budget_cost` 扣预算。预算见底时奖品池自动降级到便宜奖/安慰奖。倍数/概率只改"预算允许范围内更倾向给什么"，击穿不了预算。
- **烧预算速度**：D3①（中奖率）+ D4①（概率命中式）最省；②系列（高价值档 / 权重提升）消耗明显更快。本轮均选 ①，预算压力最小。
- **合规要点**：等级越高中奖率越高，**必须像会员权益一样公示**，"花得多更易中"才合法（§4.1）。

### D8 已知冲突（公示数值 vs 代码档位）✅ 已拍板=甲

- 协议公示为 **高 10 / 中 25 / 低 65** —— **三档、无空奖档**。

- 冲突点：①公示"high=10%" 与 代码"high 封顶 8%" 对不上；②公示无 fallback/空奖档，代码有。
- **必须二选一**（否则 §4.1"后台=公示"无法满足）：
  - 方案甲：**改公示文案**为四档、且 high ≤ 8%（贴合现状代码，改动小）；
  - 方案乙：**调代码**档位结构/上限去匹配三档公示（改动大、动安全网，慎选）。
- **✅ 2026-06-04 已拍板：方案甲**（改公示文案为四档、high≤8%）。见 §十六 P2。

## 五、目标架构（删除暗箱干预 + 收口到现成合规能力）

### 5.1 删除项

- **写入侧**：`ManagementStrategy.forceWin` / `forceLose`、`CoreService.forceWinForUser` / `forceLoseForUser`、路由 `routes/v4/console/lottery/force-control.js`（`/force-win`、`/force-lose`）。
- **per-user 概率路由**：`routes/v4/console/lottery/adjustment.js` 的 `POST /probability-adjust`（按 user_id）**删除**；B 线改由"按用户等级配置概率"的新入口承载（见 5.3）。
- **消费侧**：`LoadDecisionSourceStage._checkOverride()` 整段移除；`decision_source` 的 `override` 分支从优先级链中摘除（preset > ~~override~~ > guarantee > normal）。`PrizePickStage.js` 中 force_win（约 92-140 行）与 force_lose（约 147-184 行）分支移除。
- **结算侧**：`SettleStage._consumeDecisionSource()` 内对 override 的 `status='used'` 标记移除。
- **模型/枚举**：`lottery_management_settings.setting_type` ENUM **收窄到移除 `force_win`/`force_lose`/`probability_adjust`/`user_queue` 全部四个值**（✅ 2026-06-04 P6 拍板：`user_queue` 同属按 user_id 暗箱预设、当前 0 使用，本轮一并删除，不再保留）。

### 5.2 发奖/补偿统一收口到 `cs_compensate`（已实现，无需新建）

- 服务：`services/CustomerServiceCompensateService.js`（ServiceManager key `cs_compensate`）。
- 路由：`POST /api/v4/console/customer-service/gm-tools/compensate`（`requireRoleLevel(100)`）。
- 能力：事务内原子发放**资产**（`BalanceService.changeBalance`，`business_type='cs_compensation'`）或**物品**（`ItemService.mintItem`，`source_type='cs_compensation'`）→ 写审计日志 → 在客服聊天插入系统消息（明示）。
- 合规契合点：用户可见（明示赠送，非伪装抽中）+ 自动留痕（自证清白）+ 预留 `max_compensate_amount` 额度阈值位（对应 D2）。

### 5.3 B 线公示分级概率（按用户等级，语义已定）

- 概率配置进 `lottery_strategy_config`，由 `DynamicConfigLoader.getValue(group, key, default, { lottery_campaign_id })` 统一读取；维度为"活动 / 用户等级"，**永不按 user_id**。
- **新建"按用户等级配置概率"的管理入口**（替代已删除的 per-user `adjustment.js`），运营按等级配概率档，配置即公示口径。
- **维度 = 独立"用户等级制度"**：等级作为一等公民单独建（累计消费/积分 → 等级），抽奖 B 线只**读取**等级、不自造分级。等级亦可被折扣/特权/客服优先级等其它功能复用。
- **D3 倍数语义 = 作用于中奖率**：在 `TierPickStage` 把"命中非 fallback 档"的总概率 ×multiplier，high/mid/low 内部比例不变，封顶服从 8% 硬上限。
- **D4 指定奖品 = 概率命中式**：本次抽奖先掷独立随机数，命中则直接锁定该奖（走独立分支，不碰权重/预算/保底），否则走正常抽取。
- 作用点在 `TierPickStage`，须服从既有 8% high 档硬上限（`TierPickStage.js:394` `MAX_HIGH_TIER_PROBABILITY=0.08`）与预算分层/保底安全网。
- **预算不可击穿**：抽奖前 `BuildPrizePoolStage` 过滤付不起的奖品 + 中奖后 `SettleStage` 按 `budget_cost` 扣减，双重兜底。
- 概率必须可公示，且后台数值 = 公示数值（满足 §4.1）；"等级越高越易中"须作为会员权益明示。

### 5.4 奖品 key 统一

- 全系统（写入、消费、奖品池对象、路由、`setting_data`）只用 `lottery_campaign_prize_id`（DB 主键真名）。

## 六、成长等级制度（✅ P1=乙 拍板：新增独立体系，B 线依赖项）

> B 线的分级维度由此体系提供。它是独立的一等公民，不只服务抽奖。
> **⚠️ 重要澄清：本体系 ≠ 现有 `users.user_level enum('normal','vip','merchant')`。后者是"身份类型"（普通/VIP/商家），不是消费成长等级，且 `merchant` 是商家、不应"更易中"。B 线不复用该 enum，另建成长等级（避免语义错配 + 日后二次迁移的真技术债）。**

- **定位**：累计消费/积分 → 成长等级（如青铜/白银/黄金…），稳定、可累计、用户可见自己等级。
- **维度键**：用 `level_key`（如 bronze/silver/gold），B 线 `lottery_strategy_config` 按 `level_key` 配概率档。
- **复用面**：抽奖中奖率加成、兑换折扣、特权功能、客服优先级等均可引用。
- **与抽奖的关系**：抽奖只**读**成长等级，不自造分级。
- **合规**：各等级的中奖率加成须作为**会员权益公示**，后台数值 = 公示数值。
- **待细化（上线前必须先定，否则 B 线无法配置）**：等级划分阈值、升降级规则、各等级概率档数值——属本体系设计，可与删除/清理并行推进。

## 十五、行业设计对比（大厂 / 小公司 / 游戏 / 活动策划 / 虚拟物品交易）

> 说明：以下为公开产品形态/行业通行模式层面的归纳，不涉及任何非公开内部实现；合规细节以执业律师为准（与 §三 口径一致）。

### 15.1 共同骨架：三线分离（所有正规玩家的通用范式）

"运营想影响抽奖结果"在行业里一律拆成互不混淆的三条线，混在一起 = 同时背合规风险 + 技术债。本项目旧的 `force_win`/`probability_adjust` 恰恰把 (b)(c) 塞进了 (a) 的暗箱。

| 线 | 职责 | 决定什么 | 本项目对应 |
|----|------|----------|-----------|
| (a) 概率引擎 | 公示、版本化、可审计 RNG + 保底 | "这次中哪个档" | `UnifiedLotteryEngine` 管线 |
| (b) 人群/等级分层 | 标签/会员等级，**群体级，永不到个人** | "哪类人更优惠" | B 线（本方案新建成长等级） |
| (c) 明示发放/补偿 | 邮件/工单/GM 工具，用户可见 + 留痕 | "直接给谁发东西" | `cs_compensate`（已就绪） |

### 15.2 各类主体横向对比

| 主体 | (a) 概率引擎 | (b) 分层/定向 | (c) 发奖/补偿 | 暗箱操纵个人 |
|------|-------------|--------------|--------------|--------------|
| 游戏大厂（米哈游/腾讯游戏/网易） | 服务端权威 RNG + 保底，国内强制公示概率 | 卡池/限定 + 充值累计等级 | 邮件补偿、GM 明示发放 | ❌ 严禁 |
| 互联网大厂营销中台（阿里/腾讯/美团） | 概率配置化 + 版本 + 审计；风控只"限制"不"强中" | DMP 人群圈选 + 会员体系（88VIP/PLUS/美团会员），群体级 | 权益/券明示发放 | ❌ |
| SaaS 营销工具（有赞/微盟） | 抽奖组件概率可配 | 客户标签/会员等级、人群包 | 营销券/积分明示发放 | ❌（产品不提供该能力） |
| 小公司/外包 | 概率常硬编码或简单配置 | 常无或一个 enum | 直接改库/手动发 | ⚠️ 常见——本项目旧实现即此坑 |
| 活动策划公司 | 接第三方抽奖 SaaS，概率公示 | 报名人群/名单 | 名单内明示中奖 | ❌ |
| 虚拟物品/二手交易（Steam市场/网易BUFF/闲鱼/转转） | 非抽奖：担保交易 + 手续费 | 信用分/实名 | 担保托管、明示成交 | —（"明示交易"原则一致） |

**提炼**：规模越大，(a) 越版本化可审计、(b) 越独立成体系、(c) 越流程化留痕；"绝不暗箱操纵个人抽奖结果"是从游戏大厂到 SaaS 的无差别红线。

### 15.3 为何"大厂三线分离 + 轻量落地"最适合本项目

- 本项目是"消费→积分→抽奖"的**有奖销售**，受市场监管比游戏抽卡更严，必须按大厂合规标准做。
- 体量是小公司，但已有 Stage 管线 + `lottery_strategy_config` 动态配置 + `cs_compensate`，足以承接大厂的"三线分离"思想而无需新框架。
- 结论：采用 A+B 模型（个人走明示补偿、群体走公示分级概率），用现有技术栈轻量落地，长期技术债最低。
