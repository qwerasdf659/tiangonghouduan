# 档位门控重构：去预算门控，改资源级过滤

> **解决的问题**：当前引擎用 BUDGET_POINTS 门控整个档位（high/mid/low），导致用户预算耗尽后钻石奖品被连带锁死——即使钻石不消耗 BUDGET_POINTS、DIAMOND_QUOTA 仍有余量。经济模型设计的"钻石续航"（碎片先耗尽、钻石继续产出的四阶段体验弧线）在现有架构下无法实现。
>
> **根因**：档位门控把"概率分配"和"资格判断"耦合在同一层。两种独立资源（BUDGET_POINTS 控碎片、DIAMOND_QUOTA 控钻石）被一个统一的门控机制绑在一起，无法独立运作。
>
> **方案**：去掉档位级预算门控，改为奖品级资源检查。档位系统只管概率分配（5%/25%/70%），资格判断下沉到每个奖品各自的资源类型。
>
> 关联文档：[抽奖经济模型设计方案](./抽奖经济模型设计方案.md) 第 7.5 节

---

## 一、问题复现

### 1.1 当前架构的过滤链路

```
BudgetTierCalculator
  → 计算门槛：threshold_mid=3, threshold_low=1
  → 判定用户档位：budget >= threshold_mid → B2, budget >= threshold_low → B1, else B0
  → 输出 allowed_tiers：B0=['fallback'], B1=['low','fallback'], B2=['mid','low','fallback'], B3=全部

BuildPrizePoolStage
  → _filterByBudget (奖品级)：prize_value_points <= budget || pvp === 0 → 通过
  → _filterByAllowedTiers (档位级)：不在 allowed_tiers 里的档 → 整档删除
```

### 1.2 预算耗尽时的实际行为

用户 BUDGET_POINTS = 0，DIAMOND_QUOTA = 200（充足）：

| 过滤环节 | high 档 | mid 档 | low 档 | fallback |
|---|---|---|---|---|
| _filterByBudget（奖品级） | 钻石×600(pvp=0) ✅, 钻石×180(pvp=0) ✅ | 钻石×60(pvp=0) ✅, 碎片×150 ❌, 碎片×80 ❌ | 钻石×15(pvp=0) ✅, 碎片×6 ❌ | 积分×10 ✅ |
| _filterByAllowedTiers（档位级） | budget=0 → B0 → allowed=['fallback'] → **整档删除** | **整档删除** | **整档删除** | ✅ 保留 |
| **最终结果** | **❌ 全部钻石被杀** | **❌ 全部被杀** | **❌ 全部被杀** | ✅ 只剩保底 |

用户体验：95% 出保底，5% 偶尔出钻石（仅当 high 档无门槛时）。"钻石续航"失效。

### 1.3 期望的正确行为

| 过滤环节 | high 档 | mid 档 | low 档 | fallback |
|---|---|---|---|---|
| 按资源检查（奖品级） | 钻石×600(DIAMOND_QUOTA≥600?) → 视配额 | 钻石×60 ✅, 碎片×150 ❌, 碎片×80 ❌ | 钻石×15 ✅, 碎片×6 ❌ | 积分×10 ✅ |
| 档位门控 | **无**（不存在） | **无** | **无** | **无** |
| **最终结果** | 钻石 ✅（按 DIAMOND_QUOTA 过滤） | 钻石×60 ✅ | 钻石×15 ✅ | 积分×10 ✅ |

用户体验：37% 出钻石（和预算充足时一致），碎片全部过滤，保底补位。四阶段弧线正确实现。

## 二、方案设计

### 2.1 核心原则

```
档位系统：只管概率分配（稀有度分组）
  high(5%) / mid(25%) / low(70%) → 权重不变，不做准入门控

资格检查：下沉到单个奖品，按资源类型独立判断
  钻石奖品 → DIAMOND_QUOTA >= material_amount?
  碎片奖品 → BUDGET_POINTS >= material_amount × budget_value_points?
  保底奖品 → 永远可用

降级逻辑：不变
  选中的档位过滤后为空 → TierPickStage 自动降级到下一档
```

### 2.2 架构对比

```
之前（档位门控）：

  BudgetContext → BudgetTierCalculator → allowed_tiers
                                             ↓
  BuildPrizePool → _filterByBudget(pvp) → _filterByAllowedTiers(档位级) → 奖品池
                                                                             ↓
  TierPick → PrizePick → Settle → _deductBudget(prize_value_points)


之后（资源级过滤）：

  BudgetContext → 传递资源余额（budget_points, diamond_quota）
                       ↓
  BuildPrizePool → _filterByResourceEligibility(按资源类型) → 奖品池
                                                                ↓
  TierPick → PrizePick → Settle → _deductByResourceType(按资源扣减)
```

### 2.3 不变的部分

| 组件 | 状态 | 说明 |
|---|---|---|
| 档位概率 5%/25%/70% | **不变** | TierPickStage 按权重选档 |
| BxPx 概率矩阵 | **不变** | 体验平滑机制不受影响 |
| 档内权重选奖 | **不变** | PrizePickStage 按 win_weight 选 |
| 降级逻辑 | **不变** | 过滤后档位为空 → 向下降级 → 向上搜索 → fallback |
| 保底机制 | **不变** | GuaranteeStage 不受影响 |
| 连抽 / 批次 | **不变** | 批次逻辑不受影响 |
| 管线结构 | **不变** | 11 个 Stage 的顺序和职责不变 |

## 三、逐文件改动清单

### 3.1 BuildPrizePoolStage.js（核心改动）

**文件**：`services/UnifiedLotteryEngine/pipeline/stages/BuildPrizePoolStage.js`

**改动 1：替换 `_filterByBudget` 为 `_filterByResourceEligibility`**

当前逻辑：
```javascript
_filterByBudget(prizes, budget) {
  return prizes.filter(prize => {
    const prize_cost = prize.prize_value_points || 0
    return prize_cost <= budget || prize_cost === 0
  })
}
```

新逻辑：
```javascript
_filterByResourceEligibility(prizes, resource_context) {
  const { budget_points, diamond_quota } = resource_context

  return prizes.filter(prize => {
    if (prize.is_fallback || prize.prize_type === 'points') {
      return true
    }

    if (prize.material_asset_code === 'DIAMOND') {
      return diamond_quota >= (prize.material_amount || 0)
    }

    // 碎片类及其他 virtual 奖品：检查 BUDGET_POINTS
    if (prize.prize_type === 'virtual' && prize.material_asset_code) {
      const cost = (prize.material_amount || 0) * (prize._budget_value_points || 1)
      return budget_points >= cost
    }

    return true
  })
}
```

> `prize._budget_value_points` 需要在 LoadCampaignStage 或 BuildPrizePoolStage 加载奖品时，从 `material_asset_types` 表 JOIN 获取。如果不想改查询，可以硬编码 `budget_value_points = 1`（本方案 red_shard 的 budget_value_points 固定为 1）。

**改动 2：删除 `_filterByAllowedTiers` 调用**

在 `execute()` 方法中，删除或跳过 `_filterByAllowedTiers` 的调用。保留函数定义（标记 @deprecated）以防回滚需要。

```javascript
// 之前
const tier_filtered = this._filterByAllowedTiers(grouped, allowed_tiers)

// 之后：跳过档位门控，直接使用分组后的结果
const tier_filtered = grouped  // 不再按档位过滤
```

**改动 3：调整过滤链路中的参数传递**

`_filterByBudget` 的调用点改为 `_filterByResourceEligibility`，传入 `resource_context` 而非单一 `budget` 值。

### 3.2 BudgetContextStage.js（简化输出）

**文件**：`services/UnifiedLotteryEngine/pipeline/stages/BudgetContextStage.js`

**改动**：在输出的 `budget_data` 中增加 `diamond_quota` 字段（如果尚未包含），确保 BuildPrizePoolStage 可获取两种资源余额。

```javascript
// budget_data 输出应包含：
{
  budget_before: Number,      // BUDGET_POINTS 余额（已有）
  diamond_quota: Number,      // DIAMOND_QUOTA 余额（新增或确认已有）
  budget_tier: String,        // 可保留用于日志/分析，不再用于门控
  allowed_tiers: Array        // 可保留用于日志/分析，不再用于门控
}
```

### 3.3 BudgetTierCalculator.js（简化）

**文件**：`services/UnifiedLotteryEngine/compute/calculators/BudgetTierCalculator.js`

**改动**：

- `_calculateDynamicThresholds`：可保留计算逻辑（用于日志和监控指标），但其输出不再用于门控决策
- `_determineBudgetTier`：同上，保留 B0-B3 判定用于日志，不用于 allowed_tiers 门控
- `BUDGET_TIER_AVAILABILITY` 映射表：标记 @deprecated，不再在过滤链路中使用

> 不建议直接删除这些函数——保留它们用于日志、监控面板和 A/B 测试分析。只是切断它们与 BuildPrizePoolStage 的门控关系。

### 3.4 SettleStage.js（资源扣减修正）

**文件**：`services/UnifiedLotteryEngine/pipeline/stages/SettleStage.js`

**改动**：修正预算扣减逻辑，按资源类型扣减实际成本。

当前逻辑：
```javascript
if (budget_provider && final_prize.prize_value_points > 0) {
  budget_deducted = await this._deductBudget(
    budget_provider,
    final_prize.prize_value_points,  // 扣的是 prize_value_points（门槛值）
    ...
  )
}
```

新逻辑：
```javascript
if (budget_provider && final_prize.prize_type === 'virtual') {
  if (final_prize.material_asset_code === 'DIAMOND') {
    // 钻石：只扣 DIAMOND_QUOTA（已有 _deductDiamondQuota），不扣 BUDGET_POINTS
    budget_deducted = 0
  } else if (final_prize.material_asset_code) {
    // 碎片等：扣 material_amount × budget_value_points
    const actual_cost = (final_prize.material_amount || 0) * (final_prize._budget_value_points || 1)
    if (actual_cost > 0) {
      budget_deducted = await this._deductBudget(
        budget_provider,
        actual_cost,
        ...
      )
    }
  }
}
```

> `_deductDiamondQuota` 已存在且独立调用，不受此改动影响。

### 3.5 prize_value_points 字段处置

| 处置方式 | 说明 |
|---|---|
| **数据库字段** | 保留，不删列。设为可空（已有 DEFAULT 0） |
| **奖品配置** | 新活动可不填（默认 0）。旧活动的值保留不影响 |
| **过滤逻辑** | 不再读取此字段做过滤或扣减 |
| **日志/分析** | 可继续写入 lottery_draws 表用于事后分析 |
| **迁移** | 无需数据迁移，字段保留但语义变为"历史遗留/分析用" |

## 六、回滚方案

所有被替换的逻辑标记 `@deprecated` 但不删除代码：

- `_filterByBudget`：保留原函数，新函数命名为 `_filterByResourceEligibility`
- `_filterByAllowedTiers`：保留原函数，在调用点注释掉
- `SettleStage` 原扣减逻辑：保留在注释中

回滚时：恢复 `_filterByAllowedTiers` 调用 + 恢复 `_filterByBudget` 调用 + 恢复 SettleStage 原扣减逻辑。预计 10 分钟内完成回滚。

| 四阶段体验弧线 | 7.5.1 节 | ❌ 预算耗尽后断崖 | ✅ 碎片先尽 → 钻石续航 → 自然收尾 |
| 碎片配额消耗率 ~90-100% | 4.2 节 | ❌ prize_value_points 扣减与模型不一致 | ✅ 按 material_amount × budget_value_points 扣减 |
| 钻石配额消耗率 ~91% | 4.2 节 | ⚠️ 部分正确 | ✅ DIAMOND_QUOTA 独立扣减 |
