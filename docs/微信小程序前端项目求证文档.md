# 议题一：「申请售后」点击无反应

## 一、问题现象

「积分活动记录」页 →「消费记录」Tab → 每条记录的「申请售后」按钮，**点击后页面无任何反应**（不跳转、不报错）。

同一张卡片上的「查看详情」按钮工作正常。

---

## 二、根本原因（前端已完整排查）

前端链路代码均正确，唯一断点是「申请售后」按钮取不到记录主键 `consumption_record_id`，导致跳转条件不成立而静默失败。

### 数据流（纯文本）

```
后端 GET /api/v4/shop/consumption/me
        │  返回 data.records[]
        ▼
每条 record 是否包含字段 consumption_record_id ← 关键主键
        │
        ▼
前端 loadConsumptionRecords() 原样透传 (...record)
        │
        ▼
WXML 按钮 data-record-id="{{item.consumption_record_id}}"
        │
        ▼
点击 onApplyAfterSale(e)
   recordId = e.currentTarget.dataset.recordId
        │
        ├── 有值 → wx.navigateTo 跳转售后页 ✅
        └── 空值 → 中止，页面无反应 ❌（当前现象）
```

### 同卡片两个按钮的对比（关键佐证）

| 按钮 | 取值方式 | 结果 |
| --- | --- | --- |
| 查看详情 | `data-record="{{item}}"`（整个对象） | 对象一定存在 → 正常 ✅ |
| 申请售后 | `data-record-id="{{item.consumption_record_id}}"`（单个字段） | 字段缺失则为空 → 失败 ❌ |

**结论**：「查看详情」正常而「申请售后」无反应，说明后端 `consumption/me` 列表项**未返回 `consumption_record_id` 字段**，或该主键字段名与前端约定不一致。

# ✅ 后端核查结论（议题一）— 2026-06-15

> 核查方式：直连真实库 `restaurant_points_dev`（Node.js+Sequelize）+ 通读当前实际代码，未引用任何历史报告。
> 责任归属：**后端数据库项目的问题**（列表接口未下发标准主键字段名）。web 管理端、微信小程序前端均无代码故障。

## A1. 真实库主键确认

`consumption_records` 表主键实测为 `consumption_record_id`（bigint, PRI, 自增）。这是消费记录的权威业务主键。

## A2. 三个确认项的实测答复

| 编号 | 前端问题 | 后端实测答复 |
| --- | --- | --- |
| Q1 | `/me` 返回项是否含 `consumption_record_id`？ | **否**。当前 `models/ConsumptionRecord.js` 的 `toAPIResponse()` 只返回 `id`（= 主键值，通用脱敏字段）和 `record_id`（= 主键值，业务字段），**没有 `consumption_record_id` 这个 key**。这正是「申请售后」取不到值的根因。 |
| Q2 | 是否经 DataSanitizer？主键是 `consumption_record_id` 还是 `id`？ | `/me` 路由（`routes/v4/shop/consumption/query.js`）**未**经过 DataSanitizer，直接用模型 `toAPIResponse()` 序列化。主键当前以 `id` + `record_id` 两个名字下发，值都等于 `consumption_record_id`。 |
| Q3 | 售后接口 `order_type=consumption` 时 `order_id` 口径就是 `consumption_record_id` 吗？ | **是**。`TradeDisputeService._assertOrderOwnership()` 对 consumption 用 `ConsumptionRecord.findByPk(parseInt(order_id))` 校验，即 `order_id` === `consumption_record_id`（数字主键）。 |

## A3. 数据流真相（纯文本）

```
真实库 consumption_records.consumption_record_id (bigint PRI)
        │
        ▼
ConsumptionRecord.toAPIResponse()
   返回 { id: 主键值, record_id: 主键值, ... }   ← ❌ 没有 consumption_record_id 这个 key
        │
        ▼
GET /shop/consumption/me  →  data.records[] 里每项有 id / record_id，无 consumption_record_id
        │
        ▼
小程序 WXML data-record-id="{{item.consumption_record_id}}"  → 取到 undefined → 售后按钮静默失败
        │（而「查看详情」用整个 item 对象，所以正常）
        ▼
售后接口 POST /system/disputes  期望 order_id === consumption_record_id（数字）
```

矛盾点：**列表下发的主键 key（`id`/`record_id`）与售后接口期望的主键名（`consumption_record_id`）、与真实库主键名（`consumption_record_id`）三者不一致**。这违反项目「主键统一 `{table_name}_id`、全链路一致、不做映射」规范，是真实的命名技术债。

## A4. 后端解决方案（基于现有技术框架，不新增技术债）

修改点唯一且集中：`models/ConsumptionRecord.js` 的 `toAPIResponse()`。

- 让其返回标准业务主键 `consumption_record_id`（值 = 主键，数字类型）。
- `id` 字段：按项目「仅 DataSanitizer 为防抓包用通用 id」规范，`/me` 这条链路本不经 DataSanitizer，应去掉这里自造的 `id`，避免「一个主键三个名」。
- `record_id`：属于历史冗余别名，按「不向后兼容、清理别名」原则一并移除（见 A5 影响面与拍板项）。

可复用 / 可扩展：

- 复用现有 `toAPIResponse()` 单点序列化，无需新建任何类或工具（符合 YAGNI/KISS）。
- 售后链路（`/system/disputes` + `TradeDisputeService`）**无需改动**，它本就按 `consumption_record_id` 工作。
- 该方法被用户端 `/me`、`/detail/:id` 共用，改一处即全链路统一。

# 议题二：首页「仓库」角标改为未读提醒

## 一、问题现象

首页「仓库」入口按钮上显示「99+」红色角标。用户进入仓库页查看后再退出，角标**仍然显示 99+**，没有清零。

## 二、根因分析（前端已排查，非故障）

当前角标取的是后端 `GET /api/v4/backpack/stats` 的 `total_items`（仓库可用物品**总数**），不是「未查看物品数」。

```
进入仓库页 → 只是查看列表（GET 查询），不消耗物品 → 物品总数不变
        ▼
返回首页 onShow → 重新请求 /backpack/stats
        ▼
后端返回 total_items 仍为 100+（用户没用掉物品）
        ▼
角标仍显示 99+ ← 符合当前实现，非 bug
```

**结论**：前端、Web 管理端、后端均无故障。这是角标**业务语义**与用户预期不一致——用户期望它是「未读提醒」（看过即清零），而当前实现是「物品总数」（用掉才减少）。

## 三、产品决策（已确认）

角标语义改为**未读提醒**：只显示「新获得、尚未查看过的物品数」，用户进入仓库查看后清零。

## 四、为什么前端无法独立实现

「未读/已读」是需要持久化的状态，必须由后端记录，前端自行计算会因换设备/重装而丢失，且违反「业务数据必须由后端权威提供」的项目规则。当前 `stats` 接口也没有「未读数」这个数据。

## 五、需要后端确认/提供的事项

| 编号 | 事项 | 说明 |
| --- | --- | --- |
| Q1 | 物品表（items 或等价表）增加「是否已查看」状态字段 | 建议 `is_viewed`（BOOLEAN）+ `first_viewed_at`（DATETIME，UTC 存储） |
| Q2 | `GET /api/v4/backpack/stats` 返回中增加未查看物品数字段 | 建议 `unviewed_items`（INT），专供首页角标使用 |
| Q3 | 新增「标记已读」接口 | 建议 `POST /api/v4/backpack/mark-viewed`，用户进入仓库列表页时调用，批量标记未读为已读 |
| Q4 | 「已查看」的判定口径 | 进入仓库列表页即算已读，还是需点开单个物品详情才算？请后端与产品商定 |

## 六、字段对齐约定

遵循「以后端数据库为准，前端适配后端」原则：

- 后端提供 `unviewed_items` 字段后，前端把首页角标数据源从 `total_items` 改为 `unviewed_items`，不做映射。

# ✅ 后端核查结论（议题二）— 2026-06-15

> 核查方式：直连真实库 `restaurant_points_dev` + 通读当前实际代码，未引用历史报告。
> 责任归属：这是**产品语义变更需求**，需要**后端数据库项目新增能力**（未读状态持久化 + 接口）。微信小程序前端、web 管理端均无故障。前端「未读必须后端持久化」的判断正确。

## B1. 现状实测

- `GET /api/v4/backpack/stats`（`BackpackService.getBackpackStats`）当前返回 `total_items = items.length`，即「可用物品总数」（`status='available'`），**确无「未读数」**。前端排查正确。
- 真实库 `items` 表实测**没有任何「已查看 / 未读」字段**（无 `is_viewed`、无 `first_viewed_at`）。
- `items` 表持有关系：通过 `owner_account_id`（关联 `accounts.account_id`）+ `status` 派生，主键 `item_id`（bigint）。注意：物品归属走 account 体系，不是直接 user_id。

## B2. 关键架构约束（必须遵守，否则会「对不上账」）

项目存在「物品三表互锁」：`items ↔ item_ledger ↔ item_holds`（持有者=账本推导、状态=锁定记录）。
**「是否已读」属于纯展示状态，与持有/余额/锁定无关**，因此：

- ✅ 正确做法：把「未读」做成 `items` 表上的一个**独立展示字段**（如 `is_viewed` / `first_viewed_at`），只读改它不影响三表互锁，可直接 UPDATE。
- ❌ 错误做法：不要把已读状态塞进 `item_ledger`（账本不可变、是记账流水）或 `item_holds`（锁定记录），那会污染互锁体系。

## B3. 后端方案（基于现有技术框架，最低技术债）

围绕唯一改动载体 `items` 表 + 现有 `BackpackService`，无需新建服务类：

1. **数据库迁移（走 `npm run migration:create`，禁止手改库）**
   `items` 表新增两列（仅展示状态，不入三表互锁）：
   - `is_viewed` TINYINT(1) NOT NULL DEFAULT 0 —— 是否已查看（0=未读，1=已读）
   - `first_viewed_at` DATETIME NULL —— 首次查看时间（北京时间语义，库内 UTC 存储，符合项目时间规范）
   - 配套索引：`(owner_account_id, status, is_viewed)` 复合索引，支撑「按持有者统计未读数」的高频查询。
     创建前需检查现有索引避免重复（项目规范）。

2. **`BackpackService.getBackpackStats` 扩展（复用现有方法，不新增接口）**
   在现有返回体里增加 `unviewed_items`（INT）= 该用户 `status='available' AND is_viewed=0` 的物品数。
   `total_items` 维持不变（仍是可用总数），两者语义分离、各有用途。

3. **新增「标记已读」写操作（收口到 Service + 事务，路由不直连 models）**
   - 路由：`POST /api/v4/backpack/mark-viewed`（命名/最终以后端为准，与前端建议一致）。
   - 入参建议两种模式（二选一或都支持，见拍板项）：全量标记当前用户所有未读为已读；或按 `item_ids[]` 批量标记。
   - 实现：在 `BackpackService` 新增方法（如 `markItemsViewed(user_id, options)`），`UPDATE items SET is_viewed=1, first_viewed_at=NOW() WHERE owner_account_id=? AND is_viewed=0 [AND item_id IN (...)]`，通过 `TransactionManager` 管理事务边界、`ServiceManager` 获取服务。
   - 幂等：已读再标记不重复写 `first_viewed_at`（仅当为 NULL 时填充）。注意此写路径若纳入幂等白名单，需在 `CANONICAL_OPERATION_MAP` 注册（项目写操作白名单机制）。

可复用 / 可扩展：

- 复用 `BackpackService`（已是完整领域转换层）、复用 stats 接口、复用迁移工具链与 ServiceManager/TransactionManager，**不新建任何工具类/服务类**（符合「先查现有再扩展」与 KISS）。
- 字段设计可扩展：`first_viewed_at` 未来可支撑「新物品红点 N 天后自动消失」等运营策略，无需再加表。

## B4. 「新获得未读」初始值与产品口径

- 新增列默认 `is_viewed=0`：迁移上线后，存量老物品会全部变成「未读」，可能瞬间出现大角标。
  解决：迁移里对**存量 available 物品一次性回填 `is_viewed=1`**（视为已读），只让上线后新铸造/新到账的物品为未读。这点需产品确认（见拍板项）。
- 「新获得」入口：物品产生来源有抽奖铸造、竞价结算、兑换等（`items.source`）。这些来源创建物品时天然 `is_viewed=0`，无需逐个改造，落库即未读。

## B5. 需要你拍板的事项（议题二）

static async markItemsViewed(user_id, { transaction }) {

    { is_viewed: 1, first_viewed_at: Sequelize.literal('COALESCE(first_viewed_at, UTC_TIMESTAMP())') },

  )
  return { marked_count: affected }
}
