## 二、行业方案对比

### 方案 A：扁平 + 前缀命名（现状）

所有路由文件平铺在一个目录，靠文件名前缀区分域（`lottery-*.js`、`ad-*.js`）。

| 谁在用 | 适用规模 |
|---|---|
| 早期创业公司、10 人以下团队 | < 30 个路由文件 |

**本项目 80 个文件已远超此方案承载上限。**

### 方案 B：业务域目录，一层嵌套（行业主流）✅ 已采用

按业务域建子目录，每个目录一个 `index.js` 聚合，主 `index.js` 只挂载域级入口。

| 谁在用 | 具体做法 |
|---|---|
| **美团** 后台管理系统 | 单体阶段按域分目录（merchant/ order/ delivery/ payment/），每域 5-15 个路由文件，主入口 10-15 行挂载 |
| **阿里 Egg.js 约定** | `app/router/` 下按域分文件或目录，约定 > 配置，一般 8-12 个域 |
| **腾讯游戏 GM 后台** | 按游戏系统分（gacha/ inventory/ trading/ shop/ user/），运营后台 6-10 个顶级域 |
| **NestJS 模块模式** | 每个 Module 是一个目录（controller + service + dto），一般 5-15 个 module |
| **网易游戏运营后台** | 按玩法系统分目录（抽卡/ 背包/ 交易/ 商城/ 公告），跟你的项目结构最像 |
| **虚拟物品交易平台**（Steam Market / BUFF / C5Game） | inventory/ trading/ listing/ payment/ user/，聚焦交易流程 |
| **二手交易平台**（闲鱼 / 转转） | 按交易链路分（商品/ 订单/ 消息/ 评价/ 物流），6-8 个域 |
| **活动运营平台**（有赞 / 商派） | 按运营模块分（活动/ 优惠/ 会员/ 数据/ 配置），5-8 个域 |

**共同规律**：6-12 个顶级域，每域 3-15 个文件，没有 misc/杂项 目录。

### 方案 C：微服务拆分

每个域独立成一个服务，独立仓库、独立部署。

| 谁在用 | 适用条件 |
|---|---|
| 美团/阿里/腾讯的大规模阶段 | 多团队并行开发、独立部署需求、日请求量亿级 |

**本项目是单体 + 未上线，微服务是过度工程。**

### 对比总结

| 维度 | A 扁平 | B 域目录（推荐） | C 微服务 |
|---|---|---|---|
| 适用文件数 | < 30 | 30-200 | 200+ 或多团队 |
| 认知负担 | 文件多了找不到 | 先找域再找文件，两步定位 | 先找服务再找文件 |
| index.js 大小 | 80+ 行 | ~10 行 | 无 |
| 改动成本 | 0 | 1-2 天文件移动 | 几周重构 |
| 所有大公司最终都收敛到 | - | 这里（单体阶段） | 这里（规模阶段） |

**结论：方案 B，没有争议。** 这不是选择题，80 个文件的单体管理后台路由用域目录是唯一合理选项。所有行业（电商/游戏/交易/运营）在这个规模都走这条路。

## 五、执行前实查：import 路径影响

> **2026-03-22 实查**：连接真实数据库 `restaurant_points_dev`（115 表）+ 遍历全部 79 个根路由文件的 require 依赖。

### 5.1 路由文件无交叉依赖（安全）

0 个路由文件互相 import。每个 `.js` 文件只依赖：
- `../../../` 向上 3 层到项目根（models/services/utils/config），共 **273 处**（79 个根文件）
- `./shared/middleware` 引入鉴权中间件，共 **21 处**（21 个根文件）

路由之间完全独立，可安全逐域搬迁，不会出现 A 域移了导致 B 域编译失败。

### 5.2 文件下移一层后的路径变更

每个文件从 `console/xxx.js` 移到 `console/domain/xxx.js`，相对路径多一层：

| 原路径 | 新路径 | 影响处数 |
|---|---|---|
| `../../../models/...` | `../../../../models/...` | 273 处 |
| `./shared/middleware` | `../shared/middleware` | 21 处 |
| **合计** | | **294 处** |

这 294 处修改可用脚本批量完成（`sed` 一行命令），无需手动逐个改。

### 5.3 services/ 已有的子目录对齐情况

## 七、执行前置条件与风险评估

### 7.1 前置条件：当前未提交变更必须先落地

> **2026-03-22 实查**：git status 显示大量未提交的 Product → ExchangeItem 重命名变更。

| 新增模型 | 3 | `ExchangeItem.js` `ExchangeItemSku.js` `ExchangeItemAttributeValue.js` |

**必须先提交这批变更再开始路由重组。** 原因：
1. 路由重组涉及移动 `exchange-items.js` `categories.js` `attributes.js` 等文件，如果两批变更交叉进行会产生大量合并冲突
2. 未提交的变更涉及 `models/index.js` `services/index.js` 等核心注册文件，路由重组需要一个干净基线
3. 数据库已跑完迁移，代码不提交会造成代码与数据库不一致

### 7.2 执行风险评估

| 风险项 | 等级 | 说明 | 应对措施 |
|---|---|---|---|
| import 路径改错导致启动失败 | 低 | 294 处机械替换，脚本可一次性处理 | 迁移脚本 + `npm test` 逐域验证 |

| 子目录文件路径不需要改但误改了 | 低 | 迁移脚本只处理根目录 `.js`，子目录文件（system/ customer-service/ assets/ shared/）不动 | 脚本排除已有子目录 |

**结论：无阻塞性风险。** 唯一前置条件是提交当前的 Product → ExchangeItem 变更。

| `shared/middleware.js` | 所有域共用的鉴权/权限中间件，位置不动 |
| services/ 20 个子目录 + 80+ 根级 service | 路由只是消费者，service 不需要任何改动 |
| models/ 114 个模型 | 路由只通过 service 间接使用模型，不直接 require |
| 迁移脚本模板 | 未来新增域只需复制脚本，改域名和文件清单 |

### 9.4 可扩展性

| 场景 | 做法 | 影响范围 |
|---|---|---|
| 新增业务域（如 shipping/） | 建目录 → 写 index.js → 主 index.js 加一行 | 1 个新目录 + 1 行修改 |
| 域内新增路由文件 | 在域目录下新建 .js → 域 index.js 加两行 | 只改该域 index.js |
| 域内拆子目录 | 参考 lottery/ 已有的 management/ 子层 | 只改该域内部 |
| 跨域共享逻辑 | 放 shared/ 或提升到 services/ | 不影响路由结构 |
