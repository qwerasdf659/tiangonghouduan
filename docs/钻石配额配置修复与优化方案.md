### 1.2 消费发放三比例的实际存储现状

用户到门店消费后提交消费凭证（由商家端调用 `merchantSubmitConsumption`），管理员审核通过后（`approveConsumption`），在同一个事务中依次发放三种资产：

| 用户消费 | 获得什么 | 比例（ratio） | 举例：消费 100 元 |
|---|---|---|---|
| 消费金额 → 积分（POINTS） | 抽奖门票 | 1:1 | 100 积分 |
| 消费金额 → 预算积分（BUDGET_POINTS） | 控制能中多贵的奖 | 1:0.22（22%） | 22 预算积分 |
| 消费金额 → 钻石配额（DIAMOND_QUOTA） | 控制能赢多少钻石 | 1:1 | 100 配额 |

ratio 就是「比例」，代码里是一个乘法：`消费金额 × ratio = 发放数量`。

**三个比例分散在三个不同的地方，用三种不同的机制读取：**

| 比例 | 计算时机 | 存储位置 | 读取方式 | 代码位置 |
|---|---|---|---|---|
| 消费→积分 | 商家提交时（`merchantSubmitConsumption` 第 158 行） | 硬编码 `Math.round(parseFloat(amount))` | 无配置，直接 1:1 | `services/consumption/CoreService.js:158` |
| 消费→预算积分 | 管理员审核时（`approveConsumption` 第 346 行） | `system_settings` 表 `points/budget_allocation_ratio` | `CoreService.getBudgetRatio()` → `AdminSystemService.getSettingValue('points', 'budget_allocation_ratio', null, { strict: true })` + Redis 缓存 | `services/consumption/CoreService.js:690-702` |
| 消费→钻石配额 | 管理员审核时（`approveConsumption` 第 383 行） | `lottery_strategy_config` 表 `config_group='diamond_quota'` | 3 次 `LotteryStrategyConfig.findOne()` 裸查询（第 1 次缺少 `config_group` 过滤） | `services/consumption/CoreService.js:714-754` |

**数据库实证**：`consumption_records` 表中 `points_to_award` = `Math.round(consumption_amount)`

| consumption_amount | points_to_award | 公式 |
|---|---|---|
| 50.00 | 50 | Math.round(50.00) = 50 |
| 88.50 | 89 | Math.round(88.50) = 89 |

### 1.3 存在的问题（代码实证）

**P0-A - 钻石配额过滤功能完全失效 — 配置加载断路（后端 Bug）**

`BuildPrizePoolStage.js` 第 113 行：

```javascript
const strategy_config = campaign_data.strategy_config || {}
```

而 `LoadCampaignStage.js` 第 75-82 行返回的 `campaign_data` 结构为：

```javascript
{
  campaign: campaign.toJSON(),
  prizes: prizes.map(p => p.toJSON()),
  tier_rules: tier_rules.map(r => r.toJSON()),
  fallback_prize: fallback_prize ? fallback_prize.toJSON() : null,
  pick_method: campaign.pick_method || 'tier_first',
  budget_mode: campaign.budget_mode || 'none'
}
```

不包含 `strategy_config` 属性。结果：`strategy_config` 永远是 `{}`，`diamond_quota_enabled` 永远为 `false`，配额过滤从未生效。

**P0-B - 钻石奖品数据不完整 — 引擎无法识别钻石奖品（后端数据 Bug）**

prize #152「钻石1个」的 `material_asset_code = null`，`material_amount = null`。

而 `BuildPrizePoolStage._filterByDiamondQuota` 第 382-383 行通过以下条件识别钻石奖品：

```javascript
const diamond_prizes = prizes.filter(
  p => p.material_asset_code === 'DIAMOND' && p.material_amount > 0
)
```

`SettleStage._deliverPrize` 第 662 行也依赖同一条件触发钻石发放和配额扣减：

```javascript
if (prize.material_asset_code === 'DIAMOND') {
  await this._deductDiamondQuota(...)
}
```

两处代码均要求 `material_asset_code === 'DIAMOND'`，但数据库中 prize #152 该字段为 null。**即使修复了 P0-A，用户抽中 prize #152 时仍不会被识别为钻石奖品，不会发放 DIAMOND 资产，也不会扣减 DIAMOND_QUOTA。**

全部 16 条活跃奖品中，**没有任何一条的 `material_asset_code` 不为 null**。

**P1 - CoreService._getDiamondQuotaConfig 的 EAV 反模式（后端代码问题）**

`services/consumption/CoreService.js` 第 714-754 行对 `lottery_strategy_config` 发起 3 次 `findOne` 查询，用 `config_key` 做条件但第一次查询没过滤 `config_group`（一旦表中其他 group 有同名 key 就会错查），手动做字符串→布尔/数字类型转换。

**P2 - CRUDService.STRATEGY_DEFAULTS 缺失 diamond_quota（后端代码问题）**

`services/admin-lottery/CRUDService.js` 第 166 行的 `STRATEGY_DEFAULTS` 数组包含 14 个分组（anti_empty / anti_high / budget_tier / luck_debt / matrix / pity / pressure_tier / management / grayscale / segment / guarantee / tier_fallback / preset），不包含 `diamond_quota`。新创建的活动不会自动生成 diamond_quota 配置——现有 4 个活动的 12 条 diamond_quota 记录是通过 migration 手动种进去的。

**P3 - 三个消费比例分散管理（后端架构问题）**

三个比例全部发生在同一个业务动作（消费审核通过），却分散在三处存储、用三种机制读取。管理员调整消费发放策略时需要去三个不同的地方操作。

**P4 - diamond_quota 错误地存储在活动级策略表中（后端架构问题）**

diamond_quota 的三个配置项存储在 `lottery_strategy_config`（按活动配置），但业务上消费审核是用户到门店消费触发的，跟哪个活动无关。4 个活动的 diamond_quota 配置完全相同，按活动配置没有意义。

### 1.4 目标

- 修复 P0-A Bug（BuildPrizePoolStage 读不到 diamond_quota 配置）
- 修复 P0-B Bug（prize #152 缺失 material_asset_code 和 material_amount）
- 消除 P1 反模式（3 次裸查询→统一 `AdminSystemService.getSettingValue()`）
- 解决 P2+P4（diamond_quota 从活动级策略表迁移到全局 `system_settings`）
- 解决 P3（三个消费比例 + diamond_quota 开关/策略统一到 `system_settings` 同页管理）
- 新增用户级比例覆盖（`user_ratio_overrides` 表，三个消费比例均支持全局+个人两层配置）

## 2. 已确定的决策

| 决策项 | 选择 | 理由 |
|---|---|---|
| 架构方案 | **全局配置（system_settings）** | 消费审核是门店消费触发的，跟活动无关；4 个活动配置完全相同 |
| 默认值 | **diamond_quota_enabled 默认 `true`** | 保护类机制默认开启（运营忘配也安全，行业惯例） |
| 存储位置 | **5 个配置项全部放 `system_settings` points 分类** | 一个页面统一管理，一种读取方式 |
| POINTS 比例 | **从硬编码改为可配置** | 新增 `points_award_ratio` 到 `system_settings` |
| 迁移方式 | **Sequelize migration 文件** | 项目已有 `npx sequelize-cli db:migrate` 工作流，统一使用 |
| 白名单注册 | **`config/system-settings-whitelist.js` 中注册（min=0.1, max=5.0, step=0.01）** | 项目已有白名单+校验机制，必须注册才能通过 `validateSettingValue()` 校验 |
| POINTS 积分计算时机 | **方案 A：提交时锁定** | 商家提交时读 `points_award_ratio` 计算并写入 `consumption_records.points_to_award`，审核时直接用存储值；用户和商家看到的积分承诺一致，不会出现「提交说给 100 分，审核后变 50 分」 |
| 用户级比例覆盖 | **纳入本次范围** | 新建 `user_ratio_overrides` 表，三个消费比例均支持全局+个人两层配置；一次到位降低后续技术债 |
| P0-B 修复值 | **material_amount = 1** | 与奖品名「钻石1个」一致，后续可通过管理后台调整 |
| 钻石奖品数量 | **本次不新增，仅修复 #152** | 不影响本次修复；后续运营可按需在管理后台添加更多不同额度的钻石奖品 |
| POINTS 初始比例 | **points_award_ratio 默认 `1.0`** | 与当前硬编码行为一致（消费 100 元 = 100 积分），迁移无感 |
| 有效期字段 | **保留 effective_start / effective_end** | 成本极低（两个可空字段），支持临时活动覆盖到期自动恢复 |
| ratio min/max/step | **min=0.1, max=5.0, step=0.01** | max=5.0 覆盖行业天花板（美团/阿里同类参数上限），防运营误操作。step=0.01 对齐 budget_allocation_ratio |
| CRITICAL_SETTINGS | **加入 points_award_ratio 和 diamond_quota_ratio** | 经济参数 fail-fast（与美团/阿里/游戏公司一致），开关类不加入（有安全默认值） |
| description 文案 | **对齐 budget_allocation_ratio 的"名称（公式）"格式** | 数据库已有 description 命名模式统一 |

**迁移后 `system_settings` points 分类新增的配置项：**

| setting_key | 值 | value_type | 说明 | 状态 |
|---|---|---|---|---|
| `points_award_ratio` | 1.0 | number | 消费金额 × 1.0 = 积分数量 | 新增 |
| `diamond_quota_ratio` | 1.0 | number | 消费金额 × 1.0 = 钻石配额数量 | 新增 |
| `diamond_quota_enabled` | true | boolean | 抽奖时是否检查钻石配额 | 新增 |
| `diamond_quota_exhausted_action` | filter | string | 配额不足时：filter=移除钻石奖品，downgrade=保留最小额 | 新增 |

加上已有的 `budget_allocation_ratio=0.22`，管理员在积分设置页一个页面管理所有消费发放规则。

## 3. 已拍板的决策补充

### 3.1 prize #152 → 已确定：material_amount = 1

prize #152「钻石1个」修复为 `material_asset_code = 'DIAMOND'`，`material_amount = 1`。与奖品名一致，后续可通过管理后台调整。

### 3.2 钻石奖品数量 → 已确定：本次不新增，仅修复 #152

当前只有 1 个钻石奖品，配额过滤的 `downgrade` 策略在只有一个钻石奖品时等同于 `filter`。后续运营可按需在管理后台添加更多不同额度的钻石奖品（如钻石5个、钻石10个），届时 `downgrade` 策略才有实际区分意义。不影响本次修复。

### 3.3 POINTS 积分比例 → 已确定：默认值 1.0

`points_award_ratio` 默认值 `1.0`，与当前硬编码行为完全一致（消费 100 元 = 100 积分），迁移无感。

### 3.4 有效期字段 → 已确定：保留

`user_ratio_overrides` 表保留 `effective_start` 和 `effective_end` 字段（可为 null 表示立即/永久生效）。成本极低（两个可空字段），支持临时活动覆盖（如「双11期间所有比例×2」）到期自动恢复。

## 11. 用户级比例动态调整（纳入本次范围）

### 11.1 两层覆盖机制

三个消费比例都支持全局 + 个人两层配置，优先级从高到低：

| 优先级 | 层级 | 来源 | 示例 |
|---|---|---|---|
| 1（最高） | 个人覆盖 | 管理员手动给某个用户设定 | 用户 #12345 的积分比例设为 2.0 |
| 2（最低） | 全局默认 | system_settings 中的值 | 所有人 ×1.0 |

读取逻辑：先查个人覆盖，有就用个人的，没有就用全局默认。

三个比例的完整配置矩阵：

| 比例 | 全局默认值 | 个人覆盖示例：用户 #12345 | 最终效果：消费 100 元 |
|---|---|---|---|
| `points_award_ratio` | 1.0 | 2.0 | 给 200 积分（而非全局的 100） |
| `budget_allocation_ratio` | 0.22 | 0.5 | 给 50 预算积分（而非全局的 22） |
| `diamond_quota_ratio` | 1.0 | 1.5 | 给 150 钻石配额（而非全局的 100） |

### 11.2 个人覆盖的业务场景

- **活动奖励**：某次推广活动中给参与用户临时 ×3.0
- **补偿**：用户投诉后给予 ×2.0 补偿
- **测试**：内部测试账号设为 ×10.0
- **VIP 关怀**：大客户长期享受 ×1.5

可设置有效期（到期自动恢复为全局默认值）。

### 11.3 备用方案（如个人覆盖功能暂不可用时的临时手段）

三种资产都可以通过已有的 `BalanceService.changeBalance()` 直接给用户充值：

| 1.0 | 100 | 1 次 | 正常（当前硬编码行为） |

| 3.0 | 300 | 3 次 | 激进促销 |
